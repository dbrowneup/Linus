# python-sdk (`OpenRouterTeam/python-sdk` — OpenRouter Python SDK)

## 1. Purpose and scope

The official Python client for the [OpenRouter API](https://openrouter.ai/docs/sdks/python) — a typed toolkit covering
all OpenRouter endpoints (chat, embeddings, rerank, video generation, generations history, models/providers/endpoints
catalogues, credits, API keys, OAuth, organization, analytics, beta). OpenRouter itself is a hosted gateway that fronts
300+ models from many providers behind one OpenAI-compatible HTTP surface, and the SDK is essentially a generated façade
over that surface (Speakeasy code-gen — every file carries `Code generated by Speakeasy. DO NOT EDIT`). The package
publishes as `openrouter` on PyPI at version `0.9.1`, license Apache-2.0, requires Python ≥3.9.2, README explicitly
states **Beta with breaking changes possible between minor versions** and recommends pinning. For Linus the SDK as
software is a non-starter (it talks to a paid hosted endpoint, violating the no-paid-APIs north star); what _is_
relevant is the routing-preferences vocabulary it codifies on the wire — that vocabulary is the most mature reference
for what a "smart router in front of N model providers" needs to expose, and Linus's Phase 2a OpenAI-compatible endpoint
will eventually need to make similar choices when fronting a mix of local backends (Ollama, pmetal serve, mlx-lm, future
Linus fine-tunes).

## 2. Architecture summary

A single `OpenRouter` class in `src/openrouter/sdk.py` lazy-loads sub-SDKs from a `_sub_sdk_map` (analytics, chat,
credits, embeddings, endpoints, generations, guardrails, api*keys, models, organization, providers, rerank, beta,
video_generation, oauth) and implements the context-manager protocol with a `weakref` finalizer that closes underlying
sync and async `httpx` clients. Each sub-SDK extends `BaseSDK` and exposes paired sync/async methods (`send` /
`send_async`, `list` / `list_async`); streaming uses an `eventstreaming` utility that yields SSE events, async via
`async for`. Type safety is **Pydantic v2** (≥2.11.2) throughout — every request and response model has both a runtime
`BaseModel` class \_and* a `TypedDict` mirror in `src/openrouter/components/` (273 component files), letting callers
pass either a Pydantic instance or a plain dict and still get IDE completion. A custom `OptionalNullable[T]` / `UNSET`
sentinel distinguishes "not set" from "explicitly null" — important for OpenRouter's wire protocol. Dependencies are
deliberately minimal: `httpx`, `httpcore`, `pydantic`, `jsonpath-python`. Hooks (`_hooks/`) provide before-request /
after-success / after-error extension points. Pagination is uniform: paginated responses expose a `.next()` method
returning `None` when exhausted. Retry policy is per-call via `RetryConfig`, and a `debug_logger` parameter (or
`OPENROUTER_DEBUG=true` env) emits request/response logs.

The routing model lives in `components/providerpreferences.py`. `ProviderPreferences` is the load-bearing type: `order`
(ordered provider slugs, first-supporting wins), `allow_fallbacks` (true/false on whether to fall through to backups),
`only` / `ignore` (allow- and deny-lists merged with account-wide settings), `sort` (one of
`price | throughput | latency | exacto` from `providersort.py`), `require_parameters` (only providers supporting all
requested params), `data_collection` (allow / deny providers that may train on data), `zdr` (Zero Data Retention only),
`preferred_max_latency` and `preferred_min_throughput` (with p50/p99 percentile cutoffs), `max_price` (USD/M-token cap
on prompt and completion), `quantizations` (filter by quant level), and `enforce_distillable_text`. The chat endpoint
also accepts a top-level `models` list (multiple model IDs as cross-model fallback, distinct from provider fallback
inside a single model). That bag of knobs is the actual product of years of running a routing service in front of many
providers — and it is the most concrete prior art for what Linus's router will eventually need.

## 3. What's reusable in Linus

Almost no code. The wire vocabulary, however, is reusable as a **specification** for Linus's own router. When Linus's
Phase 2a OpenAI-compatible endpoint starts fronting more than one local backend (Ollama _and_ pmetal serve, plus a
fine-tuned Linus down the road), it will need to answer the same questions OpenRouter's `ProviderPreferences` answers:
ordered preference, fallback policy, sort criterion (latency vs. throughput vs. memory pressure on a 32 GB box),
require-parameters filtering (which backend supports tools? which supports vision? which honours `response_format`?),
and per-request overrides versus account/session defaults. Lifting that schema verbatim — even just the field names —
would give Linus an interface that any tool already speaking OpenRouter can drive without learning new vocabulary. The
sync+async-paired SDK shape (`send` / `send_async`, context-manager lifecycle, paired httpx clients) is also a clean
reference for what the eventual `linus-py` client should look like when Linus ships its own SDK in Phase 5+.

Compared to the sibling **`openrouter-skills`** repo (Group 7 cousin): they target completely different layers.
`openrouter-skills` is a pack of Agent-Skills-standard Markdown skills that teach _existing harnesses_ (Claude Code,
Cursor, OpenCode, Codex) how to use OpenRouter — no Python, no API client. `python-sdk` is the typed HTTP client for
people writing apps directly. Linus could in principle adopt the `openrouter-skills` skill format for its own
local-routing skills (Phase 7) without ever installing the SDK; conversely, the SDK's `ProviderPreferences` schema is
useful even to someone who never touches the `openrouter` package. They are complementary references, not alternatives.

## 4. What's inspiration only

Speakeasy code-gen as a workflow. The whole tree (273 component files, sync+async every method, Pydantic + TypedDict
duals) is regenerable from an OpenAPI spec. If Linus ever publishes a stable HTTP API and wants a typed Python client
without hand-writing it, this repo demonstrates the target shape. Don't adopt Speakeasy on day one — but file it as a
reference for "this is what a generated client _can_ look like when done well." The hooks system (request lifecycle
extension points), the `OptionalNullable` sentinel, and the `weakref`-based client finalizer are all small patterns
worth borrowing if Linus writes its own client by hand.

## 5. What's incompatible or out of scope

The SDK only speaks to `https://openrouter.ai/api/v1` — every request burns OpenRouter credits, which is exactly what
Linus exists to avoid. There is no offline mode, no local-provider plugin, no way to point the client at
`http://localhost:11434` and have it transparently route. (You _can_ override `server_url`, but there's no local server
that implements OpenRouter's full schema including `provider`, `plugins`, guardrails, generations history, etc.) Beta
status means breaking changes between minor versions — pinning is required if you depend on it at all. The Pydantic v2
hard requirement may conflict with other packages in the linus env that lag on v2 migration. None of this matters if the
SDK is treated as documentation rather than a dependency.

## 6. Recommendation: **Study**

Do not install. Do read `components/providerpreferences.py` and `components/providersort.py` once, in full, before
designing Linus's router-preferences schema in Phase 2a or Phase 3. Lift field names where they fit local semantics
(`order`, `allow_fallbacks`, `sort`, `require_parameters`, `only`, `ignore`); drop fields that don't (`max_price`,
`zdr`, `data_collection` — all hosted-provider concerns). Write the result as an ADR in `docs/adr/` next to the Phase 2a
endpoint design. Re-skim if Linus ever adds a paid-API escape hatch (e.g. a "burst to OpenRouter when local is
saturated" mode), at which point the SDK becomes a candidate dependency rather than a reference.

## 7. Questions for Dan

- **Vocabulary lift, yes or no?** Adopting OpenRouter's `provider` field shape verbatim (with `order`,
  `allow_fallbacks`, `sort`, `only`/`ignore`) means any client already speaking OpenRouter — including Cline and
  openclaw — can drive Linus's router with no per-client config. The cost is binding Linus's wire format to OpenRouter's
  evolution. Worth it for Phase 2a, or design a Linus-native vocabulary from scratch?
- **Sort criteria for a single-machine router.** OpenRouter's `price | throughput | latency | exacto` doesn't quite map.
  On a 32 GB M1 Max the meaningful axes are probably `latency | throughput | memory_pressure | quality_tier`. Want to
  pin the Linus sort enum now, or defer until Phase 3 when there are actually >1 local backends to choose between?
- **Burst-to-paid-API escape hatch.** The north star says "no paid APIs required for operation," not "forbidden." Is
  there a future where Linus _optionally_ falls back to OpenRouter (or hosted Anthropic) when a local backend OOMs or
  fails — and if so, this SDK becomes a real dependency. Worth flagging in ROADMAP, or off-limits?
- **Speakeasy for the eventual `linus-py`.** When Linus ships its own client in Phase 5+, would you rather hand-write
  the SDK (full control, more taste) or generate it from the OpenAPI spec (free updates, less work, the
  `python-sdk`/Speakeasy shape as the template)?
- **Differentiation from `openrouter-skills`.** Do you see Linus ever adopting the Agent-Skills standard for its own
  local-routing skill pack — i.e. publishing `linus-skills` for Claude Code, Cursor, etc. — or is that overkill given
  Linus is a backend, not an agent skill?
