# x402 (`coinbase/x402`)

## 1. Purpose and scope

x402 is an open standard for **HTTP-402-driven internet-native payments** — a single-line server middleware plus a
single-function client wrapper that turns any HTTP route into a paid endpoint. The protocol is deliberately scoped to
the **payment-authorization handshake**: a client requests a resource, the server responds with `402 Payment Required`
and a `PAYMENT-REQUIRED` header carrying a base64-encoded `PaymentRequired` object describing one or more acceptable
`(scheme, network, payTo, price)` tuples, the client signs a `PaymentPayload` for one of those tuples, retries with the
payload in a `PAYMENT-SIGNATURE` header, the server verifies it (locally or via a `facilitator` `/verify` endpoint),
executes the work, and either settles directly on-chain or POSTs to the facilitator's `/settle` endpoint. Settlement
itself is out of band — the protocol is the four-message handshake plus the typed schemas (`PaymentRequirements`,
`PaymentPayload`, `SettlementResponse`); a `scheme` is a logical way of moving money (`exact` is the v1 reference,
`upto` and `subscription` are extensible additions), and a `network` is a CAIP-2 identifier (`eip155:84532` for Base
Sepolia, `eip155:8453` for Base mainnet, Solana / Stellar / Aptos / Algorand for non-EVM chains). The repo (Apache-2.0,
mirrored from `coinbase/x402` to `x402-foundation/x402` as the canonical home; the Coinbase-hosted clone is now framed
as a development fork) ships **reference implementations across four languages** — TypeScript as the canonical
substrate, Python and Go as full-fidelity ports, and Java as an early-stage entrant — plus a documented v2 protocol
specification under `specs/x402-specification-v2.md`.

For Linus this is a **Watch-tier reference**, not a Phase 1–4 integration target. The protocol intersects three Linus
concerns simultaneously — DEC-0018 (MCP as the tool-extensibility substrate), DEC-0045 (FastMCP as the default MCP
framework), and the long-running open question of when (or whether) Linus exposes a commercial surface — and the
intersection is interesting enough to track. But Linus is in Phase 0–2; commercial-surface decisions live at Phase 5+ at
earliest, the crypto/blockchain settlement layer is out of scope for Linus's local-first orientation, and one of several
payment standards is not yet a settled enough field to commit to. The **right move** is what the entrepreneurship
synthesis already prescribes (`E13` and the seeded `DEC-NNNN agent-monetization-via-x402-mcp` ADR slot): document the
protocol now, surface the intersection as a tracked-but-deferred ADR seed, and re-evaluate when either (a) a concrete
Linus commercial-surface decision forces the question, or (b) `@x402/mcp` graduates from its current
shipped-but-recent-2.x state to a stable, widely-adopted spec with multiple compatible facilitator implementations and
visible production usage.

This note is intended to be the primary in-corpus reference for x402, integrating the framing from
[`canteen_blog_landscape_2026-05.md`](../../context/notes/canteen_blog_landscape_2026-05.md) §"x402 v2 Release
(2025-12-01)". One factual update over the Canteen note: **`@x402/mcp` is no longer roadmap-only**. The Canteen post
(dated 2025-12-01) flagged it as a TODO; on the snapshot date of this note (2026-05-10) the package ships at v2.10.0
across TypeScript (`@x402/mcp`), Python (`x402[mcp]`), and Go (`x402-foundation/x402/go/mcp`) with a complete
server-side `createPaymentWrapper` API, a client-side `createX402MCPClient` factory, hook surfaces for
`onBeforeExecution` / `onAfterExecution` / `onAfterSettlement` (server) and `onPaymentRequired` / `onBeforePayment` /
`onAfterPayment` (client), and a documented MCP-SDK workaround for embedding 402 payment-required errors in
`McpError`-converted tool results. The Canteen framing of "agent-monetization-via-MCP integration point, signaled but
not yet shipped" is therefore half-true: the integration point is shipped, but its stability and production usage are
still early enough that the Watch verdict holds.

## 2. Architecture summary

The repo is a **multi-language monorepo** with a TypeScript pnpm workspace as the canonical substrate plus full-fidelity
Python (`python/x402/`), Go (`go/`), and early-stage Java (`java/`) ports, a `specs/` directory holding the v2 protocol
specification (`specs/x402-specification-v2.md`) plus per-scheme and per-transport sub-specs, and an `examples/`
directory with end-to-end client/server demos. The TypeScript surface under `typescript/packages/` is organized into
five top-level groups: `core` (the protocol-implementation library), `mechanisms/` (per-network settlement adapters),
`http/` (per-framework middleware), `extensions` (Bazaar service-discovery and Sign-in-with-x), `mcp` (MCP integration),
plus a `legacy/` directory preserving the v1 single-package API for backwards compatibility during the v1→v2 migration.

The **`@x402/core`** package (`typescript/packages/core/src/`, organized under `client/`, `server/`, `facilitator/`,
`http/`, `schemas/`, `types/`, `utils/`) is the **transport-agnostic protocol implementation**. The client surface
exposes `x402Client` (a builder that registers `(network, scheme)` pairs to scheme implementations, e.g.,
`new x402Client().register('eip155:*', new ExactEvmScheme(evmSigner))`) plus an `x402HTTPClient` wrapper that handles
header encoding/decoding for HTTP transports. The server surface exposes `x402ResourceServer` (the same builder pattern
on the server side, with `.register()` for each supported scheme) and `HTTPFacilitatorClient` (a typed HTTP client for
the facilitator's `/verify` and `/settle` endpoints). The schemas under `schemas/` carry the canonical Pydantic-style
typed shapes — `PaymentRequirements`, `PaymentPayload`, `SettleResponse`, `VerifyResponse`, `Network` (CAIP-2-typed
literal union) — that every scheme implementation, framework adapter, and language port shares. The pluggability story
is explicit: the protocol library does not know about EVM, Solana, or any specific chain; it knows about a `scheme`
trait and delegates to whatever implementation was registered.

The **`@x402/mechanisms/*`** packages (`evm`, `svm`, `aptos`, `avm`, `stellar`) are the **per-network settlement
adapters**. Each implements the `Scheme` trait against a specific chain: `@x402/evm` covers EVM-compatible chains via
the EIP-3009 `transferWithAuthorization` signed-message pattern (the `ExactEvmScheme` is the v2 reference for "exact" on
EVM), `@x402/svm` covers Solana via SPL token transfers, and so on. Adding a new chain is conceptually one new package
implementing the scheme trait; the protocol layer is unchanged. This separation is what `@x402/core` calls
"transport-agnostic and scheme-extensible" — the same `PaymentPayload` shape works across all chains, with the chain
specifics absorbed into the scheme implementation. CAIP-2 network identifiers (`eip155:8453`, `solana:mainnet-beta`,
etc.) are the cross-package identity primitive, used in both `accepts` arrays on the server and `register()` calls on
the client.

The **`@x402/http/*`** packages (`axios`, `express`, `fastify`, `fetch`, `hono`, `next`, `paywall`) are the
**per-framework middleware integrations**. The Express integration is illustrative of the shape (`@x402/express`,
~80-line README): a server creates an `x402ResourceServer` with registered schemes and facilitator client, then calls
`paymentMiddleware({ "GET /protected-route": { accepts: { scheme, price, network, payTo }, description }, ... }, resourceServer)`
— one route declaration per protected endpoint, with the framework-specific `req` / `res` plumbing absorbed into the
middleware. Fastify, Hono, Next.js, and the standalone `@x402/paywall` (a browser-side paywall UI for human-driven
flows) follow the same pattern. The `@x402/fetch` and `@x402/axios` packages are the **client-side** integrations: a
fetch wrapper that intercepts 402 responses, derives the `PaymentRequired` from the response headers, calls the
registered scheme to produce a `PaymentPayload`, and retries the request with the `PAYMENT-SIGNATURE` header attached.
The protocol's "1 line for the server, 1 function for the client" promise is realized at this layer — the core library
does the typed-protocol work, the framework adapter does the HTTP-specific glue.

The **`@x402/mcp`** package (`typescript/packages/mcp/`, version 2.10.0, ~5 source modules under
`src/{client,server,types,utils}/` plus `index.ts`) is the **MCP integration**. The server side exposes
`createPaymentWrapper(resourceServer, { accepts, hooks })` — a higher-order function that takes an MCP tool handler and
returns a payment-aware version. A typical paid tool registration looks like
`mcpServer.tool("financial_analysis", desc, schema, paid(async (args) => { ... }))` where `paid` is the wrapper. The
wrapper is the load-bearing primitive: when the MCP server receives a `tools/call` request without a payment payload
attached (in the request's `_meta` field), it returns a structured 402 error embedded in the tool result; when the
client retries with `_meta["x402/payment"]` populated, the wrapper verifies the payment via the facilitator, executes
the tool, runs the `onAfterExecution` hook, settles via the facilitator, runs the `onAfterSettlement` hook, and returns
the result with the settle response in `_meta["x402/payment-response"]`. The `onBeforeExecution` hook is the rate-limit
/ blocklist gate (returning `false` aborts execution before any chain interaction). The client side ships
`createX402MCPClient({ name, version, schemes, autoPayment, onPaymentRequested })` as the factory, plus
`wrapMCPClientWithPayment` for layering payment onto an existing `Client` instance. The package's own README documents a
**pragmatic workaround** for an MCP-SDK limitation: the SDK converts `McpError` exceptions to tool results with
`isError: true`, losing the `error.data` field where the spec wants the `PaymentRequired` payload to live; the package
embeds the error structure in the result content as a JSON-serialized `{"x402/error": {"code": 402, "data": {...}}}`
payload that the client extracts. The README flags this as tracked-against-upstream-SDK-improvements.

The **language ports** are full-fidelity rather than nominal. `python/x402/` mirrors the TypeScript module layout
(`client.py`, `server.py`, `facilitator.py`, `mechanisms/`, `mcp/` for the MCP integration installed via `x402[mcp]`,
`extensions/`); both async and sync variants ship for the client and server (`client_async.py` / `client.py`,
`server_async.py` / `server_sync.py`). `go/` ships the same shape with idiomatic Go (`client.go`, `server.go`,
`facilitator.go`, `mcp/`, `mechanisms/`, plus `client_hooks.go` / `server_hooks.go` / `facilitator_hooks.go` for the
hook surfaces). The `java/` directory is a Maven project (`pom.xml`) with checkstyle and spotbugs configurations but is
the youngest port. There is **no Rust port** in the canonical repo as of this snapshot, which is notable for Linus given
the multi-language stance (DEC-0027) — if Linus ever needs an x402-speaking Rust component, the implementation work
falls on Linus rather than being a third-party dependency.

The **specs** (`specs/x402-specification-v2.md`, plus `specs/schemes/` and `specs/transports-v2/`) are the durable
contract. The spec is structured as Types (transport- and scheme-independent data shapes) + Logic (scheme- and
network-specific verification and settlement) + Representation (transport-specific encoding — HTTP, MCP, A2A). The **MCP
transport** is one of three documented transports (HTTP, MCP, A2A), with its own sub-spec under `specs/transports-v2/`.
The specification is implementation-agnostic — any of the four language ports can be replaced with a fresh
implementation that conforms to the spec, and the reference facilitator (`https://x402.org/facilitator`) or any
compatible facilitator can serve verification and settlement. This is the shape that makes x402 a candidate to be a real
standard rather than a Coinbase-specific protocol; whether it actually becomes one depends on adoption.

## 3. What's reusable in Linus

The honest answer is **very little, near-term**. x402 is a Watch-tier reference, not a Phase 1–4 pickup. The notable
items below are mostly Phase 5+ commercial-surface candidates and one design-pattern lift that generalizes beyond x402.

**Phase 5+ — HTTP-402 middleware shape as the reference for any future Linus paid surface (DEC-0005, DEC-0008 openclaw,
future commercial-surface ADR).** If Linus ever exposes a paid commercial surface — whether via openclaw, a future
native-app distribution, a paid-API tool registry exposed externally, or a satellite-project monetization vehicle as
flagged in the entrepreneurship synthesis — the **shape of the middleware** is directly applicable: declare routes with
their payment requirements, intercept on first request, return a typed payment-required envelope, accept the payment
payload on retry, verify before doing the expensive work, settle after. The shape is independent of the
crypto-settlement layer — `paymentMiddleware({ route: { accepts: { scheme, price, network, payTo } } }, server)` is a
useful API surface even if `accepts` eventually carries a Stripe payment-intent shape rather than an EVM
transferWithAuthorization shape. Lifting the shape, not the chain plumbing, is the right play. Reference reading:
`typescript/packages/http/express/README.md` (the most legible adapter), `typescript/packages/core/README.md` (the
underlying transport-agnostic protocol), and `specs/x402-specification-v2.md` §"Core Payment Flow" for the twelve-step
sequence diagram.

**Phase 5+ — `@x402/mcp` as the reference architecture for paid MCP tool calls (DEC-0018, DEC-0045 + the seeded
`DEC-NNNN agent-monetization-via-x402-mcp` slot from
[`entrepreneurship-synthesis.md`](../syntheses/entrepreneurship-synthesis.md) §E13).** The substrate Linus already
speaks (MCP via FastMCP per DEC-0045) is the same substrate `@x402/mcp` extends. If a Linus tool ever needs to be paid —
an external-API-deployment-tagged tool per DEC-0046 that costs money to invoke, a satellite-project tool exposed to
non-Linus agents — the `createPaymentWrapper(resourceServer, { accepts, hooks })` higher-order-function shape is the
directly applicable pattern. The hooks vocabulary (`onBeforeExecution` for blocklist/rate-limit, `onAfterExecution` for
metrics, `onAfterSettlement` for receipts) maps cleanly onto a FastMCP middleware pipeline; the `_meta["x402/payment"]`
and `_meta["x402/payment-response"]` request/response metadata plumbing is straightforward to implement against
FastMCP's existing middleware surface. **The Watch verdict holds because** this is a Phase 5+ question with a
current-Phase 0–2 cost: there is no point implementing paid tool calls before there is a tool registry to charge for,
and there is no point committing to x402 specifically before the standard's adoption is settled enough to be worth the
chain-of-custody discipline. But when the question becomes urgent, `@x402/mcp` is the prior art to read first —
`typescript/packages/mcp/src/server/paymentWrapper.ts` and the Python `python/x402/mcp/` mirror are the small, readable
surfaces.

**Phase 5+ — facilitator separation as a useful design pattern for any future Linus payment story.** The protocol's
**facilitator** abstraction — a third-party server that handles the chain-specific verification and settlement, exposed
to the resource server as a typed HTTP client — is a useful piece of architectural discipline independent of crypto. The
resource server does not know about EVM gas, RPC endpoints, or transaction confirmation polling; it just POSTs a
`(PaymentPayload, PaymentRequirements)` pair to `/verify` and `/settle` and gets typed responses back. For any Linus
future payment integration (Stripe, ACH, on-chain), the same shape is applicable: a payment-facilitator service absorbs
the payment-rail-specific complexity, and the resource server (Linus's tool-exposing surface) speaks one abstract
protocol. This is a generalizable pattern, not an x402-specific one, but x402 is the cleanest worked-out example of the
shape currently in the cloned-repo collection.

**No Phase 1–4 reusable pickup.** This is worth stating explicitly. The current Phase 1–2 work (KnowledgeBase v1, Phase
2a orchestration backend, the memory pillar implementation) does not need x402 in any form; the protocol's relevance is
entirely future-tense, gated on a commercial-surface decision Linus has not yet had to make. The repo-note exists to
**make the future intersection legible** so that when the question arrives, the prior art is already characterized
rather than discovered late.

## 4. What's inspiration only

The **layered-decomposition discipline** — protocol Types vs. scheme Logic vs. transport Representation — is the
intellectual contribution that holds up regardless of whether x402 itself becomes the dominant agent-payment standard.
The same shape ("one transport-agnostic protocol library, N pluggable scheme implementations, M pluggable transport
adapters") is the right discipline for any protocol-design task and lines up with Linus's existing typed-prediction
discipline (the BioReason-Pro shape from CLAUDE.md §"Typed structured prediction for biology skills"). It is also the
shape the LLM Wiki output-interface spec (DEC-0023) implicitly wants: typed citations and structured outputs decoupled
from how they are rendered or consumed. Reading x402's `specs/x402-specification-v2.md` for the layering shape is
worthwhile even with no commercial-surface intent.

The **multi-language reference-port discipline** (TypeScript canonical, Python / Go / Java as full-fidelity ports
against the same spec) is a useful inspiration for any Linus-side spec that needs to outlive a single implementation.
CLAUDE.md's multi-language stance (DEC-0027) commits Linus to Python core + Rust where it fits + JS at the front-end
layer; if a future Linus-internal protocol (the agent-spawner spec from DEC-0050, the AgentReport contract from
DEC-0051, the session-store conformance suite from the LangGraph repo-note) ever needs ports across languages, the x402
shape (one canonical implementation + N ports validated against a published spec + a conformance test suite) is the
right discipline. This is the same engineering pattern flagged in the LangGraph repo-note's "conformance test suite as
engineering hygiene" item — x402 is a second confirming signal for the same discipline.

The **CAIP-2 network identifier as the cross-package identity primitive** is a useful inspiration even without any chain
involvement. Using a single canonical string (`eip155:8453`, `solana:mainnet-beta`) as the identity primitive threading
every package together — registered on the client, declared on the server, dispatched on by the facilitator — is a
pattern that generalizes to any "what backend am I talking to?" identification. For Linus's tool registry (DEC-0018), a
similar canonical-string identity primitive (`tool:bio:blast`, `tool:filesystem:read`, etc.) plays the same
architectural role — one string that flows through every layer without rewriting.

The **`accepts` array supporting multiple payment options simultaneously** — a server can publish "I accept (exact, EVM,
$0.10) OR (subscription, EVM mainnet, $50/mo)" and the client picks one — is an inspirational pattern for any Linus
surface that needs to negotiate capabilities at runtime. The shape generalizes to "I accept (model:gpt-4-turbo,
provider:openai) OR (model:claude-sonnet-4, provider:anthropic)" for a future Linus router that negotiates model
selection per call. The negotiation pattern is more general than payments; x402 is a worked example of one application
of it.

## 5. What's incompatible or out of scope

The **crypto/blockchain settlement layer is the architectural mismatch with Linus.** Linus is a private,
local-orchestration AI assistant (CLAUDE.md "North Star"); the operational substrate is Apple Silicon, the user boundary
is Dan, and the data sovereignty discipline (Phase 4) is built around keeping Linus's operations off external servers.
Adopting x402 with on-chain settlement would mean every paid tool call goes through (a) a wallet held by Linus, (b) an
EVM or Solana RPC endpoint, (c) a facilitator service that sees Linus's transaction flow, (d) a public ledger that
records the settlement. The privacy posture is incompatible with Linus's local-first orientation. The HTTP-402 protocol
shape is reusable; the crypto chain plumbing is not.

The **fiat-rail support is theoretical, not shipped.** The x402 README's "Principles" section commits to "Network,
token, and currency agnostic — x402 may extend support to fiat based networks", but as of the v2.10.0 / v2-spec snapshot
the shipped scheme implementations (`@x402/evm`, `@x402/svm`, `@x402/aptos`, `@x402/avm`, `@x402/stellar`) are all
crypto-rail. There is no Stripe scheme, no PayPal scheme, no ACH scheme. A Linus-future "HTTP-402 with traditional
payment rails" use case would require **Linus to author a fresh fiat-rail scheme implementation** against the protocol
spec — the protocol does not block this, but it is build-it-yourself work, not a turnkey adoption.

The **facilitator dependency is an operational coupling Linus does not want.** The reference facilitator
(`https://x402.org/facilitator`) is run by the x402 Foundation; running an alternative facilitator is possible but
non-trivial (the facilitator carries chain RPC credentials, maintains the verification logic, and submits settlement
transactions). For a privacy-first Linus deployment, every paid tool call would be visible to whichever facilitator is
in use. Self-hosting a facilitator is a meaningful operational ask. This is an additional reason the
crypto-rail-specific layer is out of scope.

The **maintenance surface across four language ports is heavier than Linus needs.** x402 ships TypeScript, Python, Go,
and Java; each port has its own CHANGELOG, dependency tree, and release cadence. For a Linus integration that wants to
speak x402 from Python (the orchestration layer) or TypeScript (an openclaw front-end at Phase 5+), the cross-port
synchronization is a third-party concern — but if Linus ever ships an x402 client of its own, matching the four-port
discipline against the v2 spec is non-trivial work. The Watch verdict reflects this: investigate the protocol, do not
commit to it as a vendored substrate until the use case is concrete.

The **MCP-SDK-limitation workaround is a small wart that signals the early-stage state of the integration.** The
documented embedding of `{"x402/error": {"code": 402, "data": {...}}}` inside a tool result (rather than using the
JSON-RPC native `error.data` shape the spec wants) is functional but reflects the integration's freshness — the upstream
MCP SDK does not yet preserve the field shape the x402 transport spec needs, and `@x402/mcp` works around it. This is
not a deal-breaker but is a useful signal that the standard is still maturing; the Watch verdict lets Linus re-evaluate
when these rough edges are smoothed.

The **multiple-competing-payment-standards landscape means committing to x402 specifically is premature.** Pump.fun
fee-recipients, Hyperliquid HIP-3 builder codes, Polymarket V2 builder codes, Stripe-for-AI, OpenAI's API key billing,
Anthropic's metering, and several others occupy adjacent or overlapping problem spaces. x402 is the most protocol-shaped
of these, but the field's eventual winner is not yet visible. Linus's commercial-surface decision (if and when it
arrives) should be made with the field's then-current state, not with the 2026-Q2 snapshot. Watch is the right verdict
for exactly this reason.

## 6. Recommendation: **Watch**

Read x402 for the **HTTP-402 protocol pattern** and the **MCP-payment-wrapper architecture**, both as concrete prior art
for any future Linus commercial-surface decision and as reference material for the
`DEC-NNNN agent-monetization-via-x402-mcp` ADR seed flagged in
[`entrepreneurship-synthesis.md`](../syntheses/entrepreneurship-synthesis.md) §E13. Do **not** vendor x402 packages, do
**not** add x402 dependencies to the Phase 2a orchestration backend, do **not** invest in x402-aware infrastructure
ahead of a concrete commercial-surface use case. The repo earns a Watch verdict, not Study or Integrate, because:

- Linus is in **Phase 0–2**; commercial surface is **Phase 5+ at earliest** per the ROADMAP.
- The **crypto/blockchain settlement layer** is incompatible with Linus's local-first privacy orientation (§5).
- The **multiple-competing-payment-standards landscape** makes a 2026-Q2 commitment to x402 specifically premature (§5).
- The **`@x402/mcp` integration is shipped but recent** (v2.10.0; the MCP-SDK-limitation workaround documented in the
  package README is a freshness signal); committing now is committing to a still-maturing target.

The repo earns Watch rather than Ignore because the **HTTP-402 + MCP intersection is genuinely novel** and lines up
exactly with the substrate Linus already speaks (MCP via FastMCP per DEC-0045). When `@x402/mcp` ships a stable spec
with multiple compatible facilitator implementations and visible production usage, **or** when a concrete Linus
commercial-surface decision (Phase 5+ release, satellite-project monetization, paid-API tool registry) forces the
question, the seeded `DEC-NNNN agent-monetization-via-x402-mcp` ADR slot graduates to a real ADR and a real spike
against `@x402/mcp` lands. Until then, the repo lives as a tracked-but-deferred reference.

The right time to revisit x402 is on any of these triggers:

1. The Canteen hackathon kicks off with an RFB that names x402 explicitly (per
   [`canteen_outreach_2026-05.md`](../../context/notes/canteen_outreach_2026-05.md)).
2. `@x402/mcp` ships a stable v3.x spec with documented production deployments and the MCP-SDK-limitation workaround
   resolved upstream.
3. A Linus Phase 5+ planning session surfaces a concrete commercial-surface use case (paid tool registry, paid
   external-API-deployment-tagged tools per DEC-0046, satellite-project monetization vehicle).
4. A second confirming signal — independent of the Canteen corpus — establishes x402 (or a successor) as the de-facto
   agent-payment standard.

**Cluster cell:** `—` (no cluster). x402 is the awkward case in the post-fan-out cluster taxonomy: it is not an agent
framework (g11), not memory (g4), not an MCP tool server in the g6-mcp-tools sense (although it intersects that

Canteen blog landscape note's prediction-market framing is finance-adjacent but x402 itself is a payment protocol, not a

does not currently carry. The clean placement is `—` (no cluster) until either (a) more agent-monetization repos land

## 7. Questions for Dan

1. **`DEC-NNNN agent-monetization-via-x402-mcp` graduation trigger.** The entrepreneurship synthesis (§E13) seeded an
   ADR slot for "when does this graduate from seed to real ADR?" with a tentative answer of "when `@x402/mcp` ships a
   stable spec, or when a concrete commercial-surface decision forces the question." The factual update from this
   repo-note is that `@x402/mcp` is **shipped** at v2.10.0 — but with a documented MCP-SDK-limitation workaround that
   signals the integration is still maturing. Does the "stable spec" trigger therefore upgrade to "stable spec **and**
   the SDK-limitation workaround is resolved upstream **and** at least one independent production deployment is visible"
   — or stay with the looser "shipped" criterion and accept that early adoption carries the integration's freshness as a
   tax? Tentative answer: tighten the trigger to require the SDK-limitation resolution plus an independent deployment,
   since Linus's privacy posture means committing to a payment substrate is a high-stakes choice and the cost of waiting
   another two quarters for the integration to mature is small.

2. **Phase at which Linus considers a commercial surface.** The ROADMAP places interface refinement at Phase 5 (openclaw
   as front-end) and "skills + autonomy graduation" at Phase 7. Neither phase explicitly carries commercial-surface
   scope; the entrepreneurship synthesis frames it as a Phase 5+ question. Should the ROADMAP add an explicit
   "commercial surface decision" item — even just a "do we want one?" gate — to Phase 5 planning, so that when Phase 5
   lands the question is asked rather than skipped? Tentative answer: yes, add the gate question to Phase 5 planning
   explicitly; the seeded ADR slot becomes an actionable Phase 5 deliverable rather than an open seed.

   speculative for now; the Watch verdict implies "do not invest in implementations until the use case is concrete." But

5. **Is x402 the right substrate to commit to, or should Linus stay rail-agnostic at the protocol level?** The
   alternative to "Linus speaks x402" is "Linus exposes a typed-payment interface and adapters speak x402 / Stripe /
   future protocols against it." The trade-off mirrors the LangGraph-vs-pydantic-ai tension from the LangGraph repo-note
   (§4): adopt the established protocol or build a thinner Linus-native abstraction that several protocols plug into.
   Tentative answer: rail-agnostic at the protocol level — define a `PaymentRequest` / `PaymentReceipt` typed pair on
   Linus's side, adapt to x402 / Stripe / future via thin adapter packages. The x402 layered- decomposition discipline
   (§4 inspiration) is the design pattern Linus would lift, even while staying upstream of x402 itself.

6. **MCP-SDK-limitation workaround as a Linus-internal concern.** The `@x402/mcp` README documents that the MCP SDK

   transport spec wants the `PaymentRequired` payload to live. Linus's own MCP usage (per DEC-0045 FastMCP) has not yet
   hit this limitation in any published Linus skill — but if a future Linus tool needs to surface structured error data

   error data in the result content as JSON-serialized payload) becomes a Linus-internal pattern. Should the FastMCP

   first Linus tool needs it? Tentative answer: yes, fold this into the future MCP-tool-design conventions doc as a
   "structured-error-data transport workaround" pattern, with x402's `{"x402/error": {"code": 402, "data": {...}}}`
   shape as the canonical example.
