# NIST SP 800-160 Vol. 1: Systems Security Engineering

**Source:** NIST Special Publication 800-160 Volume 1 (Ross, McEvilley, Oren)
**Date:** November 2016 (updates through March 2018)
**File:** ../../context/cybersecurity/NIST.SP.800-160v1.pdf
**Relevance to Linus:** Frames security as an emergent system property, not a bolted-on feature. The publication's 32
design principles (Appendix F) and three-context framework (problem / solution / trustworthiness) are the most directly
applicable cybersecurity reference for designing Linus's Phase 2+ orchestration sandbox and the safety surface that
wraps Worker dispatch, tool-registry permissions, and audit-log accountability. Unlike the framework documents (CSF,
800-171, 800-207), 800-160v1 is _engineering_ guidance — it speaks the same language as ARCHITECTURE.md and SAFETY.md.

## Key concepts / frameworks

- **Security as an emergent property.** A system is not "secure" because each component is secure; security emerges
  from how components compose, interact, and fail. Implication for Linus: testing the inference layer, the KB, and the
  sandbox in isolation is necessary but not sufficient — the orchestration substrate must be analyzed as a whole.
- **Asset-based protection.** Security is _freedom from those conditions that cause loss of assets with unacceptable
  consequences_, where assets include intangibles (model weights, KB content, draft writing, reputation). The
  engineering question is always "which asset, which loss, which consequence" — not "what attack."
- **Three-context framework (Chapter 2.4).** Every security activity sits in the **problem context** (define
  objectives, requirements, life-cycle concepts), the **solution context** (architect, design, implement protections),
  or the **trustworthiness context** (produce evidence the solution satisfies the problem). The contexts are not
  sequential — they interleave throughout the system life cycle.
- **Proactive + reactive loss strategy.** Engineer for prevention _and_ for graceful failure. Continuous protection,
  secure failure and recovery, and reference monitor concept are the load-bearing primitives.
- **32 design principles (Appendix F), three families.**
  - _Security Architecture and Design_: clear abstractions, least common mechanism, modularity and layering, partially
    ordered dependencies, efficiently mediated access, minimized sharing, reduced complexity, secure evolvability,
    trusted components, hierarchical trust, inverse modification threshold, hierarchical protection, minimized security
    elements, least privilege, predicate permission (separation of privilege), self-reliant trustworthiness, secure
    distributed composition, trusted communication channels.
  - _Security Capability and Intrinsic Behaviors_: continuous protection, secure metadata management, self-analysis,
    accountability and traceability, secure defaults, secure failure and recovery, economic security, performance
    security, human factored security, acceptable security.
  - _Life Cycle Security_: repeatable and documented procedures, procedural rigor, secure system modification,
    sufficient documentation.
- **Reference Monitor Concept (F.4.1).** An "ideal mechanism" that is tamper-proof, always invoked, and small enough to
  analyze. Every access decision passes through it. The closest Linus analogue is the planned Phase 2 orchestration
  router + tool registry gating every Worker dispatch.
- **Defense in Depth (F.4.2), with caveat.** Layering protections is useful only if the layers fail _independently_.
  Replicating the same mechanism doesn't help — an adversary that defeats one defeats all. The principle disciplines
  Linus's "in-process safety prompt + activation monitor + sandbox + audit log" stack to make the layers different in
  kind.
- **Isolation (F.4.3).** Logical isolation (privilege domains, sandboxes) and physical isolation (separate hardware) are
  both available. Linus today uses logical isolation via macOS process boundaries, file-system gates, and the linus
  conda env; physical isolation is the eventual mobile / Mac Studio / Vision Pro story.
- **Adequate security, not absolute security.** "Things can be declared insecure by observation, but no observation
  allows one to declare an arbitrary system secure" (§2.3.4, citing Herley 2016). Linus's posture is to claim _adequate
  trustworthiness for the threat model_, not perfect security.

## Directly applicable to Linus

- **Problem-context discipline for the orchestration sandbox (Phase 2+).** Before adding any sandbox control, name the
  asset, the loss, and the consequence — exactly the discipline the safety-alignment-privacy synthesis already enforces
  for the four-axis security posture. SAFETY.md additions should explicitly answer the three problem-context questions
  rather than starting from "what attack to prevent."
- **Reference Monitor as the orchestration-router model.** The Phase 2 Linus orchestration layer (router + MCP tool
  registry + sandbox + audit log) should be designed against the reference monitor's three properties: tamper-proof
  (the registry cannot be bypassed by Workers); always invoked (every Worker dispatch passes through it, no direct
  Worker-to-tool calls); small and analyzable (reduced complexity principle, F.1.7). This sharpens the existing
  query-screening commitment from the safety-alignment-privacy synthesis.
- **Least privilege per tool (F.1.14) + predicate permission (F.1.15) for destructive actions.** Tool-registry entries
  carry per-resource, per-environment scopes (already named in the tomshardware incident note); destructive operations
  require predicate permission — explicit Dan sign-off or out-of-band confirmation, never single-actor authorization.
  Both principles compose with the biosecurity tier (DEC-0047): residue-level needs least privilege only, whole-genome
  needs predicate permission.
- **Secure defaults (F.2.5) for Worker dispatch.** "Deny unless explicitly authorized." Worker `memory_mode` defaults
  to `stateless`, sandbox defaults to confirmed-write only, network egress defaults closed, biosecurity tier defaults
  to Tier A — all already implicit in current docs; 800-160 supplies the principle name.
- **Accountability and traceability (F.2.4) as the audit-log contract.** Every security-relevant action traceable to
  the responsible entity (Worker ID, session, spec-task ID, Dan-vs-agent provenance), audit log tamper-resistant, audit
  reviewer interface distinct from audit operator interface (least privilege applied to the audit subsystem itself).
  Aligns with the workgraph JSONL session-store recommendation (G7).
- **Reduced complexity (F.1.7) as a Linus engineering rule.** "Fewer trustworthy components → fewer trust relationships
  → simpler analysis." This is The Algorithm restated for security: question every component, delete what doesn't earn
  its keep. The 16K in-context cap (DEC-0032) and the bounded-orchestration-scope decision (DEC-0020) are reduced-
  complexity moves; future security additions should be justified against this principle.
- **Self-reliant trustworthiness (F.1.16) for the local-first ethos.** A system should be trustworthy without relying
  on external services for its own security. Linus's local-first commitment is already this principle; 800-160 names
  it. The DEC-0053 KB hosted-forbidden default and the deanonymization threat model both belong under this banner.
- **Trusted components + hierarchical trust (F.1.9, F.1.10).** Open-weights Workers pulled from Ollama are _less
  trustworthy_ than the orchestration router; the router must protect itself from the Workers, not the reverse. This
  inverts the naive "trust the model" framing and matches the Maestro/Worker discipline already in CLAUDE.md.
- **Secure failure and recovery (F.2.6).** A failed sandbox check must fail _closed_ (deny the action), not _open_
  (permit and log). The tomshardware incident is the canonical fail-open failure; the Cherry-pick-to-preserve-never-
  reset-to-delete convention is a fail-closed pattern for git.
- **Secure evolvability (F.1.8).** Plan security for the system that will exist in three phases, not just the one
  shipping today. Phase 6 fine-tuning, Phase 7 widening sandbox, and Phase 8 mobile/native deployments each shift the
  threat surface; SAFETY.md and the tool registry schema need version-aware evolution paths.
- **Procedural rigor (F.3.2) commensurate with risk.** Low-trustworthiness components (an `experiments/` script) don't
  warrant the rigor of a load-bearing orchestration primitive. The autonomy-tier graduation rule (SAFETY.md) is
  procedural rigor scaled to risk.

## Not applicable

- Federal acquisition workflows (RFI, SOW, source selection) and the Acquisition / Supply Process detail in Chapter 3.
- Full ISO/IEC/IEEE 15288 life-cycle alignment — Linus uses lightweight branching + phased ROADMAP rather than
  formal life-cycle process artifacts.
- Multi-organization systems-of-systems engineering and inter-agency coordination.
- Formal assurance cases requiring independent verification and validation teams; Linus's assurance is the Maestro
  review + per-PR Dan sign-off, not a separate IV&V function.
- National-security-system overlays and FISMA authorization workflows.
- Cyber-physical system safety integration (medical devices, weapons systems, ICS) — Linus is a software-only research
  substrate.

## Cross-cutting notes for the Linus security synthesis

The 800-160v1 principle vocabulary should anchor any rewrite of `docs/syntheses/security-synthesis.md`. The current
synthesis (supply chain + prompt injection) and the safety-alignment-privacy synthesis (mechanism, values, threat-
model, design-policy) both make architectural claims that map onto specific 800-160 design principles. Naming the
principle in each SAFETY.md section ("this control instantiates F.1.14 least privilege at the tool-registry layer")
makes the design intent legible to future Worker authors and pre-empts ad-hoc additions that duplicate or contradict
existing principles. The publication is also the load-bearing reference for the eventual ARCHITECTURE.md security
section — its three-context framework (problem / solution / trustworthiness) is a natural organization for that
section, and the design-principle taxonomy is a natural checklist for the Phase 2 orchestration-substrate review.
