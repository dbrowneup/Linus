>
> **Goal:** Ship a usable, demo-able Linus you can open in a browser, ask

## Locked design choices

From the 2026-05-18 planning conversation (after Archimedes deep-dive +
clarifying questions):

1. **Q-fin corpus** = fresh KnowledgeBase instance under
   `modules/QFinCorpus/` (or similar), dedicated to quant-finance arXiv
   papers, separate from Dan's personal biochem/AI corpus.
2. **N3 (Anthropic-compat `/v1/messages`)** ships as part of the MVP,
   curl-testable, but **not wired live to Archimedes** during the
   hackathon — Archimedes integration is a post-hackathon teammate
   decision per the bridge spec.
3. **Cluster Explorer drill-down UX** = three-pane layout: left =
   breadcrumb path (Broad → Mid → Fine) with back navigation; middle =
   clickable topic list at the current level; right = detail pane with
   Sankey + 3D UMAP + paper list for the selected topic.
4. **Path-configurable outputs.** All Streamlit pages reading KB
   artifacts (`hierarchy.json`, `labels_*.json`, `topics_*.json`,
   `umap_*.npy`, `graph_sigma.html`, `graph_3d.html`, `kg_graph.graphml`,
   PNGs) take their root path from a `LINUS_KB_OUTPUTS_DIR` env var
   (default: `modules/KnowledgeBase/data/outputs/`). Lets dev point at
   Dan's personal-corpus outputs while q-fin is seeding; demo points at
   q-fin.

## Conventions for every task

- **Branch:** `feature/mvp-<task-id>` (e.g., `feature/mvp-a1-streaming`).
- **Base SHA:** captured at dispatch time; recorded in the agent prompt.
- **Worktree isolation:** every parallel-batch task gets its own
  `isolation: "worktree"` (CLAUDE.md hard rule).
- **Tests:** `pytest src/linus/tests/` must pass after every task; new
  code adds tests where the contract is non-trivial.
- **PR template:** the standard CLAUDE.md PR summary discipline format.
- **Merge:** PR → review → `gh pr merge --merge --delete-branch` (per
  the merge-strategy default).

## Track A — Chat orchestration substrate

### A.1 — N2: SSE streaming on `/v1/chat/completions`

- **Branch:** `feature/mvp-a1-streaming`
- **Files:** `src/linus/server.py` (modify), `src/linus/tests/test_streaming.py` (new)
- **Goal:** Convert the existing single-JSON-response chat endpoint into
  an SSE stream that emits Ollama's token chunks as they arrive.
- **Contract:**
  - Request: same as today, plus `"stream": true` in body activates SSE.
  - Response: `Content-Type: text/event-stream`; each event is `data: <JSON chunk>` where chunk follows the OpenAI streaming shape:
    `{"id": ..., "object": "chat.completion.chunk", "choices": [{"delta": {"content": "..."}, "index": 0, "finish_reason": null}]}`
  - Final event before stream close: `data: [DONE]`.
  - Backward compatibility: `"stream": false` (or absent) preserves the existing single-blob behavior.
  - Tool-call iterations: if the model emits tool calls mid-stream, complete the tool roundtrip server-side, then stream the post-tool assistant content. The Item-6 tool-loop logic stays; only the final-response emission becomes streaming.
- **Smoke test:** `curl -N -H "Content-Type: application/json" -d '{"model":"qwen3:8b","messages":[{"role":"user","content":"count to 5"}],"stream":true}' http://localhost:8000/v1/chat/completions` should emit one chunk per token-or-so and end with `data: [DONE]`.
- **Dependencies:** none.
- **Effort:** ~1.5 hours.

### A.2 — N3: Anthropic-compat `/v1/messages`

- **Branch:** `feature/mvp-a2-anthropic-compat`
- **Files:** `src/linus/server.py` (modify — new endpoint), `src/linus/tests/test_anthropic_compat.py` (new)
- **Goal:** Ship an Anthropic Messages API-compatible endpoint per DEC-0056 so the Anthropic Python SDK can talk to Linus via `base_url=http://localhost:8000`.
- **Contract:**
  - Endpoint: `POST /v1/messages`.
  - Request shape: Anthropic Messages format
    (`{"model": ..., "max_tokens": ..., "messages": [{"role": "user", "content": "..."}], "system": "...", "stream": false}`).
  - Response shape: Anthropic Messages format
    (`{"id": ..., "type": "message", "role": "assistant", "content": [{"type": "text", "text": "..."}], "model": ..., "stop_reason": "end_turn", "usage": {...}}`).
  - Streaming: when `"stream": true`, emit Anthropic's SSE event types (`message_start`, `content_block_start`, `content_block_delta`, `content_block_stop`, `message_delta`, `message_stop`) — at least the minimum to satisfy the Python SDK's `with client.messages.stream(...)` iterator. If full SSE shape is too much for this task, ship non-streaming first and file a follow-up — flag in the PR description.
  - Internally: translate request → Ollama call → translate response back. Reuse model-resolution + tool-loop logic from `/v1/chat/completions` if cleanly extractable; otherwise duplicate the resolution and note the cleanup as a follow-up.
- **Smoke test:** Python script using `anthropic.Anthropic(base_url="http://localhost:8000", api_key="not-used")` calling `client.messages.create(...)` returns a sensible response.
- **Dependencies:** none for the non-streaming form; if streaming, finish after A.1.
- **Effort:** ~2 hours non-streaming; +1 hour for streaming.

### A.3 — N5: Session store

- **Branch:** `feature/mvp-a3-session-store`
- **Files:** `src/linus/memory/sessions.py` (new), `src/linus/server.py` (modify), `src/linus/tests/test_sessions.py` (new)
- **Goal:** Per-conversation message history persisted in SQLite, accessible via `session_id` request param, plumbed into the existing episodic memory store so sessions become a queryable layer over `episodes.db`.
- **Contract:**
  - SQLite schema: `sessions (id TEXT PRIMARY KEY, created_at INTEGER, model TEXT, system_prompt TEXT)` and `session_messages (session_id TEXT, idx INTEGER, role TEXT, content TEXT, created_at INTEGER, PRIMARY KEY(session_id, idx))`. Single DB file: `~/.linus/sessions.db` (path configurable via `LINUS_SESSIONS_DB`).
  - Request: both `/v1/chat/completions` and `/v1/messages` accept an optional `"session_id": "<uuid>"` field. When present:
    - Server prepends stored history to `messages` before sending to Ollama.
    - Server appends the new user message + assistant response to the store after the request completes.
  - When absent: behavior is unchanged (stateless single call).
  - New endpoint: `POST /v1/sessions` returns `{"session_id": "<uuid>"}` for clients that want to create a session explicitly.
  - New endpoint: `GET /v1/sessions/{id}/messages` returns stored history (for the UI to render).
- **Smoke test:** Three sequential POSTs to `/v1/chat/completions` with the same `session_id` — turn 3 should see turn 1 in context (verified by asking "what did I ask first?").
- **Dependencies:** none.
- **Effort:** ~2 hours.

### B.3 — Cluster Explorer (three-pane drill-down) — **LARGEST TASK**

- **Branch:** `feature/mvp-b3-cluster-explorer`
- **Files:** `src/linus/app/pages/3_cluster_explorer.py` (new), `src/linus/app/cluster_data.py` (new — data loader helpers)
- **Goal:** Three-pane drill-down browser over the broad → mid → fine cluster hierarchy. Left: breadcrumb path with back nav. Middle: clickable topic list at current level. Right: Sankey diagram + 3D UMAP scatter + paper list, all scoped to the selected topic.
- **Contract:**
  - **Data inputs** (from `KB_OUTPUTS_DIR`):
    - `hierarchy.json` — Ward dendrogram structure linking broad/mid/fine cluster IDs.
    - `labels_broad.json`, `labels_mid.json`, `labels_fine.json` — sha256 → cluster_id maps.
    - `topics_broad.json`, `topics_mid.json`, `topics_fine.json` — cluster_id → topic label.
    - `umap_3d.npy` — N×3 coordinates aligned with `umap_sha256s.json`.
    - `metadata.db` (from `KB_METADATA_DB`) — paper title/year/authors for the right-pane paper list.
  - **State management:** `st.session_state` carries `current_scale` (`"broad"|"mid"|"fine"`), `current_path` (list of (scale, cluster_id) tuples representing the drilled path).
  - **Left pane:**
    - Breadcrumb path rendered as "Broad: <label> ▸ Mid: <label> ▸ Fine: <label>" with each segment a button that navigates back to that level.
    - "Back ↑" button at bottom that pops the path.
  - **Middle pane:**
    - At the broad level: list of all 28 broad topics, sorted by paper count, each clickable.
    - At mid level: only mid topics that descend from the selected broad topic (per hierarchy.json).
    - At fine level: only fine topics that descend from the selected mid topic.
    - Each list item: `<topic label>  (N papers)`.
  - **Right pane:**
    - Sankey (Plotly): three columns broad/mid/fine; flow widths = paper counts; nodes scoped to the current path (e.g., if on mid level with broad=ML selected, only ML's mid+fine subtree shown).
    - 3D UMAP scatter (Plotly `scatter_3d`): all papers in the corpus, colored by current-scale cluster, hover tooltip showing title. Papers outside the selected path are dimmed (alpha=0.1).
    - Paper list: scrollable `st.dataframe` of papers in the most-specific selected cluster, showing title/year/authors with click-to-expand abstract.
  - **Performance guard:** Initial load caches `hierarchy.json`, `labels_*`, `topics_*`, `umap_3d.npy` via `@st.cache_data`. Switching path levels triggers re-render but not reload.
- **Smoke test:** Open page → see 28 broad topics. Click "Machine Learning" → middle pane updates to ML's mid topics (e.g., NLP, CV, RL). Click "NLP" → Sankey filters to NLP subtree, 3D UMAP dims non-NLP points, paper list shows NLP papers. Click "Back ↑" → returns to ML's mid topics.
- **Dependencies:** B.0 merged. Independent of A.*. If `KB_OUTPUTS_DIR` is empty / q-fin not yet seeded, Dan can dev-point at his personal-corpus outputs.
- **Effort:** ~4-5 hours. Will likely overrun; that's expected for the most complex page.

### B.6 — Search page

- **Branch:** `feature/mvp-b6-search`
- **Files:** `src/linus/app/pages/6_search.py` (new), `src/linus/app/search.py` (new — helpers)
- **Goal:** Keyword (TF-IDF) and semantic (SPECTER2 cosine) search over the configured corpus.
- **Contract:**
  - Top of page: text input + radio "Keyword | Semantic" + result count slider.
  - **Keyword search:** load `tfidf_matrix.npz` + `tfidf_vocab.json`, vectorize the query with the same DictVectorizer trick used in KB's `vectorize.py`, compute cosine to all docs, return top-K with title + score.
  - **Semantic search:** load `specter2.npy`, run SPECTER2 on the query (`title + [SEP] + ""` since query has no abstract; fallback to query-as-title), compute cosine, return top-K.
  - Results: `st.dataframe` of (title, year, authors, score, sha256). Click-to-expand abstract.
  - First-load lazy: SPECTER2 model load is deferred until user clicks "Semantic"; cache via `@st.cache_resource`.
- **Smoke test:** Keyword search "long-read sequencing" → returns relevant titles with scores. Semantic search "PacBio HiFi" → returns adjacent semantic matches.
- **Dependencies:** B.0 merged.
- **Effort:** ~2 hours.

## Track C — Archimedes bridge

### C.1 — `papers.ingest_arxiv` MCP tool

- **Branch:** `feature/mvp-c1-arxiv-tool`
- **Files:** `src/linus/tools/arxiv_ingest.py` (new), `src/linus/tools/__init__.py` (modify — register), `src/linus/tests/test_arxiv_ingest.py` (new)
- **Goal:** A Linus tool callable from any chat completion (and from external MCP clients later) that takes an arXiv ID, downloads the PDF, runs the KB extraction pipeline on it, computes a SPECTER2 embedding, persists the result to a configurable papers-output dir, and returns a paper passport.
- **Contract:**
  - Tool name: `papers.ingest_arxiv`.
  - Input schema: `{"arxiv_id": "string (e.g., '2407.12345' or full URL)"}`.
  - Behavior:
    1. Normalize arxiv_id from URL or bare-id form.
    2. Download PDF to `~/.linus/papers/<arxiv_id>.pdf` (cached; skip if present).
    3. Extract text via PyMuPDF (lift the pattern from `KnowledgeBase/papers_analysis/extract.py`).
    4. Pull metadata from arXiv API (`http://export.arxiv.org/api/query?id_list=<id>`) — title, authors, abstract, year, primary_category.
    5. Compute SPECTER2 embedding from `title + [SEP] + abstract`.
    6. Persist to `~/.linus/papers/<arxiv_id>.json` with extracted fields + embedding shape + sha256.
  - Output (returned to caller): `{"arxiv_id": ..., "title": ..., "authors": [...], "abstract": ..., "year": ..., "category": ..., "sha256": ..., "embedding_path": ..., "pdf_path": ...}`.
  - Failure modes: arXiv API timeout → return `{"error": "..."}`; bad arxiv_id format → return `{"error": "..."}`. Don't raise.
- **Smoke test:** Call from `pytest` with arxiv_id `2407.12345` (or a stable known-good id) — returns expected fields, files exist on disk.
- **Dependencies:** none (uses Phase 2a tool-registry plumbing already in place).
- **Effort:** ~2 hours.

### C.2 — Q-fin KB instance setup

- **Branch:** `feature/mvp-c2-qfin-corpus`
- **Files:** `modules/QFinCorpus/` (new — but see below), `scripts/seed_qfin_corpus.py` (new)
- **Goal:** A fresh KnowledgeBase instance dedicated to quant-finance arXiv papers, isolated from Dan's personal corpus. Seeded with ~50-100 q-fin papers covering the kinds of strategies Archimedes' selection-bias gate evaluates.
- **Open question:** should `QFinCorpus/` be (a) a *copy* of the KB code with its own `data/`, (b) a clone of the KB submodule with a separate `data/` dir pointed via env var, or (c) a thin wrapper that imports `papers_analysis` from the existing KB submodule and points it at a different `data/` root via the existing `--cache-dir` CLI flag?
  - **Recommendation: (c).** Smallest footprint, no code duplication. `scripts/seed_qfin_corpus.py` is a Python entry-point that calls `papers_analysis.extract`, `.metadata`, `.vectorize`, `.cluster`, `.graph`, `.knowledge_graph` in order with `--cache-dir modules/QFinCorpus/data/cache/`, `--data-dir modules/QFinCorpus/data/`. If the KB CLI doesn't already accept those flags universally, add them as part of this task (small KB-side PR upstream).
  - If the CLI doesn't accept the flags and adding them requires a KB-repo PR with its own review cycle, fall back to (b) — clone, edit paths in `__main__.py` blocks, commit as a fork. Flag in PR description.
- **Contract:**
  - `modules/QFinCorpus/papers_library/` (gitignored) holds the q-fin PDFs.
  - `modules/QFinCorpus/data/` (gitignored) holds the per-instance outputs.
  - `scripts/seed_qfin_corpus.py --download-from-list scripts/qfin_arxiv_seed.txt` downloads a hand-curated list of arxiv ids (Dan provides the list — see C.2-extras below) into the library dir.
  - `scripts/seed_qfin_corpus.py --run-pipeline` then runs the full KB pipeline against `modules/QFinCorpus/data/`.
  - After seeding, Cluster Explorer with `LINUS_KB_OUTPUTS_DIR=modules/QFinCorpus/data/outputs/` shows q-fin topics.
  - **Dan's manual lane:** authoring `scripts/qfin_arxiv_seed.txt` — a list of 50-100 arxiv ids covering momentum/volatility/value/quality/macro/HFT/regime-switching/Kelly papers. Linus can suggest candidates by running C.1 on a few known papers, but the curation is Dan's editorial work. **This task ships with a starter list of 20-30 known papers (Faber 2007, Moreira-Muir 2017, MOP 2012, Bailey-de Prado DSR, etc.) so the page is demoable on day one; Dan expands over the hackathon week.**
- **Smoke test:** After seeding with the starter list, Cluster Explorer points at `modules/QFinCorpus/data/outputs/` and renders broad topics that are all finance-flavored.
- **Dependencies:** B.3 merged (so the rendered output has a consumer); C.1 useful but not strictly required.
- **Effort:** ~2 hours for the seeding script; corpus growth is ongoing Dan-time.
