# LLM in a Flash: Efficient Large Language Model Inference with Limited Memory

## TL;DR

**Apple's paper, published while we were waiting for Apple to do something about LLMs.** The premise: M1 Max (and any
modern Mac) has ~10 GB of effective DRAM available for inference but ~100 GB+ of fast flash storage. So _don't load the
whole model into RAM_. Store weights on flash, and stream only the neurons each token actually needs into DRAM. The
paper combines (a) sparsity-aware loading via low-rank predictors of which FFN neurons will activate, (b) flash-friendly
read patterns (large chunks, parallel threads), and (c) a sliding-window cache. Result: **run a 7B model on hardware
that has 4 GB free RAM**, with **4× CPU and 20× GPU speedup over naive flash-loading approaches**. This is the
conceptual foundation for "larger-than-RAM models on Linus."

## The problem (in plain language)

Standard inference loads the entire model into DRAM. A 7B FP16 model is 14 GB; a 70B is 140 GB. Most personal devices
can't hold a 70B; many can't even hold a 7B alongside other workloads. Quantization helps (BitNet 2B is 0.4 GB; 7B
BitNet is ~2 GB) but doesn't change the fundamental architecture: load all of it, run it, hope it fits.

The Apple insight: a Mac is _not_ a server. A MacBook has plenty of _flash_ (1 TB SSD, sequential reads at 6 GiB/s on M1
Max), and small _DRAM_ (16–64 GB shared between OS, apps, GPU, ANE). Flash bandwidth isn't great for random small reads
(~1 GB/s) but it's plenty for _large sequential_ reads. If we can teach inference to _predict_ which weights will be
needed and read them in big contiguous chunks, we can stream a model from flash on demand.

The biology analogue: this is exactly the protein-DNA problem of "your genome is huge, but the cell's transcription
machinery only reads what it needs at any moment, in spatially organized chunks." LLM inference, done right, is closer
to gene expression than to running a binary.

## What they propose

Three categories of techniques operating on the same hardware reality (flash is big, DRAM is small, sequential reads are
fast, random reads are slow):

**1. Reduce data transfer (sparsity exploitation):**

- ReLU/FATReLU activations produce ≥90% sparsity in FFN intermediates. So most FFN neurons aren't used per-token.
- Train a small **low-rank predictor** (per layer) that takes the current attention output and predicts which FFN
  neurons will activate. Predictor is tiny (~4 hours to train per layer on A100).
- Only load the predicted-active rows/columns from flash; never load the rest.
- Predictor false-positive rate is mostly near-zero values (small loss); false-negative rate is small. Zero-shot
  accuracy preserved across ARC-Easy, ARC-Challenge, HellaSwag.

**2. Increase transfer throughput (flash-aware I/O):**

- **Bundle columns of up-projection with rows of down-projection** — they're always used together for the same neuron.
  Doubles chunk size, halves the per-read latency overhead.
- Issue large parallel reads (32 KiB+ across multiple threads) to saturate flash bandwidth.

**3. Manage DRAM efficiently (sliding window):**

- Maintain a DRAM cache of recently-active neurons.
- On each new token, only load the _delta_ — neurons active for the new token but not for any of the previous `k`
  tokens.
- Pre-allocate a fixed-size buffer with pointer-and-scalar metadata. New rows go into freed slots from deleted ones — no
  reallocation.
- For a window of k=5 tokens, the incremental load per new token shrinks rapidly (Figure 4a in paper) because neuron
  usage exhibits strong temporal locality.

The system as a whole is built around an explicit **flash-aware cost model**: latency = (I/O cost from flash) + (DRAM
management overhead) + (compute). They don't optimize compute — that's "orthogonal" to their work — they optimize the
I/O and management.

## Key results

- **2× the available DRAM**: a 7B Llama-2 runs on a device with 4 GB free DRAM.
- **CPU**: 4× faster inference vs naive flash-loading.
- **Metal GPU**: 7× faster.
- **NVIDIA GPU**: 20× faster.
- **Sliding window k=5**: dramatically reduces incremental loads after warmup (~10–15% per token vs ~50% for k=0).
- **Bundling**: doubles chunk size from `d_model × num_bytes` to `2 × d_model × num_bytes`, increases throughput.
- **Predictor accuracy**: marginal zero-shot accuracy impact (66.1 → 66.2 on ARC-Easy with predictor; 50.3 → 49.8 on
  HellaSwag). Predictors are essentially free.

Scale: OPT-6.7B and Llama-2-7B benchmarked. Apple MacBook M1 Max specifically called out — _this is Linus' hardware in
the paper_.

## What's reusable in Linus

**Direct (this is Apple's recipe for Linus):** This paper is _the_ design doc for Apple-Silicon inference of
larger-than-RAM models. The exact techniques (predictors, bundling, sliding windows) are what
[repos/mlx-flash/](../../repos/mlx-flash/) and [repos/flash-moe/](../../repos/flash-moe/) are operationalizing. We don't
need to invent this — we need to plug into one of those repos as the inference path for Workers larger than ~10 GB.

**Direct (immediate practical leverage):** With BitNet b1.58 2B at 0.4 GB, _we don't need any of this for the small
Workers_. But the moment Linus wants to run a 30B / 70B class model — for harder reasoning tasks — flash-streaming is
mandatory on M1 Max. This paper says it works.

**Indirect (KV cache + activations):** The paper hints in a footnote that the same techniques generalize to KV cache and
activations, not just weights. KV cache is _the_ memory bottleneck for long contexts. Streaming KV from flash on demand
is the natural extension and would let Linus handle very long-context Worker queries (32K+ tokens).

**Architecture lesson:** The design philosophy here is the right one for _every_ Apple Silicon design decision in Linus:
_plan around the hardware ratios_, not the abstract algorithm. Flash-DRAM-cache hierarchy is a fact; designing for it
instead of fighting it is what makes the speedups happen.

## What's NOT applicable / hype filter

- **Requires ReLU/FATReLU sparsity** in the FFN. Modern models (Llama-3, Qwen2.5, BitNet 2B4T) often use SwiGLU or
  squared-ReLU. The paper's predictor was trained on OPT (ReLU); Falcon required a fine-tuning step to swap in ReLU.
  Predictor approach must be redone per architecture.
- **Predictor training** is one-time but non-trivial: 10K samples, 4 GPU-hours per layer. For a 30-layer model that's
  120 GPU-hours. Plausible but not casual; we'd want to consume someone else's predictors if available.
- **Flash bandwidth is M1-Max-specific** in the paper's measurements (6 GiB/s sequential). M1 Air's flash is slower. M2
  Max and M3 Max are faster. Linus on M1 Max gets the paper's exact numbers; future hardware shifts the math.
