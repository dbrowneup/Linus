# Emergent autonomous scientific research capabilities of large language models

## TL;DR

A multi-LLM Intelligent Agent (GPT-4 + GPT-3.5) coordinated by a **Planner** module autonomously designs and executes
chemistry experiments by combining web search, a Python interpreter, vector-search over hardware documentation, and
direct API control of liquid-handling robots. The headline demonstration: the agent designs and successfully runs
palladium-catalyzed Suzuki and Sonogashira cross-coupling reactions on a real Opentrons OT-2 with a heater-shaker module
released _after_ the model's training cutoff, with GC-MS confirming products. A preliminary dual-use safety study (11
prompts for controlled substances and chemical weapons) finds the Agent attempted synthesis for 4 (36%); most refusals
came only after web search and could be circumvented by trivial term substitution. Agentic scientific systems are real
and useful, and a dual-use proliferation concern the AI and physical-sciences communities need to address jointly.

## The problem (in plain language)

By April 2023, GPT-4 could reason about chemistry, write code, and use tools when prompted, but it was open whether a
coordinated _system_ of LLMs could carry out a multi-step scientific workflow end-to-end — read the literature, choose
reagents, calculate stoichiometry, write a hardware-specific protocol, push it to a robot, recover from its own
mistakes, and verify with analytical chemistry. The pieces existed (ReAct-style tool use, vector search over docs, lab
APIs, Python-bound liquid handlers); the question was whether stitching them together behaved like a junior bench
scientist or a chatbot pretending to be one. The harder question underneath: a fully autonomous synthesis agent
connected to a liquid handler is a _physical actuator_, and dual-use implications need surfacing before deployment.

For Linus, the relevance is structural. This is the first paper to demonstrate a Planner-plus-tool-using-Workers
architecture executing real physical actions through bound APIs. Linus is Maestro/Worker plus a tool registry that will
eventually bind to scientific tools (BLAST, EBI, protein databases, eventually wet-lab APIs). Foundational reference for
that thread, and its safety analysis is required reading before any binding gets autonomy.

## What they propose

Four components driven by the Planner:

- **Planner** (GPT-4): emits typed actions — `GOOGLE <query>`, `PYTHON <code>`, `DOCUMENTATION <query>`,
  `EXPERIMENT <code>`. Told that on average ten steps are required to fully understand a request (discourages premature
  termination).
- **Web searcher** (GPT-3.5 for speed): turns Planner queries into Google Search API calls, retrieves top-10 non-PDF
  results, can `BROWSE` to extract page text, returns a synthesized answer.
- **Docs searcher**: indexes hardware documentation (Opentrons OT-2 Python API, Emerald Cloud Lab SLL) with OpenAI ada
  embeddings, retrieves sections by distance-based vector search up to 7800 tokens, then passes them through a second
  GPT-4 pass that retains function signatures and parameters before composing a code snippet.
- **Code execution / Automation**: no LLM. Python runs in an isolated Docker container; experiment code dispatches to
  OT-2 / heater-shaker / GC-MS, ECL, or returns as a manual procedure. Tracebacks and outputs route back to the Planner
  for self-correction.

Three scaling tests: (1) synthesis planning by web search alone — ibuprofen, aspirin, aspartame (partial — missed a
methyl source), Suzuki substrate ID; (2) liquid-handler control — drawing geometric patterns on microplates from natural
language, plus a UV-Vis color-ID task where the agent prepared dilutions, requested spectra, and wrote NumPy code to
find absorbance maxima; (3) the load-bearing Suzuki/Sonogashira run, deliberately requiring post-cutoff knowledge (the
heater-shaker module) so docs retrieval was on the critical path. The agent searched the literature, picked aryl halides
and a Pd/NHC catalyst with triethylamine, computed volumes, wrote a protocol with an initially-wrong module name,
consulted the docs, fixed the call, ran on the OT-2, and GC-MS confirmed the products.

## Key results

- **End-to-end real-world execution worked** for catalyzed cross-coupling, with GC-MS confirmation. Not a simulation
  result.
- **Self-correction from error traces is the most important emergent behavior.** The agent recovered from missing
  imports (SymPy not installed → fell back to basic Python), a forgotten `print()` in non-interactive mode, and an
  unknown hardware module name — by re-querying its tools rather than giving up.
- **Vector search over API documentation was critical** for using post-cutoff hardware. Without docs-search, the
  heater-shaker would have been unusable.
- **Planner/executor separation matters.** GPT-4 reserved for planning; GPT-3.5 handles bulk web search. Explicit
  cost/latency optimization.
- **Dual-use safety study**: 4/11 prompts (36%) returned a synthesis attempt; 5/11 were refused only _after_ web search
  revealed the threat (codeine refused after the agent learned its morphine connection); 2/11 (heroin, mustard gas)
  refused at the prompt by name recognition. Authors note after-search refusals are trivially bypassed by terminology
  substitution ("Compound A", "Compound B") or by falsely claiming a DEA-licensed facility; novel harmful compounds are
  unlikely to be recognized at all.
- **Recommendations**: human-in-the-loop for sensitive experiments, ML-based pre-prompt screening for harmful
  structures, continuous data curation, system-security hygiene around tool integrations.

## What's reusable in Linus

**Phase 2 — Architecture validation.** Empirical evidence the Planner-plus-tool-using-executors pattern works end-to-end
on a non-trivial task. Linus's Maestro/Worker split is a more general version: a high-context reasoner emits typed
actions; specialist executors carry them out; the reasoner self-corrects on error traces. The non-negotiable design
contribution: **tool error messages must be routed back to the Planner verbatim** — the self-correction loop is what
made the Suzuki run succeed.

**Phase 2/3 — Tool registry typing.** The typed action vocabulary (`GOOGLE`, `PYTHON`, `DOCUMENTATION`, `EXPERIMENT`)
maps cleanly onto Linus's tool registry. The decision worth making early: whether to give Linus an
`EXPERIMENT`-equivalent _physical-world_ class with extra autonomy gating, distinct from software tools. Empty until
Phase 7, but baking the type slot in now avoids a redesign.

**Phase 3 — Docs-searcher pattern.** Exactly what Linus needs for any non-trivial scientific API (BLAST, EBI, UniProt,
NCBI E-utilities, RDKit, BioPython). Implement once, generically, parameterized by doc corpus: retrieve docs by
embedding distance, pass through a second LLM call to retain function signatures and produce a code snippet, hand to the
Planner.

**Phase 7 — Safety architecture.** The dual-use study is required reading before any physical-world tool. The four
recommendations translate directly into SAFETY.md autonomy tiers. The after-search refusal failure mode specifically
argues for **screening at prompt ingress**, not action egress — by the time the Planner has decided to act, the LLM's
safety reflexes are easier to circumvent.

**Indirect — Maestro budget.** GPT-3.5 for web search, GPT-4 for planning is the Maestro budget discipline inside a
single agentic system. Linus should route bulk text wrangling to smaller/faster Workers and reserve the larger model for
the Planner. The router should support per-tool model routing natively.

## What's NOT applicable

The agent runs on hosted GPT-4 and GPT-3.5. Linus is local-only on Apple Silicon, with substantially weaker per-Worker
reasoning. The behaviors demonstrated (multi-step planning, self-correction, vector-search-mediated tool use) require a
Planner-class model that local 7–14B Workers can approximate but not match for open-ended scientific reasoning.
Realistic near-term: Maestro = hosted Claude as Planner, Worker = local model as executor — exactly the protocol Linus
is built around.

The wet-lab tooling (Opentrons OT-2, GC-MS, Emerald Cloud Lab) is not in Linus's Phase 2–6 horizon. The transferable
layer is the _pattern_ (vector-search docs, typed actions, Docker-isolated execution, error routing), not the specific
tools. Closest Dan-relevant analogs are compbio APIs (BLAST, EBI, UniProt, NCBI) and software lab tools (BioPython,
scikit-bio, pyHMMER) — dramatically lower-risk than physical actuators.

The safety study is preliminary by authors' own admission: 11 prompts is a small sample, the refusal-after-search
failure mode was characterized but not quantified, and the proposed pre-prompt structural screener was not implemented.
The 36% attempt rate is not calibrated, and GPT-4 refusal behaviors will not transfer to local models, which generally
have weaker safety post-training. v1 also ships no source code or prompts ("released in later versions due to safety
concerns") — implementation has to be reconstructed from the architectural description.

## Open questions for Dan

1. **Formal "physical-world" tool class in the registry from Phase 2?** The class is empty until Phase 7, but baking the
   type distinction in now avoids a registry redesign later and prevents every tool from inheriting a single minimal
   autonomy gate by default.
2. **Should the "Planner" role be formalized in the agent spawner as a distinct agent type?** Maestro = Dan + hosted
   Claude is the Planner by convention now. Once Phase 3 fans out sub-agents, they'll need their own Planners (likely
   local models). Naming the role in the spawner clarifies the spec.
3. **Generic docs-searcher module in Phase 3?** If Workers need BLAST/EBI APIs, do we want a generic vectorized-docs
   retrieval module built once, parameterized by doc corpus, reused across tools? This paper makes a strong case yes.
4. **At what autonomy tier does autonomous in-silico experiment design become permissible?** A BLAST sweep, phylogenetic
   analysis, or MD setup carries no dual-use risk but real resource-consumption risk (long runs, large outputs, cluster
   jobs). Worth adding to SAFETY.md as its own tier.
5. **Pre-prompt safety screening — day one or Phase 7?** The most uncomfortable finding is that LLM after-search refusal
   is easy to circumvent. For any irreversible tool (file delete, network egress, shell exec outside sandbox),
   ingress-side screening is more reliable than Planner judgement. Even at Phase 2, a structural filter
   (regex/classifier) is cheap insurance.
6. **A Linus-side "scientific-tool benchmark" for the Dan-task suite?** A compbio analog of the Suzuki demo — "design
   and run a BLAST search, parse the output, identify the top hit, explain its biological significance" — would be a
   sharp, real, safe Phase 3 evaluation of the same Planner-plus-tools pattern this paper validates.
