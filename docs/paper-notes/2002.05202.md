# GLU Variants Improve Transformer

## TL;DR

Shazeer proposes replacing the ReLU-based feed-forward networks (FFN) in Transformers with gated linear unit (GLU)
variants, specifically SwiGLU. The standard Transformer FFN is `ReLU(xW1 + b1)W2 + b2`; SwiGLU replaces this with
`Swish(xW1 + b1) * (xV + c)W2`, where the gate `(xV + c)` is a learned multiplicative mask. Experiments on text-to-text
transfer show that GLU variants consistently outperform ReLU and GELU on perplexity and downstream tasks, with SwiGLU
being the strongest. This substitution is now standard in virtually every modern LLM (Llama, Mistral, PaLM, etc.).

## The problem (in plain language)

Vaswani's Transformer uses a simple two-layer feed-forward network with ReLU activation: `FFN(x) = ReLU(xW1)W2`. This is
fast but has limited expressiveness. The ReLU function is one-sided (zeros out negative values), which constrains what
the network can compute. More recent work (GELU, GLU from Dauphin et al.) suggests that gating—learning a separate
multiplicative mask—allows the FFN to be more selective about what information to pass forward. Shazeer asks: how much
do different gating and activation combinations improve Transformer quality?

## What they propose

A gated linear unit takes the form:

$$\text{GLU}(x, W, V, b, c) = \sigma(xW + b) \odot (xV + c)$$

where σ is a nonlinearity, and ⊙ is element-wise multiplication. For Transformer FFN, this becomes:

$$\text{FFN}_{\text{GLU}}(x, W, V, W_2) = (\sigma(xW) \odot (xV)) W_2$$

Shazeer tests variants with different activation functions:

- **ReGLU**: `max(0, xW) * (xV)`
- **GEGLU**: `GELU(xW) * (xV)`
- **SwiGLU**: `Swish(xW) * (xV)`

Swish is defined as `swish_β(x) = x * sigmoid(βx)` (for β=1, this is `x * sigmoid(x)`). All GLU variants have three
weight matrices (W, V, W2) instead of two (W1, W2), but to maintain parameter count, the hidden dimension is reduced
from d_ff to d_ff * 2/3 when using GLU.

## Key results

**Table 1: Heldout log-perplexity on T5 segment-filling task (Raffel et al., 2019):**

At 65,536 training steps:

- FFN_ReLU (baseline): 1.997
- FFN_SwiGLU: 1.944
- FFN_GEGLU: 1.942
- FFN_ReGLU: 1.953

At 524,288 training steps:

- FFN_ReLU: 1.677
- FFN_SwiGLU: 1.636 (best)
- FFN_GEGLU: 1.633 (best)

**Fine-tuning on downstream tasks (Table 2: SuperGLUE, GLUE, SQuAD, etc.):**

- SwiGLU and GEGLU consistently outperform ReLU and GELU baselines across language understanding benchmarks.
- Improvements are modest but consistent (~0.5-1 point on most benchmarks).

**No computational drawback**: Despite three weight matrices instead of two, the reduction in hidden dimension keeps
parameter count and compute constant compared to ReLU baseline.

## What's reusable in Linus

SwiGLU is the standard FFN activation in modern Worker models:

- **Qwen2.5-Coder, Mistral-7B, Llama 3.x:** All use SwiGLU (or equivalent GeLU variants) in the FFN, not ReLU. Linus
  fine-tuning and inference assumes SwiGLU.
- **Inference efficiency on M1 Max:** SwiGLU has three weight matrices per layer (W, V, W2) in the FFN, vs. two for
  ReLU. On streaming inference (mlx-flash), this means slightly more weight to stream per layer, but the
  quality-per-parameter gain justifies it. For a 32 GB budget, the better quality per parameter saves overall
  parameters/size, offsetting the extra matrix.
- **Quantization (BitNet path):** SwiGLU's gate structure—a separate learned mask—is amenable to quantization. The gate
  can be quantized independently from the activation, and research on GLU quantization is emerging. Understanding that
  the FFN is a three-matrix structure is critical for BitNet weight-sharing and quantization design.
- **Fine-tuning dynamics:** SwiGLU stabilizes training and provides better gradient flow than ReLU. For LoRA fine-tuning
  on Linus, this means better convergence on small target corpora.

## What's NOT applicable / hype filter

The specific configuration (T5 model, 12 layers, d_model=768, d_ff=3072) is from 2020 and doesn't directly apply to Llama
3 or Qwen (which are much larger). However, the architectural lesson (gating is better than ReLU) generalizes.

The paper doesn't address pre-norm vs post-norm, which became standard after 2020. Modern usage of SwiGLU assumes pre-norm
architecture.

The fine-tuning task (segment-filling) is not directly relevant to Linus use cases (scientific computing, code). However,
the quality and training stability improvements are general.
