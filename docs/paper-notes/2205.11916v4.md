# Large Language Models are Zero-Shot Reasoners

## TL;DR

Adding the single trigger phrase **"Let's think step by step"** to any reasoning prompt — with no task-specific
exemplars and no fine-tuning — turns a large pretrained LLM into a competent multi-step reasoner. On MultiArith,
accuracy jumps from **17.7% to 78.7%**; on GSM8K, from **10.4% to 40.7%** (text-davinci-002). The same single template
generalizes across arithmetic, symbolic, and logical reasoning tasks, and the gain only emerges at large scale,
suggesting a latent "system-2" capability that prompting unlocks rather than teaches.

## The problem (in plain language)

Pretrained LLMs are excellent at fast, intuitive, single-step "system-1" tasks (translation, summarization,
classification) but historically struggle with multi-step "system-2" reasoning — arithmetic word problems, symbolic
manipulation, multi-hop logic. The flat scaling curves on these tasks suggested that simply making models bigger would
not fix the gap.

The dominant fix at the time was **few-shot chain-of-thought (CoT) prompting** (Wei et al., 2022): hand-craft a handful
of (question, step-by-step reasoning, answer) exemplars per task, prepend them to each query, and the model imitates the
format. This works, but it is brittle and labor-intensive — every new task type needs its own exemplar set, and
performance degrades sharply if the exemplar format does not match the question format. The unanswered question: is the
multi-step reasoning capability really _taught_ by the few-shot examples, or is it already latent in the model and
merely _triggered_ by them?

## What the paper proposes

**Zero-shot Chain-of-Thought (Zero-shot-CoT)**: a two-stage prompting pipeline that needs no exemplars at all. Stage one
— _reasoning extraction_ — appends `"A: Let's think step by step."` to the question and lets the model generate a
free-form reasoning trace `z`. Stage two — _answer extraction_ — concatenates the original question, the generated
reasoning `z`, and a format-specific cue (e.g., `"Therefore, the answer (arabic numerals) is"`) and asks the model to
emit the final answer in a parseable form.

The trick is that the same single trigger sentence works _across_ arithmetic, symbolic, commonsense, and logical
reasoning — it is task-agnostic. Compared to Few-shot-CoT, which needs hand-engineered exemplars per task, Zero-shot-CoT
needs nothing but the trigger and a second forward pass to clean up the answer format. The contribution is conceptually
small but empirically loud: a single sentence, applied universally, recovers most of the few-shot CoT gain.

## Key results

- **Arithmetic.** MultiArith: **17.7% → 78.7%**. GSM8K: **10.4% → 40.7%**. AQUA-RAT: 22.4% → 33.5%. SVAMP: 58.8% →
  62.1%. (text-davinci-002.)
- **Symbolic.** Last Letter Concatenation (4 words): **0.2% → 57.6%**. Coin Flip (4 times): **12.8% → 91.4%**.
- **Other logical (BIG-bench).** Date Understanding: 49.3% → 67.5%. Tracking Shuffled Objects: 31.3% → 52.4%.
- **Commonsense.** Roughly flat or slightly negative on CommonsenseQA and StrategyQA — chain-of-thought helps less when
  the bottleneck is world knowledge rather than multi-step computation.
- **Scale dependence (Figure 3).** The gap between Zero-shot and Zero-shot-CoT is essentially zero for small models and
  opens dramatically only at large scale. On MultiArith with original GPT-3, the curve is flat from 0.3B through 6.7B
  and only diverges at 175B. Same pattern for PaLM 8B / 62B / 540B on GSM8K. **Multi-step reasoning is an emergent
  capability that prompting reveals, not creates.**
- **Prompt robustness (Table 4).** Of 16 trigger variants, all 9 "instructive" templates beat the no-trigger baseline;
  "Let's think step by step" tops the list at 78.7%, but "Let's think about this logically" (74.5%) and "First,"
  variants (77.3%) also work. Misleading triggers ("Don't think. Just feel.") and irrelevant ones ("Abrakadabra!")
  perform at or below baseline. The effect is real but template-sensitive.
- **Vs. Few-shot-CoT.** Zero-shot-CoT underperforms hand-engineered Few-shot-CoT (78.7 vs. 93.0 on MultiArith with 8
  samples) but enormously outperforms standard zero-shot. It is the strongest _zero-shot_ baseline, not a replacement
  for few-shot when exemplars are available.

## What's reusable for Linus

Read through the memory lens of
[Garrison's "Memory makes computation universal, remember?"](../../context/notes/garrison_memory_makes_computation_universal.md),
this paper is not really about prompting — it is about _externalized intermediate state_. A single transformer forward
pass sits in TC0 and cannot reliably do counting or multi-step recipes. "Let's think step by step" works because it
forces the model to commit each intermediate computation to the token stream, where the next forward pass can read it.
The reasoning trace `z` becomes durable scratchpad memory. The model is not getting smarter; it is being given a place
to remember.

This reframes Zero-shot-CoT as the _minimum viable memory mechanism_ for a Worker. Each generated token of reasoning is
both an output and an input to the next step — recursive state maintenance and reliable history access in the cheapest
possible substrate. That makes it directly load-bearing for Linus's short-term memory design in Phase 2.

**Direct, Phase 2 (MVP — short-term/episodic memory).** The Maestro/Worker protocol assumes Workers can carry state
across the steps of a delegated task. Without an explicit scratchpad, every Worker call is the o1-forgets failure mode
(see Garrison): single-shot capable, amnesiac across turns. Zero-shot-CoT is the trivially cheap default to require:
when a Worker is dispatched to a `system-2` task, the orchestration layer should inject a step-by-step trigger and
_retain the reasoning trace as a first-class artifact_ of the session — not whatever the chat template happens to
truncate. The reasoning text is durable history; it is the "tape" the next pass reads from.

**Direct, Phase 2 (router behavior).** Kojima's emergence result — the gap is zero for small models and only opens at
scale — has direct implications for Worker selection. A 0.5B–1.5B fast-terse Worker will not benefit from CoT triggers;
a 7B–14B Worker (Mistral, Qwen2.5-Coder) will. The router should know which Worker size class supports CoT and inject
the trigger only when the model is large enough to use it. Burning tokens on "Let's think step by step" with a 1B model
is pure overhead.

**Direct, Phase 1 (benchmarks).** The 17.7%→78.7% gap is the cleanest demonstration in the literature that _measurement
methodology matters more than parameter count_ on `system-2` tasks. `benchmarks/dan_tasks/` should evaluate every Worker
model in two conditions per task: with and without a CoT trigger. The delta is itself a model-property worth tracking —
it tells the router how much latent reasoning capacity a given Worker has, independent of its raw tok/s or perplexity
score. This pairs naturally with the wall-clock methodology argued for in
[Speed and Conversational LLMs (2502.16721v1)](2502.16721v1.md).

**Indirect, episodic memory substrate.** The paper's two-stage pipeline (reasoning extraction → answer extraction) is a
concrete case study in _separating thought from answer_. Linus's session/episodic store should follow the same
discipline: store the reasoning trace as one addressable record, the final answer as another, with explicit links
between them. This satisfies three of Garrison's four sub-requirements for reliable history access (addressability,
unambiguous distinction, temporal order) for free.

**Indirect, fine-tuning (Phase 6).** Trigger sensitivity (Table 4) is itself a fine-tuning target. A Linus-specific
model fine-tuned on Dan's task distribution should ideally not need a magic phrase to think — step-by-step decomposition
for `system-2` queries should be the default behavior, with the trigger sentence absorbed into the model's prior.
Measuring trigger-gap shrinkage before/after fine-tuning is a clean diagnostic.

## What's NOT applicable / hype filter

**The "let's think step by step" framing is folklore-ready but the result is narrow.** Zero-shot-CoT helps `system-2`
tasks (arithmetic, symbolic, logical) at large scale. It is **flat or negative on commonsense reasoning**
(CommonsenseQA, StrategyQA) where the bottleneck is world knowledge, not chain length. Adopting CoT-by-default for _all_
Worker calls would slow latency, inflate token counts, and not help Q&A or retrieval tasks. The router needs a
task-class signal, not a blanket policy.

**The headline numbers are 2022 GPT-3-era.** text-davinci-002 and original 175B GPT-3 are the testbeds. Modern
instruction-tuned 7B–14B models (Qwen2.5, Mistral, Llama-3) already do step-by-step reasoning by default for many tasks
because they were RLHF'd on CoT-like data. The gap Kojima measured may be substantially smaller on the actual Workers
Linus will use. **Run the smoke test before assuming the 60-point gap transfers.**

**Underperforms Few-shot-CoT.** When task-specific exemplars exist, few-shot still wins by ~10–15 points on the hardest
benchmarks. Zero-shot-CoT is the floor, not the ceiling. For Linus tasks with stable formats (e.g., a recurring
code-review or paper-summarization workflow), curating a few exemplars is still worth it.

**Trigger sensitivity is real.** "Let's think step by step" is the best of 16 templates tested, but the spread is wide
(45.7%–78.7% within the "instructive" category). Treating any one trigger as magic without re-validating per Worker
model is the kind of cargo-culting the paper itself warns against ("It remains an open question how to automatically
create better templates").

**Reasoning traces are not always faithful.** The paper's error analysis notes that Zero-shot-CoT sometimes generates
unnecessary reasoning steps after reaching the correct answer, then corrupts the answer; sometimes it just rephrases the
question. The stored reasoning trace is _a record of what the model wrote_, not a guarantee of how the model decided.
Memory infrastructure should treat traces as evidence, not as ground truth about model state.

**No Apple Silicon / local-inference content.** The paper is about prompting, not inference. The actual cost on M1 Max
is the doubled token budget (reasoning tokens + answer tokens) per Worker call, which interacts with the verbosity
discussion in [2502.16721v1](2502.16721v1.md): CoT is a deliberate verbosity choice trading wall-clock time for
accuracy.

## Connections

The natural anchor is
[Garrison's "Memory makes computation universal, remember?"](../../context/notes/garrison_memory_makes_computation_universal.md)
— Garrison cites this paper directly as the empirical bridge between the "let's think step by step" trick and the
Merrill & Sabharwal complexity result that CoT recursion lifts transformers from TC0 to UTM-equivalent. This note is
essentially the empirical companion to that synthesis.

The verbosity tradeoff connects to [Speed and Conversational LLMs (2502.16721v1)](2502.16721v1.md): CoT triggers turn a
Task-1-style minimal-output query into a Task-3-style open-ended generation, with all the wall-clock implications that
paper documents. A router that injects CoT triggers is implicitly choosing accuracy over latency, and that tradeoff
should be measured per Worker.

On the memory-substrate side, the keyphrase / structural-extraction work in [RaKUn 2.0 (2208.07262v1)](2208.07262v1.md)
is complementary: Zero-shot-CoT generates ephemeral per-turn reasoning traces; RaKUn-style extraction can index those
traces into a longer-term episodic store cheaply, satisfying "reliable history access" at the session-archive level.

The emergence-at-scale finding is methodologically adjacent to [BitNet b1.58 (2402.17764v1)](2402.17764v1.md) in one
specific way: both papers argue that capability is a function of how representation is structured, not just of parameter
count. BitNet shows ternary weights match FP16 starting at 3B; Kojima shows CoT capability emerges starting around 60B+
for original GPT-3. For Linus's Worker selection, both papers point to **3B as a meaningful capability threshold** —
below it, neither aggressive quantization nor CoT prompting reliably pays off.

Repos: no direct code dependency. The trigger phrase is ~30 characters; the orchestration layer adds it as a string. The
interesting engineering is in the _retention and indexing_ of reasoning traces, which lives in `src/linus/` (Phase 2)
and `modules/KnowledgeBase/` (Phase 3).

Phase mapping: **Phase 1** (benchmark CoT-gap per Worker), **Phase 2** (treat reasoning traces as durable session
artifacts; router decides when to inject the trigger), **Phase 6** (fine-tune to absorb the trigger so default behavior
is step-by-step on `system-2` queries).
