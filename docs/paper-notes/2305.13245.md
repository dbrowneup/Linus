# GQA: Training Generalized Multi-Query Transformer Models from Multi-Head Checkpoints

## TL;DR

Ainslie et al. propose Grouped-Query Attention (GQA), a compromise between multi-head attention (MHA) and multi-query
attention (MQA). Standard Vaswani MHA uses H independent key-value heads (one per query head); MQA uses a single K and V
head shared across all queries, saving memory but degrading quality. GQA divides query heads into G groups, where each
group shares one K-V head. Training from MHA checkpoints using mean pooling reduces K-V heads by a factor of H/G with only
5% of original pre-training compute, achieving quality close to MHA while being nearly as fast as MQA. This substitution
is now standard in Llama 3.x and other recent LLMs, directly reducing KV cache memory on Linus's M1 Max.

## The problem (in plain language)

In Vaswani's multi-head attention, each of the H query heads has its own key and value heads. For a 7B model with H=32,
this means 32 K heads and 32 V heads. During auto-regressive inference, the KV cache grows linearly with sequence length
and quadratically with the number of heads: `cache_size ∝ seq_len × H × d_head` per layer. For long-context inference on
32 GB M1 Max, the KV cache dominates memory. Multi-query attention (MQA) uses a single K and V head for all queries,
cutting KV cache by a factor of H and speeding up decoding. But MQA can lose quality, and models trained with MQA from
scratch show training instability. GQA asks: can we have an intermediate model—fewer KV heads than MHA, but more than MQA—that
achieves quality close to MHA with speed close to MQA?

## What they propose

Grouped-query attention divides H query heads into G groups, where each group shares a single key-value head:

- **MHA**: H query heads, H key-value heads (H=32, implies 32 K/V heads)
- **GQA-8**: 32 query heads, 8 key-value heads (H/4 reduction)
- **GQA-1**: 32 query heads, 1 key-value head (equivalent to MQA)

The conversion from an MHA checkpoint to GQA is non-destructive. Given an MHA checkpoint with H KV heads, construct GQA
with G groups by mean-pooling the projection matrices. For each group of (H/G) query heads, mean-pool their corresponding
KV projection matrices into a single head:

$$K^{\text{GQA}} = \text{mean}(K_1^{\text{MHA}}, K_2^{\text{MHA}}, \ldots, K_{H/G}^{\text{MHA}})$$

Then fine-tune the converted model on a small fraction (α=5%) of the original pre-training compute to allow the model to
adapt to the new structure. This uptraining procedure is faster than training from scratch and provides stability.

## Key results

**Table 1: Inference time and performance on T5 Large/XXL models with MHA, MQA, and GQA:**

- MHA-Large: 0.37 ms/sample, 46.0 avg performance
- MQA-Large: expensive in memory, lower quality
- GQA-8-Large: 0.28 ms/sample (faster than MQA-XXL), 47.1 performance (comparable to MHA-XXL)

**Table 1: Detailed T5-XXL Results:**

- MHA-XXL: 1.51 ms/sample, 47.2 perf
- MQA-XXL: 0.28 ms/sample, 47.1 perf
- GQA-8-XXL: 0.28 ms/sample, 47.1 perf

GQA achieves MHA-XXL quality at MQA speed. Inference time per sample shows negligible overhead for increasing GQA groups
from 1 to 8; beyond 8 groups, overhead increases.

**Fine-tuning on downstream tasks (Table 2: SuperGLUE, GLUE, SQuAD, MediaSum, MultiNews, WMT, TriviaQA):**

GQA-8 achieves quality close to MHA-XXL on summarization, translation, and QA tasks. MQA shows higher variance and worse
quality on some tasks.

**Uptraining efficiency (Figure 5):** GQA already achieves reasonable performance after checkpoint conversion (before
uptraining). Performance plateaus with 5-10% uptraining; diminishing returns beyond 10%.

## What's reusable in Linus

GQA is standard in modern Worker models and directly addresses Linus's M1 Max KV cache constraint:

- **Llama 3.x, Qwen2.5, Mistral:** All use GQA (or equivalent KV-head sharing) as the standard. Llama 3-8B uses 8 KV
  heads per layer (vs. 32 query heads), reducing KV cache by 4×.
- **M1 Max 32 GB KV cache budget:** This is the critical insight. For an 8B model with 32 layers, 4096 hidden dim, 8 KV
  heads: each layer's KV cache is `[batch, n_kv_heads=8, seq_len, d_head=128]` = 8×128×seq_len bytes per layer (FP16).
  With GQA, inference at 16K context is tractable on M1 Max; without it, KV cache alone at 32 heads would consume 128 GB
  at similar context. The Linus M1 Max inference architecture depends on GQA being standard.
- **Streaming inference (mlx-flash):** GQA's reduced KV heads means fewer KV projection matrices to load from SSD. When
  streaming weights layer-by-layer, KV projections are smaller, reducing I/O per layer.
- **Fine-tuning and LoRA:** GQA checkpoints can be initialized from MHA-trained models via mean-pooling and light
  uptraining. This is relevant if Linus fine-tunes on existing pre-trained checkpoints rather than training from scratch.
- **Quantization (BitNet path):** The KV projection matrices (smaller set of them in GQA) can be quantized with higher
  precision (int8) since they're compute-critical, while query projections use lower precision. GQA's grouped structure
  makes this trade-off more explicit.

## What's NOT applicable

The paper focuses on encoder-only (T5) and encoder-decoder models. Decoder-only models (GPT, Llama, Mistral, Qwen)
dominate in modern LLMs, including all Linus Worker models. However, the architectural principle (sharing K-V across query
groups) generalizes cleanly. Decoder-only Llama 3.x papers confirm GQA works equally well.

The uptraining procedure (5% of original pre-training compute) assumes access to the original pre-training recipe and
corpus. For Linus, if fine-tuning on a domain-specific corpus (scientific papers, code), the uptraining compute might need
adjustment. However, the paper's insight that GQA requires only light fine-tuning is robust.

GQA-1 (equivalent to MQA) can show training instability (Appendix A notes loss spikes and divergence on long-input tasks).
Linus should avoid GQA-1 and stick with GQA-8 or higher.

## Connections

This is not a substitution within Vaswani's architecture—it's an optimization of multi-head attention. However, it's so
fundamental to modern inference that omitting it from the corpus would be a mistake. Treat this as "Vaswani's attention
with KV-sharing."

[Llama 3 (2407.21783v3.md)](2407.21783v3.md) documents GQA as the standard attention mechanism and its interaction with
model quality and inference throughput.

[LLM in a Flash (2312.11514v3.md)](2312.11514v3.md) and streaming inference papers directly benefit from GQA's reduced KV
projection size: smaller weight matrices to stream from SSD.

[Speed and Conversational LLMs (2502.16721v1.md)](2502.16721v1.md) measures token throughput on GQA-based models, which is
the practical foundation for any Linus inference latency budget.

The KV cache memory math thread (implicit in Vaswani, explicit here) is load-bearing for all M1 Max inference planning.
Reference Vaswani's Section 3.2.1 on KV cache, then this paper on reducing it.
