# QiMeng-TensorOp: Automatically Generating High-Performance Tensor Operators with Hardware Primitives

## TL;DR

Proposes QiMeng-TensorOp, a framework that takes a one-sentence user prompt ("generate a 1024×1024×1024 GEMM on RISC-V
C910") and drives a frozen LLM to (a) extract hardware factors from architecture manuals, (b) emit a C/CUDA-C++ sketch
with tiling/reordering/layout, (c) emit a Python script that prints assembly / CuTe kernels with vectorization and
pipeline optimization at the hardware-primitive level, and (d) auto-tune the result via LLM-assisted MCTS. Covers GEMM
and Conv across heterogeneous targets (RISC-V C906/C908/C910/K1, ARM A72/A76, NVIDIA RTX 4060 with CUDA cores, A100 with
Tensor Cores). Achieves up to **1291× over vanilla prompt**, reaches **251% of OpenBLAS** on RISC-V K1, **124% of
cuBLAS** on NVIDIA RTX 4060, and **200× human development-time reduction** vs. senior engineers. Key advance over
QiMeng-GEMM: the artifact is **assembly / Tensor-Core PTX**, not C — i.e., the LLM is now driving the hardware
primitives directly, with MCTS replacing beam search for the tuning loop.

## The problem (in plain language)

Tensor operators — primarily GEMM and Conv — account for over 90% of computation in modern LLMs and DNNs. Hardware
vendors achieve peak performance by hand-writing libraries (Intel MKL, ARM ACL, NVIDIA cuBLAS / cuDNN, OpenBLAS)
directly against **hardware primitives**: assembly instructions on CPUs, CUDA-Tensor-Core intrinsics (CuTe / PTX) on
GPUs. Hand-tuned assembly for GEMM is over 62,000× faster than a naive Python triple-loop on the same hardware. But this
work takes months per (operator × architecture) pair, and the libraries don't port — every new RISC-V variant needs its
own hand-tuned BLAS.

Auto-compilers (TVM, Halide, Ansor) explore the program space without humans but consistently underperform hand-tuned
code, and even they require human experts to write the platform-specific backend rules. TVM, notoriously, does not even
support RISC-V CPUs natively in its official version. LLMs at vanilla-prompt level are far worse: a one-shot "write a
fast GEMM for C910" yields C code that hits ~2% of OpenBLAS. Chain-of-thought prompting bumps this to ~3.7%. Neither
gets close to the architecture-specific assembly that the hardware demands.

The paper's claim: with the right prompt substrate plus an MCTS tuning loop, an off-the-shelf LLM can generate
**hardware-primitive-level** tensor operators (CPU assembly, GPU CuTe / PTX) that match or beat hand-tuned vendor
libraries — across CPUs and GPUs, across operator types (GEMM and Conv), and from a single one-sentence user prompt.
This is the QiMeng-GEMM story extended down a level of the hardware abstraction stack, and broadened to a second
operator class.

## What they propose

QiMeng-TensorOp has three coupled stages.

**1. Hardware Architecture Comprehending.** A two-part front-end converts the user's one-sentence prompt into a
machine-usable hardware spec.

- _Hardware Intrinsic Optimization Hints_ — a fixed, hand-curated knowledge base mapping the **five canonical
  optimization techniques** (Tiling T, Reordering R, Vectorization V, Layout L, Pipeline P — same five as QiMeng-GEMM)
  to **four canonical hardware factors** (Memory Hierarchy MH, Instructions INST, Vector/Scalar Registers (V)R,
  Streaming-Processor info SMs). The hints describe in natural language how cache levels constrain tile size, how
  vector-register width drives the vectorization granularity, how SM count drives grid/block dimensions on a GPU, etc.
  These hints function as the LLM's prior over the optimization × architecture relationship.
- _Hardware Factor Extraction_ — given the hints plus a pre-collected (or user-supplied) corpus of vendor manuals, the
  LLM is prompted to extract the concrete factor values for the target hardware: L1/L2/L3 sizes, supported vector ISA
  (vfmacc.vv on RISC-V, NEON on ARM, CuTe on NVIDIA), register file dimensions, SM/Tensor-Core count. The output is a
  typed bundle of factor values that downstream stages consume.

**2. Tensor Operator Generation (TOG).** A two-step generator that splits the operator across two "abstraction layers."

- _Sketch Generation_ — the LLM emits a high-level C function (CPU) or CUDA C++ function (GPU) that holds the
  control-flow scaffold for the operator: multi-level for-loops with tiling/reordering/layout choices, but with the
  inner kernel reserved as opaque calls (e.g., `PACK(A, ...)`, `COMPUTE(A, B, C, ...)` on CPU; grid/block dimension
  setup on GPU). The sketch is parameterized — tile sizes BM/BN/BK, loop order — so the auto-tuner can mutate it without
  re-prompting the LLM.
- _Kernel Generation with Hardware Primitives_ — to bypass the LLM's notorious weakness at directly emitting
  syntactically correct assembly, the framework prompts the LLM to **emit a Python script** that **prints** the assembly
  or CuTe kernel. Python is a register the LLM is fluent in; the generated Python becomes a deterministic printer for
  vectorized load / FMA / store sequences. Few-shot examples plus the hardware-factor bundle guide the Python script to
  use the right register widths, the right vector-extension instructions (`vle32.v`, `vfmacc.vv` for RISC-V; CuTe
  templates for NVIDIA Tensor Cores), and the right pipeline structure (Load / Compute / Store interleaving). The Python
  script + the sketch are jointly compiled and tested on the target hardware to produce a feedback signal for both
  correctness and performance, which feeds back into refining the sketch and Python script.

**3. Auto-Tuning via LLM-assisted MCTS.** Where QiMeng-GEMM used a beam search, TensorOp uses **Monte Carlo Tree
Search**. Each node in the tree represents a (sketch, kernel) pair; the search space is sketch parameters (tile sizes,
loop orders) and instruction-block reorderings within the kernel. Two action types: parameter mutation (BM ± Δ, BN ± Δ,
…) and instruction reordering within an independent block. The MCTS loop runs Selection (UCB) → Expansion (LLM-driven) →
Simulation (compile + benchmark) → Backpropagation. The LLM does two jobs at expansion time: (a) given the path-history
of tuning actions and their performance, propose the **most valuable next action**; (b) given the global tree-history,
generate a **legal candidate action set** for the new node. The LLM is being used as the policy network — it sees enough
context to learn that, for example, "+32 to BN" tends to help on this hardware, and biases the next expansion toward
similar-granularity moves. The tree itself acts as the planner's working memory; search history is essential to
performance (ablation: no-history MCTS underperforms naive MCTS).

The headline architectural moves vs. QiMeng-GEMM (2024) are: (i) dropping down to **hardware primitives** (assembly /
CuTe / PTX) instead of C / vanilla CUDA; (ii) generating Python-as-printer-for-assembly rather than asking the LLM to
emit assembly directly; (iii) MCTS replacing beam search for tuning, with the LLM doing both the action-proposal and the
action-space generation; (iv) automatic factor extraction from manuals instead of a hand-curated hint repository keyed
on hardware. The five-primitive vocabulary (T/R/V/L/P) is preserved.

## Key results

- **Vs. vanilla prompts.** Up to **1291× speedup** (DeepSeek-V3 on A76 ARM CPU, n=4096); routinely 100–500× across the
  matrix; 1000×+ on several configurations. Vanilla LLMs hit on the order of 2% of OpenBLAS; TensorOp hits 100%+.
- **Vs. hand-tuned vendor libraries.** Reaches **251% of OpenBLAS** on RISC-V K1 (best result), **232% of OpenBLAS** on
  C910, **110% of OpenBLAS** on C908. ARM CPUs: **102–121% of ACL** on A72/A76. GPUs: **98–124% of cuBLAS** on RTX 4060;
  comparable to slightly behind cuBLAS on A100 with Tensor Cores. Conv: up to **3.89× cuDNN** on certain dimensions, up
  to **2.43× TVM** on GPU.
- **Vs. CoT prompting.** TensorOp ranges 26–47× over vanilla on K1; CoT alone ranges 1.3–2.1× over vanilla. The prompt
  substrate is doing the heavy lifting; CoT alone is not enough.
- **LLM scaling.** Performance on K1 with TensorOp is within ~10% across GPT-4o, Claude 3.5 Sonnet, DeepSeek-V3, and
  Llama-3.1-405B. The protocol is robust to LLM choice — even Llama-3.1-405B (open) lands at 47× over its vanilla
  baseline, the largest gain of the four. This is markedly better LLM-portability than QiMeng-GEMM, where Codestral-22B
  failed to follow the protocol on GPU.
- **Hardware breadth.** Validated on four RISC-V CPUs (C906, C908, C910, K1), two ARM CPUs (A72, A76), and two NVIDIA
  GPUs (RTX 4060 with CUDA core, A100 with Tensor core). The biggest wins are on RISC-V — specifically because vendor
  libraries underinvest there (RISC-V is open-ISA, fragmented across vendors, and TVM doesn't natively support it). Wins
  shrink on commercial-grade ACL/cuBLAS targets, but are still positive on most dimensions.
- **Operator breadth.** Both GEMM and Conv covered, on regular dimensions (Llama 7B / Llama 3 70B GEMM shapes) and
  irregular dimensions, plus classical CNN Conv (ResNet-50, VGG-16, U-Net) shapes. Larger and irregular dimensions show
  the biggest TensorOp wins, because hand-tuned libraries struggle there too.
- **Development cost.** ~12 minutes on A100, ~20 minutes on the slowest target, vs. 5 senior-engineer-days for a
  hand-tuned implementation — **200× human-time reduction**, similar order to QiMeng-GEMM's 240× claim but on a harder
  target (assembly vs. C).
- **Ablations.** TOG alone is necessary to break past the C-code performance ceiling (LLM C-code + AT plateaus below
  assembly + AT). AT alone (without TOG's assembly substrate) on OpenBLAS C-code also helps but plateaus below the
  TOG+AT combo. MCTS-with-history beats naive MCTS substantially; MCTS-without-history degrades to worse than naive
  MCTS, confirming that the LLM's policy-network role depends on the search trajectory as context.

## What's reusable in Linus

QiMeng-TensorOp is the QiMeng family's most direct hit on the **kernel-codegen for Apple Silicon** thread. Where
QiMeng-GEMM stays at C / vanilla CUDA and QiMeng-Kernel adds an RL-trained planner/coder split, TensorOp sits in the
sweet spot: it descends to hardware primitives (the layer Linus actually needs for Phase 6d) but keeps the LLM frozen
and the tuning loop search-based (no RL training infrastructure). The Python-as-printer-for-assembly trick is a
particularly clean engineering move that maps directly to Metal Shading Language and MLX low-level kernels. The **idea →
reality** spine for TensorOp has the cleanest possible shape:

- _Artifact the LLM produces:_ a (sketch.c, kernel_printer.py) pair, where the Python script emits MSL / MLX kernel
  source or assembly-level Metal compute-shader code targeting Apple's GPU and ANE. For an Apple-Silicon port: the
  kernel printer would emit Metal Shading Language for the GPU side, and (with much more work) emit ML-Program-format IR
  for the ANE — supported public APIs only, per CLAUDE.md §Public APIs only.
- _Downstream actor that accepts it:_ the **Metal compiler toolchain** (`xcrun metal`, `xcrun metallib`) plus Apple's
  `MTLLibrary` / `MTLComputePipelineState` runtime; for MLX, the MLX kernel-compilation path. Both are exact analogs of
  clang for RISC-V or nvcc for CUDA. No new tooling — Apple ships this. (DEC-0027 — public APIs only.)
- _What must be true for Linus to replicate the discipline locally:_ (i) a **manuals corpus** for Apple Silicon — Metal
  Shading Language Specification, Metal Performance Primitives docs, MLX architecture docs, and any usable ANE
  intrinsics surface — converted into the same factor-extraction shape (MH = unified-memory bandwidth + L1/L2/SLC
  hierarchy of M-series; INST = MSL FMA / matrix-multiply intrinsics + MLX kernel ops; (V)R = SIMD-group width on Apple
  GPU; SMs = compute-unit and threadgroup-size dimensions). (ii) a **compile-and-benchmark harness** running natively on
  the M1 Max — kernel launch latency on Metal is non-trivial (yellow flag from QiMeng-GEMM open questions), so the
  wall-clock budget for a 100-node MCTS is something to **measure** before committing architecture (per CLAUDE.md
  §Measure, don't just estimate). (iii) a **frozen Worker LLM** capable of multi-turn Python-as-printer prompting at the
  level the protocol demands — TensorOp's own LLM-portability evidence (Llama-3.1-405B gets +47× over vanilla) is
  encouraging vs. QiMeng-GEMM's harder protocol where Codestral-22B failed; **but** all TensorOp's evaluated LLMs are
  405B+ or proprietary, so Qwen3-Coder-30B-A3B / DeepSeek-Coder-V2 viability on the M1 Max is **the** open question for
  a local-only port.

This maps to **Phase 6d** ([`../specs/phase6d-streaming-target.md`](../specs/phase6d-streaming-target.md) — Linus needs
handcrafted Metal kernels for weight-streaming) and to the broader **Phase 7 idea-to-reality skill class** seeded in
[`../specs/qimeng-category-promotion.md`](../specs/qimeng-category-promotion.md). TensorOp is arguably the most
realistic Phase 6/7 spike target in the QiMeng family precisely because it stays prompt-substrate-only — no RL, no
fine-tune, just the meta-prompt + factor extraction + MCTS plumbing.

**Phase 2/3 — MCTS-with-LLM-policy as an orchestration substrate.** TensorOp's MCTS loop is a generic agent-driven
program-search pattern, more sophisticated than QiMeng-GEMM's beam search and arguably more durable as a Linus
orchestration primitive. Each MCTS node is a typed (sketch, kernel, performance) tuple; the LLM's role at expansion
matches the agent-spawner role-typed work-graph pattern (DEC-0050, DEC-0051) where the LLM proposes the next typed
action and the framework validates the result against a typed AgentReport (correctness + benchmark). This rhymes
strongly with the workgraph DAG-dispatch pattern from G7
([`../syntheses/repo-clusters/g7-harnesses.md`](../syntheses/repo-clusters/g7-harnesses.md)); the search graph itself is
a durable scratchpad-class artifact (DEC-0030). Worth considering whether Linus's Phase 2/3 scratchpad protocol should
explicitly support tree-shaped scratchpads and not just linear ones — a forward-looking design pressure that TensorOp
surfaces. _Seed: DEC-0057 candidate (tree-shaped scratchpad as a first-class shape)._

**Phase 6 — typed-structured-prediction dataset for kernel codegen.** The TensorOp protocol surfaces a clean dataset
shape:
`{operator: "gemm", platform: "metal", hardware_factors: {MH:..., INST:..., (V)R:..., SMs:...}, optimization_techniques: ["T", "R", "V", "L", "P"], sketch_template: "...", kernel_printer_python: "...", benchmark_gflops: 12.4, rationale: "..."}`.
This is exactly the typed-structured-prediction-wrapping-rationale convention from CLAUDE.md S25 (originally biology)
applied to kernel codegen. A Phase 6 LoRA fine-tune over a synthetic dataset of these tuples — bootstrapped by running
TensorOp-style prompting against a hosted teacher LLM on M1 Max targets — is the natural follow-on, and the
Python-as-printer trick means the dataset records a deterministic generator rather than fragile assembly strings.

**Idea → reality discipline.** TensorOp is the QiMeng family's clearest demonstration that an LLM can produce a real
artifact (RISC-V assembly, ARM NEON, CUDA Tensor Core CuTe / PTX) that a real downstream actor (the compiler / linker)
accepts and converts into a runnable binary at expert quality, **with no human in the loop between the LLM and the
silicon** other than the benchmark feedback signal — and the benchmark is itself automated in the MCTS simulation step.
For the broader Linus design-and-build vision per
[`../specs/qimeng-category-promotion.md`](../specs/qimeng-category-promotion.md) §"Dan's stated framing", kernel codegen
for Apple Silicon is the smallest-scope, most-auditable Apple-Silicon-relevant instance of the idea-to-reality pattern,
and TensorOp is the exemplar — the fact that it works across CPUs **and** GPUs, **and** across multiple operator classes
(GEMM, Conv), is evidence the discipline generalizes within the kernel-codegen domain.

## What's NOT applicable / hype filter

The 1291× headline is over **vanilla prompt**. The numbers that actually matter for the hand-tuned-quality assertion are
**251% of OpenBLAS** (RISC-V K1) and **124% of cuBLAS** (RTX 4060). The OpenBLAS multipliers are real — RISC-V vendor
libraries are objectively underinvested, and TensorOp's win there reflects that gap as much as it reflects TensorOp's
quality. The cuBLAS win is the durable one: matching or beating NVIDIA's hand-tuned A100 / RTX 4060 library at 124% on
certain dimensions is the strong claim, and even that is **dimension-dependent** — A100 with Tensor Core sits at
0.98–1.24× cuBLAS, and at peak A100 cuBLAS still wins.

The **manuals corpus** is hand-curated. The paper does **not** quantify the cost of building the manual corpus for a new
hardware target; the optimistic framing ("automatic factor extraction") obscures that someone curated and parsed those
vendor manuals first. For Apple Silicon, **the equivalent corpus does not exist as a clean machine-readable artifact** —
Apple ships PDFs and Markdown for MSL, the Metal Performance Primitives surface is partially documented, and ANE
intrinsics are not in the public docs at all. This is a non-trivial Phase 7 effort that the paper does not quantify and
that any Apple-Silicon port has to fund.

The five-primitive vocabulary (T/R/V/L/P) is **GEMM and Conv specific**. The paper extends from QiMeng-GEMM (GEMM only)
to TensorOp (GEMM + Conv), but the primitive set is unchanged because GEMM and Conv share the same loop-and-data shape
(after im2col). For other tensor operator classes — sparse, scan, scatter, attention with masking, MoE-dispatch,
quantized dequant — the primitive set is different. The pattern (curated primitive set + factor extraction + MCTS)
generalizes; the specific T/R/V/L/P vocabulary does not.

The Python-as-printer-for-assembly trick is brilliant but **leans on the LLM's fluency in Python plus its training
exposure to assembly-printing patterns**. For non-mainstream hardware targets where the LLM has thin training-data
exposure (Apple Silicon ANE, custom accelerators, FPGA), the printer-style approach may degrade. The paper validates on
RISC-V (open ISA, well-represented in training data), ARM (extensively trained), and NVIDIA (the dominant ML substrate
in training data). Metal Shading Language is in training data; **MLX kernels are recent and thin**; ANE intrinsics are
essentially absent. Any Apple-Silicon port should validate on Metal first, MLX second, ANE last (or not at all in a v1
spike).

The MCTS auto-tuning loop's **wall-clock cost** is the under-discussed elephant. The paper reports overall development
time (12 minutes on A100, ~20 minutes worst case) but does not break out compile-and-benchmark inner-step latency. For
Apple-Silicon Metal kernels, kernel launch latency is non-trivial and the Metal compiler is slower per invocation than
nvcc; an MCTS with hundreds of nodes could blow past the paper's 20-minute budget. **Measure before betting architecture
on it** — the same flag QiMeng-GEMM raised, now sharper because TensorOp's search depth is larger.

The ARM and NVIDIA results show the framework's **margin shrinks against well-funded vendor libraries**. The strong
RISC-V wins reflect a genuine vendor-library gap. For Apple Silicon, MPS (Metal Performance Shaders) and the new
MPSGraph / Apple Neural Engine paths are vendor-tuned; the realistic outcome of an Apple-Silicon port is not "beat
Apple's own kernels by 100%" but "match or modestly improve, on selected dimensions, with much faster development
turn-around." That's still valuable for Linus — the development-cost reduction is the durable claim — but the
performance-multiplier framing should be calibrated to "competitive with Apple-tuned, not 2.5× over."

The MCTS LLM-policy ablation shows MCTS-without-history degrades **below** naive MCTS — meaning a half-implemented port
that drops the LLM context plumbing actively hurts. The framework is not a "drop in any LLM, get gains" proposition; the
planning-context plumbing is load-bearing. For a Linus implementation, this means the MCTS loop's session-store +
audit-log integration (planned per CLAUDE.md and the workgraph JSONL convention) needs to track the path-history and
global-history with sufficient fidelity to feed the LLM at expansion time.

## Open questions for Dan

1. **Apple Silicon port — first target: Metal kernel or MLX op?** TensorOp's hardware breadth (RISC-V + ARM + NVIDIA)
   suggests the framework is platform-agnostic in principle. The choice of first Apple-Silicon target shapes the port's
   risk profile: a Metal Shading Language SGEMM kernel reuses the most-mature Apple toolchain (Xcode metal compiler,
   MTLComputePipelineState) and is closest to TensorOp's NVIDIA-Tensor-Core analog; an MLX-level operator gets us
   straight to the inference path Linus uses but exposes the framework to less-trained-data territory; an ANE intrinsics
   target gets us the most strategic Linus value (ANE is underutilized) but bets on training-data that may not exist.
   Recommend Metal SGEMM first (mature toolchain, maximum signal), MLX op second (Linus integration), ANE deferred to
   Phase 8.

2. **Manuals corpus for Apple Silicon — Phase 7 deliverable, or vendor-docs-as-Maestro-extract.** TensorOp's
   factor-extraction pipeline assumes a parseable manuals corpus. Apple's docs are PDFs + Markdown + WWDC video
   transcripts — not a clean parseable corpus. Two paths: (a) curate a Phase 7 `hint-repos/apple-silicon/` deliverable
   modeled on the typed-structured-prediction shape (per the "Reusable" section above); (b) lean on hosted Maestro
   (Claude) at Phase 6 spike time to extract factors interactively, deferring the curated corpus to a later phase when
   the protocol's value has been validated. Path (b) is faster to first signal; path (a) is the durable answer if the
   spike succeeds.

3. **MCTS wall-clock vs. M1 Max kernel-launch latency.** The paper reports overall development time but not
   per-iteration latency. For a Linus port, the MCTS inner-step (compile + dispatch + benchmark) on Metal could be
   substantially slower per iteration than nvcc on NVIDIA, because Metal's compile-and-link path is heavier and
   kernel-launch latency on M1 Max is non-trivial. **Measure first, architect second** (CLAUDE.md §Measure, don't just
   estimate, §Trust the OS page cache analog). A single-microbenchmark of Metal compile-and-launch latency for a trivial
   GEMM should precede any MCTS-port commitment, and the result determines whether MCTS is the right tuner or whether
   something lighter (beam search, hill climbing, evolutionary) fits the latency budget better.

4. **Local Worker LLM viability for the protocol.** TensorOp's LLM-portability evidence — GPT-4o, Claude 3.5 Sonnet,
   DeepSeek-V3 (236B), Llama-3.1-405B all hit similar performance — is encouraging but **all evaluated models are 405B+
   or large proprietary**. The Codestral-22B failure on QiMeng-GEMM suggested a multi-turn-protocol size threshold
   somewhere between 22B and 30B. For Linus's local-only Worker plan (Qwen3-Coder family at 7B–30B; future
   Linus-flavored Kimi-K2 derivative at 1T-MoE), a calibration run is needed: can Qwen3-Coder-30B-A3B sustain the
   factor-extraction → sketch generation → kernel-printer-Python → MCTS-action-proposal protocol across many turns
   without quality collapse? This determines whether the Phase 6/7 spike runs on a hosted teacher (Claude) for
   bootstrapping or directly on local Workers from the start.

5. **MCTS as a Phase 2/3 orchestration primitive.** Beyond the kernel-codegen application, the MCTS-with-LLM-policy
   pattern (typed action proposal + correctness/performance pruning + tree-of-prior-actions as scratchpad) is a generic
   agentic-orchestration primitive. Should Linus's Phase 2/3 agent-spawner / work-graph (DEC-0050, DEC-0051) explicitly
   support tree-shaped work-graphs alongside the workgraph-style DAG, with the MCTS variant as one realization? The
   generality argument is strong (any task with a verifiable correctness oracle and a search space of typed actions
   fits); the implementation cost is the typed-action and tree-state plumbing. _Seed: DEC-0057 candidate._

6. **Where does TensorOp sit in the g12-llm-hardware-design synthesis hierarchy?** The cluster has GEMM (C-level),
   TensorOp (assembly-level + multi-operator + multi-platform), Kernel (planner/coder + RL), CRUX (Verilog), CodeV
   (Verilog with multi-level summarization + RLVR successor), MuPa (sequential→parallel CUDA), SALV (signal-aware
   Verilog), GMEM-Attention (attention kernels), VEGA (compiler back-end gen), AutoOS (OS config tuning), Xpiler
   (transcompilation), and the cpu-v1 repo. TensorOp's spot in the cluster narrative is the **methodological middle** —
   the simplest framework that descends to hardware primitives, the broadest framework on hardware coverage. Worth
   deciding: does the synthesis frame TensorOp as the canonical Apple-Silicon-port target (because no RL, broadest LLM
   portability)? Or as one of several candidate primary spikes alongside Kernel (highest sophistication) and GEMM
   (lowest barrier)?

7. **Operator coverage roadmap beyond GEMM and Conv.** TensorOp's primitive vocabulary (T/R/V/L/P) was extended from
   GEMM-only (GEMM-paper) to GEMM+Conv (this paper), but the LLM-needs-of-Linus surface is wider: flash-attention,
   quantized dequant kernels, MoE-dispatch primitives, scan / sort / sparse. For a Phase 6/7 Linus deliverable, what is
   the operator coverage target — single Metal-SGEMM (smallest scope, fastest validation), a SGEMM + a flash-attention
   kernel (validates two operator classes), or a parameterized pipeline (Conv + GEMM + a MoE-dispatch primitive)?
   Coverage roadmap shapes both the Phase 6 fine-tune dataset shape (Open Question above) and the Phase 7
   idea-to-reality skill class scope.
