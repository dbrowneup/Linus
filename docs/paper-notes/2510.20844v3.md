# TrustResearcher: Automating Knowledge-Grounded and Transparent Research Ideation with Multi-Agent Collaboration

## TL;DR

Zhou, Zhu, Chen, Wang, and Wang's WWW Companion '26 demo paper (ACEM, Shanghai Jiao Tong University, with corresponding
authors at SJTU and UNSW; arXiv v3 dated January 2026, 6 pages) introduces **TrustResearcher** — a multi-agent
demonstration system for **knowledge-grounded and transparent research ideation**. The system addresses two named gaps
in prior automated-ideation pipelines: (1) the absence of multi-stage, granular knowledge grounding in single-monolithic
ideation steps, and (2) the opacity of agentic coordination in existing systems whose internal reasoning traces are
inaccessible to inspection. TrustResearcher decomposes ideation into **four coupled stages**: (A) **Structured Knowledge
Curation** — topic decomposition into well-formed search queries dispatched through the Semantic Scholar API under a
fixed retrieval budget, followed by an **incremental four-phase KG construction** (entity extraction → mini-batch
enrichment → degree-based expansion of top-K=10 high-degree nodes → hybrid sampling 60% high-degree + 40% random) that
combines precision with exploratory breadth; (B) **Diversified Idea Generation** — literature-informed planner agents
distill the curated KG into research blueprints across three facets (Problem Statement / Proposed Methodology /
Experimental Validation), then three parallel strategies (base variants, GoT/Graph-of-Thought variants that resample
branching-factor-3 / depth-5 KG-grounded reasoning paths scored on node quality 0.6 + edge-type diversity 0.2 + length
preference 0.2, and cross-pollination of top-ranked ideas) generate the candidate pool with α=10 over-generation and
real-time string + semantic deduplication; (C) **Multi-stage Idea Selection** — internal Jaccard-or-LLM-score-based
pruning followed by external selection comparing candidates against retrieved literature via BGE-M3 embeddings,
retaining candidates with cosine similarity below 0.7 to avoid duplicating prior work; (D) **Expert Panel Review &
Synthesis** — asynchronous reviewer agent (technical soundness, feasibility) and novelty agent (originality,
distinctness) score each idea on a 5-dimension 1-5 rubric (feasibility, expected impact, technical novelty,
implementation complexity, distinctness from prior work), an aggregator module fuses both perspectives, and ideas
scoring above ~3.5 (weak-accept threshold) are flagged high-quality. The demonstrated case study uses **k-truss breaking**
(an active graph-mining problem from the authors' own line of work) as the live stress-test topic; TrustResearcher with
GPT-5 produces three candidate ideas in ~15-30 minutes consuming 200K+ tokens, and one — "Scalable Approximation Framework
for Real-Time k-Truss Disruption in Massive Dynamic Graphs" — receives novelty 4.2 / clarity 4.5 with scalability flagged
as the limitation. The system exposes intermediate reasoning states, execution logs, configurable agents, and
JSON-archived per-component outputs through a four-region web interface (Session Management / Status Monitor / Final
Output / Real-time Activity Logs). Live demo + source code released at `github.com/valleysprings/TrustResearcher`. **For
Linus, TrustResearcher is a directly relevant Phase 7+ skill prototype** — a worked example of structured, transparent,
multi-agent scientific ideation that Linus's eventual research-skill Workers will look structurally similar to — but the
paper-as-demonstration is more **proof-of-concept than empirical-validation**: the case study is a single self-curated
topic from the authors' own research area, with no head-to-head comparison to prior ideation systems on a public
benchmark. The reusable content is the *architecture* (four-stage pipeline with transparency hooks), not the *claimed
performance*.

## The problem (in plain language)

Two interlocking gaps frame the paper. First, **existing LLM-based ideation systems collapse retrieval and reasoning
into a single monolithic stage**, leading to outputs that either hallucinate freely (when prompts are open-ended) or
parrot the literature (when prompts are tightly constrained). The authors' framing: knowledge grounding must operate at
multiple granularities — not a single retrieval step that returns a flat top-K list, but a structured, traceable
multi-phase construction that organizes retrieved papers into a coherent backbone, enriches with relations, and explores
via graph-based expansion. The §1 Introduction is explicit that "existing systems rarely achieve this balance, leaving
automated ideation bottlenecked by two fundamental gaps." The first gap is absence of multi-stage granular grounding.

Second, **the opacity of agentic coordination** in existing multi-agent ideation systems is a substantive obstacle for
researchers who want to inspect, verify, or critique the reasoning trajectory. Most multi-agent pipelines run as
black-boxes: a user submits a topic, the pipeline emits a set of ideas, and the intermediate decisions (which papers
were retrieved, which were filtered out, which KG entities were seeded, which reasoning paths were sampled, which agents
gave which scores) are not exposed. This is a problem for the **trust** dimension named in the paper's title:
researchers cannot delegate ideation to a system they cannot inspect, because the failure modes (incorrect retrieval,
biased exploration, agent collusion in scoring) are not detectable downstream. §1: "Such limitations have been widely
recognized as a core challenge for agentic research [10]." The second gap is opacity.

The authors position these two gaps as **complementary**: solving granular grounding without solving opacity yields a
better-grounded black-box; solving opacity without solving grounding yields an inspectable but un-trustworthy pipeline.
TrustResearcher's contribution is to address both — multi-stage grounding *plus* inspectable execution traces — as a
single coupled system.

## What they propose

TrustResearcher is a four-stage pipeline (Figure 1 of the paper) with explicit transparency hooks at every stage. The
authors emphasize that **each module mirrors a corresponding phase in human research iteration**: active grounding in
prior research, generating ideas, refining and evaluating hypotheses.

**Stage A — Structured Knowledge Curation (§2.1).** Given a seed topic, TrustResearcher first performs **LLM-guided
topic decomposition** to identify salient domain concepts at varying granularity. The LLM then organizes these concepts
into a set of well-formed search queries — *plural queries with different angles*, rather than a single handcrafted
query that risks under-coverage. Each query is dispatched via the Semantic Scholar API under a fixed retrieval budget.
Resulting paper sets are merged across queries, deduplicated, and further pruned based on topic-name overlap to produce
the final paper collection. The multi-granularity retrieval process **broadens coverage across related subtopics while
maintaining topical relevance** — i.e., it makes the system robust to single-query under-specification.

The retrieved papers then feed an **incremental four-phase KG construction**: (1) LLM-based entity extraction identifies
core problems, methods, and applications, forming a conceptual backbone; (2) mini-batch enrichment adds entities and
relations from paper metadata while maintaining contextual coherence (the per-paper enrichment respects the existing
backbone rather than overwriting it); (3) degree-based expansion samples top-K (default K=10) high-degree nodes to mine
adjacent or emerging concepts, providing exploratory breadth; (4) hybrid sampling (60% high-degree + 40% random)
uncovers latent methodological and theoretical links without introducing new entities. The four-phase construction
yields a structured and extensible KG that **combines precision with exploratory breadth** — the explicit design
trade-off the paper foregrounds.

**Stage B — Diversified Idea Generation (§2.2).** The KG is the substrate; the second stage transforms it into a
diverse but coherent set of candidate research ideas. Four sub-components:

*(B.1) Literature-Informed Planning.* A planner agent serves as the bridge between knowledge grounding and idea
synthesis. It analyzes high-degree entities and summarizes semantically relevant papers to extract key research cues.
Through LLM-based gap analysis, it identifies open limitations and decomposes them into three facets: **Problem
Statement / Proposed Methodology / Experimental Validation**. This structured blueprint provides the foundation for
large-scale idea generation.

*(B.2) Graph-of-Thought Exploration.* The authors adopt the GoT reasoning pattern [2] to sample KG-grounded reasoning
traces. The GoT module connects facet nodes to 20 high-degree KG entities and performs asynchronous depth-first
sampling with **branching factor b=3, depth d=5**. Reasoning paths are scored on three dimensions: node quality (0.6),
edge-type diversity (0.2), and length preference (0.2). High-quality paths are retained, each encoding a distinct
research trajectory that **enriches idea diversity**.

*(B.3) Multi-Strategy Variant Generation.* To ensure breadth, three strategies run in parallel: (1) Base variants
extend the faceted plan; (2) GoT variants reformulate high-scoring reasoning paths into structured proposals; (3)
Cross-pollination — triggered on demand — synthesizes hybrids of top-ranked ideas guided by KG cross-connections.
Redundant ideas are pruned via real-time string and semantic matching to keep the candidate pool diverse.

*(B.4) Iterative Refinement.* The refined pool undergoes multi-round critique and validation elaboration, expanding
experimental facets (datasets, metrics, performance evaluation, ablation) while preserving revision history, scores,
and reasoning traces. The resulting transparent idea pool captures reasoning traces, revision history, and supporting
literature.

The **over-generation factor α=10** is named: TrustResearcher generates 10× more ideas than the target portfolio size,
relying on subsequent selection stages to filter down. This is the standard "generate broadly, select narrowly" pattern
applied with explicit hyperparameter discipline.

**Stage C — Multi-stage Idea Selection (§2.3).** To ensure both coherence and novelty, TrustResearcher performs a
two-stage selection.

*(C.1) Internal Selection.* Each idea is evaluated across four weighted criteria — **novelty (0.30), feasibility
(0.25), clarity (0.20), and impact (0.25)** — to generate preliminary rankings. Top ideas undergo iterative merging:
pairs with Jaccard similarity > 0.85 are LLM-merged into unified proposals until convergence criteria are met. This
process eliminates redundancy via conceptual-level consolidation.

*(C.2) External Selection.* Remaining ideas are compared to retrieved literature using **BGE-M3 embeddings** [3] and
selected via cosine similarity over combined text fields. Candidates with maximum similarity below 0.7 are retained
(novel relative to the literature); the system logs top overlapping papers for transparency.

**Stage D — Expert Panel Review & Synthesis (§2.4).** Inspired by scientific peer review, TrustResearcher employs a
multi-agent evaluator to ensure that generated ideas are not only diverse and novel but also rigorous, technical, and
well-justified. Two specialized agents operate asynchronously:

*(D.1) Reviewer agent.* Focuses on technical soundness and feasibility.

*(D.2) Novelty agent.* Assesses originality and contribution relative to prior work.

Both agents follow a structured scoring template derived from standard conference and journal review rubrics,
evaluating each idea across five dimensions (feasibility, expected impact, technical novelty, implementation complexity,
and distinctness from prior work) on a **1-5 scale**, with qualitative feedback on strengths, weaknesses, and
recommended revisions.

An aggregator module fuses both perspectives by averaging dimension-level scores (equal weighting) and consolidating
textual feedback into a single meta-review. A unified score is then computed using a weighted aggregate that
emphasizes feasibility and originality, with all sub-scores preserved for traceability. Ideas with unified scores
above ~3.5 (corresponding to a "weak accept" in peer-review terms) are classified as **high-quality** and prioritized
for inclusion. If the number of high-quality ideas exceeds the target portfolio size, all are retained to avoid
discarding promising work; if fewer meet the threshold, remaining slots are filled by the generated best candidates.

**The Demonstration (§3 and §3.2 Case Study).** The interface is organized into four functional regions: (a) **Session
Control** — where users define the research topic, number of ideas, and launch new sessions; (b) **Status Monitor** —
displaying current phase, runtime progress, and metadata; (c) **Final Output** — summarizing session results with
structured downloads; and (d) **Real-time Logs** — showing detailed traces such as phase transitions, run-time, and
agent activities. This layout emphasizes transparency and interactivity: users can track the entire pipeline in
real-time, with each component's output preserved as JSON files for inspection.

The live case study demonstrates TrustResearcher on the **k-truss breaking problem** — a graph-mining primitive closely
related to community search. The k-truss structure is widely used to model cohesive subgraphs and is frequently adopted
as a building block in community-search pipelines; Zhou et al. [13] (one of the co-authors' own work) studies community
search with different cohesive structures or PPR pruning as structural priors, where truss-like cohesiveness
constraints are often used to enforce structural quality. The k-truss breaking task asks: given a graph and an integer
k, find a minimum-size edge set whose removal destroys all k-trusses. This problem is combinatorially challenging,
remains under active investigation in the graph mining community, and is a "suitable stress test for TrustResearcher's
ability to retrieve relevant literature, reason over incomplete knowledge, and explore novel research directions."

For the case study, TrustResearcher was prompted with:

> "Design scalable and robust algorithms for the k-truss breaking problem that bypass global trussness updates via
> localized, incremental, and approximation methods, enabling near-real-time interventions on large-scale graphs."

Using GPT-5, the system produced **three candidate research ideas within approximately 15-30 minutes (depending on
external LLM services), consuming over 200K tokens in total**. Among these, the authors sample a technically coherent
and novel one for in-depth analysis:

> **Scalable Approximation Framework for Real-Time k-Truss Disruption in Massive Dynamic Graphs** — a streaming-graph
> framework integrating streaming graph analytics with probabilistic estimation and distributed computing principles.
> Key components: (1) incremental k-truss maintenance via sketch-based triangle estimation; (2) hierarchical graph
> partitioning for computation locality; (3) locality-sensitive edge prioritization with adaptive scoring of disruption
> potential; (4) adaptive deletion budgeting aligned with temporal streaming windows; (5) distributed integration with
> Apache Flink / Spark Structured Streaming for near-linear scalability. The proposal includes scalability tests on
> R-MAT, Barabási-Albert, and SBM graphs up to 10^9 edges; streaming experiments on Twitter and citation networks;
> benchmarking against static and dynamic baselines; robustness under adversarial noise and delayed updates; ablation
> studies; distributed stress tests on real-world crisis networks.

Reviewer agents rated this idea: **novelty 4.2, clarity 4.5**, with scalability flagged as a potential limitation. The
aggregated portfolio of five high-quality ideas (average score ≈4.1/5) covers algorithmic, theoretical, distributed,
and learning-based perspectives, demonstrating that the system can produce technically grounded, diverse research
directions on a topic at the active research frontier.

## Key results

The paper does not report empirical comparisons to prior ideation systems on a public benchmark. The results are
**demonstrative**:

**Workflow correctness.** The full four-stage pipeline runs end-to-end on the k-truss breaking topic, producing three
to five candidate ideas with structured outputs, full traceability, and reviewer-rated quality. The interface exposes
the workflow at every step, and each agent's output is preserved as JSON for downstream inspection.

**Reviewer-rated idea quality.** The selected case-study idea scored **novelty 4.2 / clarity 4.5** on the 5-point rubric,
with scalability identified as a limitation. The aggregated portfolio averages ~4.1/5 across five ideas. The score range
(2-3 for weak candidates rejected during selection, 3.5-4.5 for accepted candidates) is consistent with standard
conference review distributions.

**Resource cost.** Approximately **15-30 minutes wall time and 200K+ tokens** to produce three ideas using GPT-5 as the
LLM substrate. The variability is attributed to external LLM service latency.

**Transparency artifacts.** Every component's output is **preserved as JSON files for inspection**, the real-time
activity log shows per-agent token costs (e.g., from the screenshot Figure 2: `idea_generator $0.9880 / 200,250 tokens`,
`idea_generator_self_critique $0.2519 / 39,210 tokens`, `idea_generator_refinement $0.2296 / 38,825 tokens`), total
session runtime is timestamped (`Total Execution Time: 1401.66 seconds` ≈ 23 minutes for the example session), and
phase transitions are tracked (`research_executor completed successfully!` / `frontresearcher completed successfully!`).
The audit trail covers what was retrieved, how the KG was built, which papers were closest matches, which agents
contributed to each score, and the full text of each generated idea with its revision history.

**Domain generality claim.** The authors emphasize that the four-stage pipeline is **domain-agnostic**: it can be
instantiated in any scientific field, with the k-truss breaking case serving as an illustrative example. The pipeline
itself does not embed graph-mining-specific knowledge; the Semantic Scholar API, the LLM, the BGE-M3 embeddings, and the
generic four-stage architecture are all domain-independent. The implicit claim is that the same system would work for
bioinformatics, chemistry, materials science, or any other research area with adequate Semantic Scholar coverage.

The honest read: this is a **proof-of-concept demonstration**, not a comparative empirical validation. The single case
study from the authors' own research domain is presented as evidence of architectural feasibility, not of generalized
performance. Whether TrustResearcher actually produces better-quality ideas than simpler baselines (e.g., a single-shot
GPT-5 prompt to "generate research ideas on k-truss breaking") is **not measured in this paper**. The paper does not
claim it does; the claim is more modest — that the architecture is transparent, reproducible, and produces
domain-coherent outputs on a stress-test topic. The empirical validation is deferred to future work.

## What's reusable in Linus

**Phase 7+ — TrustResearcher as a directly relevant prototype for Linus's eventual research-ideation skill.** Linus's
Phase 7 biology-skills roadmap ([biology-phase7-roadmap.md](../specs/biology-phase7-roadmap.md)) and the broader
Phase 7+ "domain skills" surface (per DEC-0047 biosecurity tier control, DEC-0046 external-API tool registry, DEC-0048
KB model-prediction edges) commit to Linus eventually shipping skill Workers that operate over Dan's KnowledgeBase
to produce structured scientific outputs. TrustResearcher is the most directly analogous published system in the
cloned-corpus collection: a multi-agent skill that takes a seed topic, retrieves relevant literature, constructs a
working KG, generates structured candidate research directions, applies internal + external selection, and produces
reviewer-scored outputs with full traceability. For Linus's Phase 7+ "research-ideation skill" target, this is the
**reference architecture** to study and adapt. The translation: replace Semantic Scholar with Linus's KnowledgeBase
(paper-qa per DEC-0044 as the retrieval engine), replace the generic GPT-5 substrate with Linus's Worker registry
(Qwen3 family + Phase 6+ fine-tuned variants per DEC-0014, DEC-0043), keep the four-stage architecture (Knowledge
Curation → Idea Generation → Idea Selection → Expert Panel Review) as the skill's pipeline shape, keep the
transparency-by-design discipline (every intermediate state preserved as JSON, every agent's per-stage cost logged,
every score traceable) as the audit-log integration pattern, and keep the typed-rubric scoring (5 dimensions, 1-5
scale, weighted aggregate, weak-accept threshold) as the AgentReport output shape per DEC-0051. Phase 7+ skill spec
should reference TrustResearcher as the foundational design reference for "what a Linus research-ideation skill
looks like."

**Phase 7+ — four-stage decomposition as the canonical scientific-skill pipeline shape.** TrustResearcher's
A-Curation / B-Generation / C-Selection / D-Review decomposition is structurally identical to **what most
human-scientist research-ideation workflows look like** — gather literature, generate hypotheses, prune candidates,
review the survivors. For Linus's Phase 7+ skills more broadly (not just research ideation — also literature review,
experimental design, results interpretation, manuscript drafting), the four-stage shape is a directly portable
architectural template. The variations: literature-review skill might skip Stage D (no expert review of a literature
summary) but emphasize Stages A+C heavily; experimental-design skill might compress Stage A (the relevant literature
is already curated) and expand Stage B (the design space is the focus); results-interpretation skill might invert the
shape (start with results in Stage A, work backward through the literature in Stage B, generate interpretations in
Stage C, review in Stage D). The pattern is "decompose any scientific workflow into the four canonical stages, adapt
each stage's depth to the task." Worth a Phase 7+ skill-design-pattern ADR.

**Phase 3 — Graph-of-Thought (GoT) exploration as a reasoning-pattern primitive for multi-agent Workers (DEC-0050).**
TrustResearcher's GoT module is a worked Phase 3+ reference for **knowledge-graph-grounded reasoning** in a
multi-agent Worker setting. The pattern — facet entry nodes → 20 high-degree KG seeds → asynchronous depth-first
sampling (branching factor 3, depth 5) → multi-criteria scoring (node quality 0.6 + edge-type diversity 0.2 +
length preference 0.2) → retained high-quality paths fed forward as research trajectories — is directly applicable
to any Linus Worker that needs to explore an entity-relationship-rich substrate (Linus's KnowledgeBase, the Phase
3+ memory graph if Cognee-augmentation is adopted per the [ElephantBroker paper-note](2603.25097v1.md) §"What's
reusable in Linus", the dependency graph of a codebase under analysis). The GoT pattern is more disciplined than
"prompt the LLM to think about the topic broadly"; the parameterization (b=3, d=5, scoring weights) is empirically
tunable and the search trace is auditable. Worth a Phase 3+ reasoning-primitive spec; the Linus Worker registry
can flag specific Workers as "GoT-capable" (similar to function-calling-reliability tags in DEC-0033's CoT-gap
fingerprint extension).

**Phase 7+ — typed five-dimension review rubric as a canonical AgentReport scoring shape (DEC-0051 extension).**
TrustResearcher's reviewer + novelty agents produce structured 5-dimension scores (feasibility, expected impact,
technical novelty, implementation complexity, distinctness from prior work) on a 1-5 scale plus qualitative
feedback. This is structurally aligned with the typed-structured-prediction discipline (CLAUDE.md §"Typed
structured prediction for biology skills", DEC-0051 AgentReport): a machine-queryable structured head with a
free-text rationale field. For Linus's Phase 7+ AgentReport schema, the rubric is directly portable — a Linus
research-ideation skill's AgentReport carries the same 5-dimension scores plus rationale, plus optional
domain-specific dimensions (e.g., for biology skills: experimental tractability, ethical acceptability). The
rubric's weighted-aggregate-with-weak-accept-threshold pattern (~3.5 = weak accept) is a useful concrete
calibration point. Worth a Phase 7+ AgentReport extension that defines the canonical 5-dimension rubric for
research-ideation outputs alongside the BioReason-Pro / Trading-R1 examples that anchor the convention.

**Phase 7+ — over-generation factor (α=10) plus multi-stage filtering as a quality-vs-diversity trade-off pattern.**
TrustResearcher generates 10× more ideas than the target portfolio size, then filters via internal
(Jaccard-or-LLM-score-based) + external (BGE-M3-similarity-vs-literature) + expert-panel review. This is the
classic "generate-broadly-select-narrowly" pattern applied at three levels (internal-vs-internal redundancy,
internal-vs-external novelty, internal-vs-rubric quality), with the α parameter making the over-generation cost
explicit. For Linus's Phase 7+ skill design, the pattern is portable: any skill that produces a portfolio of
candidate outputs (research ideas, experimental designs, code refactorings, manuscript revisions) should
over-generate by a measured factor and apply multi-stage filtering. The α=10 is a starting point; the right value
depends on the task's noise floor (more diverse candidate quality → higher α; lower noise → lower α). Worth a
Phase 7+ skill-design-pattern note.

**Phase 2+ — transparency-by-design as the canonical audit-log discipline.** TrustResearcher's transparency
posture — every component's output preserved as JSON, every agent's per-stage cost logged with token counts and
dollar amounts, every phase transition timestamped, every score traceable — is the most thorough published
example of "design the system so its full reasoning trace is inspectable." For Linus's audit log
(`~/.linus/audit.jsonl` per ARCHITECTURE.md), the relevant lessons are: (1) per-stage / per-agent cost and token
counts as load-bearing audit fields, (2) JSON preservation of intermediate state (not just final outputs) as the
discipline that enables post-hoc debugging and quality analysis, (3) timestamped phase transitions as a structured
event stream rather than ad-hoc log messages. The ElephantBroker 100+-metric surface (per the
[ElephantBroker paper-note](2603.25097v1.md) §"What's reusable in Linus") and TrustResearcher's transparency
discipline together set the standard for what a Phase 2+ Linus audit-log schema should cover. Worth integrating
into the Phase 2 audit-log-schema spec.

**Phase 7+ — knowledge-grounded reasoning as a defense against hallucination, with KB integration.** The paper's
central engineering claim — that **multi-stage granular knowledge grounding is the antidote to both unconstrained
hallucination and rigid parroting** — is a directly portable design principle. The discipline of "ground every
generation step in retrieved evidence with explicit provenance" is what makes the audit trail meaningful and the
output trustworthy. For Linus's Phase 7+ skills, the equivalent pattern is: every research-skill output must be
grounded in KnowledgeBase retrievals via paper-qa (DEC-0044), with explicit citation of which papers / passages
support which claims. The four-state evidence verification model from ElephantBroker plus TrustResearcher's
explicit per-stage retrieval discipline together define what Linus's Phase 7+ skill-grounding pattern looks like.
Worth integrating into the Phase 7+ skill spec template.

## What's NOT applicable / hype filter

**The case study is a single self-curated topic from the authors' own research area; generalization claims are
not empirically validated.** TrustResearcher's k-truss breaking demonstration is a topic the authors are
themselves expert in (Zhou et al. [13] is cited as one of the co-author's own prior work on community search
with cohesive structures). This is **the most charitable possible setting** for the system: the authors know
which papers should appear in retrieval, which directions are genuinely novel, which scores are appropriate. The
paper does not run TrustResearcher on a domain the authors are non-expert in; the paper does not compare
TrustResearcher to simpler baselines (e.g., single-shot GPT-5 with the same prompt); the paper does not test
TrustResearcher on a held-out benchmark of human-rated research-ideation quality (such an evaluation suite would
require crowdsourced expert annotation, which is non-trivial). The architecture-as-feasible claim is honest; the
implicit "this produces higher-quality ideas than baselines" is **not supported by the evidence presented**. For
Linus, the takeaway is: **adopt the architecture, do not adopt the performance claim**. Phase 7+ skill quality
assessment will require Linus-specific evaluation (per Dan's task suite) and cannot rely on TrustResearcher's
case-study reading.

**The 15-30 minute / 200K-token cost is GPT-5-shaped and may be larger for local-model substrates.** The
reported costs assume GPT-5 as the LLM substrate. For Linus's Phase 7+ deployment on local models (Qwen3-4B,
Qwen3-32B, or Phase 6+ fine-tuned variants), the relevant questions are: (a) does the local model produce
comparable-quality outputs at each stage? (b) what is the wall-time cost on M1 Max with the Linus inference stack
(Ollama, pmetal, mlx-lm)? (c) does the 200K-token cost translate to a measurable energy/heat footprint that
constrains continuous use? None of these are addressed by the paper. For Linus's Phase 7+ planning, expect
substantial re-measurement on local substrates; the GPT-5 numbers are upper-bound references, not lower bounds.

**The Semantic Scholar API dependency is not a Linus dependency.** TrustResearcher's Stage A (Structured Knowledge
Curation) is built on the Semantic Scholar API. Linus's substrate is the KnowledgeBase (paper-qa per DEC-0044
plus Dan's personal corpus). For Linus's Phase 7+ research-ideation skill, **replace** the Semantic Scholar
dependency with paper-qa-mediated KB retrieval. The substitution is structurally clean (both substrates return
relevant papers + metadata), but the corpus content differs: Semantic Scholar is comprehensive across all
academic literature; Dan's KB is biased toward Dan's research areas (bioinformatics, computational biology, LLM
infrastructure, agentic systems). For research-ideation tasks within Dan's areas of expertise, the KB substitution
is plausibly an upgrade (deeper coverage of relevant prior work); for tasks outside Dan's domains, the KB will
under-cover and a Semantic-Scholar-equivalent retrieval source may be needed. Worth a Phase 7+ design decision on
whether to integrate an external retrieval source (Semantic Scholar, arXiv API, Google Scholar) alongside the
KB, or to constrain Phase 7+ research-ideation skills to Dan's research areas only.

**The four-criteria internal scoring (novelty 0.30 / feasibility 0.25 / clarity 0.20 / impact 0.25) is empirically
unmotivated.** The weights are presented as a configuration, not as a tuned outcome. Whether these specific
weights produce better idea selection than uniform weights (each at 0.25) or different distributions (more
weight on novelty, less on clarity) is not addressed. For Linus's Phase 7+ skill, the weights should be a tunable
hyperparameter, with empirical tuning against Dan's task suite as data accumulates. The TrustResearcher weights
are a starting point, not a settled configuration.

**The GPT-5 substrate dependency is API-only.** TrustResearcher's case study used GPT-5 (a proprietary
API-only model from OpenAI). Linus's commitment is to local-first operation. Two implications: (a) the
TrustResearcher case-study results are not directly reproducible on Linus's local substrate without
re-measurement; (b) the per-component token costs (Stage A uses ~50K tokens, Stage B uses ~100K tokens, Stage
D uses ~50K tokens) are GPT-5-API-shaped and may differ on local models (Qwen3 family typically requires more
tokens per equivalent output due to less-efficient tokenization for some domains, plus the local model's
context-handling characteristics). Phase 7+ planning should expect 2-5× more tokens on local substrates as a
working estimate.

**The "transparency claim" applies to the demo system, not to the underlying LLM's reasoning.** TrustResearcher
exposes the multi-agent coordination layer (which agent contributed what, at what cost, with what intermediate
state) — but it does not expose the LLM's own internal reasoning. The agents are still LLM-driven and their
internal chain-of-thought is whatever the LLM produces. The transparency is at the **system layer**, not at the
**LLM-reasoning layer**. For Linus, the equivalent transparency claim is: Linus's audit log exposes the
orchestration-layer decisions (which Worker ran which task, with what prompt, producing what output, at what
cost) — but the Worker's internal reasoning is still the LLM's reasoning, subject to the usual limitations of
LLM interpretability. The TrustResearcher contribution is at the system layer; do not over-claim it as a
solution to LLM interpretability.

**The 5-dimension scoring rubric is empirically unvalidated for Linus's domain.** The dimensions are derived
from "standard conference and journal review rubrics" — i.e., generic academic peer-review templates. For
Linus's Phase 7+ skills targeting specific scientific domains (bioinformatics, biology, materials science), the
dimensions may need domain-specific augmentation (experimental tractability for wet-lab biology, sample size
adequacy for clinical studies, mechanistic plausibility for chemistry). The 5-dimension rubric is a starting
point; Phase 7+ skill spec should plan for domain-specific extensions. Worth a Phase 7+ scoring-rubric design
note.
