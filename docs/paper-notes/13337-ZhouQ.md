# QiMeng-GEMM: Automatically Generating High-Performance Matrix Multiplication Code by Exploiting Large Language Models

## TL;DR

Proposes QiMeng-GEMM, a meta-prompt + auto-tuning framework that lets off-the-shelf LLMs (GPT-4o, Claude 3.5 Sonnet,
DeepSeek-Coder-V2, Codestral-22B) emit hand-tuned-quality GEMM code across heterogeneous hardware (RISC-V CPU, NVIDIA
RTX 4070 / A100). Five reusable optimization primitives (tiling, reordering, vectorization, layout, pipeline) are each
encoded as a meta-prompt with a platform-agnostic description plus a platform-specific hint repository; an auto-tuning
beam search iteratively recombines them. Achieves up to 113× over vanilla prompts on CPU and 9.65× on GPU; reaches 211%
of OpenBLAS on RISC-V and 115% of cuBLAS on NVIDIA — while reducing development time from senior-engineer-months to ~10
minutes (240× cost reduction).

## The problem (in plain language)

GEMM (general matrix multiplication) is the computational backbone of LLMs and most scientific computing — it accounts
for >90% of deep-learning FLOPs, and a well-optimized implementation can be over 1000× faster than a naive nested-loop.
Hardware vendors hand-tune libraries (Intel MKL, NVIDIA cuBLAS, AMD rocBLAS, OpenBLAS) at enormous human cost, and these
libraries are platform-specific by construction. Automatic compilation approaches (TVM, polyhedral compilers) explore
the program space without humans but consistently underperform hand-tuned code.

LLMs can generate functionally correct code for simple tasks, but they fail on high-performance GEMM because they don't
understand the architectural characteristics of specific hardware: GPU memory hierarchies (global → shared → local),
warp-level execution patterns, RISC-V vector intrinsics, cache-line layout, and which optimization primitives to combine
in which order. A vanilla "write me a fast GEMM for an A100" prompt yields code that hits ~3% of cuBLAS performance.
Even chain-of-thought prompting only modestly closes the gap. The question the paper attacks: can prompt engineering,
structured around domain-specific optimization primitives and a search procedure, push an off-the-shelf LLM to match or
beat hand-tuned vendor libraries?

## What they propose

**QiMeng-GEMM** has two coupled mechanisms:

1. **Meta-prompts.** Five canonical GEMM optimization primitives (tiling, reordering, vectorization, layout, pipeline)
   are each encoded as a meta-prompt template with two components:
   - **Platform-agnostic description** — natural-language statement of the optimization concept (e.g., "tiling
     partitions the matrix into blocks that fit into cache to enhance memory access efficiency").
   - **Platform-specific hints** — a curated repository keyed by `(primitive, target hardware)` returning natural-
     language details and code skeletons. Example: tiling on a RISC-V CPU returns a single-level cache-block-tile
     for-loop nest; tiling on an NVIDIA GPU returns a warp/thread/block multi-level tile with a CUDA skeleton.

   The meta-prompt is **instantiated** at run time by retrieving the relevant hint and integrating it into the template.
   This decouples the universal optimization concept from the platform-specific realization and lets the same primitive
   set apply across CPU and GPU back-ends.

2. **Auto-tuning beam search.** Optimization is treated as a tree search over permutations of primitives. Starting from
   a naive GEMM root, each iteration: (a) the LLM proposes candidate next-primitives based on the current code's
   characteristics; (b) each candidate is instantiated with platform-specific hints and used to ask the LLM to transform
   the code; (c) the new code is compiled and benchmarked; (d) functionally incorrect or low-performance branches are
   pruned; (e) the top-k branches survive into the next iteration. The search terminates on saturation or a
   max-iteration budget. The pseudo-code (Algorithm 1) reads as a textbook beam search over a code-edit graph where the
   edits are LLM-generated under prompt control.

The overall stance is: don't fine-tune the LLM, don't build a new code-generation model, **engineer the prompt
substrate** so a frozen LLM can climb the optimization landscape. The meta-prompt template + hint repository encodes the
same kind of expert knowledge that goes into a hand-tuned BLAS library, but in natural-language and code-skeleton form
rather than in finished code.

## Key results

- **Vs. vanilla prompts.** Up to **113×** speedup on RISC-V (XuanTie C910) with GPT-4o; up to **9.65×** on RTX 4070; up
  to **3.92×** on A100. Speedup grows with matrix size (113× on N=4096, 34× on N=512), tracking the LLM-context GEMM
  workload trend.
- **Vs. hand-tuned vendor libraries.** Reaches **211% of OpenBLAS** on the C910 RISC-V CPU (10.23 vs 4.85 GFLOPS at
  N=4096) and **115% of cuBLAS** on RTX 4070 GPU (10.55 vs 9.14 TFLOPS at N=768). Trails A100 cuBLAS by ~5% at peak but
  matches at most matrix sizes.
- **Vs. CoT prompting.** Substantially better — CoT alone gets only 1.4–4.5× over vanilla on CPU; QiMeng-GEMM gets
  52–101× on the same configurations.
- **LLM scaling.** Larger / stronger LLMs benefit more. GPT-4o and Claude 3.5 Sonnet show big gains; DeepSeek-Coder-V2
  (236B, open) lags the closed models but still meaningfully helps; Codestral-22B is too small to reliably follow the
  multi-turn meta-prompt protocol on GPU.
- **Development cost.** A senior coder takes ~5 days for a tuned GPU GEMM and ~3 days for CPU; QiMeng-GEMM takes 10
  minutes (GPU) and 7 minutes (CPU) — **~240× speedup** in human-time. Output beats junior coders outright and matches
  or beats senior coders.
- **Ablations.** Without platform-specific hints, the LLM cannot generate correct warp-tiled, thread-tiled, or
  float4-vectorized GPU code at all (functional failures). Auto-tuning produces functionally correct code on 76% of
  search trails; 36% beat OpenBLAS; the best trail is 2× OpenBLAS.

## What's reusable in Linus

QiMeng-GEMM is the earliest paper in the QiMeng family and arguably the cleanest expression of the **idea → reality**
spine for kernel code: the LLM produces high-performance matmul source; a downstream actor (a compiler — clang for
RISC-V, nvcc for CUDA) accepts it and turns it into a binary that runs at vendor-library quality. No new model, no
fine-tune, no RL — just a structured prompt substrate plus beam search. This makes it the most directly liftable
paradigm in the QiMeng cluster for Phase 6+ Linus work, because there is no training infrastructure to reproduce.

**Phase 6/7 — MTMC's earlier sibling for Apple Silicon kernel codegen.** The macro/micro decomposition of
[QiMeng-Kernel (2511.20100v1.md)](2511.20100v1.md) is a more sophisticated descendant of the QiMeng-GEMM paradigm —
QiMeng-Kernel adds an explicit planner LLM and RL-trained coder, where QiMeng-GEMM uses prompt engineering and beam
search alone. For a Phase 6/7 Linus spike on Apple Silicon kernel optimization, QiMeng-GEMM is the right starting point
precisely because it is simpler: no RL training, no policy network, just (a) a meta-prompt template, (b) a hint
repository, (c) a benchmark harness, and (d) beam search. The minimum-viable port is:

- _Artifact the LLM produces:_ MLX kernel source or Metal Performance Shaders (MPS) kernel source for a small target (a
  single GEMM variant, a flash-attention micro-kernel, a quantized dequant kernel).
- _Downstream actor that accepts it:_ the MLX / Metal compiler toolchain, exactly analogous to clang/nvcc.
- _What must be true for Linus to replicate the discipline locally:_ a curated hint repository for Apple Silicon's
  unified memory model + Metal tile/thread group structure + ANE primitives where applicable; a compile+benchmark
  harness running on the M1 Max; a frozen Worker LLM (Qwen3-Coder, DeepSeek-Coder-V2, or hosted Claude during
  development) playing the prompt-driven coder. No fine-tuning is required for the v1 spike — that's the point.

This maps to **Phase 6d** (`docs/specs/phase6d-streaming-target.md` already implies Linus needs handcrafted Metal
kernels somewhere) and to the broader **Phase 7 idea-to-reality skill class** seeded in
[`qimeng-category-promotion.md`](../specs/qimeng-category-promotion.md). The auto-tuning beam search is structurally
identical to the agent-spawner's role-typed work-graph pattern (DEC-0050, DEC-0051) — primitives are roles, the search
tree is the work-graph, prune-on-correctness maps directly to typed AgentReport validation.

**Phase 6 — meta-prompts as a fine-tuning substrate.** Even without porting to Apple Silicon, the meta-prompt template +
hint repository is itself a high-quality dataset shape for a kernel-codegen fine-tune. The pattern "primitive name +
agnostic description + platform-specific hint + correct output" is exactly the
typed-structured-prediction-wrapping-rationale convention from CLAUDE.md (S25, biology-skills) applied to kernel code:
`{primitive: "tiling", platform: "metal", hint: "...", code: "...", rationale: "..."}`. A LoRA fine-tune over a
synthetic dataset of (primitive, platform, hint, correct-code) tuples — bootstrapped by running QiMeng-GEMM-style
prompting against a strong teacher LLM — is a plausible Phase 6 deliverable.

**Phase 2/3 — auto-tuning beam search as an orchestration substrate.** Stripped of the GEMM specifics, QiMeng-GEMM's
auto-tuning loop is a generic agent-driven program-search pattern: a tree of code states; each step a typed action
proposed by the planner LLM; branch evaluation against a correctness + performance oracle; top-k prune; iterate. This is
a Phase 2/3 orchestration primitive, not a Phase 6 thing — it's the same pattern as workgraph's DAG-based dispatch (G7
synthesis) extended with a fitness function and pruning. Worth noting that **the search graph is a durable artifact** in
the sense of DEC-0030 (scratchpad as first-class) — prune trails are episodic-memory candidates, not just throwaway
state.

**Idea → reality discipline.** Per [`qimeng-category-promotion.md`](../specs/qimeng-category-promotion.md) §"Dan's
stated framing": QiMeng-GEMM is the cluster's purest demonstration that an LLM can produce a real artifact (matmul
source code) that a real downstream actor (compiler) accepts and converts into a runnable binary at expert quality.
There is no human-in-the-loop between the LLM and the compiled binary other than benchmark feedback — and the benchmark
feedback is itself automated (the auto-tuning loop runs the compiled code and reads GFLOPS off the benchmark harness).
For the broader Linus design-and-build vision, GEMM kernel codegen is the smallest-scope, most auditable instance of the
pattern; it is also the one most directly relevant to Apple Silicon viability.

## What's NOT applicable / hype filter

The 113× headline number is over **vanilla prompts**, not over a strong baseline. Beating cuBLAS by 15% and OpenBLAS by
100%+ are the claims that actually matter for a hand-tuned-quality assertion, and those are platform-dependent (only on
certain matrix sizes). On A100, QiMeng-GEMM trails cuBLAS at peak; the result is "competitive on a wide range of sizes"
rather than "uniformly faster than cuBLAS."

The hint repository is **hand-curated by the authors** — it encodes domain expertise about RISC-V vector intrinsics,
NVIDIA warp programming, and so on. The paper does not address how this repository is built or maintained for a new
hardware platform; for Linus targeting Metal/MLX, this curation work is a cost the paper does not quantify. The
optimistic read is "you write a hint repo once per platform"; the realistic read is "this is non-trivial bioinformatics-
of-hardware work and is the actual unit of effort being amortized."

The five-primitive set (tiling, reordering, vectorization, layout, pipeline) is **GEMM-specific**. For other kernel
classes (flash-attention, scan, sort, sparse) the primitive vocabulary is different. The general principle — hand-curate
a primitive set + hint repo + auto-tune — generalizes; the specific dataset does not.

The Codestral-22B failure (1.0× across all sizes on GPU) is informative: smaller models cannot follow the multi-turn
meta-prompt protocol reliably. For Linus's local-Worker plan (Qwen3 family at 7B–32B on M1 Max), this is a yellow flag —
the protocol may need adaptation (shorter prompts, more aggressive primitive scaffolding) for sub-30B local models. A
Phase 6/7 Apple-Silicon port would need to test whether Qwen3-Coder-30B-A3B or DeepSeek-Coder-V2-Lite can sustain the
protocol; the paper's evidence suggests this is not a given.

The paper does not address **portability of the LLM-emitted code itself**. A QiMeng-GEMM-generated CUDA kernel does not
run on Apple Silicon. The reusable component is the methodology (meta-prompts + auto-tuning), not the artifacts.

The auto-tuning loop's compile-and-benchmark inner step assumes a fast benchmark harness. For Apple Silicon Metal
kernels, kernel launch latency is non-trivial; the search wall-clock could be substantially worse than RISC-V/CUDA.
Worth measuring before committing to an MTMC-style or QiMeng-GEMM-style auto-tuner architecture.

It connects most strongly to its later sibling [QiMeng-Kernel (2511.20100v1.md)](2511.20100v1.md) — the

## Open questions for Dan

1. **Apple Silicon port — minimum viable target.** The paper's RISC-V + NVIDIA results suggest the meta-prompt + hint
   repo + auto-tune pattern is platform-agnostic. What's the smallest meaningful Apple Silicon target for a Phase 6/7
   spike — a single GEMM variant in MLX, a Metal SGEMM kernel, a flash-attention micro-kernel, a quantized dequant
   kernel? The smaller the target, the cheaper to validate that the pattern ports; the larger the target, the more the
   result speaks to Phase 6d weight-streaming kernel needs.

2. **Frozen Worker LLM viability.** Codestral-22B failed to follow the multi-turn meta-prompt protocol on GPU.
   QiMeng-GEMM's headline results all use closed/large models (GPT-4o, Claude 3.5, DeepSeek-Coder-V2-236B). For a
   local-only Linus port using Qwen3-Coder (7B–32B) or a future Linus base model, is the protocol viable, or does it
   need adaptation (shorter prompts, fewer primitives per turn, more explicit scaffolding)? Worth testing before
   committing to the architecture — and worth testing **before** any Phase 6 fine-tune, because if a pre-fine-tune local
   model can't follow the protocol there's no signal to fine-tune toward.

3. **Hint repository as a Phase 7 deliverable.** The hint repo is the actual unit of expert effort being amortized (the
   meta-prompt template is generic; the platform-specific hints are where the knowledge lives). Should Linus carry a
   curated `hint-repos/metal-kernels/` and `hint-repos/mlx-kernels/` as a first-class Phase 7 artifact, structured as
   typed-prediction records (primitive, platform, hint, code-skeleton, rationale)? This is the kind of
   Dan-knowledge-base content where biochem-style enzyme-cofactor-substrate tuples and kernel-primitive-platform tuples
   have surprisingly similar shapes.

4. **Auto-tuning loop as a generic orchestration primitive.** The beam-search-over-LLM-emitted-code pattern is not
   GEMM-specific. Is there a Phase 2/3 case for adopting it as a Linus orchestration primitive (alongside or underneath
   the workgraph DAG-dispatch pattern from G7), with the search graph as a durable scratchpad-class artifact (DEC-0030)?
   If yes, the v1 substrate could be the QiMeng-GEMM port from Open Question 1 — same code base serves both the
   kernel-codegen skill and the orchestration-primitive validation.

5. **Where to place QiMeng-GEMM in the synthesis hierarchy.** With QiMeng-Kernel (2511.20100v1) as the macro/micro
   sophistication of the same idea, the in-flight g12-llm-hardware-design cluster synthesis needs to decide: does
   QiMeng-GEMM anchor the "earliest exemplar / cleanest minimum-viable port target" thread, with QiMeng-Kernel anchoring
   the "fully realized planner/coder + RL" thread? Or are they collapsed into a single arc? The decision shapes how the
   synthesis presents the family's design space to readers.

6. **240× development-cost claim — relevance to Linus.** The paper's headline development-cost reduction (10 minutes vs
   5 senior-engineer-days) is over a single GEMM variant. For Linus, where the long-tail of kernel needs is much wider
   (every quantization scheme × every attention variant × every MoE topology × every hardware target), does the claim
   hold? Or does the cost shift to building and maintaining the hint repository — i.e., the human work moves from
   "writing the kernel" to "writing the hint that lets the LLM write the kernel"? Either way, the claim is worth
   measuring on a Linus-relevant target before betting Phase 6/7 architecture on it.
