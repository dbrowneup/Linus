# Speed and Conversational Large Language Models (LLMs): Not All Is About Tokens per Second

## TL;DR

Tokens per second is the standard speed metric for LLMs, but it is **systematically misleading** as a proxy for how fast
a model actually completes real tasks. The authors run five open-weights ~7B models (LLaMA-2-7B, LLaMA-3-8B, Mistral-7B,
Yi-6B, Gemma-7B) on 660 multiple-choice questions across three task designs — pick the answer, paraphrase the question,
explain the answer — and find that **task-completion time rankings frequently invert the tokens-per-second rankings**.
For instance, LLaMA-3-8B and Yi-6B are slow in tokens/sec but fast to complete tasks because they generate fewer output
tokens for the same prompt. Gemma-7B, which is slow on tokens/sec for Task 3 (open explanation), is actually the fastest
to complete that task because it generates substantially fewer tokens. The core message: **token throughput conflates
tokenizer efficiency, verbosity, and generation speed into a single number that can mislead model selection**.

## The problem (in plain language)

When choosing a local LLM Worker — for Ollama, for the Linus orchestration layer, for any deployment — the go-to metric
is tokens per second (tok/s) as reported by inference servers and benchmarking sites. Higher is better, right? The paper
argues no — or at least, not always.

Three independent sources of noise corrupt the comparison. First, different tokenizers split the same text into
different numbers of tokens: a tokenizer that encodes "biochemistry" as one token versus three changes both the input
token count and the speed number, even though the model does the same amount of reasoning. Second, for the same prompt,
models differ wildly in how verbose their responses are — some pad with hedges and explanations, others answer tersely.
The model that answers in 50 tokens looks twice as slow if measured per-token, but finishes in half the wall-clock time.
Third, none of this is constant — it depends on the specific task: a model fast at short-answer tasks may be slow at
open-ended explanation tasks, and vice versa.

The practical consequence is that a user picking Ollama models by the tok/s leaderboard number may install a model that
is demonstrably slower on the tasks they actually run. For Linus, where Worker model selection directly affects session
throughput, latency seen by Dan, and the Maestro-budget cost of waiting for results, this is a directly operational
concern.

## What they propose

The paper proposes **task-completion time** as an additional measurement axis alongside (not replacing) tokens per
second. The methodology uses 660 miscellaneous multiple-choice questions drawn from MMLU as a stable, fixed prompt set,
then defines three task variants that span the verbosity spectrum:

- **Task 1 (minimal generation):** "Select the right answer (a, b, c, d) only." The model is instructed to produce the
  letter alone. This taxes the prefill phase heavily relative to generation; output token count is nearly zero.
- **Task 2 (fixed-length generation):** "Paraphrase the question without the answer choices." Output length is
  determined by the input — roughly a 1:1 token ratio. This is a constrained generation task.
- **Task 3 (unconstrained generation):** "Explain why the answer is correct." The model writes as much as it wants.
  Verbosity differences between models explode here.

All five models run on an NVIDIA A100 40 GB using 16-bit floating point, batch sizes swept from 1 to 128. Three
measurements are reported for each model × task × batch: average time per token (the canonical metric), total time to
complete all 660 questions, and input/output token counts.

The approach is deliberately low-tech — no new architecture, no new framework, no new training. The contribution is a
_measurement argument_: that the community should report task-completion benchmarks alongside tok/s, and that
task-specific speed benchmarks (translation, summarization, Q&A) would give users more actionable guidance than the
current single-number tok/s leaderboards.

## Key results

The headline finding is the consistent rank inversion between per-token speed and task-completion speed. Concretely:

- **Task 1 (short answer):** LLaMA-3-8B and Yi-6B are among the two **slowest** in tokens/sec, yet are the **fastest**
  to complete all 660 questions — because they produce the fewest output tokens. Gemma-7B generates substantially more
  tokens for the same short-answer prompt, taking longer despite reasonable tok/s.
- **Task 2 (paraphrase):** The correlation between tok/s and task-completion time improves (output length is more
  constrained), but Mistral-7B shows a large relative speed advantage on task completion that its raw tok/s rank
  wouldn't predict.
- **Task 3 (open explanation):** LLaMA-2-7B is fast in tok/s but slow at completing the task. Gemma-7B is the **fastest
  to complete** despite mediocre tok/s, because it generates far fewer explanation tokens than the others for the same
  questions.
- **Batch size matters:** All models improve substantially with larger batch sizes (the generation curves are steep from
  batch=1 to batch=8, then flatten). The relative ordering is mostly stable across batch sizes, but batch-size effects
  interact with task type.
- **Absolute numbers:** The paper doesn't publish clean table-form numbers for all conditions, but figures show total
  task-completion times ranging from roughly 100–600 seconds across models and tasks at batch=1, collapsing to 50–200
  seconds at batch=64.

The conclusion the authors draw is not that any specific model is universally best, but that **the right model for a
given deployment depends on the distribution of tasks it will actually run**, and tok/s alone cannot tell you that.

## What's reusable in Linus

**Direct, Phase 1 (Recon & Baselines):** The paper's methodology translates directly into a Linus benchmark design. The
`benchmarks/dan_tasks/` directory should include at minimum three task variants analogous to their three tasks — a
minimal-output task (structured extraction, label classification), a fixed-length task (paraphrase or summary), and an
open-ended task (explain, synthesize) — measured by wall-clock completion time per task, not just tok/s. Implementing
this for Ollama-served models on Dan's M1 Max would give Linus a hardware-local baseline that leaderboard numbers can't
provide.

**Direct, Phase 1:** When evaluating which Ollama model to use as the default Linus Worker, run a task-completion
benchmark on the actual tasks Linus will perform: code generation, scientific Q&A, literature summarization. The paper
makes a strong case that this will rank models differently than the Ollama leaderboard, and that the difference is
predictable from output-length tendencies, which can be measured cheaply in a 50-item smoke test.

**Direct, Phase 2 (MVP):** The orchestration layer's router needs to dispatch tasks to Workers. If Linus eventually
hosts multiple Workers (e.g., a fast-but-terse Mistral for short answers and a more expansive Qwen-Coder for code
generation), task-completion-time profiles per task type are exactly what the router needs to make good dispatch
decisions. This paper's three-task decomposition is a minimal viable schema for that profiling.

**Indirect, Phase 6 (Fine-Tuning):** Verbosity (output token count per task type) is a tunable behavior. A fine-tuned
Linus model should be calibrated to be terse where terseness is correct (structured outputs, labels) and expansive where
expansion is correct (synthesis, explanation). Measuring verbosity across task types before and after fine-tuning is a
diagnostic the paper makes easy to justify.

**Indirect:** The paper's argument is also a caution against over-optimizing tok/s during inference setup. Apple Silicon
and MLX inference speeds will be reported in tok/s; Linus should always pair those numbers with a small task-completion
benchmark before committing to a quantization level or a model variant, because a more aggressive quantization might
gain 20% tok/s while losing 40% in task verbosity calibration.

## What's NOT applicable

The paper runs on a single NVIDIA A100 40 GB — a GPU that is architecturally and memory-bandwidth-wise very different
from the M1 Max unified memory system. The absolute throughput numbers (seconds per 660-question benchmark) will not
transfer. The _rank orderings_ between models might transfer partially, but model behavior under Metal/MLX can differ
from CUDA due to kernel implementations, quantization strategies, and batch handling. Linus should treat this paper as
motivation to _run the benchmark_, not as a source of transplantable numbers.

All five models tested are in the 6–8B parameter range. Linus will likely also run models in this range
(Qwen2.5-Coder-7B, Mistral-7B derivatives), so the scope is reasonable — but the paper says nothing about 0.5B, 1B, or
1.5B models, which may be relevant if Linus deploys a very fast terse Worker for short tasks.

The evaluation uses floating-point 16-bit. Most Linus inference will use quantized formats (4-bit, 8-bit) to fit within
32 GB. The tok/s vs. task-completion-time discrepancy may widen or narrow under quantization, depending on how much the
quantization changes output-token verbosity (which it can, through its effect on sampling distributions near decision
boundaries).

The paper covers only conversational completion tasks from a single multiple-choice benchmark. Scientific reasoning,
code generation, and tool-use (Linus' primary Worker tasks) are not evaluated. The principle generalizes, but the
specific model rankings may not.

Finally, this is a measurement paper, not an optimization paper. It diagnoses a problem but does not propose a new
inference technique, architecture, or training method to fix it. The actionable output for Linus is a benchmarking
practice, not a new component.

## Connections

The paper's argument is structurally similar to the point [FineWeb (2406.17557v2)](2406.17557v2.md) makes about data
curation: the headline metric (token count for FineWeb, tok/s here) is easy to measure but fails to capture the thing
that matters (model quality / task completion time). Both papers are "the community is measuring the wrong thing"
interventions backed by systematic ablations.

It also connects to [LLM in a Flash (2312.11514v3)](2312.11514v3.md) on a different axis: flash-streaming inference is
explicitly about reducing wall-clock time for large models by trading memory bandwidth for latency, and the
justification for that engineering effort would be stronger if framed in task-completion time rather than tok/s. If
Linus eventually deploys mlx-flash for larger-than-RAM models, measuring task-completion time is the right way to
evaluate whether the streaming overhead is worth it.

The batching discussion (batch size 1 → 128, steep improvement then plateau) is a concrete datapoint for how Linus
should configure Ollama when running parallel Workers — batch size matters significantly up to roughly 8–16 requests,
which bounds the optimal degree of Worker parallelism for throughput on a single machine.

Nothing in this paper directly relates to the BitNet / quantization papers (2310.11453v1, 2402.17764v1, 2411.04965v1,
2504.18415v2), the KnowledgeBase embedding work (2408.08073v2), or the hypercube geometry papers (2401.00422v3, Horiike)
— it's an evaluation methodology paper that sits orthogonally to those threads.

## Open questions for Dan

1. **Benchmark design decision:** Should `benchmarks/dan_tasks/` be structured from the start around the three-task
   schema (minimal, fixed-length, open-ended) with wall-clock completion time as the primary metric, rather than being
   organized by topic or capability? Getting the measurement axis right now costs nothing and prevents the tok/s trap
   later.
2. **Worker selection experiment:** Would you like a smoke-test spec (50 items, three task types, wall-clock time
   measured) comparing whichever models you currently have pulled in Ollama — as a Phase 1 deliverable? The paper's
   methodology is simple enough that a Worker could write the script from a spec.
3. **Router implications:** If Linus eventually dispatches different task types to different Workers (fast-terse vs.
   slow-expansive), does the orchestration layer need to classify tasks before dispatching, or is the distinction
   between "short-answer" and "open-ended" tasks something the caller always knows at dispatch time? The architecture
   answer changes how complex the router needs to be.
4. **Verbosity as a fine-tuning target:** Is calibrating output verbosity (terse for structured tasks, expansive for
   synthesis) a Phase 6 fine-tuning objective worth adding to the roadmap explicitly? This paper makes a strong case
   that verbosity is a first-class model behavior, not a side effect — and that it directly determines task throughput
   independent of raw tok/s.
5. **Apple Silicon specifics:** The paper's A100 results are a motivation, not a number. Who runs this benchmark on
   Apple Silicon with MLX/Ollama and publishes the results? Is there a community resource (MLX Discord, Hugging Face
   discussions) where M1/M2/M3 task-completion benchmarks are being collected, or is this a gap Linus could fill with a
   small public benchmark release?
