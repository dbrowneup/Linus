# QiMeng-MuPa: Mutual-Supervised Learning for Sequential-to-Parallel Code Translation

## TL;DR

Proposes QiMeng-MuPa, a Mutual-Supervised Learning (MuSL) framework that trains two language-agnostic LLMs — a
**Translator** that converts sequential C into parallel CUDA (and back), and a **Tester** that generates unit tests for
both — in a closed loop alternating between **Co-verify** (run both source and target on real CPU/GPU and discard any
triplet whose outputs disagree) and **Co-evolve** (fine-tune both models on the verified triplets via back-translation).
Bootstrapped from unaligned monolingual C and CUDA corpora (BabelTower), the loop converges in four iterations to a
verified dataset of 10,563 (C, CUDA, 5 unit tests) triplets. When applied to Qwen2.5-Coder-7B it lifts Pass@1 by 28.91%
and Tester Valid-Tests by 68.90%, beats CodeRosetta by 1.56 BLEU / 6.92 CodeBLEU, and reaches Pass@k parity with
DeepSeek-R1 and GPT-4.1. The released Qwen3-0.6B Translator achieves Pass@1 of 84.44 on C→CUDA — a frontier-comparable
sub-billion-parameter parallelizer trained without any human-aligned parallel corpus.

## The problem (in plain language)

GPU programming with CUDA is fundamentally harder than C: threads, blocks, memory hierarchy, shared/global memory, warp
execution. Writing or porting code to CUDA is expert work, and the sequential→parallel translation problem is the gating
step between "code that works" and "code that exploits the hardware." Automating it would unlock a large amount of
existing C / Fortran / scientific code for GPU acceleration.

Two prior families of attempts:

1. **Back-translation NMT** (TransCoder, TransCoder-IR, BabelTower, CodeRosetta). Train an encoder-decoder on monolingual
   corpora using cycle-consistency. Works for syntactic / lexical fidelity but cannot guarantee **functional
   equivalence** — the translated code compiles but doesn't compute the same function as the original. BabelTower adds a
   compile filter; CodeRosetta adds AST entity recognition; both still fail Pass@k. The fundamental problem is that
   compile-success is a necessary but not sufficient correctness signal.
2. **Static / formal verification.** Rice's theorem says semantic equivalence is undecidable, so authors fall back to
   testing — but generating unit tests for CUDA is itself hard, because CUDA datasets are tiny (26k repos vs 41M for C),
   so any model trained to generate CUDA tests has the same data-scarcity problem as the translator.

The deeper issue is data alignment. Unit tests would let you verify functional equivalence by execution, but you don't
have a tester that produces good CUDA tests, because you don't have aligned (C, CUDA, test) triplets to train one on.
Meanwhile the translator can't reliably produce functionally-equivalent CUDA, because its training data was filtered
only on compilation. **The two problems are mutually-conditional**: a good translator would produce parallel code whose
tests you could trust; a good tester would produce tests that filter the translator's output to functionally-correct
triplets. Neither can bootstrap alone from monolingual corpora.

## What they propose

QiMeng-MuPa breaks the chicken-and-egg by making the two models bootstrap **each other** in a closed loop. Let
$\mathcal{F}$ be the Translator and $\mathcal{G}$ be the Tester; both start from a base code LLM (Qwen2.5-Coder-7B,
Llama3-8B-Instruct, or Qwen3-0.6B in the released checkpoint).

**Co-verify step.** For each source program $x$ in the unaligned BabelTower monolingual set:

1. The Translator emits $y = \mathcal{F}(x)$ (e.g., a CUDA kernel for a C function, or a C function for a CUDA kernel —
   both directions are trained simultaneously because the model is language-agnostic).
2. The Tester emits a unit test set $T = \mathcal{G}(x) = \{t_1, \ldots, t_n\}$.
3. Both $x$ and $y$ are wrapped (with a Function Call Wrapper for C, or a CUDA Kernel Wrapper that adds memcpy + grid
   launch + memcpy-back) so that they can be invoked uniformly under the test harness.
4. Each $t \in T$ is executed against both wrapped versions on real CPU/GPU. If any test diverges, or any version fails
   to compile or runs into a runtime error, the entire triplet $(x, y, T)$ is discarded.

The output of Co-verify is a high-quality verified triplet set $\mathcal{S}_i^{\models}$ in iteration $i$.

**Co-evolve step.** Fine-tune both models on $\mathcal{S}_i^{\models}$ via back-translation:

- Translator loss: $\mathcal{L}_{\text{trans}}(\theta_i) = -\mathbb{E}_{(x,y,T)}[\sum_j \log p(x_j \mid x_{<j}, y;
  \theta_i)]$ — i.e., teach $\mathcal{F}$ to recover the **source** from the verified target (back-translation).
- Tester loss: $\mathcal{L}_{\text{test}}(\phi_i) = -\mathbb{E}_{(x,y,T)}[\sum_j \log p(T_j \mid T_{<j}, y; \phi_i)]$ —
  teach $\mathcal{G}$ to generate the verified tests given the verified target code.

Iterate. Each iteration the verified-triplet count grows (from 759 C / 2459 CUDA at iteration 1 to 2079 / 8484 at
iteration 4 for the joint MuPa loop), saturation arrives around iteration 4, and the framework terminates.

**Why both directions and both languages share weights.** The Translator does C↔CUDA translation in both directions and
the Tester generates tests for both C and CUDA, all from a single set of weights. Empirically, the language-agnostic
shared-weight setup beats per-language separate models — the authors attribute this to total-training-data scaling
dominating per-language specialization at this scale.

The architecture in one sentence: **co-verification by execution gives you the supervision signal that monolingual
back-translation lacks; the two models that produce that signal also consume it, and the loop is closed by alternating
fine-tuning on filtered triplets.**

## Key results

- **Headline metrics (Qwen2.5-Coder-7B + QiMeng-MuPa, C→CUDA, BabelTower test set, 233 paired samples × 5 tests).**
  BLEU 84.96 (vs 72.07 baseline), CodeBLEU 84.98 (vs 47.79), Compile Pass 95.71% (vs 72.10%), Pass@1 85.15% (vs
  56.24%), Pass@5 95.73%, Pass@10 97.00%. Pass@1 lift of **+28.91 absolute**, Pass@10 lift of **+16.10**.
- **Llama3-8B-Instruct + QiMeng-MuPa.** BLEU 86.68, CodeBLEU 86.63, CPass 92.70%, Pass@1 85.56% (+29.44 vs baseline
  56.12%). Confirms the framework is base-model-agnostic.
- **Tester evolution.** Valid-Tests metric (VT, % of generated unit tests that compile + execute correctly) climbs
  68.90 absolute on Qwen2.5-Coder, ending at ~85% VT for both C and CUDA — comparable to GPT-4.1 (64.91 / 65.35) on the
  same task and **better** than DeepSeek-R1.
- **Vs. SOTA back-translation (CodeRosetta, 0.8B).** QiMeng-MuPa beats CodeRosetta by **1.56 BLEU and 6.92 CodeBLEU**,
  the gap that the paper's headline emphasizes for the existing-baseline narrative.
- **Vs. closed-source frontier models on Pass@k.** Qwen2.5-Coder-7B + QiMeng-MuPa lands within ~3% Pass@1 of
  DeepSeek-R1 (671B) and within ~2% of GPT-4.1, despite being two orders of magnitude smaller.
- **Sub-billion-parameter regime.** Qwen3-0.6B fine-tuned only on the QiMeng-MuPa-filtered dataset hits Pass@1 84.44
  on C→CUDA — first sub-billion-parameter parallelizer to match GPT/DeepSeek-class frontier on this task.
- **Speedup of generated CUDA.** Translated parallel code achieves up to **15,259× speedup** vs the original sequential
  C on suitable workloads (Table 5 in the paper). This is the "did the parallel code actually parallelize" check beyond
  Pass@k; functional equivalence at execution level _and_ meaningful HPC speedup.
- **Verified-triplet dataset.** 10,563 (C, CUDA, 5 unit tests) triplets after 4 iterations — released as a public
  artifact, useful well beyond MuPa itself.
- **Ablations.** (a) Without Co-verify (compile-filter only or no filter), performance peaks early and degrades — the
  iterative fine-tuning on functionally-incorrect data poisons subsequent rounds. (b) Freezing the Tester while
  evolving the Translator (or vice versa) loses ~5.7% Pass@1 / 3.15% VT — both directions of the loop are required.
  (c) Test coverage stays >95% across iterations (line / branch / branch-taken-once on C; ~95% on CUDA), so the Tester
  doesn't degrade in coverage as it adapts.
- **Error type analysis.** Iterating MuPa for 3 rounds eliminates almost all Type-9 (Logical-sequence / block-index /
  shared-memory) errors — the CUDA-specific failure mode that pure NMT methods can't catch — and reduces Type-4 / Type-5
  (syntax / undefined-identifier) to negligible levels.

## What's reusable in Linus

QiMeng-MuPa fits the **idea → reality** spine cleanly. Mapping the triplet:

- _Artifact the LLM produces:_ parallel CUDA source code (the Translator) and CUDA-compatible unit tests (the Tester).
- _Downstream actor that accepts it:_ `nvcc` + the CUDA runtime (executes the wrapped kernel), and the test harness
  itself (executes the unit tests on actual hardware to verify functional equivalence). The "compiler" is the
  immediate downstream; the "execution oracle" (running both versions and comparing outputs) is the verification
  downstream.
- _What must be true for Linus to replicate the discipline locally:_ (a) a real toolchain to compile and execute both
  source and target on the same hardware; (b) wrappers that paper over the calling-convention difference between the
  two languages; (c) a triplet store + filter pipeline; (d) base LLMs capable of multi-turn structured output for both
  translation and unit-test generation.

**Phase 6 — local-LLM training as a Worker self-improvement substrate.** This is the cleanest demonstration of an
agentic-self-supervision loop in the QiMeng family and the most directly portable to Linus. The Mutual-Supervised
Learning shape generalizes well beyond C↔CUDA: any task where you have two coupled abilities and a real-world execution
oracle to filter both outputs simultaneously is a candidate. For Linus, the natural targets are: (a) Python↔Rust
translation for the Linus codebase itself (an idea Dan has flagged in `experiments/` — Worker-pair could co-evolve a
Python-to-Rust translator and a property-based test generator using `pytest` and `cargo test` as the execution oracle);
(b) **MLX kernel translation** — see Phase 7/8 below. The MuSL pattern is in scope for Phase 6 because the
training-loop infrastructure is conventional fine-tuning (LLaMA-Factory + back-translation cross-entropy); the novelty
is the data-curation loop, not the optimizer. Tag this as a _Seed: DEC-NNNN_ for the Worker-pair self-supervision ADR
when it lands.

**Phase 7/8 — MLX/Metal kernel translation as the Apple-Silicon analog.** This is the most exciting Linus-specific
read. CUDA is to NVIDIA as MLX (or hand-written Metal) is to Apple Silicon. The corpus has very few (sequential C ↔
MLX kernel) aligned pairs because MLX is a young framework, exactly the data-scarcity asymmetry MuPa was designed to
break. A Linus-flavored MuPa-on-MLX would: train Translator on monolingual C / Python corpus and monolingual MLX
kernel corpus; train Tester to emit Python+MLX execution harnesses; Co-verify by compiling and executing both versions
on the M1 Max and comparing outputs. The execution oracle is the Metal compiler + MPS / MLX runtime. This is
structurally identical to MuPa and is arguably the highest-leverage application of the framework for Linus's specific
hardware context — it directly addresses the Phase 6d / Phase 7 question "where do the hand-tuned Metal kernels for
Linus come from?" without requiring Apple to release them or Dan to write them all by hand. Surface as **Phase 7
candidate skill** in the LLM-hardware-design synthesis (Tier 5 of the parent fan-out spec).

**Phase 6 — typed-triplet record as fine-tuning data shape.** The (source, target, unit-tests) triplet is exactly the
typed-structured-prediction-wrapping-rationale convention from CLAUDE.md (S25, biology-skills) generalized to code-pair
+ verification: `{source: "...", target: "...", tests: [...], oracle: "executed_match", rationale: "..."}`. The Linus
KB should adopt this shape natively for any code-translation-skill ingested into the corpus, and the released
10,563-triplet dataset is itself a Phase 6 fine-tune candidate even outside the Apple-Silicon-port question.

**Phase 2/3 — co-verify as an orchestration primitive.** Stripped of the C↔CUDA specifics, the Co-verify step is a
generic two-agent quorum-by-execution pattern: Translator agent emits artifact; Tester agent emits test; the
orchestrator runs both and accepts the triplet only on cross-validation. This is a natural extension of the workgraph
DAG-dispatch substrate (G7 synthesis) and the role-typed agent-spawner pattern (DEC-0050, DEC-0051) — the verifier role
becomes first-class rather than ad-hoc. Worth surfacing as a Phase 2/3 orchestration primitive separate from the
Phase 6/7 fine-tuning use; the loop without the fine-tune step is itself a useful agent pattern (multi-agent
self-consistency by execution).

**Phase 6 — language-agnostic shared-weight finding.** MuPa's empirical result that one model trained on both
directions and both languages beats four specialized per-(direction × language) models is a useful Phase 6 datapoint.
For Linus's eventual fine-tuned base, multi-skill rather than per-skill training is favored — consistent with the
"total-data-volume dominates per-task specialization" finding. Connect to DEC-0043 (memory-mode-specific fine-tuning
targets in Phase 6).

## What's NOT applicable / hype filter

The 28.91% Pass@1 lift is on a **tiny test set** (233 BabelTower paired samples). The set is well-coverage'd by the
generated tests (>95% line coverage), so the result is honest, but absolute headcount is small enough that confidence
intervals on Pass@k differences could swallow the SOTA-by-1.56-BLEU claim against CodeRosetta.

The CUDA Kernel Wrapper is a **hand-engineered scaffold** — a prompt template that asks the base LLM to generate
control-flow + memcpy + grid-launch boilerplate around the kernel body. It is non-trivial domain knowledge ("here's how
you make a CUDA kernel callable from C++"), comparable in shape to QiMeng-GEMM's hint repository. For Linus's
Apple-Silicon port, the analog wrapper for Metal/MLX is not free — somebody (Dan, hosted Maestro, or a separate
QiMeng-GEMM-style hint-repo effort) writes that wrapper before any MuPa-on-MLX loop can run. The paper does not
quantify that cost.

The framework requires **a working toolchain on real hardware** for both source and target during co-verify. For
NVIDIA + GCC, this is fine — both available on a single Linux box with `nvcc`. For Apple-Silicon MuPa, the analog
requires an MLX / Metal toolchain that can compile and execute candidate kernels on the M1 Max with low-enough latency
to make iterative co-verify tractable. CUDA kernel launch is microseconds; Metal kernel launch is non-trivial; the
search wall-clock could be substantially worse. Worth measuring a Metal-kernel-compile-and-execute latency profile
before committing to a MuPa-on-MLX architecture.

The CUDA-only test set means the framework's generality across **other** sequential→parallel pairs (OpenMP, MPI, SYCL,
HIP, Vulkan compute, WebGPU) is asserted but unproven. The language-agnostic claim is empirical from C↔CUDA only.

The released code is **partial as of 2026-05** — the README explicitly says "We will release the end-to-end training
and evaluation code in the near future." The shipped artifacts are the Co-verify infrastructure (`unit_test/`), the
inference scaffolds (`trans/vllm_predict.py`, `trans/vllm_predict_ut.py`), the eval scripts, and the BabelTower-derived
training data. The Co-evolve fine-tuning side relies on the user cloning LLaMA-Factory separately and registering the
filtered dataset — i.e., training is an external dependency.

The data scarcity ratio cited (26k CUDA repos vs 41M C) is a 2024 GitHub snapshot. By 2026-05 it's plausible the
absolute volumes have grown, especially with LLM-emitted CUDA flooding the open-source ecosystem; the data-scarcity
case for MuPa specifically against C↔CUDA may be eroding faster than the paper's framing suggests. The general
mutual-supervision pattern still applies to genuinely-low-resource pairs (Metal/MLX, novel ASIC ISAs, scientific DSLs).

The Translator + Tester pair is **two networks of similar size**. For Linus's 32 GB unified memory budget, running both
during inference (for an agentic self-verifying translation skill at runtime) requires careful memory planning — likely
sequential rather than concurrent. Co-verify at training time isn't subject to that constraint because the two models
each generate offline; runtime co-verify (e.g., as a Worker-pair pattern at inference) is.

## Open questions for Dan

1. **MuPa-on-MLX as the highest-leverage Phase 7 skill candidate.** If the C↔CUDA result transfers to C/Python ↔ MLX (or
   C ↔ Metal Shading Language), this is the cleanest path the corpus has so far to a Linus-native kernel-codegen skill
   that doesn't require Apple to release hand-tuned kernels. What's the smallest meaningful first target — a single
   element-wise kernel, a small fused-attention micro-kernel, a quantized-dequant kernel? And what's the wrapper-cost
   estimate (the MLX/Metal analog of the CUDA Kernel Wrapper) before any MuPa loop can run?

2. **Frozen Worker LLM viability for the Translator/Tester roles.** Codestral-22B failed the QiMeng-GEMM protocol;
   QiMeng-MuPa works on Qwen2.5-Coder-7B and Llama3-8B and even Qwen3-0.6B (after fine-tune). For Linus's local-Worker
   plan with Qwen3 on 32 GB unified memory, can two simultaneous fine-tuned 7B-class models run side-by-side at inference
   time for a runtime self-verifying translation skill, or is the realistic deployment "single Worker doing both roles
   sequentially"? This affects whether Co-verify is a training-time-only pattern or an inference-time orchestration
   primitive.

3. **Triplet KB integration.** The 10,563 verified (C, CUDA, 5-tests) triplets are a high-quality public dataset. Should
   the Linus KnowledgeBase ingest these as first-class typed records (per the Phase 6 typed-prediction shape), making
   them retrievable as exemplars for downstream skills (kernel-codegen, code-review, parallelization-suggestion)? The
   record shape is consistent with the model-prediction-edge-class spec (DEC-0048).

4. **Co-verify as a runtime primitive, not just a training-time loop.** The Translator/Tester quorum-by-execution
   pattern is generic. Is there a Phase 2/3 case for adopting it as a Linus orchestration primitive (alongside or
   underneath the workgraph DAG-dispatch pattern from G7), with the Tester-emitted unit-tests serving as a
   typed-AgentReport validator (DEC-0051) for Translator output? If yes, this is a Phase 2/3 deliverable that would
   _then_ provide the substrate for a Phase 6/7 MuPa-on-MLX fine-tune, rather than the other way around.

5. **Python↔Rust as a Linus-internal MuPa target.** Dan is learning Rust. The Linus codebase is mostly Python with some
   Rust components (pmetal, claw-code, claw-code-local). A Python↔Rust MuPa loop using `pytest` + `cargo test` as the
   execution oracle would (a) generate Python-to-Rust translations of bottleneck Linus orchestration code; (b) train a
   Worker that can do that translation on demand; (c) double as a learning aid for Dan (typed exemplars of "how this
   Python ends up in Rust"). Is this a reasonable Phase 6 sub-project, scoped much more tightly than the MuPa-on-MLX
   ambition?

6. **Language-agnostic shared-weights vs per-skill specialization for Phase 6 base.** MuPa's empirical result favors
   shared-weight multi-skill training over per-skill specialization at the 7B–8B scale. This is consistent with DEC-0043
   (memory-mode-specific fine-tuning) but in tension with a "many small specialized Workers" architecture. What's the
   synthesis: Phase 6 base model trained on a heterogeneous corpus including code-translation triplets, with
   memory-mode being the dispatch axis rather than per-skill weights?

7. **Co-verify execution oracle as a biosecurity-tier-control parallel.** DEC-0047 introduced biosecurity tiers gating
   generative biology. The Co-verify step here is structurally a "execution-gated tier" — output is admitted only after
   real-hardware execution agrees with the source. Is there a generalization where Phase 7 skills graduation uses
   execution-or-equivalent oracle gates as a uniform admission criterion (kernel-codegen → execution match;
   chemistry-skill → simulation match; bio-skill → predicted-vs-measured match)? Worth surfacing in the SAFETY.md
   autonomy-tier discussion.
