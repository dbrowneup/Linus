# A generative language model decodes contextual constraints on codon choice for mRNA design

## TL;DR
Trias is a **47M-parameter encoder-decoder (BART) language model** trained on **10 million full-length vertebrate coding sequences** from 640 species (NCBI RefSeq) to **reverse-translate proteins into species-specific codon sequences**. The model conditions on a species tag prepended to the protein input and generates codon sequences up to ~2,046 amino acids long. The headline architectural claim — and the one that matters most for Linus' design philosophy — is that Trias was trained **only on natural sequences, with zero exposure to expression-data supervision**, and yet its sequence likelihood scores correlate strongly with experimentally measured GFP protein output (ρ = -0.76), mRNA half-life (ρ = -0.84), and ribosome load (ρ = 0.82) on a held-out 30-variant benchmark. On the same benchmark, Trias-generated GFP sequences resemble the five highest-expressing variants more closely (76% codon similarity) than sequences produced by three commercial codon optimizers (GenScript, Twist, IDT). Code and weights are released openly (HuggingFace + GitHub) covering 600+ vertebrate species. For Dan's domain, this is one of the most directly useful nucleotide-language-model papers in the corpus: codon optimization for mRNA therapeutics and protein expression is a high-frequency biotech use case, and Trias is small, open, and ready to wrap as a tool.

## The problem (in plain language)
The genetic code is degenerate: most amino acids can be specified by two to six synonymous codons. So for any protein you want to express in cells — a vaccine antigen, a therapeutic enzyme, GFP for a reporter assay — there are astronomically many DNA sequences that encode the same protein. The SARS-CoV-2 spike alone admits more than 10^632 synonymous variants. Codon choice is not neutral. It changes how fast ribosomes translate the message, how stable the mRNA is, how much protein the cell ends up making, and even how the protein folds.

Commercial codon optimizers (GenScript, IDT, Twist) handle this with hand-engineered rules: pick the most-frequent codon for each amino acid in the target organism, smooth out runs of GC, avoid certain motifs. These rules are useful but coarse — they ignore long-range sequence context, position effects, domain structure, and the fact that organisms sometimes deliberately use rare codons (to slow translation in folding-sensitive regions, for example).

The deep-learning alternative has been to train models on experimental ribosome-profiling or protein-expression datasets, which are scarce and noisy. Trias asks a different question: **can a model trained purely on natural coding sequences — never told anything about expression measurements — learn the contextual rules that govern good codon choice?** If yes, you get a codon optimizer that scales with sequence data (which is abundant) rather than with expression assays (which are not), and you get a model whose recommendations reflect natural selection's accumulated wisdom about which codon goes where.

## What they propose
**Trias** is a sequence-to-sequence transformer based on the HuggingFace BART implementation: 6 encoder layers, 6 decoder layers, 8 attention heads each, hidden FFN size 2,048, ~47M trainable parameters total. The vocabulary is small and biology-aware: 64 codon tokens, 20 amino acid tokens, 640 species-specific tokens, and 4 special tokens. Input length cap is 2,048 tokens — a species tag plus up to 2,046 amino acids.

The training task is **reverse translation under masking**. The encoder receives a protein sequence with the species tag prepended and 15% of amino acid positions replaced with a `<mask>` token. The decoder autoregressively generates the corresponding codon sequence. Loss is the reconstruction loss against the true codon sequence. Because the codon table is never given to the model explicitly, learning to emit valid synonymous codons is itself a learned behavior (the paper confirms it acquires the codon table essentially perfectly during early training).

Training data is 10M coding sequences from 640 vertebrate species drawn from NCBI RefSeq. The test set is constructed carefully: an initial random split is followed by **BLASTp-based homology filtering** that moves any test sequence with significant protein-level similarity to a training sequence into the test set. This prevents the model from succeeding via sequence-similarity lookup rather than learned codon rules — a discipline that should be standard but often is not.

Compute is modest by foundation-model standards: a single NVIDIA A100 80 GB, batch size 8 with gradient-accumulation factor 4 (effective batch 32), Adam optimizer with cosine schedule, learning rate 1e-4, weight decay 0.01, dropout 0.1, flash-attention enabled, 650K steps (~2 epochs).

Decoding for sequence design uses **greedy search** — pick the most probable codon at each step conditional on the protein, the species, and the codons emitted so far.

## Key results
- **The model learns a meaningful codon-protein representation.** t-SNE of encoder hidden states clusters amino acids by biochemical property (hydrophobic, polar, charged) — analogous to what ESM-1b shows for protein-only models. Cross-attention maps show a dominant diagonal (each codon attends to its target amino acid), a vertical band at the species tag (species conditioning is global), and off-diagonal signal indicating long-range context use.
- **Generated sequences beat naïve baselines.** Mean codon identity to wild-type human sequences is **50% for Trias, 44% for a most-frequent-codon baseline, 37% for a uniform-random synonymous baseline**.
- **A bimodal generation pattern.** On ~6,000 human test sequences, Trias outputs split into a large bulk of high-sRSCU (frequent-codon) sequences and a smaller population that preserves the rare-codon character of low-sRSCU wild-types. GO analysis on the predicted-low-sRSCU set is enriched for transcription factors, suggesting Trias picks up on functional reasons rare codons cluster in some genes. Notably, **predicted sRSCU does not correlate with gene expression** in GTEx — consistent with recent evolutionary work that disputes the simple "highly expressed genes use frequent codons" rule for vertebrates.
- **Position, context, and domain effects.** Chi-squared analyses on 4.35M codon predictions show predictions depend significantly on previous codon identity (rare codons follow rare codons), on relative codon position (rare codons enriched in the first 10%, common-as-rare misclassifications enriched 5'-end and 3'-end), and on Pfam-domain location (codon usage is more constrained inside annotated domains).
- **Internal consistency in autoregressive generation.** A targeted in-silico experiment shows that as the prefix of rare codons grows, the model's probability of choosing a rare codon next increases — an autoregressive momentum effect. This is presented as both a feature (matches natural rare-codon clustering) and a caution (can degrade generation quality if the prompt happens to be rare-codon-heavy).
- **The expression-correlation result.** Scoring 30 GFP variants from Bicknell et al. 2024 (transfected as N1-methylpseudouridine-modified mRNAs into HEK293), Trias' negative-log-likelihood scores correlate at **ρ = -0.76 with GFP fluorescence AUC, ρ = -0.84 with mRNA half-life, and ρ = +0.82 with mean ribosome load**. sRSCU alone is not a sufficient predictor, so the correlation reflects context Trias has learned beyond raw codon frequency.
- **Beats commercial optimizers on the GFP benchmark.** The Trias-greedy GFP sequence has likelihood score 0.20; GenScript 0.86, Twist 1.13, IDT 1.22. Trias-generated GFP shows 76% codon similarity to the five highest-expressing experimental variants (significantly higher than the commercial tools) and only 16% to the five lowest. The comparison is honest in scope — it is one protein (GFP), one cell line (HEK293), one set of 30 variants — but it is the central head-to-head claim.

## What's reusable in Linus
**Direct, Phase 7 (Skills) — high priority.** A `codon_optimize(protein_sequence, organism)` skill is one of the highest-value domain tools Linus could expose for Dan's actual workflow. The model is small (47M params), released under permissive HuggingFace hosting, and runs against a HuggingFace BART implementation that already has CPU/MPS paths. Wrapping Trias as a Linus tool — given a protein sequence and a species tag from the 600+ supported vertebrates, return an optimized codon sequence plus a likelihood score — is a near-term concrete deliverable that would be used regularly.

**Direct, Phase 7.** A scoring tool is just as valuable as a generation tool: `score_codon_sequence(cds, organism)` returning Trias' negative log-likelihood per codon would let Dan rank candidate sequences from any source (commercial optimizer outputs, hand-edited variants, literature sequences) using the same calibrated metric the paper validates against expression data.

**Direct, Phase 6 (Fine-Tuning).** The encoder-decoder BART backbone is well-suited to LoRA. A LoRA fine-tune on a species or tissue not well-represented in vertebrate RefSeq — or on a curated set of high-expression sequences from a particular system Dan cares about — is computationally tractable on the M1 Max (47M base + small adapter, well under the 32 GB ceiling) and would extend Trias beyond its native scope. The discussion section of the paper itself flags fine-tuning on expression data as the natural next step to reduce the rare-codon-momentum bias.

**Direct, Phase 3 (KnowledgeBase).** Codon-optimized sequences plus their provenance — input protein, organism, design timestamp, model version, score — are first-class KG content for a project-history log of designed mRNAs. Every time Linus designs a sequence for Dan, the (protein, organism, sequence, score, rationale) tuple should land in the KnowledgeBase, indexed for later retrieval ("show me all the GFP variants we designed last spring," "what was the optimized sequence for the SpyCatcher fusion?"). This is the kind of personal-project memory that turns an off-the-shelf model into a *Dan-specific* tool.

**Indirect, architecture.** The "trained on natural sequences only, beats supervised commercial tools" result is empirical support for the **sequence-pretraining-as-foundation** thesis that underwrites the whole Group A nucleotide-FM corpus. Trias is the smallest, narrowest data point in the family, and the pattern still holds: pretrain on enough natural sequence and functional regularities fall out for free.

**Indirect, evaluation.** The BLASTp homology-filtered test split is a methodological habit Linus' benchmark suite should adopt for any sequence-based evaluation. Trivial-similarity leakage is the silent killer of biological ML benchmarks.

## What's NOT applicable
The model is **vertebrate-only** — its 640-species vocabulary doesn't cover *E. coli*, yeast, insect cells, or plants. A LoRA fine-tune is plausible but unverified; for prokaryotic optimization the cleaner alternative is a different model (Constant et al.'s E. coli encoder-decoder, or a generalist nucleotide FM like Evo 2 or LucaOne with appropriate prompting).

The training data is **mRNA coding regions only**. Trias does not model 5' UTRs, 3' UTRs, cap, or polyA signals — all of which matter for full mRNA construct design. Trias handles the CDS; UTR design needs a separate model or rule set.

The expression-correlation evaluation is a **single benchmark**: 30 GFP variants in HEK293, from one paper. The correlations are strong and beating commercial tools is meaningful, but a Linus-internal evaluation on Dan's own sequences (or on a broader public benchmark) is needed before treating "Trias is the best codon optimizer" as established.

The autoregressive **rare-codon momentum** is a known failure mode the authors flag: a prefix of rare codons biases continued rare-codon emission. Greedy decoding without temperature or sampling tuning may not be best for all proteins; worth probing in the wrapper.

The paper does not address **regulatory/safety questions**: codon-optimized therapeutic sequences must pass GMP, FDA review, IP clearance. Trias is a research tool — the Linus skill should be framed for design-for-experimentation, not design-for-clinic.

The model is **CUDA-trained on an A100 80 GB**. Inference should run fine on M1 Max via HuggingFace transformers MPS (47M params is trivial); anyone planning a fine-tune from scratch should not assume the training recipe transfers cleanly to MLX.

## Connections
The most direct connection is to **GenNA** (`paper-notes/2026.04.22.720063v1.md`). Trias and GenNA carve up the constrained-generation design space: Trias is **constrained by a fixed amino acid sequence** and **conditioned on a categorical species tag**; GenNA is **conditioned on a free-text functional description** with no protein constraint. Trias is narrower, smaller, and ready to deploy today. GenNA is broader and more speculative. Both could be exposed as Linus tools — `codon_optimize(protein, organism)` (Trias) and `design_sequence(text_description)` (GenNA) — covering complementary biotech needs. Trias' reverse-translation framing is closer to a classical bioinformatics tool than GenNA's natural-language conditioning, and so it is the safer Phase 7 bet.

Trias also sits inside the **Group A nucleotide foundation model** thread (Evo 2, LucaOne, AlphaGenome, RiNALMo) as a **specialist counterpoint**. The Group A models are large multi-genome multi-task backbones; Trias is a narrow specialist (vertebrate CDS, 47M params, single task). For codon optimization specifically, the specialist wins on inference cost, deployment ease, and — on the GFP benchmark — task quality against commercial tools. Generalist nucleotide FMs may eventually dominate via fine-tuning; Trias is the right tool today.

Methodologically, the **homology-filtered test split** is a discipline shared with strong protein-language-model evaluations (ESM, AlphaFold). The **likelihood-as-zero-shot-fitness-proxy** finding parallels the long-established result in protein language models that pretrained likelihoods predict variant effects.

Architecturally, BART encoder-decoder + species token + 2,048-position context is **cheap enough to run locally on the M1 Max within minutes of pulling weights**, and small enough to fit easily alongside a Worker LLM in unified memory. It is one of the cheapest scientific-FM additions to Linus possible.

## Open questions for Dan
1. **Phase 7 priority.** Codon optimization is a high-frequency biotech use case for your work. Should `codon_optimize` and `score_codon_sequence` be elevated to top-priority Phase 7 skills — meaning we wrap Trias as a Linus tool before, say, more general bioinformatics skills? My bias is yes; the model is small, open, and the use case is frequent enough that the per-use value compounds quickly.
2. **Linus vs. commercial benchmark.** The paper compares Trias against IDT, GenScript, and Twist on one GFP construct. Would it be worth setting up a Linus-internal benchmark that runs the same comparison on a handful of proteins you actually care about (a model enzyme, an antibody fragment, a SpyCatcher fusion), using whatever expression data we can pull from the literature or from your own assays? This would directly stress-test whether Trias generalizes beyond GFP/HEK293.
3. **Beyond vertebrates via LoRA.** Trias is vertebrate-only. Do any of your workflows need codon optimization for organisms outside the training distribution (E. coli, yeast, insect cells, plants)? If yes, a LoRA fine-tune on a curated CDS set for that organism is a tractable Phase 6 experiment — and a good test of whether the encoder-decoder pattern adapts cleanly.
4. **Rare-codon-momentum mitigation.** The authors flag that autoregressive decoding biases toward continued rare codons. Should the Linus wrapper expose decoding-strategy controls (greedy vs. nucleus sampling vs. temperature, vs. constrained beam search that penalizes long rare-codon runs)? Or is it cleaner to ship a single sensible default and let Dan inspect outputs for anomalies?
5. **Encoder-decoder pattern transfer.** Trias' template — protein-to-codon constrained reverse translation — is one instance of a broader "constrained sequence generation conditioned on a categorical context tag" pattern. Are there other biotech tasks in your workflow that fit the same template (promoter design given a target expression level and host, ribozyme design given a target substrate, signal peptide design given a target subcellular compartment)? If yes, the same training recipe at the same parameter scale might work, and we'd have a family of small specialist models rather than one.
6. **KnowledgeBase integration.** Should the design log (every protein-organism-sequence-score tuple Linus generates) be a first-class KnowledgeBase entity from day one, with provenance fields (model version, decoding strategy, timestamp, project tag)? My bias is yes — the long-term memory of "what we designed and why" is where personal-AI value compounds.
