# Nested Learning: The Illusion of Deep Learning Architecture

## TL;DR

Behrouz et al.'s NeurIPS 2025 paper from Google Research reframes deep learning as a **nested system of associative memories**, each with its own context flow and update frequency, rather than as a stack of architecturally heterogeneous layers. The core claim is sharp: a Transformer trained with Adam, an MLP trained with momentum-SGD, and a linear-attention block trained with gradient descent are **all instances of the same primitive** — an ordered nested system of optimization problems, where every level (including the optimizer itself, which the paper formally treats as an associative memory compressing past gradients) is a key-value mapping fitted by gradient descent at some characteristic frequency. From this viewpoint they derive three contributions: (1) **Expressive Optimizers** — Adam, AdaGrad, and momentum-SGD decompose into two-level nested associative-memory problems, and the paper introduces **Delta Gradient Descent (DGD)**, an update rule that conditions on the current weight state rather than only on the current input; (2) **Self-Modifying Learning Modules** — a sequence model that, given context, generates its own update rule (a step beyond hypernetworks), realized as deep self-referential Titans where every projection (k, v, q, η, α) becomes an in-context-adapted memory; (3) **Continuum Memory System (CMS)** — a generalization of the long-term/short-term memory dichotomy to a spectrum of update frequencies, implemented as the **Hope** architecture (self-modifying Titans plus a stack of MLP-as-memory blocks at logarithmically-spaced update frequencies). Hope clears Transformer baselines on RULER needle-in-a-haystack, BABILong, MK-NIAH multi-key retrieval, and class-incremental learning on CLINC / Banking / DBpedia using Llama3-3B/8B backbones with continual pre-training of MLP blocks. The **Multi-Scale Momentum Muon (M3)** optimizer demonstrates CMS-as-optimizer; M3 stacks Adam + Muon + CMS into multiple momentum buffers at different frequencies. For Linus, this paper is the **closest 2025-era theoretical scaffold for the Phase 2 memory pillar** (DEC-0028) — it gives names and equations to the multi-timescale update story that the Linus memory architecture (Layer A intra-step latent through Layer E semantic) already commits to, and it surfaces a Phase 8-tier research direction (CMS-style architecture with logarithmic-frequency blocks) that is more promising than minGRU + BitNet (DEC-0041) for the substrate-experiment slot.

## The problem (in plain language)

Modern deep learning, the authors argue, has hit four interconnected ceilings that adding layers cannot overcome. First, **computational depth doesn't grow with stacking** in the formal sense — Merrill et al. 2024 and Sanford et al. 2024 show that deep Transformers without CoT still live in TC0; stacking layers doesn't change what the network can express in a fixed forward pass. Second, **parameter capacity saturates**: scaling depth past the Chinchilla-optimal regime yields marginal improvements at exponential compute cost. Third, **the training process converges to suboptimal minima** that the choice of optimizer materially affects — the same Transformer trained with SGD versus Adam ends up at different solutions, suggesting the optimizer is doing real work that the architecture story ignores. Fourth, and most operationally important for the paper's thesis, **deep models cannot continually learn**: after pre-training, an LLM's MLP blocks are frozen for all practical purposes, in-context learning is the only adaptable component, and any attempt to update weights post-deployment risks catastrophic forgetting (Akyürek 2024a; Eyuboglu 2025).

The framing analogy is **anterograde amnesia** (the neurological condition where new long-term memories cannot be formed). The authors argue that LLMs after pre-training experience exactly this condition: the "knowledge in MLPs" is the long-past memory laid down before "end of pre-training"; the in-context window is the immediate present; nothing in between gets consolidated. The hippocampus-to-cortex consolidation pipeline that lets humans continually learn — first online consolidation during wakefulness, then offline consolidation during sharp-wave ripples and sleep spindles, both gated by neural oscillations at distinct frequencies (delta 0.5-4 Hz through gamma 30-100 Hz) — is precisely what modern deep architectures lack.

The neurophysiological observation that grounds the contribution: brain oscillations show that **different brain regions update at different characteristic frequencies**, and the slow-frequency consolidation (delta, theta) is what enables transfer from short-term to long-term storage. In a Transformer, by contrast, every parameter has only two effective update frequencies — **infinity** (attention, which recomputes from scratch every forward pass and so has "infinite" update rate over the context) and **zero** (MLP/feedforward weights, which never update during inference). This is, in the paper's terminology, a degenerate two-level nested system. The argument is that closing the gap requires intermediate frequencies — a spectrum of memory components updating at different timescales — and that gradient descent on associative memory at multiple levels is the unifying primitive that delivers it.

## What they propose

The paper builds in four conceptual moves, each formalizing a different aspect of the unifying view.

**Move 1: Optimizers are associative memories (§4).** The first non-obvious move. Take ordinary SGD: $W_{t+1} = W_t - \eta \nabla \mathcal{L}(W_t; x_t)$. Rewrite the gradient via the chain rule as $\nabla_{y_t} \mathcal{L} \otimes x_t$ — the "Local Surprise Signal" outer product with the input. The paper observes that gradient descent itself is solving an optimization problem: it maps each data point $x_t$ to its corresponding local surprise signal $u_{t+1} = \nabla_{y_t} \mathcal{L}$, with the linear layer $W$ acting as the operator that performs this mapping. Formally, $W_{t+1} = \arg\min_W \langle W x_{t+1}, u_{t+1} \rangle + (1/2\eta_{t+1}) \|W - W_t\|^2$ — exactly an associative memory's optimization problem (Definition 1) at the dot-product objective. Therefore **gradient descent on a linear layer is itself solving an associative-memory problem**, where the memory $W$ learns to map inputs to their LSS values. Adding momentum yields a **two-level nested system**: the inner level uses gradient descent to compress past gradients into the momentum buffer $m_t$ (which is the inner-level associative memory mapping past inputs to gradient values), and the outer level uses the momentum-as-gradient to update the slow weights $W_t$ (the outer-level memory). Adam, AdaGrad, and other standard optimizers fall out as different choices of objective and update rule at the inner level.

**Move 2: Delta Gradient Descent (DGD, §4.5).** Once you see gradient descent as associative-memory learning under a dot-product objective, you can replace the dot-product objective with an $L_2$-regression objective and pick a different update rule. The resulting **Delta rule** has update $M_{t+1} = M_t - \eta_t (M_t k_{t+1} - v_{t+1}) k_{t+1}^\top$, which depends on **both** the input and the current weight state. The paper proves this is the correct generalization for capturing dependencies across data samples (the i.i.d. assumption breaks for sequence tokens). DGD becomes the building block for Move 4.

**Move 3: Continuum Memory System (CMS, §7).** Move 1 reveals that Transformers have only two update frequencies (attention = ∞, MLP = 0). CMS generalizes the long-term/short-term dichotomy by introducing a **spectrum of MLP-as-memory blocks**, each updating at a different characteristic frequency $f^{(k)}$ — block $k$ runs gradient descent on its own context flow with its own learning rate, with frequencies in a logarithmic ladder (e.g., a Hope-3 architecture has four memory levels at frequencies 1, $f_2$, $f_3$, and $f_4$ with respective chunk lengths 16, $\geq 16$, 1M, 16M tokens). The paper argues this directly addresses catastrophic forgetting: when the high-frequency block is being updated, the knowledge possibly forgotten from it is still stored in lower-frequency blocks, and gradient flow through backpropagation can circle back via the initial-state pathway.

**Move 4: Self-Modifying Titans + Hope architecture (§8).** Self-Modifying Titans extends the associative-memory framework to make every projection $W_k, W_v, W_q, W_\eta, W_\alpha$ in a Titans-style block its **own** in-context-adapted memory. Each projection becomes a memory $\mathcal{M}_{k,t-1}(x_t)$, $\mathcal{M}_{v,t-1}(x_t)$, etc., that updates online during the forward pass. To enable self-modification beyond just key/value generation, the model **generates its own training values** $\hat{v}_{\Box,t} = \mathcal{M}_{\Box,t-1}(v_t)$ for each memory. The result is a fully adaptive memory where every component can adapt itself in-context. **Hope** combines self-modifying Titans (low-capacity, complex update rule) with CMS (high-capacity, simple update rule) — these two are complementary: Titans for fast contextual adaptation, CMS for persistent long-term storage. The training is parallelizable via chunk-wise updates (chunks of size $C$, gradients precomputed before chunk processing).

**Move 5 — implementation: Multi-Scale Momentum Muon (M3, §7.2, Algorithm 1).** As a proof-of-concept that CMS works in optimizer space, M3 fuses Adam + Muon + CMS into a single optimizer with two momentum buffers: $M^{(1)}$ at high frequency (per-step Adam-like updates) and $M^{(2)}$ at lower frequency (chunked accumulation every $\hat{C}$ steps). Newton-Schulz iteration maps gradients to a proper metric space (Muon contribution). The final weight update is $\Theta_t \leftarrow \Theta_{t-1} - \eta (O_t^{(1)} + \alpha O_t^{(2)}) / \sqrt{V_t + \epsilon}$ — a weighted combination of fast and slow momentum.

## Key results

**Class-Incremental Learning (§9.1, Figure 6).** Hope-enhanced Llama3-8B versus baseline ICL, EWC, and InCA (Momeni 2025) on CLINC, Banking, DBpedia. Hope wins all three across both 3B and 8B backbones. The mechanism: Hope's multiple in-context-learning levels (i.e., different frequency updates for MLP blocks) prevent catastrophic forgetting in a way that EWC's regularizer and InCA's external learner cannot replicate. Specifically on the Banking dataset (77 fine-grained intents), Hope-8B reaches ~85% accuracy versus ~70% for ICL and ~75% for InCA at Llama3-8B scale.

**Long-context Q&A (§9.1, Figure 7).** On LongHealth (long-form clinical QA), QASPER (NLP research-paper QA, ~5K QA pairs over 1.6K papers), and MK-NIAH (multi-key needle-in-a-haystack from RULER 64K), Hope shows **monotone improvement with more memory levels** — more memory levels lift in-context capability, and the lowest-frequency memory dominates long-context understanding. Hope outperforms both ICL and DuoAttention (Xiao 2025) across all three datasets.

**RULER Needle-in-a-Haystack (§9.2, Table 1).** Trained from scratch on 50B FineWeb-Edu tokens with a 32K vocabulary, Hope clears all comparable attention-free methods (RWKV-7, Comba, DLA, Titans) across S-NIAH-{1,2,3}, MK-NIAH, MQ-NIAH, MV-NIAH at 4K/8K/16K. Notably, Hope-Attention (which keeps softmax attention but adds CMS) outperforms the pure Transformer baseline, demonstrating that **CMS contributes independently of the attention/recurrence choice**.

**BABILong long-context reasoning (§9.2, Figure 9).** Hope is the only fine-tuned small-model entry that climbs to large-model performance levels (Llama-8B + RAG, GPT-4o, GPT-4) at sequence lengths beyond 1M tokens, holding ~80-90% accuracy where baselines collapse.

**Continual Translation of a Novel Language (CTNL, Figure 8).** A two-language manual learning setup combining MTOB and Manchu benchmarks. ICL collapses under continual learning (catastrophic forgetting); Hope-3 (3 memory levels) almost fully recovers the ICL-only performance under sequential learning, demonstrating CMS's continual-learning property in a translation setting.

**Released artifacts.** No open-source code release in the v1 paper; M3 algorithm pseudocode is in Algorithm 1, full Hope training details in Appendix. The paper has been published at NeurIPS 2025 (per the footnote on page 1).

## What's reusable in Linus

**Phase 2 — multi-timescale update vocabulary for the memory pillar (DEC-0028, DEC-0031, DEC-0043).** The most load-bearing fold-in. Linus's five-layer memory architecture (Layer A latent through Layer E semantic) currently has implicit timescales — Layer A updates every forward pass, Layer B every Worker call within a session, Layer C across sessions, Layer D within an investigation, Layer E essentially never. The Behrouz et al. framing gives this **explicit characteristic frequencies** ($f_A = $ per-token, $f_B \sim 1/n_{\text{turn}}$, $f_C \sim 1/n_{\text{session}}$, $f_D \sim 1/n_{\text{investigation}}$, $f_E \sim 0$ until next training pass) and a unifying theoretical claim: each layer is an associative memory minimizing some objective at its own update frequency, with backpropagation-like knowledge transfer across layers possible. Phase 2 spec ([`memory-architecture.md`](../specs/memory-architecture.md)) should adopt the "spectrum of update frequencies" framing as the explicit unifying principle, with citations to this paper as the 2025-era theoretical scaffold. The MemGPT paper-note [`Letta-2310.08560.md`](Letta-2310.08560.md) is the 2023 precedent; Behrouz et al. is the 2025 theoretical generalization.

**Phase 2 — optimizer-as-associative-memory view as the substrate for online consolidation experiments (DEC-0028, DEC-0029).** Move 1 (optimizers are nested associative memories) is more than vocabulary — it implies that the **training/inference distinction is artificial**. Linus's Layer C (cross-session episodic) is currently a content-addressed text store with no learned consolidation; the Phase 6+ TTT-style parametric consolidation (DEC-0037) is the upgrade path. Behrouz et al. provide the bridge: an online consolidation Worker could implement Move 1 directly, treating each Layer-C episode as an associative-memory entry that the Worker learns to compress into faster-frequency storage via gradient descent at session boundaries. This is closer in spirit to test-time training (DEC-0037) than to RAG, and it sits naturally between the current text-store substrate and the eventual fine-tuned consolidator. Worth a Phase 6 spike.

**Phase 8 — Hope-style CMS as a substrate-experiment alternative to minGRU + BitNet (DEC-0041, DEC-0042).** DEC-0041 currently names minGRU + BitNet as the Phase 8 substrate-experiment direction; DEC-0042 names COCONUT as the Phase 6 substrate experiment. Hope is now the **third candidate**, and arguably the strongest for the multi-timescale-update story specifically. The argument: Hope's CMS demonstrates that multi-frequency MLP-as-memory blocks plus a self-referential adaptive front-end matches or exceeds Transformer + RAG on BABILong at 10M-token contexts, with the architecture itself doing the work rather than an external retrieval system. For Linus's Phase 7+ biology/long-document Workers (per [`biology-phase7-roadmap.md`](../specs/biology-phase7-roadmap.md)), this matters: long-paper analysis is currently bottlenecked by either RAG retrieval quality or the Worker's effective context window, and a CMS-style architecture would shift both bottlenecks to the model's own parametric memory. A Phase 8 reproduction at small scale (Hope-1 on Llama3-3B) is a concrete experiment.

**Phase 6 — DGD (Delta Gradient Descent) as a fine-tuning recipe candidate (DEC-0043).** Move 2's Delta Gradient Descent is a drop-in alternative to AdamW for fine-tuning small models on sequence data. The paper claims DGD captures cross-sample dependencies that standard SGD/Adam cannot (because they assume i.i.d. samples, which is false for sequence tokens). The DGD update depends on both the current weight state and the input; the gradient term is $M_t k_{t+1} - v_{t+1}$, which the paper claims is the principled generalization of the perceptron-style Delta rule. For the Phase 6 fine-tuning lane on Dan's domain corpus (LoRA on metagenomics texts), DGD is worth benchmarking against AdamW LoRA on the same task — if DGD's promised cross-sample-dependency capture is real, it should show up as cleaner contiguous-token completions and lower perplexity on sequential targets.

**Phase 7+ — multi-scale momentum Muon (M3) as an optimizer for fine-tuning long-context workers.** The M3 optimizer (Algorithm 1) is interesting independently of CMS-the-architecture: it adds a low-frequency momentum buffer that accumulates gradients over $\hat{C}$-step chunks, then weighted-sums with a high-frequency Adam-like buffer. For long-context fine-tuning on the M1 Max (32 GB unified memory), where memory pressure forces small batches and many small steps, M3's slow-momentum component would absorb noise across many noisy minibatches. The cost is two momentum buffers per parameter (more memory), but in the LoRA regime (per DEC-0027 fine-tuning lane scoping) the parameter count is small enough that doubling momentum-buffer storage is cheap. Worth a Phase 6+ spike against AdamW + Muon at LoRA scale.

**Methodology — RULER MK-NIAH and BABILong as Linus benchmarks (Phase 1c+, DEC-0035).** Hope's evaluation on MK-NIAH (multi-key needle-in-a-haystack from RULER 64K) and BABILong at 1M-10M token contexts gives Linus a concrete, well-defined benchmark for measuring Layer C consolidation effectiveness. DEC-0035 names ARC-AGI as the memory diagnostic; MK-NIAH/BABILong are the long-context-retention complements. The Linus Phase 1c spike ([`phase1c-spike.md`](../specs/phase1c-spike.md)) should include MK-NIAH at 16K / 32K against the Worker's effective in-context cap (DEC-0032's 16K default) and measure where Worker performance falls off — that's the empirical anchor for the cap policy. The Hope paper provides the comparison baseline (their Transformer + DLA + Titans numbers in Table 1) and the dataset (publicly available from Hsieh 2024 and Kuratov 2024).

**Phase 3 — self-modifying Titans as a Layer D investigation-memory architecture (DEC-0052).** Move 4's self-modifying Titans is a precise mechanism for "an agent that adapts its own update rule during an investigation": every projection becomes its own in-context-adapted memory, and the model generates its own training targets via $\hat{v}_{\Box,t} = \mathcal{M}_{\Box,t-1}(v_t)$. This is exactly the multi-agent investigation memory pattern Layer D is designed for (DEC-0052) — agents in an investigation accumulate and modify their own state in ways that depend on the investigation's evolving context. The Phase 3 spawner spec ([`phase3-spawner.md`](../specs/phase3-spawner.md)) should reference Hope's self-modifying Titans alongside MemGPT's heartbeat pattern and Letta's memory blocks as the architectural precedents.

## What's NOT applicable / hype filter

**The "Transformers and Adam are the same thing" framing oversells the unification.** Move 1 (optimizers as associative memories) is mathematically valid for linear layers with dot-product objectives — the equivalence in Equation 9 ($W_{t+1} = \arg\min_W \langle W x_{t+1}, u_{t+1}\rangle + (1/2\eta) \|W - W_t\|^2$ is a one-line algebraic restatement of $W_{t+1} = W_t - \eta \nabla \mathcal{L}$). The paper extends this to "training an architecture with a specific optimizer can also be represented as a set of nested and/or parallel optimization problems," which is true in the formal sense but does not mean Adam-as-an-associative-memory has different behavior from Adam-as-an-optimizer. The unification is a useful conceptual organizer, not a new computational object. Linus should adopt the multi-timescale framing without claiming that "Linus optimizers are associative memories" — that framing is true but operationally vacuous at the orchestration layer.

**Hope is a fine-tuned-specific result, not a from-scratch breakthrough.** Section 9.2 trains Hope from scratch on 50B FineWeb-Edu tokens, which is a small scale by 2025 standards. Section 9.1's stronger results come from continual pre-training on top of Llama3 backbones — taking Llama3-3B/8B and adding 15B tokens of MLP-block-targeted continual pre-training with the CMS structure swapped in. The improvements over baselines (Figure 6) are real but **conditional on having the pre-trained backbone**. Linus's Phase 8 reproduction should be honest about this: Hope-from-scratch on M1 Max is intractable (32 GB unified memory caps useful from-scratch training at ~3B parameters); Hope-as-MLP-block-modification on top of an existing 3B-7B Llama or Mistral is tractable but requires a continual-pretraining pass which itself burns ~$10K of cloud compute if outsourced.

**Self-modifying Titans is a memory-hungry architecture and the paper acknowledges this.** Section 9 explicitly excludes Cartridges (Eyuboglu 2025) from the Hope comparison "due to the fact that Hope has higher memory usage and there are fundamental differences in their computational costs." Self-modifying Titans requires every projection to be its own MLP-as-memory; for a Llama-7B model with 32 attention heads per layer, this is 5 × 32 × num_layers extra memory blocks. On M1 Max (32 GB unified memory), this is a hard constraint — Hope-on-Apple-Silicon is probably feasible only at Hope-1 (one extra memory level) on a small backbone (3B parameters). The Phase 8 spike should scope tightly.

**M3 is a proof-of-concept optimizer, not a benchmarked production tool.** Section 7.2 explicitly flags: "this optimizer is designed as a proof-of-concept to support the design of CMS. The M3 optimizer per se, however, might suffer from computational overhead and so face challenges when scaling to larger networks (see Figure 12)." Linus should treat M3 as a candidate worth benchmarking but not as a drop-in replacement for AdamW until the scaling story is better-understood. Concretely: the Phase 6 fine-tuning lane should keep AdamW as the default and run M3 only on a controlled ablation.

**The "anterograde amnesia" analogy is rhetorically strong but architecturally vague.** The paper's framing — LLMs experience anterograde amnesia, where in-context is the "immediate present" and MLPs are the "long past before end of pre-training" — is a vivid hook but doesn't actually do mechanical work in the construction. The mechanical work is done by the Continuum Memory System's logarithmic frequency ladder, which is a separable engineering claim. Linus's memory pillar already commits to the multi-timescale story via DEC-0028; reading the paper as "we have a name for what we already decided" is the right disposition.

**No public open-source code as of v1.** The paper appendices include pseudocode (Algorithm 1 for M3; details on Hope training in Section 8.3) but no reproducible repository. Linus's reproduction would have to implement from the paper text. This is a non-trivial cost — the chunk-wise parallelization details in Section 8.2 are sketched but not fully specified, and Hope-2 / Hope-3 training-stability tricks may be in the unreleased code. Worth tracking the authors' future open-source release before committing to a Phase 8 reproduction.

## Open questions for Dan

1. **Adopt "characteristic update frequency" as the explicit unifying primitive in the memory-architecture spec?** Currently [`memory-architecture.md`](../specs/memory-architecture.md) defines the five-layer model with implicit timescales. The Behrouz et al. framing makes the timescales explicit and gives them theoretical grounding. The cost is some additional vocabulary in the spec; the benefit is that Layer A/B/C/D/E become instances of a single primitive at different frequencies rather than five disjoint architectural choices. Worth a DEC?

2. **Phase 8 substrate-experiment slot: rank Hope vs. minGRU+BitNet vs. COCONUT.** Three candidates now exist (DEC-0041, DEC-0042, this note). They are not mutually exclusive but the experimental budget (one substrate spike, ~$5K compute) probably accommodates only one or two reproductions. Hope's strongest argument: the most published evidence at the largest scale, with cleanest benchmark comparisons on RULER/BABILong. minGRU+BitNet's strongest argument: by far the smallest memory footprint, native-Apple-Silicon-feasible. COCONUT's strongest argument: the most direct fit for the existing Worker / CoT architecture. Which to fund first?

3. **Phase 6 — DGD vs. AdamW for LoRA fine-tuning on Dan's metagenomics corpus.** Move 2's Delta Gradient Descent has a claim (cross-sample-dependency capture in sequence data) that should be benchmarkable cheaply on Dan's domain corpus. A controlled run: same Llama3-3B base, same LoRA rank/dropout, AdamW vs. DGD, evaluated on Dan's task-specific eval. If DGD shows a 2%+ improvement, it earns a place in the Phase 6 fine-tuning recipe; if not, treat as honored.

4. **M3 optimizer for Phase 6 long-context fine-tuning?** M3's slow-momentum buffer is theoretically attractive for noisy-minibatch regimes (M1 Max forces small batches). The cost is one extra momentum buffer per LoRA parameter and the implementation effort (Algorithm 1 is well-specified but not in any existing optimizer library). For a Phase 6 spike — once we have a baseline AdamW LoRA fine-tune working — is benchmarking M3 against it worth the implementation cost?

5. **MK-NIAH and BABILong adoption for the Phase 1c+ benchmark suite.** DEC-0035 names ARC-AGI as the memory diagnostic; the multi-key NIAH and BABILong benchmarks are the long-context-retention complements. Should Linus adopt both for Phase 1c+ benchmark coverage? The datasets are public and small enough to run locally (BABILong at 4K-64K is tractable on M1 Max; the 10M-token regime needs care). The argument for both: ARC-AGI tests reasoning; MK-NIAH tests recall; BABILong tests long-context reasoning under retrieval pressure. They're orthogonal axes.

6. **Activation-hooks API as a substrate for online consolidation experiments (DEC-0054).** The Behrouz et al. paper's Move 1 (optimizer-as-associative-memory) implies that online weight updates during inference are the natural primitive for cross-session consolidation. Linus's activation-hooks API stub (DEC-0054) is currently scoped for read-only observability; the Hope paper suggests it should evolve into a read/write surface where an "online consolidator" Worker can update specific MLP blocks at session boundaries. Should the activation-hooks API surface graduate from observability-only to read/write in Phase 2+?

7. **Adding cited foundational works to the corpus.** The paper cites Akyürek 2022 (in-context learning for transformer-based models, formal), Schmidhuber 1992 (fast weight programmers — historical anchor), and Sun et al. 2024 (the modern parallelizable RNN line that Hope builds on). None are currently in `context/papers/`. The first two are tier-1 candidates for foundational corpus expansion; Sun 2024 is paired with the existing minGRU / TTT thread and would be a natural pickup with the next Phase 8 substrate-experiment spec.
