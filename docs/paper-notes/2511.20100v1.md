# QiMeng-Kernel: Macro-Thinking Micro-Coding Paradigm for LLM-Based High-Performance GPU Kernel Generation

## TL;DR
Proposes MTMC (Macro Thinking Micro Coding), a hierarchical LLM-based framework that decouples GPU kernel generation into semantic-level optimization strategies (Macro Thinking) and stepwise low-level code implementation (Micro Coding). Achieves 100% accuracy and 70% speedup over general-purpose LLMs on KernelBench, with 7.3x speedup over hand-optimized PyTorch Eager kernels.

## The problem (in plain language)
Developing high-performance GPU kernels is critical for AI but extremely challenging. LLMs struggle because they need to understand both high-level hardware characteristics and low-level implementation details simultaneously, making it hard to generate correct and efficient kernels. Existing approaches fail on CUDA/Triton implementations and lack comprehensive hardware-specific optimization knowledge.

## What they propose
**Macro Thinking Micro Coding (MTMC)** framework with two key components:

1. **Macro Thinking**: Lightweight LLMs generate semantic optimization actions (tiling, fusion, pipelining, reordering) using domain-specific knowledge and hardware characteristics, decomposing complex optimization into interpretable steps
2. **Micro Coding**: General-purpose LLMs iteratively implement these steps as GPU code, with RL-trained policy to guide optimization and rule-based reward shaping to maximize correctness

The pipeline decouples optimization strategy from implementation, enabling efficient policy learning on a compact human-curated dataset without prohibitive data requirements.

## Key results
- **KernelBench**: 100% accuracy, 70% speedup vs general-purpose LLMs; 1.2x over 50% speedup on SOTA; 1.3x over domain-specialized LLMs
- **TritonBench**: 96% accuracy, 34x speedup over baselines
- **Hardware platforms**: Evaluated on Intel CPUs (V100, A100, H100) with consistent generalization
- **Comparison**: MTMC is only approach approximating expert-optimized PyTorch Eager kernels while maintaining correctness

## What's reusable in Linus

**LLMs designing real hardware (the load-bearing thread).** QiMeng-Kernel is the second QiMeng entry in the corpus and
sits inside an emerging cluster Dan is actively building out: LLM-driven hardware design. The plan is to promote QiMeng
to its own category and add ~4 more QiMeng-related repos and ~8 more papers as the thread matures. The category's
center of gravity is _LLMs that design the things underneath the LLMs_ — kernels, processor variants, accelerators —
which closes a feedback loop that, if it lands on Apple Silicon, would let Linus eventually optimize its own substrate.

**The macro/micro orchestration paradigm transfers cleanly to Apple Silicon.** The MTMC pattern — a lightweight planner
LLM emitting semantic optimization actions (tiling, fusion, pipelining, reordering) and a heavier coder LLM realizing
them step-by-step under RL-trained policy — is platform-agnostic. The CUDA/Triton output is a back-end choice; the
front end (planner action space, RL reward shaping, rule-based correctness gating) ports cleanly to Metal kernels via
pmetal or MLX kernel work. Phase 2-3 relevance: this hierarchical decomposition is directly applicable to Linus's
Worker orchestration — macro-level action spaces map to the agent-spawner's role types (DEC-0050), micro-level code
generation to typed AgentReports (DEC-0051), and the planner/coder split mirrors Maestro/Worker discipline.

**Closing the loop on Apple Silicon kernel optimization (Phase 6d+).** Linus's flash-streaming inference target
(`docs/specs/phase6d-streaming-target.md`) already implies handcrafted Metal kernels somewhere downstream. If the
QiMeng paradigm is genuinely platform-agnostic, a Linus-internal MTMC pipeline targeting MLX or pmetal kernels would
be the kind of self-improving loop the project's north star points at — Linus designing kernels to make Linus run
better. Out of scope today, on the radar for Phase 6d+ research.

**Cross-paradigm sibling: Sketch2Simulation.** The deeper "LLMs designing physical things" thread —

## What's NOT applicable / hype filter

The CUDA / Triton output syntax is NVIDIA-specific and does not run on Apple Silicon — but this is a back-end choice
in the MTMC pipeline, not an architectural constraint. The transferable components (action space design, RL reward
shaping, rule-based correctness gating, planner/coder decomposition) are platform-agnostic.

The training infrastructure assumes large-scale offline RL on NVIDIA hardware. Reproducing the full RL training is
out of scope for Linus; the realistic path is to lift the macro/micro pattern as an architectural template and apply
it to a Linus-relevant kernel domain (Metal / MLX) at much smaller scale, possibly with a frozen planner and
fine-tuned coder.

The KernelBench / TritonBench evaluation harnesses are NVIDIA-flavored. Apple Silicon equivalents would need to be
built — a non-trivial but bounded effort, and a natural deliverable of the QiMeng synthesis Dan is planning.
