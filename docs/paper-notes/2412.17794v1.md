# Memory Makes Computation Universal, Remember?

## TL;DR

This paper formalizes a foundational principle: memory (recursive state maintenance + reliable history access) is not
just a feature but the critical substrate enabling universal computation. Complex behavior emerges from state management,
not from sophisticated processing units—a principle applicable across biological cells, neural networks, and AI systems.

## The problem (in plain language)

Standard computational theory emphasizes sophisticated processing mechanisms (logic gates, attention heads, etc.). Yet
biological systems and neural networks achieve complex computation through inherently limited components. The paper asks:
what makes parallel systems with minimal processing units Turing-complete? The answer isn't more complex operations—it's
robust memory.

## What they propose

Formal proof (Theorem 1) that any system with two properties achieves universal computation with logarithmic overhead:

1. **Recursive state maintenance**: ability to read current state, compute updates, and maintain new state
2. **Reliable history access**: ability to access past states unambiguously and maintain temporal ordering

The proof constructs a Universal Turing Machine simulation using state encoding, tape-operation simulation via history
entries, and position tracking with logarithmic overhead in both space and time.

## Key results

- **Theorem 1 (Universality)**: Systems with recursive state maintenance + reliable history access simulate a UTM with
  O(log t) space and time overhead
- **Corollary 4**: Such systems are Turing-complete
- Demonstrated across domains: Rule 110 cellular automata, neural networks via attention windows, biological cells via
  epigenetic state, developmental embryo morphogenesis

## What's reusable in Linus

**Phase 3–6 relevance (Knowledge integration, agent orchestration, fine-tuning)**:

- Session-store design: Linus orchestration layer should treat state maintenance as the primary abstraction, not tool
  outputs. Workgraph's `.workgraph/graph.jsonl` append-only DAG already embodies this principle.
- Agent continuity: multi-agent parallel execution requires reliable history access across agents. This formalism
  justifies audit logging as a first-class citizen, not a feature add-on.
- Memory-efficient inference: on 32 GB unified memory, privileging state compaction over model size aligns with this
  principle; trading computation for memory-coherence may improve Metal/ANE efficiency.

## What's NOT applicable / hype filter

- Not directly about LLM training or inference (no model architectures discussed)
- No hardware-specific optimization; the overhead analysis is asymptotic, not suited to Apple Silicon profiling
- Does not address fine-tuning or LoRA constraints on 32 GB systems

## Open questions for Dan

1. **Layer A (intra-step latent) governance.** Garrison treats memory as orchestration-layer machinery, but Layer A
   is opaque — whatever the dispatched Worker's hidden state happens to be. Should Linus carry an explicit
   "preferred-architecture" register for Layer A in the router (DEC-0031) — e.g., "for memory-mode = stateful, prefer
   architectures with externally addressable hidden states" — or is opacity at Layer A the correct contract?
2. **Logarithmic-overhead claim, in practice.** Theorem 1 promises O(log t) space and time overhead for a UTM
   simulation. The Layer C episodic substrate (DEC-0029) is the place where this overhead lands in practice. Is it
   worth a Phase 1c benchmark spike to characterize actual scratchpad-write and history-retrieval latency on the
   chosen substrate (SQLite + content hashes), to confirm the asymptotic story holds at the scales Linus hits?
3. **Biology analogue worth pursuing.** Garrison repeatedly cites cellular epigenetics, neural time cells, and
   morphogen gradients as biological instantiations. For Dan's metagenomics workflows, is there a useful framing
   where a sample's processing history (read QC -> assembly -> annotation) is treated as the same kind of
   reliable-history-access object the memory pillar describes? Worth a session-summary or an episodic-memory
   case-study artifact?
4. **What this paper does NOT settle.** Garrison proves universality given the two requirements; he does not say
   how to *implement* them efficiently or how to *measure* whether a given system meets them. Should Linus carry an
   internal property test — a "Garrison conformance check" — that any new Worker integration must pass to be
   considered eligible for `memory_mode != stateless` dispatch? The current ADR cluster assumes Garrison's
   framework but does not yet ship a check.
5. **Phase 6 fine-tuning targets.** [DEC-0043](../adr/0043-memory-mode-finetuning-targets-phase6.md) commits to
   memory-mode-aware fine-tuning. Is Garrison's framework strong enough to dictate which signals the fine-tuning
   loss includes (e.g., explicit history-recall accuracy as a separate term), or is "the model just gets better
   at using the substrate" the right contract?
