# QiMeng-CRUX: Narrowing the Gap Between Natural Language and Verilog via Core Refined Understanding eXpression

## TL;DR

CRUX is a structured intermediate representation bridging free-form hardware design descriptions (natural language) and
formal Verilog code. Two-stage training (Joint Expression Modeling + Dual-Space Optimization) enables LLMs to generate
syntactically correct, functionally faithful hardware specifications—a domain-specific code-generation pattern
applicable to Linus scientific workflow automation (e.g., bioinformatics pipeline DSL generation).

## The problem (in plain language)

LLMs struggle with hardware code generation because natural language hardware descriptions lack explicit emphasis on
critical design details (module interfaces, control logic, timing constraints). Free-form descriptions are ambiguous;
Verilog is unforgiving. Existing LLM approaches either generate incorrect code or require extensive post-hoc correction.

## What they propose

**CRUX (Core Refined Understanding eXpression)**: A structured intermediate space capturing essential circuit semantics:

- **Module Interface**: input/output ports with bit widths
- **Core Functions**: behavioral logic (FSMs, combinational circuits)
- **Key Considerations**: subtle constraints (bus arbitration, reset logic, timing)

Two-stage training framework:

1. **Stage I: Joint Expression Modeling (JEM)**: LLM learns to generate CRUX from natural language descriptions via
   supervised fine-tuning on curated (description, CRUX) pairs and reference code
2. **Stage II: Dual-Space Optimization**: CRUX-enhanced GRPO training with Veriloga code as supervision signal,
   rewarding functional correctness (Code-Reward) and narrowing intent-implementation gap

## Key results

- QiMeng-CRUX achieves state-of-the-art on VerillogEval-v1 (76.8 → 82.9%) and RTL-LLM-v1 (65.2 → 69.0%) benchmarks
- CRUX space proves robust and semantically meaningful; transferable when used as prompt for other code models
- Two-stage approach reduces under-specification errors, improves code synthesis fidelity
- CRUX-enhanced GRPO outperforms baseline RL on both challenging (Spec-to-RTL) and standard (code completion) tasks

## What's reusable in Linus

**Phase 2–3 relevance (Orchestration, knowledge substrate, domain-specific tool generation)**:

- **Intermediate representation pattern**: CRUX as a domain-specific abstraction layer is directly applicable to
  bioinformatics workflow DSL. Linus could define an intermediate representation for genomic pipelines (e.g., assembly
  workflow: inputs, reference genomes, parameters, output validation rules) and teach Workers to generate both CRUX
  and downstream WDL/Nextflow code.
- **Supervised + RL training**: JEM + GRPO pattern is a blueprint for fine-tuning Linus Workers on Dan's domain corpus.
  Curate (natural query, structured intermediate, generated tool code) triples from KnowledgeBase and experiment logs.
- **Correctness feedback**: CRUX-Reward (functional correctness via automated verification) maps to Linus tool
  verification loop. Sandbox policy (phase 7) could privilege tools that include CRUX-like validation specs.
- **Multimodal grounding**: CRUX includes graphs, state diagrams, reference code—analogous to pairing genomic pipeline
  descriptions with assay protocols, sequencing data quality metrics, and assembly quality thresholds.

## What's NOT applicable / hype filter

- Hardware verification domain expertise (Verilog syntax, circuit design patterns) doesn't transfer to bioinformatics
- RTL-specific benchmarks and metrics (Verilog syntax correctness, simulation pass@k) need biology equivalents (pipeline
  correctness, assembly contiguity, gene call sensitivity/specificity)
- GPU-based fine-tuning; Linus must use LoRA on 7–14B models with MLX/Metal
- Assumes extensive labeled hardware design corpus; biology equivalent (curated protocols + assembly benchmarks) is
  smaller and noisier
