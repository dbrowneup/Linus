# VEGA: Automatically Generating Compiler Backends using a Pre-trained Transformer Model

## TL;DR

Proposes VEGA, an AI-driven system that automatically generates LLVM compiler back-ends for new processor targets from
their target description files (`*.td`, `*.h`, `*.def`) alone. Mines 101 existing LLVM back-ends from GitHub, groups
target-specific implementations of each compiler-interface function (e.g. `getRelocType`) into "function groups",
abstracts each group into a function template with common code plus placeholders, then maps every statement to a feature
vector capturing target-independent and target-dependent properties. A fine-tuned UniXcoder-based model (CodeBE, 125M
params) emits target-specific code with per-statement confidence scores. Evaluated on RISC-V, RI5CY (a customized ULP
processor), and xCORE (an IoT processor): generates an entire back-end in under an hour, with pass@1 accuracy of 71.5% /
73.2% / 62.2% across all generated functions — vs. <8% for the traditional fork-flow baseline. The confidence scores
localize manual remediation effort to ~25–35% of generated functions.

## The problem (in plain language)

Building a compiler back-end for a new processor is one of the most labor-intensive tasks in systems software. The ARM
back-end in LLVM 12.0 contains 3,628 functions and 83,704 lines of code; across the 101 LLVM back-ends VEGA's authors
mined from GitHub, manually-written back-end code totals ~37.8M lines vs. ~16.7M lines of LLVM-provided infrastructure.
The traditional workflow ("fork-flow") is to copy an existing back-end (say MIPS) and edit it line by line for the new
target — slow, error-prone, and requiring deep architecture-specific knowledge.

LLM code generation has been demonstrated for general programming tasks (AlphaCode, Codex, etc.) and even for hardware
description languages (the QiMeng family — CodeV, QiMeng-Kernel, etc.). But compiler back-ends are a harder shape: each
back-end is enormous, the target-specific bits are interleaved with massive target-independent boilerplate, and
"correctness" means passing the LLVM regression suite (16,000+ test cases) — not just compiling. The question the paper
attacks: can a relatively small pre-trained transformer learn the structure of LLVM back-end code well enough to emit a
complete back-end for a never-seen target, given only that target's description files?

The deeper question — and the one that makes the paper interesting beyond compiler engineering — is how to factor the
code-generation problem so a 125M-parameter model can solve it. Vanilla "write me an LLVM back-end for RISC-V" against
GPT-class LLMs is not the framing; this is structured retrieval and template-instantiation, with the model doing the
target-specific value selection that statistical methods cannot.

## What they propose

**VEGA** decomposes back-end synthesis into three stages:

1. **Code-Feature Mapping (preprocessing + Stage 1).** Ingests 101 existing LLVM back-ends. Groups every target-specific
   implementation of each compiler interface function (e.g., the 101 different versions of `getRelocType`) into a
   **function group** `FG_M`. For each group, synthesizes a **function template** `FT_M` containing the
   longest-common-subsequence common code plus placeholders (`SV_k`) for target-specific values. Uses GumTree AST
   diffing to align statements across implementations.

   Then identifies two property classes per template:
   - **Target-independent properties** (Boolean): whether a class, enum, or global variable referenced in the common
     code exists in this target's implementation. Located via parsing tokens against `LLVMDIRs` (`llvm/CodeGen`,
     `llvm/MC`, `llvm/BinaryFormat`, `llvm/Target`) and the per-target `TGTDIRs` (`lib/Target/<Proc>`,
     `llvm/BinaryFormat/ELFRelocs`).
   - **Target-dependent properties** (string): the actual target-specific value to fill into a placeholder. Located via
     TableGen `*.td` / `*.def` / `*.h` files and partial-match heuristics against enum members and assignments.

   Every statement in every target-specific function becomes a **feature vector** `FV_k = <T_k, V_k>` where `T_k` is the
   tokenized statement template and `V_k` is the property-value tuple for that target.

2. **Model Creation (Stage 2).** Fine-tunes UniXcoder (a 12-layer, 125M-parameter encoder-decoder transformer
   pre-trained on multi-language code) into the **CodeBE** model. Inputs are feature vectors; outputs are
   `<confidence_score, statement_tokens>` sequences. Training set: 75% of functions per group → 7,902 functions /
   107,718 statements / 825 function templates. Verification: the remaining 25% (3,338 functions / 52,615 statements).
   Held-out targets (RISC-V, RI5CY, xCORE) are excluded. Hyperparameters: lr 6e-5, batch size 256, max sequence length
   512, 50 epochs, Adam + cross-entropy, end-to-end backprop.

   The confidence-score formula is structurally interesting:
   `CS(S_k) = (|T_k^com| / |T_k| + Σ 1/(|T_k| × N(SV))) × has(S_k)`, where `N(SV)` is the number of possible
   target-specific values for placeholder `SV` and `has(S_k)` is 1 if the statement is present in this target's training
   implementation. This makes confidence a function of how many tokens are common-code (high-certainty) vs. how many
   target-specific values the placeholders have to choose among.

3. **Target-Specific Code Generation (Stage 3).** Given a new target, VEGA reads its description files, populates
   property values `V_k` for each function template using the same identified-site / update-site mechanism used during
   training, and feeds the resulting feature vectors into CodeBE. CodeBE emits target-specific code with per-statement
   confidence scores; statements with score ≥ 0.5 are kept, < 0.5 flagged for manual review or deletion. The confidence
   score of a function's first statement (the function definition) acts as the function- level confidence.

The key methodological move — and the reason a 125M model works at all — is that VEGA does not ask the LLM to invent
back-end structure from scratch. The structure is mined from the existing 101 back-ends as templates; the LLM's job is
the much narrower task of selecting the right target-specific values from the target description files and deciding
which statement templates apply. This is structured retrieval with neural value-completion, not free-form code
synthesis.

## Key results

- **Generation time.** RISC-V: 1,383 sec. RI5CY: 1,664 sec. xCORE: 424 sec. All under one hour for a complete back-end.
  The Code-Feature Mapping stage runs in ~1,200 sec automatically; only the one-time Model Creation stage (fine-tuning)
  takes ~72 hours, amortized across all future targets.
- **Function-level accuracy (pass@1, regression-test based).** 71.5% (RISC-V), 73.2% (RI5CY), 62.2% (xCORE) of all
  generated functions are accurate without manual modification. Per-module breakdown: OPT 83.3% / 84.2% / 78.3%, SCH
  80.5% / 76.7% / 76.7%, EMI 78.0% / 78.1% / 85.7%. The weakest module is SEL (instruction selection) at 48.4% / 49.6% /
  44.6% — the most architecture-dependent.
- **Statement-level accuracy.** 55.0% (RISC-V), 58.5% (RI5CY), 38.5% (xCORE) of all generated statements are accurate
  without manual modification, vs. 14.8% / 14.6% / 12.6% for ForkFlow.
- **Vs. ForkFlow.** ForkFlow (manually editing the most-similar existing back-end — MIPS in all three cases) yielded
  accuracy of 7.9% / 6.7% / 2.1%. VEGA delivers a ~9–30× function-level accuracy improvement over the traditional
  baseline.
- **Cross-target reuse.** 14.8% / 13.5% / 4.0% of accurate functions integrate statements from multiple existing
  back-ends — e.g., the RISC-V `applyFixup` function pulls elements from PowerPC and AMDGPU based on byte-order and
  instruction-format similarity. CodeBE learned cross-architecture reuse implicitly.
- **Performance of generated back-ends.** After patching the inaccurate functions with their LLVM-canonical
  counterparts, the resulting back-ends execute SPEC CPU2017 (RISC-V), the PULP regression suite (RI5CY), and Embench
  (xCORE) at performance comparable to the manually-written LLVM back-end (Fig. 10).
- **Error taxonomy.** Three error classes: Err-V (incorrect target-specific value, 1.1–3.9%), Err-CS (statement emitted
  with wrong confidence, 10.1–11.6%), Err-Def (missing necessary statement, 22.9–37.2%). The dominant error is Err-Def —
  incomplete generation, not incorrect generation.
- **Ablations.** UniXcoder-based VEGA outperforms RNN-based VEGA by 35.3–77.7% and BERT-based VEGA by 32.1–67.0%;
  function-group-based dataset partitioning beats backend-based partitioning by 11.1–26.2%. The pre-trained UniXcoder,
  with no VEGA fine-tuning, generates zero correct functions on the held-out targets — confirming the fine-tune carries
  the back-end-specific knowledge and the held-out targets are not contaminating training.

## What's reusable in Linus

VEGA is the QiMeng-family member that points furthest into compiler infrastructure rather than kernels or hardware
itself. The **idea → reality** spine, per [`qimeng-category-promotion.md`](../specs/qimeng-category-promotion.md): the
LLM produces an LLVM back-end (compilable C++ source); the downstream actor is the LLVM build system + the regression
test harness; the realization is a working compiler that targets a real (or simulated) processor. There is no human in
the loop between CodeBE's output and the binary that runs on RISC-V QEMU other than ~25–35% of functions needing manual
remediation, and the confidence scores localize that effort precisely.

**Phase 6 — small-model + structured-retrieval as a fine-tune target.** The most surprising fact about VEGA is that a
125M-parameter UniXcoder fine-tune handles this. Compare the QiMeng-GEMM (paper-note [13337-ZhouQ.md](13337-ZhouQ.md))
result that Codestral-22B was too small to follow GEMM's multi-turn meta-prompt protocol — VEGA's task is harder in
absolute terms (back-end >> kernel) but factored differently: structure mined from corpus, not generated; LLM only
completes target-specific values. For Linus's Phase 6 fine-tuning lane (DEC-0014), this is an existence proof that small
fine-tuned models can outperform large frozen models on narrow-but-structured code-gen tasks _when the structure is
extracted upstream_. A 125M-class fine-tune is well inside the M1 Max envelope; the ~72-hour fine-tune on 8× V100 maps
roughly to a few days on the M1 Max for an equivalent model. _Artifact:_ a Linus fine-tune for a specific
structured-retrieval task. _Downstream actor:_ whatever build system / harness validates the structured output. _What
must be true for Linus locally:_ a corpus- mining stage that extracts templates + property schemata from existing
artifacts (Worker-orchestratable), then a small-model fine-tune.

**Phase 7 — confidence-scored generation as the audit substrate.** VEGA's per-statement confidence score is the single
most reusable technique in the paper. The audit-log convention (CLAUDE.md §Memory pillar discipline, DEC-0030) records
reasoning tokens and tool calls; layering per-token or per-statement confidence onto generated output is the natural
extension for any Linus skill that produces structured artifacts a downstream actor must trust. The confidence formula
itself — common-code-fraction plus inverse-cardinality of placeholder choices — generalizes beyond compiler back-ends to
any template-instantiation task: "how much of this output is mechanical vs. how many choices did the model make, and how
big was each choice's space?" A reusable Linus utility. This pairs with the
typed-structured-prediction-wrapping-rationale convention (CLAUDE.md §Typed structured prediction for biology skills,
S25): the rationale explains _why_, the confidence quantifies _how sure_.

**Phase 7/8 — the Apple Silicon Metal back-end question.** This is the most provocative open question. VEGA was trained
on 98 LLVM back-ends and held out three (RISC-V, RI5CY, xCORE). LLVM today already has a Metal/AIR back-end indirectly
via Apple's AIR (Apple Intermediate Representation), and a Mojo / MLX kernel-codegen pipeline that ends in Metal Shading
Language. For a hypothetical novel Apple Silicon hardware variant (an ANE successor, a new GPU ISA, a Mac M5 with
extended SIMD), could a VEGA-style pipeline auto-generate the LLVM back-end? The corpus is public (101 back-ends in
GitHub), the technique is published, and the variant-specific knowledge would be exactly the description-files VEGA
needs. Worth flagging as a Phase 7/8 idea-to-reality candidate, even though Apple's internal compiler team almost
certainly won't open-source the answer. _Artifact:_ an LLVM (or AIR-equivalent) back-end for a new Apple Silicon
hardware variant. _Downstream actor:_ Apple's compiler toolchain. _What must be true for Linus locally:_ access to a
hardware variant's description files (or a reverse-engineered approximation, per the ANE work in `repos/ANE/`); a
Linus-internal corpus of LLVM back-ends; and a CodeBE-class fine-tune. This is the speculative-but-pointed direction
VEGA opens up for Linus. (CLAUDE.md §Public APIs only constrains how Linus itself ships; it does not constrain Linus
from _producing_ a back-end if the description files become available.)

**Phase 2/3 — function-group + template-extraction as an orchestration primitive.** Stripped of the LLVM specifics,
VEGA's preprocessing pipeline is a generic pattern: cluster N existing implementations of a common interface; extract a
template via longest-common-subsequence + AST alignment; identify the variable points (placeholders); identify the
choice space at each variable point. This is the same shape as the agent-spawner role-as-typed-pattern (DEC-0050,
DEC-0051): a role is a function group, an AgentReport is a feature vector, the typed action space at each role is the
placeholder choice space. Could be a Phase 2/3 orchestration primitive: given N existing solutions to a class of
problems, automatically extract the typed action schema. Worth a memo.

## What's NOT applicable / hype filter

The 71.5–73.2% pass@1 numbers are **per-function**, not **per-back-end**. A back-end with 28.5% of functions failing is
a back-end that does not compile or fails the regression suite — it requires the manual-remediation pass before it works
at all. The "in under an hour" headline is the auto-generation step; the manual remediation downstream is real work. The
paper acknowledges this and reports cleanly (post-patching performance vs. canonical LLVM); a casual reader might miss
it.

The training corpus is **101 existing LLVM back-ends from GitHub**. Coverage of unusual targets (highly customized
ASICs, non-LLVM frameworks like GCC or MLIR, hardware with no instruction-set similarity to anything in the corpus) is
not addressed. The xCORE result (62.2% function accuracy) is the lowest of the three — and xCORE is the most divergent
target tested (IoT processor with domain-specific instructions). For a truly novel hardware variant with no similar
precedent in LLVM, the generation accuracy is an open question.

The approach is **LLVM-specific**. Function templates, identified-site / update-site logic, and the `LLVMDIRs` /
`TGTDIRs` directory schema all assume LLVM's project structure. For GCC, MLIR, or a custom compiler infrastructure, the
same methodology applies but the engineering is non-trivial. The paper does not argue for general portability.

The 125M-parameter UniXcoder choice is **historically grounded**: this is a 2025 CGO paper, and 125M was a sensible size
when UniXcoder was the SOTA encoder-decoder for code in 2022. Whether scaling to a larger code model (Code-LLaMA at
7B-34B, DeepSeek-Coder-V2, or a Qwen3-Coder fine-tune) materially improves results is not measured. The
upstream-template-extraction does most of the heavy lifting; large-model gains may saturate. Worth measuring before
betting Phase 6 architecture on either size.

The **confidence threshold of 0.5** is asserted, not derived. The paper uses 0.5 as the keep/discard cutoff with no
ROC-curve analysis or per-target tuning. For Linus uses of confidence-scored generation, a calibration step on a
held-out validation set is the right discipline — VEGA's threshold may not transfer to other artifact classes.

The auto-tuning + benchmark-feedback loop that QiMeng-GEMM uses is **absent in VEGA**. VEGA is one-shot generation with
confidence scoring; there is no iterative refinement loop where the failed regression tests feed back into the model.
This is a significant simplification that the paper does not problematize. The follow-up question — does a VEGA +
regression-feedback loop close the remaining ~30% accuracy gap? — is left open.

## Connections

VEGA sits inside the QiMeng family (ICT/CAS, SKLP) and is methodologically the closest cousin to QiMeng-GEMM
([13337-ZhouQ.md](13337-ZhouQ.md)) — both treat code generation as structured retrieval with a curated substrate (hint
repository in QiMeng-GEMM; function templates + property schemata in VEGA), both use a frozen-or-fine-tuned model inside
a hand-engineered framework rather than asking a large LLM to do free-form synthesis. The
[QiMeng-Kernel (2511.20100v1.md)](2511.20100v1.md) macro-thinking / micro-coding paradigm is the more sophisticated
descendant: explicit planner LLM, RL-trained coder LLM, iterative refinement. VEGA / QiMeng-GEMM / QiMeng-Kernel
together trace a clean evolutionary line from "small fine-tune + structured-retrieval" → "frozen large model +
prompt-engineered substrate + beam search" → "planner LLM + coder LLM + RL". Useful pedagogically.

The closest hardware-design siblings in the QiMeng family are
[Push the Limits — CPU design via AI (2306.12456v2.md)](2306.12456v2.md) and
[Cheng et al. superscalar processor design (0549.md)](0549.md): these target the processor itself; VEGA targets the
compiler that compiles for the processor. Together with [QiMeng-CRUX (2511.20099v4.md)](2511.20099v4.md) (Verilog
code-gen) and [QiMeng: Fully Automated HW+SW Design (2506.05007v1.md)](2506.05007v1.md) (the integrating overview), the
family covers an end-to-end stack: design the processor, design the kernels, design the compiler back-end, design the
runtime. VEGA is the compiler-toolchain piece.

The repo-side anchor is [QiMeng-cpu-v1 (repo-note)](../repo-notes/QiMeng-cpu-v1.md) — behavioral synthesis of a RISC-V
CPU. Cross-paradigm: [Sketch2Simulation (2603.24629v1.md)](2603.24629v1.md) (HYSYS process flowsheets) is the
non-hardware exemplar of the same idea-to-reality spine. The cluster home is the in-flight g12-llm-hardware-design
synthesis under [`docs/specs/qimeng-category-promotion.md`](../specs/qimeng-category-promotion.md).

UniXcoder (the underlying pre-trained model) is cited extensively in the function-annotation-discovery and biological
foundation-models corpus as a code-pretraining baseline; the small-fine-tune-on-narrow-task pattern VEGA exemplifies is
the same shape used by several biology specialist models (BioReason-Pro and others — see
[biological-foundation-models-synthesis.md](../syntheses/biological-foundation-models-synthesis.md)). The
template-extraction + property-schema pipeline is structurally analogous to the agent-spawner role-typed pattern
([DEC-0050](../adr/0050-role-first-class-type-agent-spawner.md),
[DEC-0051](../adr/0051-agent-report-typed-inter-agent-message.md)), and the confidence-scored output substrate is a
natural pairing with the typed-structured-prediction convention from CLAUDE.md S25. The pmetal evaluation gate
([DEC-0006](../adr/0006-pmetal-phase1-evaluation.md), [DEC-0012](../adr/0012-pmetal-primary-inference-candidate.md)) is
conceptually adjacent — pmetal is a back-end implementation; VEGA is back-end generation; the convergence point would be
auto-generated Apple Silicon back-ends.

## Open questions for Dan

1. **Apple Silicon Metal/AIR back-end as a Phase 7/8 idea-to-reality target.** Could a VEGA-style pipeline auto-generate
   an LLVM back-end (or AIR-equivalent) for a hypothetical novel Apple Silicon hardware variant — an ANE successor, a
   new GPU ISA, an M5-class extended SIMD? The corpus (LLVM back-ends from GitHub) is public, the technique is
   published, and the variant-specific knowledge is exactly the description-files VEGA needs. The honest constraint:
   Apple does not ship description files for unreleased hardware. But for a reverse-engineered variant (the methodology
   in `repos/ANE/`) or a hypothetical/educational target, this is squarely on the idea-to-reality spine. Worth a Phase
   7/8 spike?

2. **Small-model fine-tune as the right Phase 6 shape for narrow code-gen.** VEGA's 125M-parameter UniXcoder fine-tune
   outperforms the frozen pre-trained model by 100× on this task. For Linus's Phase 6 fine-tuning lane (DEC-0014), is
   the right pattern (a) a single large LoRA on Qwen3-30B-class base + broad-task instruction-tune, or (b) a
   constellation of small fine-tunes — VEGA-style — each ~125M-class, each targeting one structured artifact class
   (compiler back-ends, kernel codegen, schematic synthesis, MCP-tool wrappers, etc.)? The small-constellation pattern
   is more memory-friendly on M1 Max (multiple cohabiting models fit in 32 GB) and maps cleanly to the Worker-orchestra
   metaphor; the large-base pattern has better cross-task transfer. Empirical question worth measuring on the first
   Linus fine-tune.

3. **Confidence-scored generation as a Linus-wide convention.** Should every Linus skill that emits a structured
   artifact also emit per-element confidence scores, calibrated against a held-out validation set? VEGA's formula
   (`(common_fraction + inverse_choice_cardinality) × has`) is a sensible default for template-instantiation tasks; for
   free-form generation, log-probabilities or per-token entropy would substitute. This is the natural pairing with
   CLAUDE.md S25 (typed-structured-prediction-wrapping-rationale): rationale + confidence is a strong audit substrate.
   Worth seeding as a convention candidate before the first Phase 6 skill ships.

4. **Function-group + template-extraction as a Phase 2/3 orchestration primitive.** VEGA's preprocessing pipeline
   (cluster N implementations → extract template via LCS + AST diff → identify placeholders + property schemata) is the
   inverse of the agent-spawner role-as-typed-pattern (DEC-0050/0051): _given_ N existing solutions, automatically
   extract the typed action schema. Could Linus carry this as a generic "schema-from-corpus" utility, applied first to
   the role library bootstrap (mine from existing LangGraph / CrewAI / etc. agent code) and later to user- specific
   corpora (mine Dan's own writing, code, or notebooks for typed action schemata)? Phase 2/3 case worth thinking about;
   it converts the "where do action schemas come from" question into a search problem with a known algorithm.

5. **VEGA + regression-feedback loop as the unification point.** VEGA is one-shot; the QiMeng-Kernel paradigm is
   iterative-with-RL; QiMeng-GEMM is iterative-with-beam-search. The paper does not measure what happens if VEGA's ~30%
   inaccurate functions feed back into the model via the regression-test failures. Is the natural Phase 7 experiment to
   graft QiMeng-Kernel's RL-trained coder + correctness-reward shape onto VEGA's structured-retrieval substrate, closing
   the accuracy gap? This is the obvious next move and the paper-as-published does not take it.

6. **Confidence threshold calibration on novel artifact classes.** The 0.5 threshold is asserted without ROC analysis.
   For Linus uses of confidence-scored generation, the threshold matters: a too-low cutoff floods the review queue with
   false positives; a too-high cutoff silently drops correct output. Should the Linus skill library carry a calibration
   step (held-out validation per skill, ROC-tuned threshold per artifact class) as a first-class skill-deployment
   requirement? This is a small addition with outsized correctness-engineering payoff.
