# Survey and Evaluation of Converging Architecture in LLMs Based on Footsteps of Operations

## TL;DR

Traces the evolution of LLM architectures (RNNs → Transformer → modern variants) through the operational lens: which layers, operations, and hyperparameter choices drive performance. Benchmarks Llama 2/3, Gemma on RTX 6000 GPUs across layer configurations, inference time, and memory. Focused on identifying which architectural components are essential vs. redundant. Directly applicable to Linus Phase 2: choosing which operations to prioritize for MLX/Metal implementation on M1 Max.

## The problem (in plain language)

LLM architectures have converged on Transformer decoder-only with attention, FFN, normalization. But which design choices are load-bearing? Does every model need grouped-query attention (GQA)? How much does multi-head attention (MHA) vs. single-head cost? Survey dissects the operational lineage from RNNs through BERT to modern LLMs, measuring latency/memory per component on modern GPUs.

## What they propose

Five architectural stages traced through operations:

1. **RNNs** (sequential, no parallelism): LSTM, GRU
2. **Encoder-Decoder** (seq2seq, bidirectional): attention bridges encoder/decoder
3. **Transformer** (parallel, self-attention): multi-head attention (MHA), FFN, layer norm
4. **Pre-trained LMs** (causal masking, decoder-only): masked language modeling, generative pretraining
5. **Modern LLMs** (Llama, Gemma, Qwen): grouped-query attention (GQA), RMSNorm, SwiGLU/GeGLU FFN variants, flash attention, rope positional embeddings

Benchmarks on RTX 6000 (Ada): layer-by-layer latency, inference time, weight kernel size, memory footprint.

## Key results

- **Feed-Forward Networks (FFN)**: ~60-65% of inference time in Transformer blocks. SwiGLU (gated) no faster than standard FFN; choice driven by training stability, not inference speed.
- **Attention variants**: MHA → MQA (multi-query) → GQA (grouped-query) reduces KV cache by 8-16x. GQA adds minimal latency; MQA sometimes slower due to poor kernel fusion.
- **Normalization**: LayerNorm vs. RMSNorm has <5% latency difference on modern GPUs. RMSNorm slightly faster (no mean subtraction).
- **Rope vs. ALiBi**: Rope cheaper if precomputed; ALiBi requires per-sequence computation but saves memory.
- **Model size scaling**: Llama 2 (7B→70B) and Gemma (2B→7B) show linear scaling in latency with parameters; no emergent discontinuity.

## What's reusable in Linus

**Phase 1-2 (immediate):**
- Operation prioritization: FFN is the bottleneck. Optimize Metal/ANE shaders for GeMM (general matrix-multiply) first. Attention is secondary.
- GQA adoption: reduces KV cache 8-16x. Essential for fitting larger models (13B+) on 32 GB unified memory. Paper validates GQA adds <2% latency.
- RMSNorm over LayerNorm: simpler, same speed. Use in MLX models.
- Rope positional embeddings: proven generalization to longer sequences; precompute for inference.

**Phase 2-3 (architecture decisions):**
- Layer configuration guidance: survey data shows 32-40 layers is typical sweet spot (7-13B). Linus should default to Llama 2/3 or Mistral 7B family as base.
- Attention kernel choices: if implementing custom Metal attention, GQA is the default for modern models. Single-head MQA has gotchas; avoid unless necessary.

**Phase 6 (fine-tuning):**
- FFN scaling laws: if LoRA-tuning only attention, recognize that FFN dominates cost. Consider full FFN fine-tuning or LoRA on FFN weights.

## What's NOT applicable

- **GPU-specific benchmarks**: RTX 6000 (Ada architecture) does not translate to M1 Max Metal/ANE. Memory bandwidth, cache hierarchy, warp scheduling are completely different. Relative timings will not hold.
- **Large-batch inference**: Benchmarks assume industrial inference server (batch size 16-32). Linus is single-device, batch size 1-2 max.
- **Distributed training**: multi-GPU optimization (tensor parallelism, pipeline parallelism) not applicable to MacBook.
- **Quantization details**: paper does not cover bfloat16 vs. float16 vs. int8 trade-offs, which are critical for M1 unified memory fitting.
