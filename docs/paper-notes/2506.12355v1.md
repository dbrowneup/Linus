# QiMeng-Attention: SOTA Attention Operator is Generated by SOTA Attention Algorithm

## TL;DR

Proposes **LLM-TL** (LLM-friendly Thinking Language), a two-step DSL-mediated workflow that lets a frozen LLM
(DeepSeek-V3, DeepSeek-R1, Claude 3.5 Sonnet, GPT-4o) emit FlashAttention-quality CUDA kernels on heterogeneous NVIDIA
GPUs (A100, RTX8000, T4, L40S) — including configurations that the official FlashAttention library does not support
(Turing architecture, FP8 datatype, MLA operator). The trick is decoupling the work into (1) a TL Sketch / TL Code
generation phase where the LLM reasons over abstract `Copy` (data movement) and `Compute` (computation) statements that
hide PTX-level details, and (2) a TL→CuTe translation phase that lowers those statements into NVIDIA CuTe / Tensor-Core
code. Hits **up to 35.16× speedup over vanilla LLM-generated CUDA**, **2.15× over cuDNN on the novel MLA operator**
(175.6 vs 81.7 TFLOPS), and reduces kernel development time **from months to ~10 minutes** while exceeding the human
expert's TFLOPS (175.6 vs 162.7).

## The problem (in plain language)

Attention is the dominant runtime cost of modern LLMs — quadratic in sequence length, and the long-context regime makes
it the dominant memory-bandwidth cost too. FlashAttention (Dao 2022) is the canonical fix, but each new variant of
attention (MHA → GQA → MQA → MLA → NSA) and each new GPU generation (Turing → Ampere → Ada → Hopper → custom datatypes
like FP8) requires a fresh hand-tuned implementation. The official FlashAttention v2 only supports a subset of GPUs and
attention shapes; FP8 MHA and MLA on Turing (RTX8000) currently have no official kernel at all. The labor cost is months
per port.

The earlier QiMeng-GEMM paradigm ([13337-ZhouQ.md](13337-ZhouQ.md)) showed that meta-prompts + auto-tuning over GEMM
optimization primitives could match cuBLAS / OpenBLAS using a frozen LLM. Attention is harder. Three reasons the paper
identifies: (1) the multi-step data flow (Q/K/V → S = QK^T → softmax → S·V) needs to be **fused** at the register level
to avoid global-memory round-trips — the entire FlashAttention insight — and naive prompt-driven LLM code can't see the
fusion opportunity; (2) Tensor-Core programming requires PTX `mma` / `ldmatrix` instruction families and CuTe
abstractions that LLMs reproduce inconsistently from training data alone; (3) attention variants and GPU specs combine
into a large matrix of (operator × architecture × datatype) targets, each needing its own fused kernel. Vanilla LLMs
either generate Torch-level high-level code (no Tensor-Core fusion → memory-bound) or attempt a fused CUDA kernel and
get the warp-tile / thread-tile / shared-memory / reshape choreography wrong (incorrect output, OOM, or 0.1× cuDNN
performance). Even chain-of-thought prompting only inches forward; the paper's Table 5 shows DeepSeek-V3 + CoT reaches
0.52 TFLOPS on a 2k-token A100 MHA kernel, while LLM-TL reaches 154.7 TFLOPS — an **895× margin**.

## What they propose

**LLM-TL** is a two-stage thinking-language workflow with a hand-designed mid-level DSL — TL — sitting between the
abstract algorithm and the concrete CuTe / PTX code.

1. **The TL DSL.** Two statement primitives — `Copy` (data movement between memory hierarchies: global ↔ shared ↔
   register) and `Compute` (computation: GEMM, softmax, reshape, multiply/divide, custom ops like `Compute Softmax S`) —
   plus a `Loop` construct (`for i = 0:N ... end`). Each statement carries parameters (compute layout, transfer size,
   allocate / reshape behaviour). The DSL is deliberately small: roughly a dozen lines of TL replaces hundreds of lines
   of CuTe/CUDA. The whole point is that the LLM can _reason about data flow and fusion_ at this layer without drowning
   in PTX index arithmetic. Crucially, fusion is encoded **structurally** — consecutive `Compute` statements at the same
   memory hierarchy with no intervening `Copy` are by-construction fused.

2. **TL Code Generation (Stage 1, in two sub-steps).**
   - _Sketch Generation:_ the LLM reads the user request ("generate a performant GQA kernel with causal mask on A100")
     and produces a **TL Sketch** — a sequence of `Copy` and `Compute` statements describing the algorithmic skeleton,
     no parameters yet. This step's whole job is to surface fusion choices.
   - _Parameter Analysis & Reasoning:_ a second LLM pass adds block sizes (BM, HeadDim), allocate clauses, reshape
     statements (the load-bearing one — without a `Reshape` between two GEMMs the layouts mismatch and either the code
     fails to compile or silently produces wrong output), and coordinate offsets (`L = block_idx`). The output is a
     complete TL Code program.

3. **TL Code Translation (Stage 2, Adaptive Translation).** Given finalized TL Code and a target GPU spec (Tensor-Core
   instruction set + datatype), the LLM emits CuTe code directly. The CuTe framework is itself a high-level abstraction
   over PTX `mma` / `ldmatrix`, so the LLM's translation target is "TL → CuTe", not "TL → PTX". Each TL statement maps
   to a small CuTe idiom (e.g., `Copy A from global to shared` → `cute::copy(gmem_tiled_copy, tQgA, tQsA)`). For GPUs
   CuTe doesn't yet support (the paper validates on T4 / Turing where CuTe MMA atoms are missing), the LLM is few-shot
   prompted to synthesize the missing MMA APIs from the architecture's documented PTX instructions; the rest of the
   pipeline is unchanged.

The key empirical claim driving the design is from their **TL designation ablation** (Appendix B): if you skip the
two-step decomposition and ask the LLM to generate TL Code in a single pass, the LLM consistently makes _Reshape
omission_ errors (forgetting layout transforms between GEMMs) and _GEMM transpose-vs-physical-layout_ errors. The
hierarchical Sketch → Code split forces the LLM to commit to the data-flow structure first, then layer parameters on
top, so the reshape and transpose constraints are visible at the right reasoning step.

## Key results

- **Vs. vanilla LLM-generated CUDA.** Up to **35.16×** on common attention operators (Table 1), **9.41×** on MLA (Table
  2). Vanilla DeepSeek-V3 hits OOM at long sequences; LLM-TL stays in memory and scales.
- **Vs. cuDNN / FlashAttention.** Outperforms cuDNN in most scenarios; reaches **2.15× cuDNN on the novel MLA operator**
  (175.6 vs 81.7 TFLOPS at A100 / 2k tokens / head-dim 128). On well-supported MHA/GQA/MQA shapes the result is
  "competitive or faster on most sizes" rather than uniformly faster — at peak A100 sizes flash-attn-v2 sometimes leads
  by ~10%, but LLM-TL leads at long-context (16k tokens, with causal mask) by 1.3–1.5× consistently.
- **Vs. human expert.** On A100 / head-dim 64 / 1k tokens MHA: human expert ~months / 162.7 TFLOPS; LLM-TL **10 minutes
  / 175.6 TFLOPS**. Both expert and LLM-TL use CuTe, so the comparison is "expert wrote CuTe by hand vs. LLM wrote CuTe
  via TL" — the ~1000× development-time reduction is the headline.
- **Generality across LLM backbones.** DeepSeek-R1 > DeepSeek-V3 > Claude 3.5 ≈ GPT-4o + DeepSeek-V3 > GPT-4o-only
  (which fails to translate TL → CuTe correctly because of training-cutoff CuTe scarcity). DeepSeek-R1 reaches 194.7
  TFLOPS at 16k seqlen / head-dim 128 / A100 MHA-causal — the highest of the four backbones.
- **Generality across GPUs.** Validates A100 (Ampere), RTX8000 (Turing), T4 (Turing), L40S (Ada). On Turing —
  unsupported by official FlashAttention v2 — LLM-TL produces a working FP8 MHA kernel at 224.8 TFLOPS (16k seqlen,
  L40S). This is the strongest "extends supported hardware/datatype" claim in the paper.
- **Long-context win is the load-bearing one.** The 34.94× speedup at 16k seqlen on A100 with causal mask is the result
  that genuinely matters for modern LLM inference. Most modern serving workloads sit in the long-context regime; the
  speedup curve grows with sequence length (28.13× at 2k, 34.94× at 16k), tracking the bottleneck shift from compute to
  memory-bandwidth.

## What's reusable in Linus

QiMeng-Attention is the **second cleanest expression** of the **idea → reality** spine in the QiMeng family, after
QiMeng-GEMM. The artifact is a fused attention kernel; the downstream actor is the CuTe → CUDA → PTX → SASS toolchain
(equivalent to MLX → AIR → Metal-IR → ISA on Apple Silicon); the oracle is a kernel benchmark harness. Crucially —
unlike QiMeng-GEMM which uses prompt engineering + beam-search auto-tuning over a static primitive set —
QiMeng-Attention uses **a hand-designed DSL as the LLM's reasoning substrate**. This is a different, more powerful
pattern: the DSL is the "typed structured prediction wrapping rationale" convention from CLAUDE.md (S25, biology skills)
applied to kernel codegen, where the structured part is the TL program AST and the rationale is the LLM's reasoning
between sketch and code.

**Phase 6/7 — direct Apple-Silicon analogue: Metal Flash-Attention via TL→Metal.** The whole pipeline transposes onto
Apple Silicon with the same shape but different terminal:

- _Artifact the LLM produces:_ a Metal compute-kernel source (or MLX `mx.fast` custom kernel) implementing fused
  attention.
- _Downstream actor that accepts it:_ the Metal compiler toolchain (`xcrun metal`, AIR, AGX) — exactly analogous to
  CuTe + nvcc.
- _What must be true for Linus to replicate the discipline locally on Apple Silicon:_
  1. **A "Metal-TL" DSL** that mirrors LLM-TL's `Copy`/`Compute`/`Loop` triplet over Apple's memory hierarchy (unified
     memory ↔ tile memory ↔ thread-local registers) and Apple's tensor-compute units (the AMX-style SIMD-group matrix
     instructions on M-series, plus ANE primitives where applicable). The TL designation lesson transfers verbatim — a
     single-pass "LLM, write a Metal flash-attention kernel" prompt fails the same way vanilla CUDA prompts fail on
     RTX8000.
  2. **A hint repository for the AGX architecture** in the QiMeng-GEMM sense — `simdgroup_matrix`, `threadgroup`
     allocation rules, the `barrier(mem_flags::mem_threadgroup)` semantics, the unified-memory `mtl::buffer<T>` idioms.
     This is the actual unit of human work being amortized; the DSL is generic but the lowering is
     architecture-specific.
  3. **A frozen Worker LLM that can sustain the two-step protocol.** The paper's GPT-4o failure case (couldn't translate
     TL → CuTe correctly because CuTe was under-represented in training) is a yellow flag for any pre-Phase-6-fine-tune
     local model: Metal-Performance-Shaders / MLX custom-kernel code is similarly under-represented in Qwen3-Coder /
     DeepSeek-Coder training corpora. A Phase 6 fine-tune over a synthetic `(TL-program, Metal-kernel)` corpus is the
     natural answer.

This maps directly to **Phase 6d** (`docs/specs/phase6d-streaming-target.md`), which already implies Linus needs
hand-crafted Metal kernels for the weight-streaming MoE path; LLM-TL is the methodology that lets a Worker emit those
kernels rather than Dan hand-writing them. The cross-reference to [flash_moe.md](flash_moe.md) is exact: Flash-MoE's
custom Metal kernel for fused dequant-of-2-bit + GEMM is precisely the kernel class LLM-TL would target if ported. The
Flash-MoE result (5.74 tok/s sustained on M3 Max for a 397B MoE via NVMe streaming) requires fused custom Metal kernels;
QiMeng-Attention is the recipe to generate those kernels under Worker control. Combined, the pair is a Phase 6d → Phase
7 progression: Flash-MoE proves the streaming substrate; LLM-TL-on-Metal lets Linus regenerate optimized kernels for new
variants without months of human work.

**Phase 7 — DSL-as-rationale-substrate is a generic Linus pattern.** The deeper reusable contribution is the recipe:
**hand-design a small DSL that hides the messy lower-level details, and use it as the LLM's reasoning substrate, then
translate.** This is not just a kernel-codegen pattern. It generalizes to any domain where the gap between
"natural-language description of what to do" and "syntactically and semantically correct artifact" is too large for a
single-shot LLM. Candidate Linus applications: (a) MLX training loops (TL-train DSL → MLX trainer source); (b)
CRISPR-edit design specifications (TL-edit → guide-RNA + repair-template artifacts); (c) Sketch2Simulation-style HYSYS
process flowsheets (TL-process → flowsheet XML). Each uses a domain-specific TL with a domain-specific lowering; the
two-step Sketch → Code pattern transfers. _Seed: an ADR for "Domain-DSL as LLM rationale substrate" alongside the MTMC
ADR seed already in [qimeng-category-promotion.md](../specs/qimeng-category-promotion.md)._

**Phase 6 — TL programs as a fine-tuning corpus shape.** The pair (TL Sketch, TL Code) and the pair (TL Code,
target-arch CuTe) are both clean structured-prediction targets. A fine-tune over
`(natural-language request, TL Code, CuTe code)` triples — bootstrapped by running QiMeng-Attention against a strong
teacher LLM and harvesting the intermediate TL — is a plausible Phase 6 LoRA target. This is the same logic as the
QiMeng-GEMM Phase 6 hint-repo fine-tune ([13337-ZhouQ.md](13337-ZhouQ.md)), but the TL DSL is a richer, more reusable
scaffold: a single TL fine-tune covers MHA/GQA/MQA/MLA/NSA + multiple GPU targets, where the GEMM hint-repo fine-tune
was per-primitive.

**Phase 2/3 — TL as an inter-Worker message format.** Stripping out the kernel-codegen specifics, TL is also a candidate
**typed AgentReport payload** (DEC-0051) when one Worker hands a kernel-design task to another: the Sketch-generating
Worker emits TL Sketch as its AgentReport.payload; the parameter-reasoning Worker consumes the sketch and emits enriched
TL Code; the translation Worker consumes TL Code and emits a Metal kernel artifact. The DSL gives the inter-agent
protocol a typed shape, not just a free-form string. This is a Phase 2/3 orchestration insight, not a Phase 6/7 codegen
insight.

## What's NOT applicable / hype filter

The 35.16× headline is **vs. vanilla LLM CUDA**, not vs. handcrafted FlashAttention. The fair-comparison numbers — vs.
cuDNN and flash-attn-v2 on shapes where flash-attn-v2 actually supports the GPU — are "competitive or modestly faster on
most sizes, sometimes 10-15% slower at peak compute-bound regimes, decisively faster (1.3-1.5×) at long context with
causal masks." The honest takeaway: LLM-TL is **as good as a senior expert spending months** on well-supported
configurations, and **the only available solution** on under-supported configurations (Turing FP8, MLA on any GPU, T4
with FlashAttention v2). The under-supported claim is the load-bearing one for Linus too — Apple Silicon is the entire
under-supported configuration for FlashAttention.

The TL DSL itself is **GPU-and-Tensor-Core-shaped**. The `Copy` / `Compute` primitives over global/shared/register
memory, the MMA tile structure, the `cute::copy` / `cute::gemm` vocabulary in the lowering — all of this assumes the
NVIDIA SIMT + warp-of-32 + Tensor-Core programming model. Apple Silicon's GPU is SIMD-group-of-32-on-M3 / 32-on-M1Max,
but the tile-memory semantics, the matrix-instruction availability (M3 added simdgroup_matrix; M1 has it via the
Performance Shaders compiler but the semantics differ), and the unified-memory hierarchy (no explicit global-vs-shared
distinction in the same sense) are different enough that the Apple-Silicon-TL DSL needs to be **re-derived from
scratch**, not transliterated. The methodology ports; the DSL design does not.

The paper does not test on **the latest H100** (resource constraint, acknowledged in §Limitations) and does not test on
**non-attention operators**. The "self-optimizing paradigm for attention-centric algorithms" framing is broader than the
evidence warrants — the empirical claim is about flash-attention-shaped fused operators specifically. Generalizing to
non-attention fused kernels (mamba scan, sparse attention with structural sparsity, ring-attention distributed
primitives) is conjecture.

The CuTe-API-generation step (the "for newer architectures CuTe may not include MMA APIs, so we few-shot prompt the LLM
to generate them" subsection) is **a nontrivial trust gap.** If CuTe hasn't shipped an MMA atom for an architecture, the
LLM-generated MMA atom is unverified by the CuTe team's correctness reviews; it's only verified by the paper's benchmark
harness, which checks numerical correctness on a finite test set. For a Linus-internal use this is fine (the kernel runs
in Linus's own sandbox with bounded blast radius), but for any kernel that ships into a production deployment outside
Dan's hardware, this is a process-engineering gap.

The development-time claim ("months → 10 minutes") collapses **expert effort into a single number** that doesn't account
for the time spent _writing the TL DSL_, _curating the few-shot CuTe examples_, and _building the benchmark harness_.
Those are upfront fixed costs amortized across all kernels generated thereafter. For Linus's first kernel target the
upfront cost is the dominant cost; the amortization argument only holds after several kernels have been generated
against the same DSL + harness.

The paper does not address **autoregressive generation regressions** — i.e., does an LLM-TL-generated kernel give the
same numerical output as the reference attention to within the expected float-precision tolerance, on a representative
suite of input distributions, including pathological cases (mixed-precision overflow, denormals, masked positions)? The
paper reports TFLOPS, not full numerical-equivalence audits. For Linus production use this gap matters; a Phase 6/7
spike port should add a regression suite before trusting the kernel.

## Connections

This is the **kernel-codegen sibling** of QiMeng-GEMM ([13337-ZhouQ.md](13337-ZhouQ.md)) — same group, same
"frozen-LLM-as-kernel-author" stance, but with a hand-designed DSL replacing GEMM's meta-prompt + auto-tuning beam
search. Where QiMeng-GEMM proves the methodology on a single operator (matmul) across CPU + GPU back-ends,
QiMeng-Attention extends it to a fused multi-operator artifact (Q·K^T → softmax → ·V) and to a wider GPU surface (four
NVIDIA architectures). The **two together cover the kernel-codegen frontier of the QiMeng family**: GEMM-class operators
via primitive-search prompts; fused attention-class operators via DSL-mediated reasoning.

The closest descendant is [QiMeng-Kernel (2511.20100v1.md)](2511.20100v1.md) — same lineage advanced one step further:
where QiMeng-Attention uses prompt + DSL for both planning and lowering, QiMeng-Kernel adds an **explicit planner LLM
(Macro Thinking)** that emits semantic optimization actions and a **separate RL-trained coder LLM (Micro Coding)** that
lowers them, with rule-based reward shaping. The "LLM emits TL Sketch then TL Code" two-step in QiMeng-Attention is the
primitive form of the planner/coder split; QiMeng-Kernel's MTMC formalizes it as an architecture and adds RLVR training.
For Linus, QiMeng-Attention is the Phase 6 spike target (no RL training to reproduce), QiMeng-Kernel is the Phase 7
target (RL feedback loop).

Within the QiMeng family the immediate thematic neighbors are
[Push the Limits — CPU design via AI (2306.12456v2.md)](2306.12456v2.md) — the earliest CAS-group exemplar of LLMs
designing real hardware artifacts; [Cheng et al. superscalar processor design (0549.md)](0549.md) — the closely related
processor-architecture exemplar; [QiMeng: Fully Automated HW+SW Design (2506.05007v1.md)](2506.05007v1.md) — the

The cross-paradigm Apple-Silicon connection is to [flash_moe.md](flash_moe.md) — which proves the weight-streaming

## Open questions for Dan

1. **Metal-TL DSL design — minimum viable shape.** LLM-TL's `Copy` / `Compute` / `Loop` triplet is defined over NVIDIA's
   three-tier global/shared/register memory hierarchy and warp-tile MMA structure. Apple Silicon's tile-memory +
   simdgroup-matrix model is similar but not identical (no explicit global-vs-shared partition in unified memory;
   SIMD-group sizes differ across M1/M2/M3 generations; AMX is CPU-side, the GPU has separate simdgroup_matrix
   instructions on M3+ only). What is the smallest-scope Metal-TL DSL that captures the necessary abstractions for fused
   attention while remaining cleanly generated by a frozen LLM? Should it parameterize over M-series generation (M1 vs
   M3) explicitly, or punt that to the lowering phase?

2. **DSL design → Phase 6/7 ADR or experiment-spec?** A "Domain-DSL as LLM rationale substrate" decision is a plausible
   ADR seed (alongside the existing MTMC ADR seed in
   [`qimeng-category-promotion.md`](../specs/qimeng-category-promotion.md)). But the DSL design space is wide enough
   that committing to an ADR before running a Metal-TL spike risks freezing decisions on no evidence. Should the
   sequence be: (a) experiment-spec for a Metal-TL spike on a single fused kernel (e.g., the Flash-MoE dequant + GEMM),
   (b) measure outcome, (c) author the DSL-as-substrate ADR _after_ the spike validates the pattern? This is the
   flash-moe / measure-don't-estimate convention applied to architectural commitments.

3. **Fine-tune vs. frozen Worker.** GPT-4o failed on TL → CuTe translation due to training-cutoff CuTe sparsity. For a
   local-only Linus, every candidate Worker (Qwen3-Coder, DeepSeek-Coder-V2-Lite) has the same problem at the Metal
   level — Metal Performance Shaders + MLX custom-kernel code is under-represented in their training corpora even more
   than CuTe is in GPT-4o's. Is the right path: (a) hosted-Claude-Sonnet during development to validate the substrate,
   then (b) Phase 6 LoRA over harvested `(TL-Sketch, TL-Code, Metal-kernel)` triples to localize the skill onto a Linus
   Worker? This is a concrete Phase 6 fine-tune target with a clear measurement gate (does the fine-tuned Worker reach
   the hosted-Claude TFLOPS within X%?).

4. **Numerical-correctness audit gap.** The paper reports TFLOPS but not numerical-equivalence audits. For Linus
   production use — particularly Phase 6d weight-streaming where a kernel generates output that flows into Dan's actual
   scientific work — this gap matters. Should every LLM-TL-generated kernel land with a mandatory regression suite
   (matched outputs vs reference within float-precision tolerance, including masked / denormal / overflow cases)? This
   is the _kernel-codegen analogue_ of the smoke-test convention (CLAUDE.md §Smoke-test gates) — never ship a kernel
   without a numerical-equivalence sample first.

5. **LLM-TL as inter-agent typed payload.** The TL Sketch / TL Code / CuTe-code triplet is a clean typed-payload chain
   across three agent roles (Sketch-author, Parameter-reasoner, Translator). Is there a Phase 2/3 case for making TL
   programs a first-class **AgentReport.payload** type in the agent spawner (DEC-0050, DEC-0051), with the corresponding
   role types pre-registered? If yes, the Metal-TL spike from Open Question 2 doubles as a spawner-architecture
   validation, exactly like the QiMeng-GEMM auto-tuning loop doubles as one in [13337-ZhouQ.md](13337-ZhouQ.md) Open
   Question 4. Same code base serves both the kernel-codegen skill and the orchestration-primitive validation.

6. **Cross-domain DSL transfer — schematic / process / biology.** Per Dan's stated framing in
   [`qimeng-category-promotion.md`](../specs/qimeng-category-promotion.md) §"Dan's stated framing", the **idea →
   reality** spine extends beyond hardware to schematics, process flowsheets, and biology design specs. QiMeng-Attention
   is evidence that DSL-as-substrate works for kernel codegen. Does the same recipe work for: (a) a TL-process DSL
   emitting HYSYS flowsheets (Sketch2Simulation-adjacent); (b) a TL-edit DSL emitting CRISPR guide-RNA + repair-template
   specs; (c) a TL-schematic DSL emitting EDA design files for fab-ready PCBs? Each of these is a Phase 7 idea → reality
   skill class candidate. Worth scoping which has the cleanest downstream-actor acceptance criterion (compiler ↔ HYSYS
   interpreter ↔ Cas9 wet-lab protocol ↔ EDA tool) — that's the gating criterion for which to spike first.
