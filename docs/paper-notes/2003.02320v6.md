# Knowledge Graphs

## TL;DR
The definitive 100+ page tutorial on knowledge graphs, written by 17 leading researchers as the academic-community consensus output of the 2018 Dagstuhl seminar that crystallized the field. It is the *textbook* for everything [modules/KnowledgeBase/](../../modules/KnowledgeBase/) is trying to be: it walks through data models (directed edge-labelled graphs, heterogeneous graphs, property graphs), querying (SPARQL, Cypher, graph patterns, regular path queries), schemata (semantic, validating, emergent), identity and context, deductive knowledge (ontologies, rules, OWL, Description Logics), inductive knowledge (graph analytics, embeddings, GNNs, rule/axiom mining), creation and enrichment (text/markup/structured-source extraction, ontology engineering), quality assessment, refinement (link prediction, entity matching, fact validation), publication (FAIR, Linked Data, access protocols, encryption, anonymisation), and a survey of real-world open and enterprise knowledge graphs (DBpedia, Wikidata, YAGO, Google KG, Amazon Product Graph, etc.). For Linus, this is **the single most KB-actionable paper in the folder** — every decision Dan will make while building [modules/KnowledgeBase/](../../modules/KnowledgeBase/) has a corresponding section in this paper.

## The problem (in plain language)
Modern AI systems — including everything Linus is being designed to do — depend on a *substrate of knowledge* that is integrated, queryable, reasonable-over, and enrichable from many heterogeneous sources. A knowledge graph is the data structure that lets you do all of this: nodes are entities (Santiago, Pearl Jam, the protein hemocyanin, Dan's PhD thesis), edges are relations (capital, headlined, contains, cites), and the result is a flexible, schema-optional graph that can grow incrementally as you add new sources without the costly re-modelling that relational databases demand.

The problem is that "build a knowledge graph" is not one decision — it is a tower of decisions. Which graph data model? Property graph, RDF, or directed edge-labelled? How do entities get globally unique identifiers? How is context (time, provenance, geography) attached to edges? Which ontology language do you use, if any — and at what point does ontological reasoning become more burden than benefit? How do you extract knowledge from text, from tables, from existing knowledge graphs? How do you assess quality? How do you refine the graph after creation? How do you publish it? Each of these has a research literature behind it, and most working data scientists encounter knowledge graphs without a coherent framework that connects all of these decisions to each other. This paper *is* that framework — written by the people who wrote much of the underlying research.

The biology analogue: this is the *Molecular Biology of the Cell* of knowledge graphs — comprehensive enough to be a reference, structured enough to be a tutorial, opinionated enough to teach a perspective.

## What they propose
**A definition first** (Section 1): "a knowledge graph is a graph of data intended to accumulate and convey knowledge of the real world, whose nodes represent entities of interest and whose edges represent relations between these entities." Schema, identity, and context are *optional* enhancements; ontologies and rules are *optional* deductive layers; embeddings, GNNs, and symbolic learning are *optional* inductive layers. The flexibility of this definition — and its alignment with how the major industrial knowledge graphs (Google, Amazon, Wikidata) actually work — is the paper's first major contribution.

**Then a structured tour through ten sections**, each of which is publishable on its own:

- **§2 Data graphs**: Directed edge-labelled graphs (what RDF uses), heterogeneous graphs, property graphs (what Neo4j uses), graph datasets (named graphs). Also: query languages — basic graph patterns, complex graph patterns with relational operators, navigational graph patterns with regular path expressions. SPARQL, Cypher, G-CORE, Gremlin all situated in one framework.

- **§3 Schema, identity, context**: Semantic schema (RDFS-style: classes, subclasses, properties, domains, ranges) for *inference*; validating schema (SHACL, ShEx) for *constraint checking*; emergent schema (quotient graphs, bisimulation) for *automatic structure discovery*. Identity: persistent identifiers, IRIs, external identity links, datatype handling, lexicalisation, existential/blank nodes. Context: direct representation, reification (RDF reification, n-ary relations, singleton properties), higher-arity (named graphs, property graph attributes, RDF*), and annotation domains (semi-rings) for reasoning *over* context (temporal, fuzzy, provenance).

- **§4 Deductive knowledge**: Ontologies and entailment. Tables 3-5 enumerate every OWL/DL feature with semantics in plain English. Reasoning via rules (Datalog-style, OWL 2 RL) vs Description Logics (ALC, SROIQ, the OWL 2 DL fragment). The fundamental tradeoff: expressivity vs decidability.

- **§5 Inductive knowledge**: The chart in Figure 23 organizes inductive techniques by representation (numeric/symbolic) × paradigm (unsupervised/self-supervised/supervised). Graph analytics (centrality, community detection, connectivity, node similarity, path finding) for unsupervised numeric. Knowledge graph embeddings (TransE through TuckER through ComplEx through ConvE — Table 8 lists 15+ models) for self-supervised numeric. Graph neural networks (RecGNNs, ConvGNNs) for supervised numeric. Rule mining (AMIE, NeuralLP, DRUM) and axiom mining (DL-Learner) for self-supervised symbolic. Each gets a working example, a formal definition, and discussion of strengths/weaknesses.

- **§6 Creation and enrichment**: Human collaboration (Wikidata-style), text sources (NER, EL, RE — full IE pipeline), markup sources (wrapper-based, web tables, deep web), structured sources (R2RML mappings from relational, GRDDL/JSON-LD from trees, SPARQL CONSTRUCT for graph-to-graph), and ontology engineering (METHONTOLOGY, DILIGENT, XD, ontology design patterns, ontology learning).

- **§7 Quality assessment**: Five dimensions (accuracy: syntactic/semantic/timeliness; coverage: completeness/representativeness; coherency: consistency/validity; succinctness: conciseness/representational-conciseness/understandability) with metrics for each.

- **§8 Refinement**: Completion (general link prediction, type-link prediction, identity-link prediction / entity matching) and correction (fact validation against external sources, inconsistency repair via minimal hitting sets).

- **§9 Publication**: FAIR principles, Linked Data principles, access protocols (dumps → node lookups → edge patterns → graph patterns, with bandwidth-vs-CPU tradeoffs), and usage control (ODRL licensing, WAC access control, encryption with cryptOntology, k-anonymity / l-diversity / k-automorphism / ε-differential privacy).

- **§10 Practice**: Open KGs (DBpedia, YAGO, Freebase, Wikidata, BabelNet, Cyc, NELL) and enterprise KGs (Google, Microsoft Bing, Amazon, eBay, Airbnb, Uber, Facebook, LinkedIn, Pinterest, Bloomberg, Thomson Reuters, IBM, Bosch, AstraZeneca).

- **§B Formal definitions**: 50+ pages of appendix with formal semantics for everything. This is where the paper transitions from tutorial to handbook.

## Key results
This is not a results-driven paper, so "key results" maps better to "load-bearing organising principles":

- **The deductive/inductive split** is the cleanest taxonomy in the literature for knowledge-graph reasoning, and it directly governs the architecture choices every KB project has to make.
- **The schema typology** — semantic vs validating vs emergent — clarifies that schemata serve at least three different *purposes* and shouldn't be conflated. SHACL is not RDFS is not a quotient graph.
- **The 17-co-author consensus on the definition of "knowledge graph"** ends years of confusion in the literature. (Appendix A walks through that history — useful if you want to understand why other papers' definitions are weirdly narrow.)
- **The unification of 15+ knowledge-graph embedding models** in Table 8 with a common scoring-function framework is, by itself, more useful than most of the original embedding papers.
- **The framework for context (semi-ring annotation domains)** is the right abstraction for representing time, provenance, geography, and confidence on edges in a unified way — and it's already standardised through W3C's PROV and Time ontologies.

Scale: 100+ pages of body, 50+ pages of formal appendix, ~600 references. This is the field's reference document.

## What's reusable in Linus
**Direct, foundational, this-is-the-spec-for-KnowledgeBase**:

The paper *is* the design document for a well-built knowledge graph. Almost every section maps to a concrete decision Dan needs to make in [modules/KnowledgeBase/](../../modules/KnowledgeBase/):

1. **Data model**: Use RDF (directed edge-labelled graphs) for interoperability with the existing tooling ecosystem (rdflib, blazegraph, virtuoso, Apache Jena, SPARQL endpoints). The paper's §2 makes the case clearly. Property graphs are tempting for richer per-edge metadata but cut you off from the Semantic Web stack.

2. **Identity**: HTTP IRIs (per Linked Data principle 2) so that nodes can be dereferenced over the Web. Use PURL services for persistence. Distinguish information-resource IRIs (the document) from non-information-resource IRIs (the entity).

3. **Schema**: Combine semantic schema (RDFS for inferred subclass/subproperty hierarchies on Dan's domain ontologies) + validating schema (SHACL for ingest-time quality checks on incoming papers). Per §3.1, these aren't redundant — they serve different purposes.

4. **Context**: Reify edges (RDF*-style or with named graphs) to attach provenance, source paper, ingest timestamp, and confidence. Section 3.3 shows the alternatives and their tradeoffs.

5. **Querying**: SPARQL 1.1 with property paths is plenty for most KB queries. The paper's §2.2 is the right reference for query design.

6. **Deductive layer**: Lightweight RDFS-level reasoning is almost certainly enough; OWL 2 DL is overkill and computationally expensive. §4.3 makes this case.

7. **Inductive layer**: For embeddings, use modern techniques (TuckER, ComplEx, or sentence-transformer–based dense embeddings as in [2408.08073v2](2408.08073v2.md)). For link prediction and entity matching across paper sources, the techniques in §8 (with their citation graphs to the underlying papers) provide a complete starting set.

8. **Creation**: §6 is a multi-month research agenda for KB ingestion: NER + EL + RE for unstructured text (Dan's papers), wrapper-based extraction for HTML (web-table papers), R2RML for any relational sources, SPARQL CONSTRUCT for ingesting from external KGs (Wikidata for author/affiliation info, ORCID for author IDs).

9. **Quality**: §7's five dimensions form a natural KB quality dashboard. Worth implementing as a standing report.

10. **Publication**: Even for a private KB, the protocols hierarchy in §9.2 (dumps → lookups → edge patterns → graph patterns) is the right way to think about *internal* APIs that Linus Workers consume.

**Direct (synthesis with the embeddings paper [2408.08073v2](2408.08073v2.md))**: The combination of this paper's §5.2 (knowledge-graph embeddings — entity-relation pairs) with the Stankevičius embeddings paper (text-based dense embeddings of nodes from their textual labels and descriptions) gives you the two complementary embedding views every modern KB needs: structural embeddings from the graph, and semantic embeddings from the text. They serve different downstream tasks (link prediction vs similarity search) and ideally co-exist.

**Direct (Maestro/Worker as KG-curation collaboration)**: Most of the curation tasks in §6 (entity linking, relation extraction, ontology mapping) are exactly the kind of well-specified work that should be delegated to a Linus Worker, with the Maestro reviewing the schema and edge cases. The KB grows by *running this loop*. The paper's §6.1 explicitly notes that human-in-the-loop curation is the most quality-effective creation pattern.

**Indirect (publishing/sharing)**: Dan may eventually want to publish parts of his KB (the bibliographic graph of his papers, e.g.) as Linked Data so that others can reuse it. §9 is the playbook.

## What's NOT applicable
- **Enterprise KG scale (§10.2)**: The Wikidata, Google, Bloomberg, Bosch examples involve teams and infrastructure beyond a single-person project. Useful for inspiration; don't over-engineer toward.
- **OWL 2 DL reasoning**: Tableaux-based DL reasoners are complex software and rarely earn their keep on a personal KB. Stick to RDFS-level inference + custom rules.
- **ε-differential privacy / k-automorphism (§9.3)**: These matter for KGs with personal data being publicly published. Linus' KB is private and contains scientific content — the threat model doesn't justify the complexity.
- **Federated SPARQL queries (§9.2)**: Useful eventually for joining your KB with Wikidata at query time. Not Phase 1.
- **Some of the embedding-model survey content (§5.2)**: The paper covers TransE through ConvE comprehensively, but BitNet-era and modern dense-encoder embeddings (E5, BGE, etc.) are post-publication. The principles transfer; the specific recommendations need updating.

## Open questions for Dan
1. **Choice of graph data model for KnowledgeBase**: RDF (Semantic Web stack, full ecosystem) or property graph (richer attributes, more performant for some workloads)? §2 lays out the tradeoffs. My read: RDF wins for a KB whose primary purpose is knowledge integration and SPARQL queryability; property graph wins if you imagine KB primarily as a graph database for analytics. Worth deciding *now* before the schema design hardens, because converting later is painful.
2. **Schema-first vs schema-emergent**: Build a top-down ontology (using ontology-design patterns from §6.5) before ingesting content? Or ingest first, then mine the emergent schema (§3.1.3) and formalize? My recommendation: schema-first for the *spine* (papers, authors, concepts, citations) using existing standards (BIBO, SKOS, FOAF), schema-emergent for the long tail.
3. **Entailment regime**: RDFS-only (sub-class, sub-property, domain, range) or OWL 2 RL (richer, but heavier)? §4.3 makes the case; my read is RDFS-only is plenty for KB v1, with OWL 2 RL revisited if/when Phase 3 reasoning needs justify it.
4. **Worth a `docs/specs/kb-architecture.md`** that walks through this paper section by section and records the design choice + rationale for each? It would make the KB design decisions auditable and would pay back across the project lifetime.
