# QiMeng-SALV: Signal-Aware Learning for Verilog Code Generation

## TL;DR

QiMeng-SALV (Signal-Aware Learning for Verilog) shifts reinforcement learning for Verilog code generation from
**module-level** rewards to **signal-level** rewards. The key insight: Verilog describes structural interconnections
between hardware gates and wires, so different output signals are largely independent — a module that fails as a whole
can still contain individually correct signal implementations. SALV uses (1) random-stimulus simulation to verify which
output signals match the reference module per-signal, (2) AST analysis to extract the precise code segment driving each
signal, and (3) a modified DPO loss that computes token probabilities only over the contrast signals' code segments
(both in the preferred and dispreferred samples). A 7B Qwen2.5-Coder fine-tuned with this recipe matches DeepSeek-V3
(671B) on RTLLM v1.1 (62.6% pass@1, 75.1% pass@5) and substantially outperforms CodeV trained on the identical dataset.

## The problem (in plain language)

LLM-based Verilog code generation pipelines typically combine SFT on Verilog corpora with an RL post-training stage
(PPO, GRPO, DPO) using a functional-correctness reward. Two structural problems block this from working well:

1. **The reward signal is sparse.** A Verilog module either passes the testbench or it doesn't. Modules that are
   _almost_ correct — the address-decoder logic is right but the data-path register is off-by-one — provide no learning
   signal under module-level rewards. RL training collapses because too few samples earn positive reward.
2. **SFT data quality is heterogeneous.** Public Verilog code crawled from GitHub is uneven; the resulting SFT model
   produces partially-correct modules far more often than wholly-correct ones. The RL stage then has very few
   wholly-correct preferred samples to anchor preference optimization against, and the available preferred samples
   themselves contain incorrect signals that get reinforced as a side effect.

Prior RL approaches for Verilog (RTLCoder, CodeV, Origen, VeriSeek, VeriPrefer) all operate at module granularity. The
closest precedent, VeriPrefer, uses module-level preference rewards and addresses some of the bias issues but still
treats the module as the atomic unit of preference. The question SALV attacks: can we exploit Verilog's _structural
parallelism_ — the fact that distinct output signals are described by largely independent code blocks — to derive
fine-grained, signal-level functional rewards from partially-correct modules, expanding the effective training set and
producing a denser learning signal?

## What they propose

SALV's three-stage pipeline transforms the Verilog RL problem from "module pass/fail" to "per-signal pass/fail with
code-segment isolation":

1. **Signal-aware Verification.** For each prompt, the SFT model emits k candidate modules. Yosys analyzes the
   reference module's interface to auto-generate N sets of random input stimuli. The candidate and reference modules
   are simulated against the same stimuli; output signals are compared one-by-one. A signal is "correct" iff all N
   stimuli produce matching output. This yields, for each module, a per-signal correctness vector. A preference pair
   `(y_w, y_l, c)` is constructed when there exists a contrast signal `c` such that `c` is correct in `y_w` and
   incorrect in `y_l`. Crucially, the entire module need not be correct — partially-correct modules contribute as long
   as some signal is verifiably right somewhere. The implementation uses iverilog for simulation.

2. **Signal-aware Code Extraction.** Given a module's source and a target output signal, parse it into an AST (via
   Yosys), build the **signal dependency graph** (each output node's dependents — intermediate wires/registers, case
   selectors, always-block conditions), do a backward traversal from the output signal's leaf node, and retain the AST
   subtree's covered code lines. This produces `S^c_w` (the code segment driving contrast signal `c` in the preferred
   sample) and `S^c_l` (the corresponding segment in the dispreferred sample). The decomposition is precisely possible
   because Verilog is a description of static interconnect rather than sequential execution: signal-implementing code
   blocks are largely self-contained.

3. **Signal-aware DPO.** Standard DPO computes token probabilities over _all_ tokens in `y_w` and `y_l`. SALV's
   modified loss restricts the token-probability sum to tokens in `S^c_w` and `S^c_l` only:

   ```
   L = -E[log σ(β · Σ_{y_t ∈ S^c_w} log π_θ(y_t|y_{<t},x)/π_ref(y_t|y_{<t},x)
                - β · Σ_{y_t ∈ S^c_l} log π_θ(y_t|y_{<t},x)/π_ref(y_t|y_{<t},x))]
   ```

   This selectively reinforces correct signal-implementing tokens in the preferred sample and selectively penalizes
   incorrect signal-implementing tokens in the dispreferred sample, while ignoring all other tokens. The model never
   gets credited for correct-by-accident tokens elsewhere in a partially-correct preferred module, and never gets
   penalized for correct-by-accident tokens elsewhere in a partially-correct dispreferred module. LLaMA-Factory was
   customized to support the selective-loss variant.

The base model is Qwen2.5-Coder-Instruct 7B; SFT runs full-parameter for 2 epochs on 135k cleaned CodeV samples;
signal-aware DPO runs ~7000 LoRA steps with a 5e-6 learning rate. Random stimuli generation, simulation (iverilog),
and AST parsing (Yosys) are all fast — 0.65 ms simulation + 1.59 ms AST per sample with 60-core parallelism.

## Key results

- **VerilogEval 1.0 (open-source SOTA).** 81.4% pass@1 / 90.8% pass@10 on Machine; 60.4% pass@1 / 71.2% pass@10 on
  Human. Comparable to GPT-4o on Human, surpasses DeepSeek-V3 and GPT-4o on Machine.
- **VerilogEval 2.0.** 57.1% (T=0) and 62.2% (T=0.8) on Specification; 56.2% / 58.8% on Completion. New open-source
  SOTA at the 7B scale.
- **RTLLM v1.1.** 62.6% pass@1 / 75.1% pass@5 / 81.1% pass@10. Matches DeepSeek-V3 (671B) on pass@1 with two orders of
  magnitude fewer parameters; surpasses DeepSeek-V3's 72.4% pass@10.
- **RTLLM v2.0.** 62.0% pass@1 / 71.7% pass@5 / 76.0% pass@10. Beats GPT-4o's 56.5% pass@1 by a wide margin.
- **vs CodeV (same base, same data).** ~59.7% improvement in pass@1 on RTLLM v1.1 vs CodeV-Qwen2.5; 27.4% (T=0) and
  50.2% (T=0.8) improvement on VerilogEval2.0 Specification. The signal-level-vs-module-level recipe is the
  responsible variable; everything else is matched.
- **Ablations.** (a) Training only on _complete_-correct modules (the standard DPO setting) is _better_ than training
  only on partial-correct modules without filtering. Mixing partial-correct without filtering _hurts_ vs.
  complete-only — the noise from incorrect signals dominates. With SALV's signal filtering, however, both partial-only
  and mixed-data settings now beat complete-only, and the full mix beats any subset (62.6% vs 57.4% pass@1 on RTLLM
  v1.1). (b) Stage ablation: base Qwen2.5 30.1% → +SFT 48.3% → +SALV-DPO 62.6% pass@1. Each stage adds 14–18 points;
  SALV-DPO contributes the larger marginal gain. (c) Dataset scaling: pass@1/5/10 all increase monotonically with the
  preference-dataset size from 50k to 350k pairs; no plateau visible.
- **Throughput.** Per-sample 0.65 ms iverilog simulation + 1.59 ms Yosys AST parsing — overhead is negligible vs.
  forward/backward training cost.

## What's reusable in Linus

SALV is the cluster's clearest demonstration of the **idea → reality** spine specifically applied to RL training: the
LLM produces Verilog source; simulation + AST analysis (downstream non-LLM actors) extract a fine-grained reward that
the next training step accepts. No human in the loop after the dataset is curated; the reward derivation is fully
automated and runs in milliseconds per sample. For Linus, this is the cluster's most directly useful pattern for
**Worker self-improvement** because it generalizes beyond Verilog: any task whose output decomposes into independent
sub-artifacts (each with its own oracle) can use the same recipe.

**Phase 6 — signal-aware DPO as a general Worker fine-tuning substrate (DEC seed).** SALV extends naturally to any
domain where (a) outputs are structurally decomposable, (b) per-substructure correctness oracles exist, and (c) SFT
quality is heterogeneous so fully-correct samples are rare. Examples for Linus:

- _Code generation generally._ Functions are independent; per-function pytest oracles exist; partial-correct files are
  the norm. Signal-aware DPO ↔ "function-aware DPO": preferred/dispreferred sliced to function-body token spans, with
  per-function pass/fail from pytest. The CodeV-R1 paper (`2505.24183v5.md`, paired-agent) covers RLVR with module-
  level rewards; SALV's signal-level extension is the obvious next refinement for code domains. _Seed: DEC-NNNN
  (signal-aware DPO as Worker fine-tuning substrate)._
- _Bioinformatics pipeline output._ A multi-section analysis report (BLAST hits, alignment QC, taxonomic
  classification, abundance table) is structurally decomposable; per-section oracles can be defined. SALV-style would
  reward correct sections and penalize incorrect sections without conflating.
- _Typed-structured-prediction outputs (CLAUDE.md S25)._ The
  `{gene, predicted_function, confidence, evidence, rationale}` shape is _itself_ a decomposable structure — a model
  can be right about `predicted_function` and wrong about `evidence`. Per-field DPO is a direct SALV analog.

The recipe maps to: (a) the LLM produces a structurally-decomposable artifact; (b) parse the artifact into its
substructures (AST, JSON tree, RST sections, etc.); (c) per-substructure oracle votes correct/incorrect; (d) DPO loss
restricted to substructure-spanning token slices using the pair-with-contrast-substructure construction. What must be
true for Linus to replicate: a parser that produces structurally-aligned token spans (the AST→token-span step is the
non-trivial part), per-substructure oracles, and a customized loss that consumes the token-span masks. The LLaMA-
Factory fork in the repo demonstrates the feasibility of the loss customization on a public training framework.

**Phase 6/7 — Verilog/RTL skill class proper.** If Linus ever needs to generate hardware-description code (e.g., for a
Linus-flavored Apple-Silicon-coupled accelerator design experiment), SALV's recipe is the closest open precedent. The
released model (`TabCanNotTab/SALV-Qwen2.5-Coder-7B-Instruct` on HuggingFace) is a 7B model that fits easily in 32 GB
on M1 Max under MLX or via Ollama (quantized to Q4_K_M would be ~4 GB; bf16 ~14 GB). The released SALV-dataset
(`TabCanNotTab/SALV-dataset` on HuggingFace) plus the open recipe means a full reproduction on Apple Silicon is
tractable. Linus would _not_ generally use this skill, but it's a valuable skill-class probe for the
**idea → reality** spine: how does Linus learn to produce artifacts a non-LLM actor accepts? SALV is the smallest
realistic case study with a published recipe.

**Phase 2/3 — Signal-aware verification as a generic test-generation pattern.** SALV's "use the reference's interface
to auto-generate random stimuli, simulate both, compare per-signal" is a clean specification of property-based
testing applied to LLM-emitted code. The pattern transfers to any executable LLM output: the reference is
human-written, the candidate is LLM-emitted, the differential test is "do they agree on N random inputs?" For Linus's
agent-spawner (DEC-0050) and AgentReport schema (DEC-0051), this is the same shape as a typed correctness-oracle
plugged into the agent loop — generalizable as a Phase 2/3 orchestration primitive.

**Phase 7 — Worker self-improvement substrate (RLVR successor).** Per the CodeV-R1 paired note, the QiMeng family's
RL-with-verifiable-rewards thread is a candidate Linus convention for letting Workers self-improve on pure-Python
tooling tasks where pytest oracles exist. SALV's signal-level refinement says: don't stop at module/function-level
rewards; if your output decomposes further, push the granularity down. _Seed: DEC-NNNN (Worker self-improvement uses
signal-/substructure-aware preference optimization, not module/function-level)._

**Phase 6 — partial-correct sample filtering as a hyper-parameter knob.** The ablation table is striking: mixing
partial-correct samples without filtering hurts the model; with SALV's filtering, the same mixing helps it. For
Linus's fine-tuning lane (Phase 6), this is a concrete convention: when curating SFT/DPO datasets from heterogeneous
sources, signal-/substructure-level filtering should be the default for partially-correct samples, and complete-only
filtering should be the fallback (not the default).

## What's NOT applicable / hype filter

The "matches DeepSeek-V3 671B with 7B parameters" headline is a domain-specific claim — Verilog generation, narrow
benchmark suite (VerilogEval, RTLLM), specific SFT base. It's not evidence that 7B SALV would match DeepSeek-V3 on
general-purpose code or anything outside the trained distribution. The training data is 135k cleaned CodeV samples; the
benchmarks (RTLLM, VerilogEval) draw from a similar circuit-design distribution. Out-of-distribution Verilog (analog
mixed-signal, asynchronous-design idioms) is unaddressed.

The signal-decomposition assumption hinges on Verilog's structural-interconnect semantics. For sequential-execution
languages (Python, C++, Rust), per-output-signal decomposition doesn't apply directly; the analogous decomposition is
function-level or AST-subtree-level, but functions in sequential code _do_ have inter-function dependencies (mutable
state, side effects) that Verilog signals don't have. The "largely independent" assumption that lets SALV slice the
DPO loss is much stronger for HDLs than for general-purpose code; the function-aware-DPO transfer is plausible but
needs measurement.

The reward source is **differential simulation against a reference**. SALV requires that the training data have
reference modules paired with each prompt. For Linus's general use, this means SALV-style fine-tuning needs a
reference dataset structure: paired (spec, reference-implementation) examples that the simulation compares against.
Pure RLHF (no reference, just preferences) doesn't fit. The reference dependence also bounds the diversity of behaviors
the recipe can teach — the model learns to match reference idioms, not to invent them.

The Yosys + iverilog dependency is **Linux-native**. macOS support exists (Yosys via Homebrew, iverilog via Homebrew)
but is less battle-tested. For an Apple-Silicon-native reproduction, the simulation-throughput claim
(0.65 ms/sample with 60-core parallelism) is not directly transferable; the M1 Max's 10 CPU cores would need ~6×
longer wall-clock per epoch's worth of stimuli, which is still tractable but a meaningful constraint.

The 350k-pair preference dataset is large. Generating it required (a) running the SFT model k times per prompt to
sample candidates, (b) iverilog-simulating each, (c) Yosys-AST-parsing each. For a 135k-prompt corpus this is
comfortably within the paper's resources; for a hypothetical Dan-corpus fine-tune, the simulation/parse step assumes
a meaningful prompt corpus exists with reference implementations. This is fine for narrow domains where Dan has
reference code; less straightforward for novel domains.

LLaMA-Factory customization is required for the selective DPO loss. The fork is published in this repo, but the loss
customization is an integration cost rather than a library install. Linus's Phase 6 fine-tuning lane would need to
either fork LLaMA-Factory or implement the selective loss from scratch in MLX-LM-FT. The latter is plausible (the loss
masking is conceptually simple) but is yet-another implementation step.

The 7B model release is **bf16**. For local M1 Max inference at low memory pressure, quantization to Q4_K_M or Q5_K_M
via Ollama/llama.cpp is the natural step; quantization-induced regression on Verilog-specific outputs is unmeasured.
This is the same caveat as any 7B-model deployment; not specific to SALV.

The paper does not address **out-of-vocabulary Verilog idioms** (e.g., `generate` blocks, parameterized modules,
SystemVerilog assertions). The signal dependency graph extraction is described for the case of straightforward
combinational + always-block Verilog; complex generate/parameterize patterns may not parse cleanly into signal-aligned
code segments. Worth measuring on a wider Verilog corpus before assuming the recipe generalizes within HDL itself.

## Open questions for Dan

1. **Function-aware DPO transfer to general-purpose code.** SALV's recipe ports obviously in shape to Python/Rust:
   parse to AST, identify functions, per-function pytest oracle, function-aware DPO loss restricted to function-body
   token spans. The non-trivial assumption is that functions in sequential code are "largely independent" the way
   Verilog signals are. Mutable state, side effects, and cross-function dependencies could mean the SALV trick reduces
   noise less or even introduces it. Worth a Phase 6 spike: take a Python code dataset (e.g., CodeContests, MBPP), run
   SFT, then run function-aware vs. file-aware DPO and compare. Smallest viable case might be `assert`-decorated
   competitive-programming problems where the contract is fully local.

2. **Signal-aware DPO for typed structured prediction outputs.** Per CLAUDE.md S25, biology/skill outputs are typed
   structured predictions like `{gene, predicted_function, confidence, evidence, rationale}`. SALV's recipe says: per-
   field oracles, per-field DPO. Is this the right way to fine-tune the typed-structured-prediction skill class — i.e.,
   should Linus's Phase 6 Worker fine-tunes for biology skills use SALV-style per-field DPO rather than whole-record
   DPO? If yes, this becomes a CLAUDE.md convention. If the per-field oracle is the bottleneck (rare in biology where
   ground-truth is itself partial), the recipe degrades to whole-record DPO.

3. **Worker self-improvement loop convergence.** SALV achieves signal-level rewards via a _fixed_ reference dataset.
   For Worker self-improvement (Linus's Workers improving themselves on pytest-oracle tasks), the reference is the
   pytest suite, not a paired reference implementation. Does the recipe still work when "reference" is replaced by
   "pass/fail oracle without a paired correct example"? The DPO contrast pair construction — `(y_w, y_l, c)` where
   `c` is correct in `y_w` and incorrect in `y_l` — requires both samples to exist. Self-generated rollouts produce
   plenty of failed samples, but the preferred sample needs to actually exhibit a correct sub-structure. For
   sufficiently capable Workers, this is fine; for early-stage Workers that fail wholesale on most tasks, signal-aware
   DPO may have nothing to learn from. Worth modeling pre-fine-tune vs post-fine-tune dynamics before committing
   architecture.

4. **Apple Silicon reproduction cost.** The released model is 7B bf16; the dataset is 350k pairs. Reproducing the
   training run on M1 Max requires (a) iverilog/Yosys throughput, (b) ~7000 LoRA steps, (c) the LLaMA-Factory
   selective-loss customization. The simulation throughput (0.65 ms/sample at 60-core parallelism in the paper) drops
   to ~3.9 ms/sample with M1 Max's 10 cores — still negligible. The LoRA fine-tune itself fits in 32 GB unified memory
   for a 7B base. This is a tractable, single-MacBook reproduction; the main cost is the LLaMA-Factory port to MLX-LM-
   FT (or running PyTorch + MPS, which the upstream framework supports but at unmeasured throughput). Should this be
   a Phase 6/7 reproduction target — i.e., the first published RL-fine-tune Linus reproduces end-to-end on Apple
   Silicon as a methodology validation? The benefits: a complete RLVR-via-DPO recipe that Linus can adapt for general
   code; a published model on a published dataset; a non-trivial but bounded scope. _Seed: DEC-NNNN
   (SALV-on-Apple-Silicon Phase 6/7 reproduction target)._

5. **Selective-loss generalization beyond DPO.** The loss customization in SALV is conceptually "compute the DPO term
   only over a token-span mask." The same masking pattern would apply to GRPO, PPO, or SFT objectives — wherever per-
   token-span correctness can be derived, the objective can be restricted to that span. Should Linus's training-recipe
   library carry "selective-objective" as a first-class abstraction (token-mask + objective-fn → masked-objective-fn),
   so SFT/DPO/GRPO/PPO all consume the same mask source? This is a Phase 6+ Linus-internal convention question; SALV
   is the existence proof but the abstraction is broader.

6. **Heterogeneous-quality data filtering as a default convention.** The ablation that mixing partial-correct without
   filtering hurts but with filtering helps is empirically striking and matches Dan's flash-moe instinct (DEC-0027):
   measurement beats intuition. Should "always filter heterogeneous-quality data at the substructure level when an
   oracle exists" become a CLAUDE.md convention for the Phase 6 fine-tuning lane? The cost is the substructure-oracle
   plumbing; the benefit is reliably positive ROI from partially-correct samples that would otherwise need to be
   discarded. For domains where substructure oracles are easy (Verilog signals via simulation, Python functions via
   pytest, typed-structured-prediction fields via per-field validators), this seems like a no-brainer; for domains
   where they're hard (free-form prose), it's not applicable.

7. **MTMC ↔ SALV synthesis question for the cluster.** The QiMeng-Kernel macro-thinking/micro-coding paradigm
   (`2511.20100v1.md`) is a planner/coder split with RL on the coder. SALV is signal-aware DPO without the explicit
   planner. The cluster synthesis (g12-llm-hardware-design.md, pending) needs to articulate whether MTMC and SALV are
   complementary (MTMC = decomposition, SALV = per-decomposition-element preference) or alternative (MTMC implicitly
   carries the signal-level decomposition in the planner's output schema). The cleanest reading is complementary:
   MTMC's macro outputs are the decomposition input to SALV's per-substructure DPO. If that's right, the synthesis
   should foreground "MTMC + SALV as a Worker self-improvement stack" as the QiMeng family's contribution to Linus
   Phase 6/7.
