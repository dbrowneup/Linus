# A Practical Guide for Evaluating LLMs and LLM-Reliant Systems

## TL;DR

Three Google practitioners write the field guide they wish existed: how to evaluate an LLM system that real users hit,
not a leaderboard. Their framework rests on three pillars — Datasets (the "5 D's": Defined scope, Demonstrative of
production, Diverse, Decontaminated, Dynamic), Metrics (no silver bullet — combine term-overlap, semantic similarity,
NLI/entailment, autoraters; perplexity is a trap), and Methodology (handle non-determinism via self-consistency and
noise-bound baselines, probe for hallucinations with fictitious entities, measure undesirable non-response). The
recurring theme is that benchmarks decay, single metrics mislead, and evaluation must mature in lockstep with the system
it measures.

## The problem (in plain language)

An LLM-reliant system has, in principle, infinite inputs and infinite outputs. Multi-turn agent workflows compound
errors at every step. Tiny prompt perturbations swing responses; the same prompt twice can return different answers;
grounding pipelines are flaky. In this regime, picking metrics from a paper and running a public benchmark gives you a
number that is easy to compute and almost useless for deciding whether to ship.

The authors argue practitioners chronically pick from a "plethora of de-facto evaluation techniques" with no consistent
strategy tied to actual deployment goals. The result is a familiar pathology: glowing offline scores, miserable user
satisfaction. Their paper exists to give practitioners a checklist — what to do at each of the three pillars, what
mistakes to avoid, what hypothesis test to use when an autorater says response A beat response B.

It is explicitly a _practical_ guide. There are no novel benchmarks, no new metric, no model. The contribution is a
framework you can lift wholesale into a project plan.

## What they propose

The framework, summarized in a flowchart they label Figure 1, decomposes evaluation design into three pillars connected
by an iterative loop with "Define Scope and Objectives" upstream and "Analysis and Iteration" downstream.

**Datasets.** A good evaluation dataset satisfies the _5 D's_. It has a **Defined Scope** matched to specific tasks. It
is **Demonstrative of Production Usage** — its prompts mimic what real users send. It is **Diverse**, capturing the
variability of the problem space (measured via clustering, n-gram or embedding similarity). It is **Decontaminated**
from training data (checked via continuation testing, perplexity inspection, hash/substring matching when training data
is accessible). And it is **Dynamic** — versioned, audited, refreshed.

Datasets come from three complementary sources: public benchmarks (cheap, often contaminated, rarely use-case specific),
human-annotated _golden_ datasets (expensive, slow, but irreplaceable for specialist domains), and synthetically
generated _silver_ datasets (scalable but require oversight; built via distillation from frontier models, persona
prompting, temperature/top-p variation, or scraping pipelines with quality filters). Metadata is first-class: tags for
diversity tracking, grounding information for autoraters, and "expected information" — the specific terms or steps the
response must contain. The recommended pattern is to bootstrap from benchmarks or human annotations into a silver set
and iteratively turn it golden.

A useful concrete number: the standard sample-size formula `n = z² · m̂(1 − m̂)/ε²` gives ~246 evaluation items for 95%
confidence at ε=5% on a metric expected at 80%. This is the right back-of-envelope for sizing `dan_tasks/`.

**Metrics.** No metric is a silver bullet, so you carry a balanced scorecard. Term-overlap metrics (ROUGE for
summarization, BLEU for translation, plus their many variants — SacreBLEU, NIST, METEOR) are simple and
language-agnostic but reward syntax over substance. The paper's Apollo 11 worked example is sharp: a factually wrong
"Mars" candidate scores high on ROUGE because most n-grams match the golden Moon-landing reference, while a correct
paraphrase scores lower because it uses synonyms ("moon rocks" vs. "lunar samples"). Semantic-similarity metrics —
embeddings plus cosine — fix the paraphrase problem but say nothing about factuality. NLI/entailment metrics (premise
vs. hypothesis classified entailment / neutral / contradiction) do address factuality but inherit the limitations of
whatever entailment model you use. Autoraters — LLMs scoring other LLMs — handle nuance and can do point-wise or
side-by-side comparison, but bring biases (verbosity, position, self-enhancement, list-preference, stylistic) and cost.
Side-by-side comparisons need real hypothesis testing: McNemar for binary, paired t-test for continuous, Wilcoxon
signed-rank for Likert. **Perplexity is dangerous as a quality proxy** — a fluent nonsense response can score low PPL
because the model's own probabilities favor it.

**Methodology.** Three big problems to confront. First, _non-determinism_: stochastic sampling, chained calls, and
grounding inconsistency. Address it via self-consistency (sample n times, take the mode — quality up, cost up linearly)
and via direct estimation of system noise (run n≥10 times on fixed inputs, establish an error band, only treat metric
deltas outside that band as real). Second, _prompt sensitivity_: case-swapping, whitespace perturbation, character
swapping, LLM-rewriting — measure how each test set's metrics shift, isolate brittle subgroups, identify individual
high-variance items as RAG/upstream-component bug indicators. Third, _complex failure modes_. Measure hallucination
directly by probing with **fictitious entities** ("Tell me about CVE-2037-1234567" — a future-dated CVE that cannot
exist). Measure undesirable _non-response_ by asking questions you know the model can answer (with grounding) and
counting "I don't know" replies. The two failure modes are in tension: hallucination-suppression instructions raise
non-response rates.

For composed/agentic systems they push _unit-style_ component evaluation: don't only evaluate end-to-end, evaluate each
LLM call and grounding API independently. Use traditional retrieval metrics (P, R, F_β@k, MRR, NDCG) to score grounding
modules. Run with-and-without-grounding ablations to attribute correctness to context vs. prior knowledge.

## Key results

Because this is a methodology paper, the "results" are recommendations and worked examples rather than numbers. The
standout artifacts to lift verbatim:

- **The 5 D's** as a dataset checklist. Print it on the wall above `benchmarks/dan_tasks/`.
- **The Apollo 11 example table** demonstrating how four candidate responses (factually-wrong, correct-paraphrase,
  incomplete, irrelevant) get differently ranked by ROUGE, embeddings, NLI, and autoraters — making the "use multiple
  metrics" argument visceral.
- **The hypothesis-test selection table** for autorater A/B comparisons (McNemar / paired t / Wilcoxon).
- **The fictitious-entity hallucination probe** — a cheap, automatable test that doesn't need ground truth.
- **The non-response measurement protocol** — paired with hallucination measurement, makes the prompt-engineering
  trade-off measurable instead of vibes-based.
- **The sample-size formula** — anchor for sizing evaluation sets.

The unifying meta-result: evaluation is not a one-off validation step, it's a continuous loop where the eval suite
matures with the system. Benchmarks that stop changing rot.

## What's reusable in Linus

**Direct, Phase 1 (Recon & Baselines), into `benchmarks/dan_tasks/`.** This paper is essentially the design document for
that directory. Concretely:

1. Tag every `dan_tasks/` item against the 5 D's. Defined Scope = which Linus capability it targets (code gen,
   scientific Q&A, paper synthesis, tool use). Demonstrative = the prompt is something Dan would actually send. Diverse
   = topic/difficulty/format spread, audited via embedding clustering against the existing KnowledgeBase embedding
   model. Decontaminated = collected from Dan's own work, post-cutoff or never-public — a structural advantage of a
   private benchmark suite. Dynamic = versioned in git, with a "task added on YYYY-MM" header, and an explicit policy to
   refresh quarterly.
2. Size each task category to ~250 items (the paper's ε=5% / 80% / 95% number). For Phase 1 smoke-tests, the existing
   10–50 item gate stays — that's a sanity check, not a measurement.
3. Curate metadata per item: tags, grounding context (for KB-RAG items), expected information (specific terms or facts
   the response must contain). The "expected information" field is exactly the schema needed to compute
   keyword-selection metrics later.

**Direct, Phase 1, metric scorecard.** When evaluating Worker candidates (Qwen2.5-Coder, Mistral-7B, BitNet, future
Linus), run a _balanced scorecard_, not a single metric. Suggested minimum: ROUGE-L (or keyword-overlap) for summary
tasks, embedding-cosine for semantic similarity, an NLI score for factuality on Q&A, and an autorater (using hosted
Claude as the judge during development, a local Worker as the judge once one is good enough) for nuanced quality. Track
all four; never average them into a single number.

**Direct, Phase 2 (Linus MVP), orchestration layer.** The methodology section maps onto the orchestration layer's
audit-log design. Every Worker call should be logged with enough detail that the non-determinism estimation procedure
(run-n-times-on-fixed-inputs, establish error band) can be retrofitted from logs without re-running. The audit-log
schema needs: prompt hash, response, sampling parameters, latency, model+version. This is the substrate for a
continuous-eval loop.

**Direct, Phase 3 (Knowledge & Parallel Agents).** RAG-specific. Evaluate KnowledgeBase grounding using P/R/F_β@k, MRR,
NDCG on a held-out query set. Run paired with-grounding / without-grounding evaluations on the same prompts to confirm
that KB retrieval is actually contributing — the paper warns this is the only reliable way to detect when a model is
silently relying on prior knowledge instead of grounding. Sensitivity testing (whitespace/case noise injection) on
KB-RAG queries will surface brittle entries.

**Direct, all phases, hallucination + non-response gates.** Add two tests to the CI gate: (1) a fictitious-entity probe
set ("explain protein XYZQ-9000", "what does the gene LINUS-FAKE-1 do?") with a heuristic "I don't know" classifier to
compute hallucination rate; (2) a known-answerable set to compute non-response rate. Track both as paired metrics — the
prompt-engineering knob that improves one will degrade the other, and you want the trade-off visible.

**Indirect, Phase 6 (Fine-Tuning).** The decontamination discipline matters most here. Any fine-tuning corpus from
KnowledgeBase needs explicit separation from the eval set, with hash-based contamination checks before training.
Continuation-testing the fine-tuned Linus model against `dan_tasks/` would catch leakage early.

**Indirect, Phase 7 (Skills & Autonomy).** Behavioral / robustness testing (prompt perturbation) becomes the
autonomy-graduation gate. A skill graduates from Tier-N to Tier-N+1 only after its eval set survives noise injection
without metric collapse. Concrete and falsifiable.

## What's NOT applicable

The paper assumes a Google-scale team with budget for human annotators, third-party labeling platforms (Labelbox,
SageMaker Ground Truth), UXR surveys, and a panel of SMEs. Linus is one user. The "golden" annotation pipeline collapses
to "Dan checks the output and writes the gold standard," which is fine but means golden-set growth is rate-limited by
Dan's attention — a Maestro-budget item. Synthetic silver datasets generated by hosted Claude become more important,
traded off against the cost-of-Maestro-tokens caveat.

The framework treats autoraters as a heavyweight component, sometimes a separate fine-tuned model. For Linus the natural
autorater during development _is_ hosted Claude (Maestro), used sparingly. Long term, a local Worker becomes the
autorater — but that requires the autorater itself to be evaluated, recursively, which the paper acknowledges only in
passing.

The hypothesis-testing table assumes you have enough A/B comparisons to do statistics. For a one-user system, n is
small; treat the tests as guidance for batched eval runs (each commit, each model swap), not per-interaction.

Some of the dataset-curation infrastructure (Labelbox, SageMaker) is irrelevant to a private offline system. Open-source
equivalents (doccano, Prodigy) are mentioned and would be a better starting point if a labeling UI is ever needed.

Nothing in the paper addresses Apple Silicon, Metal/MLX, quantization, or unified memory constraints. It is silent on
local inference. That's fine — those concerns belong to a different paper class.

## Connections

This paper sits **upstream** of [Speed and Conversational LLMs (2502.16721v1)](2502.16721v1.md). 2502.16721v1 is a
single-axis intervention: "tok/s is misleading, measure task-completion time." 2506.13023v1 is the larger framework that
explains _why_ such single-metric arguments matter — speed-as-tok/s is exactly the kind of "de-facto metric without a
real-world strategy" the Google authors are pushing back against. Together they give Linus both the framework (this
paper) and a worked example of metric replacement (the speed paper). The natural Linus deliverable is one design
document for `benchmarks/dan_tasks/` that uses 2506.13023v1's three-pillar structure and folds 2502.16721v1's
task-completion-time methodology in as the canonical speed metric.

The decontamination discussion connects to the FineWeb-style data-curation papers in the corpus and to any future
fine-tuning work — both share the principle that data hygiene is load-bearing and easy to get wrong silently.

The component-level evaluation argument (LLM call + RAG + grounding API as separately measurable units) maps directly
onto the orchestration layer in `ARCHITECTURE.md` — router, tool registry, agent spawner are exactly the kind of
intermediate components the paper says need their own metrics.

The non-determinism estimation procedure (n=10 baseline runs, error band) is the right pattern for the eval CI gate
Linus will need before any model swap is approved by Maestro.

## Open questions for Dan

1. **`benchmarks/dan_tasks/` design pass.** Should we draft a one-page design doc that codifies the 5 D's, the
   four-metric balanced scorecard (ROUGE-L / embedding-cosine / NLI / autorater), the fictitious-entity probe set, and
   the non-response set as the canonical structure — and have a Worker scaffold the directory layout from that spec?
   This is a Phase 1 deliverable that pays off across every later phase.
2. **Autorater identity.** During Phase 1–2, who is the autorater? Hosted Claude (high quality, Maestro-budget cost,
   requires online), or a top local Worker (free, lower quality, eval-the-evaluator problem)? A hybrid (local default,
   escalate to Claude on disagreement or low-confidence) is plausible — does it merit an ADR?
3. **Sample-size policy.** Is ~250 items per task category the right standard target for `dan_tasks/`, or do we want
   tighter (ε=2–3%, ~1000+ items) for headline categories Linus will be benchmarked on publicly, and looser (ε=10%, ~70
   items) for exploratory tasks? The answer drives how much annotation effort is committed.
4. **Hallucination probe set as a public artifact.** A fictitious-entity probe set targeted at biochem/genomics ("the
   LINUS-fake-gene", "the XYZQ-9000 enzyme") is exactly the kind of small public release that would put Linus on a few
   people's radar. Worth doing in Phase 1, or hold for later?
5. **Continuous-eval CI gate.** When does `pytest` start running a small `dan_tasks/` slice on every PR that touches
   `src/linus/`? Phase 2 seems natural (you have an MVP to test against), but the dataset infrastructure could be in
   place earlier so the gate flips on the moment the first inference path lands.
