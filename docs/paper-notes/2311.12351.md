# Advancing Transformer Architecture in Long-Context Large Language Models

## TL;DR

Comprehensive survey of recent architectural advances to enable LLMs to handle longer context windows efficiently. Covers attention optimizations (local, hierarchical, sparse), memory extensions, and pre/post-processing techniques. Critical for Linus Phase 1-2: designing inference backends that balance long-context capability against 32 GB unified memory constraint.

## The problem (in plain language)

Transformer attention has O(n²) quadratic complexity in sequence length: processing a 100k-token document requires 10 billion attention operations. Standard LLMs plateau at 4k-8k tokens; extended context hurts latency and memory linearly. Yet many real tasks (scientific papers, codebases, conversations) exceed that. This survey catalogs the engineering solutions that make long-context practical.

## What they propose

The survey organizes advances into four dimensions:

1. **Efficient Attention** (Section 3): Local attention (windowed), Hierarchical attention (multi-scale), Sparse attention (pattern-based), Approximated attention (low-rank), ID-Aware attention (structured indexing)
2. **Memory Mechanisms** (Section 4): Internal cache optimization (KV compression), External memory storage (retrieval-augmented)
3. **Positional Embeddings** (Section 5): RoPE, ALiBi, extrapolation methods for sequences longer than training
4. **Context Processing** (Section 6-7): Pre-processing (context selection, compression), Post-processing (reranking, filtering), mixture-of-experts for selective token routing

## Key results

- Local attention (e.g., ring attention) reduces complexity to O(n·sqrt(n)) or O(n·k) where k is window size
- Sparse patterns (block-sparse, pattern-matrix) achieve 50-70% compute savings with <2% accuracy drop
- ALiBi positional embeddings generalize to 8x longer sequences than training without retuning
- Context compression (summarization, token merging) reduces memory 3-5x with <5% quality loss
- Hybrid approaches (local + sparse + memory) handle 128k-1M tokens on single 80GB GPU

## What's reusable in Linus

**Phase 1-2 (immediate):**
- Local + sparse attention patterns: low-hanging fruit for MLX or Ollama inference. Ring attention implementable in Metal shaders.
- ALiBi or NTK scaling for positional embeddings: enables 32k+ context on M1 Max with minimal overhead.
- KV cache quantization (int8): 4x memory savings. Critical for fitting larger models within 32 GB.

**Phase 3 (knowledge integration):**
- External memory + retrieval (Sec 4): direct pipeline to KnowledgeBase integration. Store paper embeddings, retrieve relevant context pre-inference.
- Context compression: compress Dan's personal notes into a compressible summary layer before reasoning.

**Phase 6 (fine-tuning):**
- The survey's taxonomy of positional embedding strategies informs LoRA adaptation for domain-specific long-context tasks.

## What's NOT applicable / hype filter

- GPU-specific optimizations (CUDA Tensor Cores, Triton kernels): survey relies on NVIDIA infrastructure. Metal/ANE equivalents require reimplementation.
- Billion-parameter fine-tuning studies: assumes 80GB+ VRAM. Linus is 32 GB unified memory; only LoRA/QLoRA on 7-13B models is tractable.
- Very large batch inference: survey benchmarks on multi-GPU clusters. Linus is single-device; batch size constrained to 1-4 by memory.
- Multi-node distributed training: not applicable to MacBook Pro.

## Open questions for Dan

1. **MLX-side support for these tricks.** The survey's most attractive items for Linus (KV cache int8 quantization,
   RoPE/ALiBi extrapolation, ring/local attention) presume CUDA / Triton implementations. Which of them already have
   MLX-native primitives in mlx-lm, and which would Linus have to port? A short scout of the MLX repo and Bonsai-demo
   would establish the surface area cheaply.
2. **Long-context vs. retrieval trade-off.** The survey separates long-context attention from external retrieval
   (KnowledgeBase / RAG) as if they're alternatives. For Dan's actual workflow (papers + notes + protocols), is the
   plan to push context to 32k–128k natively, or to keep the context modest and lean harder on KnowledgeBase? Both
   have memory implications on M1 Max.
3. **Position-extrapolation need.** ALiBi and NTK-RoPE generalize to longer sequences than training. For Linus, the
   choice of base model determines this implicitly (Llama 3 / Mistral / Qwen all use RoPE variants). Is there a case
   for preferring a particular base model on extrapolation grounds, or is "use whatever Ollama serves best" the
   right policy until benchmarks say otherwise?
4. **Compression-as-summarization for memory.** The survey's "context compression" line (token merging, learned
   summarization) maps onto the Layer B (within-session scratchpad) summary-rollup pattern in
   [`docs/specs/memory-architecture.md`](../specs/memory-architecture.md). Is there a Phase 2 spike worth doing on
   summary-rollup quality at fixed budget, using one of the survey's compression primitives as the baseline?
