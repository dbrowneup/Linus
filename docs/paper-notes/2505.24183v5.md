# QiMeng-CodeV-R1: Reasoning-Enhanced Verilog Generation

## TL;DR

Direct RLVR (reinforcement learning with verifiable rewards) follow-up to [CodeV (2407.10424v5)](2407.10424v5.md),
sharing the [`repos/CodeV`](../repo-notes/CodeV.md) code release. Attacks the three structural blockers that prevented
RLVR from being applied to Verilog generation: (1) no automated, accurate verification environment for sequential
hardware; (2) scarce high-quality natural-language ↔ Verilog pairs; (3) prohibitive RLVR compute cost. CodeV-R1's three
contributions answer those blockers in order: a rule-based testbench generator using Yosys for circuit-structure
analysis (96.1% fewer false negatives than LLM-generated testbenches; 62.5% more injected errors detected); a
**round-trip data synthesis** loop (code → NL → code, equivalence-checked by the testbench, theoretically proven to
yield high-quality pairs with high probability under strong-LLM + ideal-verification assumptions); a **two-stage
distill-then-RL** training pipeline using **adaptive DAPO**, an extension of DAPO with dynamic batch sizing that yields
1.25× training speedup. The resulting **CodeV-R1-7B** is the first Verilog-domain LLM to demonstrate strong reasoning
behavior with consistent test-time scaling, achieving 68.8% pass@1 on VerilogEval v2 and 72.9% on RTLLM v1.1 — surpassing
671B DeepSeek-R1 by 8.1% on RTLLM v1.1 and 3.3% on RTLLM v2 from a 7B parameter footprint, with total training cost ~2,656
A100-GPU-hours. Published at NeurIPS 2025.

## The problem (in plain language)

CodeV (the predecessor paper) established that supervised fine-tuning on real-world-Verilog + LLM-summarized-descriptions
produces strong 7B-class HDL models, with VerilogEval-Machine pass@1 in the 80% range. RLVR (DeepSeek-R1, OpenAI-o1) had
shown that adding a reinforcement-learning phase on top of an SFT-distilled model unlocks emergent **reasoning**
capability — extended chain-of-thought, test-time scaling, much higher pass rates on hard problems. The natural question
is whether RLVR can similarly unlock reasoning for Verilog generation, taking the CodeV-class model from "good HDL
autocompletion" to "Verilog-domain reasoner."

The answer is structurally harder than for Python or math RLVR, for three concrete reasons the paper quantifies:

1. **Verification environment.** RLVR needs an oracle that returns "correct" or "incorrect" for each candidate generation
   so the policy gradient has signal. For software code, unit-test-based verification is the standard tool — but unit
   tests are themselves often LLM-generated and noisy (the APPS benchmark has 60% false positives at average 20 unit
   tests). For sequential hardware (Verilog with clocks and resets) the LLM-generated-testbench approach fails harder:
   even GPT-class models lack the hardware-specific knowledge to handle reset signals, clock polarities, and sequential
   state machines correctly. Without a high-quality verification oracle, RLVR has no signal to optimize.

2. **Data scarcity.** Verilog NL ↔ code pair scarcity is the same structural problem CodeV addressed for SFT, but RLVR
   stresses the corpus harder — RLVR explicitly trains on _difficult_ problems where the base model fails some fraction
   of the time, so the corpus needs to be large after filtering. CodeV's 165K-module corpus collapses to ~3K after the
   RL-grade filters that CodeV-R1 applies (challenging, solvable, error-free).

3. **Compute cost.** SFT on 1K Verilog examples × 5 epochs on a 32B model takes 0.5 hours on 16 H100s. RLVR on 24K
   coding problems with a 14B model takes 2.5 weeks on 32 H100s. The cost difference between SFT and RLVR is roughly
   four orders of magnitude per training example — and a Verilog-domain RLVR run that can't lean on existing
   software-coding pipelines has to absorb the full cost.

## What they propose

**CodeV-R1** is a five-stage RLVR pipeline that resolves all three blockers in a coupled way (an automated testbench
that the data synthesis loop relies on; a data synthesis loop that produces the RL training corpus; an adaptive DAPO
algorithm that compresses the RL training cost):

1. **Code-to-NL.** Crawl Verilog from GitHub (~150K modules), summarize each with DeepSeek-V3 to produce
   (description, code) pairs `(x_i, y_i*)`. (Same as CodeV's Stage 1.)

2. **NL-to-Code.** Send each NL `x_i` to **DeepSeek-R1** (the strong reasoning teacher) and ask for both a "thought"
   `c_i'` and Verilog `y_i'`, producing 150K (NL, thought, code) triples — the seed corpus for SFT.

3. **Supervised Fine-Tuning (CodeV-R1-7B-Distill).** Filter the triples to retain only the difficult cases (Qwen2.5-Coder
   bases fail to generate equivalent code in any of 5 attempts, equivalence checked by the automated testbench from §2.1
   below) and synthesizable cases (Yosys can synthesize `y_i*`). Decontaminate against VerilogEval and RTLLM via
   Rouge-L > 0.5 filter. Resulting 87K-example dataset is used to SFT Qwen2.5-Coder-7B-Instruct into **CodeV-R1-7B-Distill**
   on 8 A100-80G for ~78 hours. The model is now a Verilog-flavored reasoner that emits chain-of-thought before code.

4. **Equivalence Checking.** Run the automated testbench against pairs `(y_i*, y_i')` from the round-trip; discard
   non-equivalent pairs. The retained `(x_i, y_i*)` pairs are guaranteed (under strong-LLM + ideal-verification
   assumption) to be high-quality NL ↔ Verilog data — a theoretical claim the paper proves explicitly.

5. **Reinforcement Learning (CodeV-R1-7B).** From CodeV-R1-7B-Distill, apply RLVR using **adaptive DAPO** on the
   filtered ~3.1K-example corpus. The reward function is rule-based and binary: a response gets reward 1 iff (a) the
   response is properly formatted (chain-of-thought between `<think>` tags, code between `<answer>` tags) and (b) the
   code passes the automated testbench. The RL stage runs on 16 A100-80G for ~127 hours.

The two pipeline pieces that carry the contribution weight:

**Automated testbench framework (the verification oracle).** Three phases: (1) **Circuit-structure analysis** — Yosys
extracts I/O ports, bit widths, clock signals + edge polarity, and reset signals + sync/polarity classification. (2)
**Simulation** — feed M=100 random input sequences of N=1000 cycles to both the generated and reference code. For
sequential circuits with reset signals, run a dual-stage validation (deterministic-reset stage tests core functionality;
random-reset stage validates reset consistency); for circuits with multiple reset signals, exhaustively test
non-conflicting combinations. (3) **Verification** — assess equivalence after each clock toggle, producing 2MN
assessments for sequential circuits and MN for combinational. Throughput: 15 instances/second on 32-way parallelization.
Quality validation: 96.1% fewer false negatives than LLM-generated testbenches and 62.5% more injected errors detected
in fuzzing tests for sequential circuits — meaning the testbench is the first Verilog verification oracle reliable
enough for RLVR.

**Adaptive DAPO (the cost-compression algorithm).** Standard DAPO uses dynamic sampling — for each prompt, generate G
responses and discard prompts whose responses are all-correct or all-incorrect (no learning signal). The fixed
generation batch size `b_gen` is suboptimal: too small → costly re-sampling when too few partially-correct samples land
in `b_train`; too large → wasted generation. Adaptive DAPO estimates the valid-sample rate `r_valid` from a sliding
window of past batches and sets `b_gen = ⌈b_train / r_valid⌉` adaptively per step. This preserves DAPO's accuracy
guarantees while accelerating training by ~1.25× — measurable wall-clock savings without changing the loss surface.
Adaptive DAPO is independently interesting as a generic RLVR speedup; it is not Verilog-specific.

## Key results

- **VerilogEval v1.** CodeV-R1-7B reaches 76.5% pass@1 on -Machine and 69.9% on -Human — outperforming the prior 7B
  open-source SOTA (CraftRTL-DS-6.7B) by 4.5% on -Human pass@1. Surpasses BetterV, RTLCoder, and the original CodeV
  series on -Human; trails CraftRTL-SC2-15B (a 2x-larger model) marginally on -Machine.
- **VerilogEval v2.** CodeV-R1-7B reaches 68.8% pass@1 on Specification-to-RTL and 69.9% on Code-Completion. Beats
  GPT-4o by 4.7% / 12.3% on the two splits respectively. The DeepSeek-R1-Distill-Qwen-7B baseline collapses to 0.6% /
  2.0% pass@1 — meaning naive distillation from a math/code teacher does not transfer to Verilog, and CodeV-R1's
  Verilog-specific distillation is doing real work.
- **RTLLM v1.1.** CodeV-R1-7B reaches **72.9% pass@1**, surpassing 671B DeepSeek-R1 (64.8%) by 8.1%. This is the
  headline result: a 7B Verilog specialist beats a 671B general reasoner on a Verilog-specific benchmark by training the
  small model with high-quality verifiable Verilog data.
- **RTLLM v2.** CodeV-R1-7B reaches 68.0% pass@1, surpassing DeepSeek-R1 (64.7%) by 3.3%. The advantage is consistent
  across both RTLLM versions; not a benchmark-specific artifact.
- **Distill vs RL.** CodeV-R1-7B-Distill (SFT-only) reaches 65.7% / 65.5% on VerilogEval v2 and 57.4% on RTLLM v1.1; the
  RL stage adds 4.2% / 4.4% / 15.5% on those benchmarks. RL is contributing substantively, not marginally.
- **Test-time scaling.** CodeV-R1-7B's pass@10 vs pass@1 gap is consistent across all benchmarks — the model exhibits
  the test-time-scaling signature of a true reasoner (more samples → better pass rate, at predictable rates), which the
  paper claims is the first such demonstration in the Verilog domain.
- **Compute cost.** Total ~2,656 A100-GPU-hours (78 SFT-hours × 8 GPUs + 127 RL-hours × 16 GPUs ≈ 624 + 2032 = 2,656).
  This is well below the 14B-on-32-H100s × 2.5-weeks reference point the introduction sets up — adaptive DAPO + the
  small filtered corpus + the 7B base together compress the RLVR cost into a research-lab-tractable budget.
- **Testbench quality (ablation).** 96.1% fewer false negatives vs LLM-generated testbenches, 62.5% more injected errors
  detected in fuzzing — the data the RLVR loop trains on is meaningfully cleaner than a vanilla pipeline could produce.

## What's reusable in Linus

CodeV-R1 is the QiMeng family's exemplar of **fine-tuned-Worker-self-improvement-via-RLVR** — the cleanest case in the
entire Linus reference corpus of an LLM that improves itself via reinforcement learning against a verifiable oracle, on
a domain where data is scarce. As such it is the load-bearing candidate for a Linus self-improvement ADR seed.

**Phase 6 — RLVR loop as a Worker self-improvement substrate (candidate ADR seed `_Seed: DEC-NNNN_`).** Per
[`qimeng-category-promotion.md`](../specs/qimeng-category-promotion.md) §"What remains to be done" task 7c, CodeV-R1's
RLVR loop is exactly the architecture a future Linus could adopt to improve a fine-tuned Worker on Dan's verifiable
tasks. The pattern factors into four reusable pieces: (a) **automated verification oracle** — for Linus, this would be
pytest output, ruff lint passes, KB-canary-test passes, or paper-qa retrieval-faithfulness checks; (b) **round-trip data
synthesis** — generate (Dan-spec, Linus-output) pairs, verify equivalence between original artifacts and Linus
regenerations through the oracle, retain only equivalence-passing pairs; (c) **distill-then-RL** — bootstrap reasoning
via SFT on teacher-LLM-generated traces (hosted Claude as the teacher), then sharpen on RL; (d) **adaptive DAPO** — a
generic RLVR speedup, applicable to any Linus RL run.

The strong claim: this is the first paper in the Linus reference corpus that gives a working blueprint for "Worker
improves itself on a domain Linus cares about, in compute envelopes a research lab can afford." The 2,656 A100-GPU-hours
total is on the high end of "expensive but bounded"; LoRA-only RLVR on a 7B Worker may be cheaper still. As a Phase 6
ADR seed: **adopt the CodeV-R1 four-component pattern (automated oracle + round-trip data synthesis + distill-then-RL +
adaptive DAPO) as the canonical Linus Worker self-improvement architecture, with the per-domain oracle as the unit of
effort being amortized**. Tagged inline with `_Seed: DEC-NNNN_` per
[`qimeng-category-promotion.md`](../specs/qimeng-category-promotion.md); ADR number to be claimed at the time the
synthesis lands and the architectural commitment becomes durable.

**Phase 6 — adaptive DAPO as a generic RLVR speedup convention.** Independent of the Verilog-specific contributions, the
adaptive batch-size mechanism for DAPO is a general improvement to any RLVR run with a dynamic-sampling-based loss. If
Linus ever fine-tunes a Worker via DAPO or a DAPO-derivative, the adaptive batch sizing is a free 1.25× training speedup.
This is the kind of lift-without-architecture-cost that belongs in a Phase 6 fine-tuning conventions doc.

**Phase 6 — automated-oracle-first as a fine-tuning convention.** The deeper insight from CodeV-R1 is that **the oracle
is the bottleneck, not the training algorithm**. The paper's dominant contribution by impact is the rule-based testbench
generator (96.1% false-negative reduction, 62.5% more injected-error detection); the adaptive-DAPO speedup is secondary.
For Linus Phase 6, the analogous principle: **before fine-tuning a Worker on a Dan-task class, build an automated
verification oracle for that task class first**. This generalizes the smoke-test-gate convention (CLAUDE.md
§Smoke-test gates) to fine-tuning data: you don't fine-tune on a corpus until you have an oracle that can validate it.

**Phase 7 — idea-to-reality discipline at the language level.** Same framing as the parent CodeV note — the artifact is
Verilog source, the downstream actor is the synthesis toolchain (Yosys → DC → fab), the oracle is the testbench. The
distinction CodeV-R1 adds: the artifact is now **reasoned about** before it is generated, with explicit
chain-of-thought visible to the user. For Linus's Phase 7 idea-to-reality skills (process flowsheets, expression-vector
designs, BOMs), the chain-of-thought is the auditability layer — it lets Dan inspect why Linus produced a particular
design before committing it to a downstream actor. CodeV-R1 is the cleanest existing instance in the corpus of "reasoned
artifact generation against a verifiable oracle."

**Phase 6 — concrete Worker checkpoint.** Like CodeV, CodeV-R1-7B is published as an open checkpoint (project page
https://iprc-dip.github.io/CodeV-R1; code + dataset released per the abstract). Linus can host it as a frozen Worker via
Ollama / mlx-lm and expose Verilog generation + reasoning as an MCP tool. Whether this matters for Dan's actual work is
the same question as for CodeV (probably not directly), but the artifact is small enough (~7B → ~4 GB int4) to keep on
hand. The more interesting use of the checkpoint is as a **substrate for studying** the RLVR-trained reasoning behavior
on a domain where the oracle is well-defined — Linus's eventual self-improvement loop will need this kind of probe.

**KV-cache continuity (Phase 2/Phase 6).** CodeV-R1's training emits chain-of-thought in `<think>` tags before producing
code in `<answer>` tags. The CoT is durable and addressable per DEC-0030 (scratchpad as first-class artifact); the
KV-cache continuity convention (DEC-0036) applies — silent CoT truncation between turns would invalidate the model's
reasoning behavior. For Linus integrating any reasoning-trained Worker, this is the canonical case where the scratchpad
discipline matters at runtime, not just at training time.

## What's NOT applicable / hype filter

**The "7B beats 671B" framing is benchmark-specific.** CodeV-R1-7B beats DeepSeek-R1-671B on **RTLLM** specifically;
DeepSeek-R1 still substantively beats CodeV-R1 on **VerilogEval-v1-Machine** (81.0% vs 76.5%) and is ahead on most
VerilogEval-v1-Human metrics. The right reading is "specialized 7B beats general 671B on the harder benchmarks where
specialization helps most" — not "7B uniformly dominates 671B." For Linus's broader fine-tuning question, this
nuance matters: specialist Workers beat generalist behemoths on tasks where the specialist's training data is
demonstrably higher-quality than the generalist's exposure, not universally.

**Total compute ~2,656 A100-GPU-hours is still substantial.** Adaptive DAPO and the small 3.1K-example RL corpus push
the cost down considerably from the introduction's 2.5-weeks-on-32-H100s reference point, but ~2.6K A100-hours remains
beyond what a single research-grade workstation can deliver in any reasonable wall time. For a single M1 Max Linus
target, even LoRA-only RLVR on a 7B Worker is at the edge of viable; the realistic Linus self-improvement plan probably
needs to either (a) lean on cloud bursts for occasional RL runs or (b) restrict RL to small-corpus, short-horizon
updates. The CodeV-R1 four-component pattern is reusable; the absolute compute cost is not "free."

**The synthesis toolchain still has closed-source segments.** Same caveat as CodeV: Synopsys / Cadence are required for
production ASIC tape-out. CodeV-R1 doesn't change this — the testbench is iverilog + Yosys + custom Python, all
open-source, but the back-end synthesis flow is not.

**Adaptive DAPO requires a sliding-window estimator that can be noisy.** The paper claims the adaptive batch-size
mechanism preserves DAPO's accuracy guarantees, but small-sample windows could introduce bias if the valid-sample rate
shifts rapidly during training. For Linus replicating the speedup, this is a small risk to monitor; the paper does not
include a worst-case-behavior ablation.

**Verilog still has back-end gaps.** Same as CodeV: PPA optimization, formal verification, timing closure are not
addressed. CodeV-R1 explicitly flags PPA optimization and analog-circuit synthesis as future work in §G "Limitations and
Future Work" — meaning even the QiMeng family's strongest current paper does not yet close the full RTL → tape-out loop
end-to-end.

**The DeepSeek-R1 teacher is an external dependency.** CodeV-R1's distillation stage uses DeepSeek-R1 as the
chain-of-thought teacher. For a fully-local Linus pipeline, this is a structural dependency on a hosted (or
prohibitively-expensive-to-run-locally) model. The realistic Linus path: use **hosted Claude** (or a future
Claude-equivalent) as the teacher during the distillation phase, treating the teacher cost as a one-time supervised data
generation step rather than a runtime requirement. This matches the Maestro budget discipline (CLAUDE.md §Maestro budget
discipline) — pay tokens once for high-value supervised data, not repeatedly for runtime.

**The "first reasoning Verilog LLM" claim is a NeurIPS-2025 timestamp claim.** It will age; expect later RLVR-Verilog
work to land in the corpus as the field matures. The structural contributions (automated testbench, round-trip data
synthesis, adaptive DAPO) will outlast the SOTA claim.

## Open questions for Dan

1. **Linus self-improvement ADR — when to commit.** CodeV-R1's RLVR loop is the cleanest blueprint in the corpus for a
   Worker self-improvement substrate, but committing to it as a Phase 6 ADR (`_Seed: DEC-NNNN_`) requires Dan to first
   identify a Linus-relevant verifiable-oracle domain. The candidate domains: (a) pytest-passing Python — easy oracle,
   broad relevance; (b) ruff-clean Python — easy oracle, narrow improvement; (c) paper-qa-faithful answers (DEC-0044) —
   harder oracle, high relevance to Dan's KB workflow; (d) bioinformatics-pipeline correctness (Nextflow / WDL passing
   small synthetic inputs) — high effort to build the oracle, high relevance to Dan's actual work. Which of these is
   the right first target for the RLVR loop?

2. **Apple Silicon RLVR feasibility.** The total ~2,656 A100-GPU-hours for CodeV-R1 training is well above what a single
   M1 Max can deliver. For Phase 6 viability, the realistic Linus path is LoRA-only RLVR on a 7B Worker with a much
   smaller corpus. Is there a Phase 6c spike worth running — take CodeV-R1-7B-Distill as a frozen starting point, run a
   tiny LoRA RLVR on M1 Max (or borrow cloud A100-hours for a one-off run), measure throughput on the adaptive-DAPO
   inner loop? The point is to establish the per-step cost before committing to the architecture.

3. **DAPO vs alternative RLVR algorithms.** Adaptive DAPO is a specific enhancement of DAPO. The broader RLVR landscape
   (PPO, GRPO, DAPO, OREO, RLOO) is moving fast. Should the Phase 6 ADR seed (Open question 1) commit to adaptive DAPO
   specifically, or to "a dynamic-sampling RLVR algorithm with adaptive batch sizing" abstractly? The latter preserves
   optionality at the cost of forcing a per-iteration choice.

4. **Round-trip data synthesis for Dan-tasks.** The CodeV-R1 round-trip is: (real artifact → teacher-LLM NL summary →
   teacher-LLM regenerated artifact → equivalence check via oracle → retain pair if equivalent). For Dan-task RLVR,
   the equivalent is: (Dan's real task output → Maestro-generated Dan-spec → fine-tuned-Worker-regenerated output →
   equivalence check via oracle → retain spec-output pair if equivalent). Does this workflow have legs, or does the
   Dan-spec generation step (Maestro paying tokens to summarize Dan's outputs) blow the budget? Worth a paper-napkin
   token-cost estimate before any commitment.

5. **Testbench-as-skill.** CodeV-R1's testbench generation framework is itself a Phase 7 candidate skill — Linus
   producing a testbench given a Verilog reference, exposing it via MCP to Maestro. This is independently useful (any
   user wanting to verify Verilog could call it) and reusable (the underlying Yosys + iverilog pipeline ports to
   anywhere Yosys runs). Worth filing as a Phase 7 skill candidate alongside the broader CodeV-R1 model deployment.

6. **CodeV / CodeV-R1 anchoring in the g12 synthesis.** Per
   [`qimeng-category-promotion.md`](../specs/qimeng-category-promotion.md) §"What remains to be done" Task 5.1, the new
   cluster synthesis (`g12-llm-hardware-design.md`) needs to decide whether the CodeV → CodeV-R1 progression anchors
   "the language-design thread" as a coherent two-paper arc, or whether CodeV-R1 anchors a separate "RLVR-for-hardware"
   thread that the synthesis presents as a maturity layer above the SFT-only language thread. Same shape question as
   CodeV's Open question 5 from the other side.

7. **Kernel-side RLVR follow-up.** CodeV-R1 is RLVR-for-Verilog. There is no published RLVR-for-GPU-kernels paper in the
   corpus yet — QiMeng-Kernel uses RL but with rule-based reward shaping, not the full distill-then-RLVR pipeline.
   Worth watching for a "QiMeng-Kernel-R1" follow-up that applies the CodeV-R1 RLVR template to GPU-kernel codegen; if
   it lands, the QiMeng family will have RLVR-trained instances at all three levels (kernel, language, microarchitecture).
