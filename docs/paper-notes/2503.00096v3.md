# BixBench: a Comprehensive Benchmark for LLM-based Agents in Computational Biology

## TL;DR

BixBench is a 61-capsule, 205-question benchmark for evaluating LLM agents on real-world bioinformatics analysis. Each
capsule pairs heterogeneous input data with a research hypothesis and was assembled by PhD-level bioinformaticians who
recapitulated published or de novo analyses in Jupyter (Python/R/bash). The agent scaffold is built on FutureHouse's
Aviary gymnasium and exposes only three tools — `edit_cell`, `list_workdir`, `submit_answer` — running inside a
pre-built Docker container with bioinformatics packages preinstalled. Two frontier models (GPT-4o, Claude 3.5 Sonnet)
are evaluated; **Claude 3.5 Sonnet tops out at ~21% accuracy in open-answer mode and barely beats random in
MCQ-with-refusal mode**. Performance does not exceed a pure-recall baseline (questions answered without notebook access)
in the MCQ regime, suggesting the models are mostly leaning on memorized knowledge rather than the analyses they
perform. The benchmark, the eval harness, and the agent framework are all open-source — the dataset on Hugging Face
(`futurehouse/BixBench`) and the eval code at `github.com/Future-House/BixBench`.

## Problem

Bioinformatics is one of the most plausible early targets for autonomous AI scientists: the "lab" is computational, the
data is digital, and the analytical workflow has clear stages (hypothesis → data exploration → statistical analysis →
interpretation). But the field lacks a benchmark that captures what bioinformaticians actually do. Existing benchmarks
either constrain agents to specific environments and tools (BLADE, ScienceAgentBench), test isolated code-generation
skills (BioCoder, HumanEval), reduce success to a scalar reward like training loss (RE-Bench, MLE-Bench), or test recall
over real research workflow (LAB-Bench, BioLPBench). None of them ask: given raw heterogeneous data, a research
question, and a working Jupyter environment, can an agent figure out what analysis to run and arrive at a defensible
answer?

The authors argue this open-ended, ambiguous, multi-step character is precisely what makes bioinformatics work hard —
and what makes it the right next benchmark frontier. The practical consequence of getting a real benchmark in place is
that progress on autonomous bioinformatics agents becomes measurable, which lets the community focus engineering effort
on the remaining gaps rather than chasing leaderboards on toy tasks.

## What they propose

The paper has three artifacts. First, the **dataset**: 61 "capsules", each a self-contained analytical scenario
consisting of (a) a hypothesis or research question, (b) input data files in their native messy formats (CSV, RDS,
FASTQ, etc.), (c) a reference Jupyter notebook implementing one valid analytical trajectory, (d) a textual result
describing the conclusion, and (e) a true/false flag for whether the analysis supported the hypothesis. The capsules
were built by contracted PhD-level bioinformaticians using a structured Colab interface, then peer-reviewed by other
analysts before merging. A median capsule contains roughly 75 lines of code and operates on substantial input data
(sometimes pulled live via wget, persisted separately for the benchmark).

Second, **the question set**: for each capsule, Claude 3.5 Sonnet was prompted to draft eight MCQs in two rounds;
analysts were given full edit/approve/reject authority and could see prior reviews to catch their own errors; a final
LLM-driven duplicate-filter pass (run in triplicate with manual concordance checks) deduplicated until clean. The result
is 205 questions, averaging 3.8 per capsule, designed by construction to be unanswerable without performing the analysis
(i.e., not solvable from training-set recall alone).

Third, **the agent scaffold and eval harness**: built on Aviary (Narayanan et al., 2024; FutureHouse's open agent
gymnasium). The scaffold is deliberately spartan — three tools only:

- `edit_cell` — select, modify, and execute a cell in a Jupyter notebook; each modification triggers a full notebook
  rerun, so the agent sees fresh tracebacks, tables, and plots after every edit.
- `list_workdir` — recursively inspect the workspace to discover input files and their layout.
- `submit_answer` — finalize the open-answer response, ending the trajectory.

All execution happens in a pre-built Docker container (`BixBench-env:v1.0`) with the standard bioinformatics
Python/R/bash package set. The agent must still find and import what it needs; it just doesn't have to install it. Two
evaluation regimes: **open-answer**, where a judge LLM (Claude 3.5 Sonnet) compares the agent's free-form response
against the ground-truth `result` field and scores binary correctness; and **MCQ**, where after the agent finishes its
open-answer trajectory, a second LLM is given the notebook plus the question with multiple-choice options (and
optionally an "Insufficient information" refusal option) and picks. Each capsule is run five times in parallel to handle
stochasticity; in MCQ mode they additionally do majority voting.

## Key results

Headline accuracy is bleak. **Claude 3.5 Sonnet hits 21% in open-answer mode; GPT-4o sits at 15%.** That's the better of
the two frontier models, with the better-fit task framing, on questions specifically designed to be answerable from the
analysis. In MCQ-with-refusal mode, both models perform very close to random because they preferentially refuse rather
than commit to an answer when uncertain — an "ethics" effect the authors attribute to the models recognizing they don't
actually have the evidence. When the refusal option is removed, MCQ performance rises above random, but the authors
argue this gain is largely driven by recall-based guessing on options the model hasn't earned through analysis. As a
calibration, they evaluated both models on the same questions **without any notebook access at all** (pure recall) and
found that MCQ performance with notebooks does not exceed the no-notebook baseline — meaning the agents' analyses are
not adding net signal in the MCQ setting. Majority voting across the five parallel trajectories does not significantly
help; vote concentration is flat as votes accumulate.

A vision ablation tested whether plot-generation hurts (since multimodal models reason poorly over scientific figures):
when agents were prompted to suppress image generation, performance was unchanged — neither helped nor hurt by removing
visual artifacts. The authors note this is consistent with prior observations (LAB-Bench, FigQA) that frontier
multimodal models still reason better over textual representations than over plots they themselves produced.

The reasoning-model story is interesting in its absence: the authors tested o1 and DeepSeek R1 in preliminary work and
found they "struggled" with the long contexts and structured tool-use outputs the agentic loop required, so they were
excluded from main results. This is a notable data point — the long-trajectory + structured-tool-use combination is not
yet a strength of the current reasoning-model lineage, even though reasoning models top many other benchmarks.

## What's reusable in Linus

**Direct, Phase 1 (Recon & Baselines):** BixBench is a runnable, Apple-Silicon-compatible benchmark for the kind of work
Dan actually does. Pull the Hugging Face dataset, run a Linus-Worker (Qwen2.5-Coder-7B, then later Mistral /
Linus-fine-tuned variants) through the Aviary scaffold against a small smoke subset (5-10 capsules), and we get a real
number for where local models sit relative to frontier models on bioinformatics analysis. The 21% Claude ceiling is the
upper-bound reference point; whatever Linus-7B scores defines the gap to close.

**Direct, Phase 1:** The capsule schema (hypothesis + data + reference notebook + result + truth-flag) is also a clean
template for `benchmarks/dan_tasks/`. Dan can author 5-10 capsules from his own genomics work and reuse the BixBench
eval harness more or less unmodified — capsules are the kind of artifact a PhD bioinformatician produces naturally as a
byproduct of analysis.

**Direct, Phase 2 (MVP):** The three-tool scaffold (`edit_cell`, `list_workdir`, `submit_answer`) is a minimal viable
spec for the Linus orchestration layer's first agentic tool surface. It demonstrates that a useful scientific agent can
be built on a tiny tool count — the design pressure to keep the tool registry small (per the Algorithm) is empirically
validated. The "Jupyter notebook with full rerun on edit" interaction model is a strong default for any code-execution
tool Linus exposes.

**Direct, Phase 3 (Knowledge & Parallel Agents):** The five-parallel-trajectories pattern with downstream judge or
majority vote is a concrete prior-art template for Linus' agent fan-out. The fact that majority voting doesn't help here
is itself a useful finding: parallelism without diversity in approach (different prompts, different models, different
toolings) doesn't pay. Linus' fan-out should aim for _trajectory diversity_, not just sample replication.

**Direct, Phase 7 (Skills & Autonomy Graduation):** Aviary itself is worth a `repos/` clone and a one-pager.
FutureHouse's framework is permissively licensed, designed for tool-using agents, and already validated by the
bioinformatics-skill capsules. If Linus adopts Aviary as the inner agentic loop (alternative to writing one from
scratch), the BixBench eval harness becomes a continuous regression test for Worker-skill quality.

**Indirect, Phase 6 (Fine-Tuning):** The capsule format is also a fine-tuning target. A LoRA on Qwen2.5-Coder using
BixBench capsules (hypothesis + data layout → reference notebook trajectory) is a plausible Phase 6 experiment — the
data is open, the format is standardized, and there's an evaluation harness ready. This is the kind of small,
well-scoped fine-tuning task that fits within 32 GB unified memory.

**Indirect, safety:** The ethics observation — that Claude 3.5 Sonnet preferentially refuses when uncertain and that
this hurts MCQ-with-refusal accuracy — is a useful data point for Linus' SAFETY.md. A locally-served Worker with an "I
don't know" output channel is safer than one trained to always commit; Linus should preserve that property.

## What's NOT applicable

The benchmark is built around a Docker container with bioinformatics packages. **Docker on macOS doesn't pass through
Metal/ANE** (per CLAUDE.md), so running the BixBench scaffold _as-is_ would mean the Worker model runs natively on Apple
Silicon while the analysis environment runs in the Docker VM — a perfectly fine split for reproducibility, but it means
the eval harness as shipped uses Docker for the _analysis sandbox_, not for inference. We need to confirm that the
scaffold can be configured to talk to an Ollama / mlx-served local model rather than the OpenAI/Anthropic APIs the paper
used.

The two frontier models tested are hosted (GPT-4o, Claude 3.5 Sonnet); the paper explicitly excluded reasoning models
because they struggled with long-context structured tool use. So the absolute 21% number is not transferable as a "what
Linus should hit" goal — local 7B-class models will almost certainly start much lower. The right framing is
gap-to-frontier as the metric, not absolute accuracy.

The judge LLM is Claude 3.5 Sonnet itself, a hosted call. For Linus to run BixBench in fully-private mode, we'd need to
swap in a local judge — possibly a larger local model or a specialized judge fine-tune. Judge-LLM substitution is a
known source of evaluation noise; this is an experimental risk.

The scaffold is single-agent. Linus' Maestro/Worker decomposition (where Maestro plans and Workers execute) maps cleanly
onto BixBench capsules but isn't what the paper measures. We can use BixBench to evaluate Worker-only performance;
evaluating the full Maestro/Worker stack against frontier-as-Maestro requires harness modifications.

Finally, the paper does not address training data leakage with strong rigor — the recall baseline shows the questions
weren't _trivially_ memorized, but it doesn't preclude that the underlying analytical patterns are well-represented in
training data. For local models trained on different data mixes, performance distribution may differ in ways the paper
can't predict.

## Connections

This sits squarely in the FutureHouse thread alongside [Kosmos (2511.02824v2)](2511.02824v2.md), the same group's full
"AI scientist" with world-model and parallel rollouts. BixBench is the _evaluation_ artifact that motivates Kosmos;
Kosmos is what FutureHouse built once they had a way to measure progress. Linus should treat BixBench as the metric and
Kosmos as architectural inspiration for the multi-agent loop.

It also pairs naturally with the
[bioinformatics multi-agent documentation paper (2026.02.09.704801v1)](2026.02.09.704801v1.md): both use multi-agent
setups for bioinformatics-adjacent work, both report fragility, and together they suggest that the _combination_ of
bio-domain knowledge + long-trajectory tool use is where current frontier models break.

The "MCQ-vs-open-answer" methodology connects to the broader measurement-rigor thread that includes
[Speed and Conversational LLMs (2502.16721v1)](2502.16721v1.md): both papers argue that the headline single-number
metric (tok/s in one case, MCQ accuracy in the other) misleads, and that the more honest measurement (task-completion
time, open-answer accuracy) reveals real model weaknesses. Linus' benchmark suite should default to open-answer wherever
possible.

The recall-vs-analysis distinction echoes [Sparks of AGI (2303.12712v5)](2303.12712v5.md) and the Chain-of-Thought paper
[(2205.11916v4)](2205.11916v4.md) — what looks like reasoning is often retrieval, and rigorous evaluation has to control
for that. BixBench's no-notebook recall baseline is exactly the kind of control Linus should bake into its own
benchmarks from the start.

The Aviary framework connection is also relevant to the harness-vs-orchestration distinction in CLAUDE.md and
GLOSSARY.md. Aviary is a harness; Linus is an orchestration layer. Linus could use Aviary the way it would use Cline —
as one inner-loop tool surface that the orchestration layer dispatches to.
