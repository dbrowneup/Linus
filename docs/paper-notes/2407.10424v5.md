# CodeV: Empowering LLMs with HDL Generation through Multi-Level Summarization

## TL;DR

Proposes CodeV, an instruction-tuning pipeline for HDL (Verilog + Chisel) code generation that flips the usual
data-synthesis direction: instead of asking GPT to generate Verilog from synthesized prompts (low quality, GPT-4 gets
only 43.5% pass@1 on VerilogEval-Human), CodeV harvests 1.5M Verilog and 24K Chisel modules from GitHub, filters them
to 165K + 18.7K self-contained syntax-correct modules, and asks GPT-3.5 to **summarize** each module at multiple
abstraction levels — exploiting the empirical observation that GPT-3.5 can describe Verilog (64.7% pass@1) far better
than it can generate it (33.5% pass@1). Pairs the resulting description-code corpus with a Chat-FIM-Tag supervised
fine-tuning method (chat for natural-language → HDL, fill-in-middle for code completion, explicit `<verilog>` / `<chisel>`
language tags). The headline 7B-class CodeV-Verilog-QC checkpoint reaches 80.1% / 59.2% pass@1 on VerilogEval-Machine /
-Human and 96.6% / 51.7% syntax / functional pass on RTLLM, surpassing GPT-3.5 by 35.2% on VerilogEval-Machine. CodeV-All
extends the same recipe to multilingual (Verilog + Chisel) and multi-task (chat + FIM) generation.

## The problem (in plain language)

Modern processor design pipelines are bottlenecked on hardware-description-language code authoring — Verilog for RTL,
Chisel for higher-level FIRRTL-flavored Scala-embedded HDL. The bottleneck is human-expensive: a digital designer
hand-writing Verilog for a non-trivial module is a time and skill drain comparable to the senior-engineer-days that
QiMeng-GEMM displaces for kernels. The natural play is "let an LLM write the Verilog from a natural-language spec," but
that breaks for a structural reason that the CodeV authors quantify: HDL training data is 40-80x scarcer than Python
training data on The Stack v2 (1.91M Verilog tokens vs. 80.6M Python; Chisel is a small fraction of 10.1M Scala). The
direct consequence is that even GPT-4 — the strongest software-coding LLM at the paper's writing — gets 43.5% pass@1 on
VerilogEval-Human, less than half its 88.4% on Python HumanEval.

The standard remedy in the prior literature was to **synthesize** Verilog training data with GPT (RTLCoder, AutoVCoder,
OriGen, BetterV, CraftRTL): generate (description, code) pairs by prompting GPT to write both. This inherits GPT's HDL
weakness — the generated code carries syntax errors, illegal combinational loops, and semantic disconnects from the
descriptions (the paper's Figure 2 shows a synthesized "barrel-shifter + 2-to-1 mux" example whose description maps to a
circuit with an illegal feedback loop and whose generated Verilog fails to integrate the two modules). Existing work also
focused on Verilog-only and chat-only — no Chisel, no fill-in-middle. The question CodeV attacks: can the data-synthesis
direction be reversed so that LLMs are used where they are strong (summarizing existing real-world HDL) rather than where
they are weak (generating new HDL), producing a high-quality corpus that survives both Verilog and Chisel and supports
both chat and FIM generation?

## What they propose

**CodeV** is an instruction-tuning pipeline with three coupled components:

1. **High-quality HDL collection + filtering.** Crawl GitHub for Verilog and Chisel modules. Apply deduplication
   (MinHash-LSH; near-duplicate threshold 0.85), self-containedness checks (modules importing only stdlib / declared
   inputs), and syntax-correctness gates (compile with iverilog for Verilog, sbt + Chisel for Chisel). The crawled
   1.5M Verilog modules collapse to 165K high-quality modules; 24K Chisel modules collapse to 18.7K.

2. **Multi-level summarization data synthesis.** Each filtered module y\* is sent to GPT-3.5 with a few-shot prompt that
   asks for a **fine-grained description** capturing implementation details, then asks for a **high-level functional
   specification** that abstracts away from the implementation (closer to how a real designer would write a spec in
   English). The pairing of (high-level description x, real-world code y\*) becomes a high-quality SFT training tuple.
   The choice to summarize rather than generate exploits an empirical asymmetry the authors quantify directly (Table I):
   GPT-3.5 hits 64.7% pass@1 when asked to summarize Verilog and only 33.5% pass@1 when asked to generate Verilog from a
   description. Summarization is the structurally easier task, so the data quality is much higher on the summarization
   side.

3. **Chat-FIM-Tag supervised fine-tuning.** A single instruction-tuning run mixes two task formats:
   - **Chat**: `[Instruction <verilog>] Design a register file… → [Response] module regs … endmodule`
   - **Fill-in-middle (FIM)**: `[prefix] <verilog> module mux_MDR … [suffix] … endmodule [middle] always …` using PSM
     (Prefix-Suffix-Middle) ordering with the base model's native FIM tokens.

   Explicit `<verilog>` / `<chisel>` language tags are inserted in both task formats; the paper's ablation (Table XI)
   shows the tags substantively help when one of the two languages has scarcer data — the model otherwise blurs the two
   distributions. Fine-tuning runs on top of four base models (DeepSeek-Coder-6.7B, CodeLlama-7B-Python, CodeQwen-1.5-7B,
   Qwen2.5-Coder-7B) using the Magicoder training infrastructure with bf16, flash-attention, accelerate + DeepSpeed.

The framework yields two model series: **CodeV-Verilog** (single-language, single-task: chat-Verilog only) and
**CodeV-All** (multi-language Verilog + Chisel, multi-task chat + FIM). The empirical claim is that CodeV-All matches
or exceeds CodeV-Verilog on Verilog-only benchmarks despite the broader scope — the multi-task / multi-language
fine-tuning is a free generalization, not a tax. Two new benchmarks — ChiselTutorial and ChiselEval (translated from
VerilogEval to Chisel) — and three FIM benchmarks (VerilogEval-FIM-Machine/Human, RTLLM-FIM, ChiselEval-FIM) are
released alongside the models to fill the evaluation gap.

## Key results

- **VerilogEval (Verilog-only).** CodeV-Verilog-QC (Qwen2.5-Coder-7B base) reaches 80.1% pass@1 on VerilogEval-Machine
  and 59.2% on VerilogEval-Human. Outperforms GPT-3.5 by 35.2% on -Machine and 22.5% on -Human; outperforms RTLCoder-DS
  (the prior open-source SOTA at the time) by 18.9% / 17.6%; surpasses GPT-4 on VerilogEval-Human (59.2% vs 43.5%).
- **RTLLM.** CodeV-Verilog-QC reaches 96.6% syntax pass and 51.7% functional pass — a > 30% absolute gain over RTLCoder
  on functional pass.
- **CodeV-All vs CodeV-Verilog.** CodeV-All-QC reaches 81.9% / 56.6% on VerilogEval-Machine / -Human and 96.6% / 55.2%
  on RTLLM — matching or beating the single-language CodeV-Verilog despite covering Chisel and FIM as well. The paper's
  Section V argument: multi-language / multi-task SFT is regularizing rather than diluting on this corpus.
- **Chisel.** CodeV-All-CL reaches 76.2% / 80.0% / 81.7% pass@1/5/10 on ChiselEval, establishing a baseline open-source
  Chisel-generation capability the prior work entirely lacked.
- **FIM.** CodeV-Verilog-CQ reaches 81.5% / 92.6% / 95.6% pass@1/5/10 on VerilogEval-FIM-Machine, 37.9% / 55.1% / 63.4%
  on -Human; CodeV-All performs comparably on the same FIM benchmarks while also handling Chisel-FIM.
- **Data-size ablation (Section V.E).** Performance scales smoothly with summarized-corpus size up to the full 165K
  modules; the Chat-FIM-Tag method itself (vs. plain chat-only SFT) accounts for a measurable share of the gain.
- **Cross-model robustness.** All four base models (DS, CL, CQ, QC) gain substantially from the CodeV recipe; the
  Qwen2.5-Coder-7B base produces the strongest CodeV checkpoint, consistent with its strong code-pretraining baseline.

## What's reusable in Linus

CodeV is the QiMeng family's exemplar of the **idea → reality** spine in its language-design framing: the LLM produces
Verilog or Chisel source code (the artifact); a downstream actor (the synthesis toolchain — Yosys, Synopsys DC, Cadence
Genus — followed by the FPGA/ASIC tape-out pipeline) accepts it; the realized artifact is a working RTL netlist or a
fabricated processor. CodeV is the first "language" entry in the family, complementing
[QiMeng-GEMM (13337-ZhouQ.md)](13337-ZhouQ.md)'s "kernel" entry, [QiMeng-Kernel (2511.20100v1.md)](2511.20100v1.md)'s
"GPU kernel" entry, and [QiMeng-cpu-v1 (../repo-notes/QiMeng-cpu-v1.md)](../repo-notes/QiMeng-cpu-v1.md)'s "CPU
microarchitecture" entry. The cluster home is in-flight at
[`docs/specs/qimeng-category-promotion.md`](../specs/qimeng-category-promotion.md).

**Phase 6 — multi-level summarization as a Linus fine-tuning data-synthesis convention.** The structural insight that
**summarization is easier than generation for low-resource code domains** is reusable far beyond Verilog. Many of Dan's
biology-skill Phase 7 targets share the same scarcity profile: bioinformatics scripts in WDL/Nextflow, KEGG-pathway
SBML models, MaxQuant proteomics configs, _Botryococcus_-flavored metabolic modeling — none have Python-scale public
corpora, but Dan's career-accumulated artifacts (LanzaTech metagenomics SOPs, PacBio internal docs from his FAS / PO
years, his _B. braunii_ thesis assembly scripts) constitute the high-quality real-world equivalent of CodeV's GitHub
Verilog harvest. The Phase 6 fine-tuning convention should be: **collect real artifacts, ask a strong teacher LLM to
summarize them at multiple abstraction levels, pair (summary, artifact) as training data, fine-tune on the pair set**.
This generalizes the typed-structured-prediction-wrapping-rationale convention (CLAUDE.md S25) by making the rationale
itself a teacher-LLM-generated artifact rather than a hand-authored one. Maps to a candidate Phase 6 convention seed:
`_Seed: DEC-NNNN_` for "multi-level summarization data synthesis" as a Linus fine-tuning pattern, distinct from the
straight DPO / SFT recipes already considered.

**Phase 6 — Chat-FIM-Tag as a Linus instruction-tuning ablation.** The Chat + FIM joint training with explicit language
tags is a small but well-validated trick: a single SFT run produces a model that handles both natural-language → code
and code-completion / mid-code-edit tasks. For Linus's eventual fine-tuned Worker, the equivalent split is "Dan-spec →
artifact" plus "edit / extend an existing artifact in place." A Phase 6 ablation using the Chat-FIM-Tag recipe over
Dan's task corpus should test whether a single SFT yields both behaviors, matching the CodeV finding that multi-task
training does not dilute single-task quality.

**Phase 7 — HDL skill class as a downstream candidate.** When Linus eventually grows a hardware-design skill (per the
idea → reality framing in [`qimeng-category-promotion.md`](../specs/qimeng-category-promotion.md)), CodeV is the first
out-of-the-box artifact: open-source 7B model, MIT license, HuggingFace distribution (`yang-z/CodeV-DS-6.7B` and four
sibling checkpoints). Linus can host the model via Ollama / mlx-lm and expose it as an MCP-wrapped Verilog-generation
tool to Maestro, with the synthesis toolchain as the downstream actor. Whether this is genuinely useful to Dan's
workflow is an open question — the realistic Linus user is Dan-the-biochemist, not Dan-the-chip-designer. But the
artifact is small enough (7B → ~15 GB FP16, ~4 GB int4) to keep on hand.

**Phase 8 — extension target for future Linus base.** CodeV is a fine-tune of standard 7B code bases. As Linus grows a
fine-tuned base model (Phase 6+), the CodeV recipe — multi-level summarization, Chat-FIM-Tag SFT, multi-language tags —
can be re-applied on top of Linus's own base to specialize for HDL or any other low-resource code domain Dan cares
about. The recipe's generality (DS, CL, CQ, QC all gain from it) suggests it would also work on a Qwen3 / Linus-base
substrate.

**Idea → reality discipline (load-bearing thread).** CodeV is the cleanest "language" instance of the family's
idea-to-reality spine. The artifact is text (Verilog source); the downstream actor is a deterministic synthesis
toolchain (Yosys → DC → fab); the oracle is functional simulation (VerilogEval testbenches, RTLLM functional pass). No
human-in-the-loop between LLM output and synthesized netlist beyond the testbench gate. This is exactly the discipline
Dan wants Linus to bring to non-hardware design domains: a process flowsheet that HYSYS accepts (Sketch2Simulation
sibling), a building-information-model that a contractor accepts, an expression-vector design that a wet lab accepts.
CodeV's evidence is that the discipline works at 7B parameters when the training data is clean; the data-cleanliness
step is the actual unit of effort.

## What's NOT applicable / hype filter

**The 80.1% headline is on VerilogEval-Machine, the easier benchmark.** CodeV-Verilog-QC's 59.2% on VerilogEval-Human is
a more honest baseline — humans write tougher specs than the auto-generated -Machine variant. The gap to a senior digital
designer remains substantial (the paper does not claim parity); CodeV is a tool for productivity gain, not a replacement
for the human designer. RTLLM functional pass at 51.7% similarly says "half the modules work, half need debug."

**No mention of post-synthesis quality.** Functional pass via simulation means the Verilog compiles and produces correct
outputs on the testbench. It does not mean the synthesized gate-level netlist achieves competitive PPA (power /
performance / area) targets — the part of HDL design that absorbs the largest share of expert effort in industry. CodeV
addresses the front-end (RTL authoring), not the back-end (synthesis quality, timing closure, formal verification).
This is acknowledged implicitly by CodeV-R1's "Limitations and Future Work" pointing to PPA optimization as the next
frontier.

**Real-world Verilog filtering is a hidden cost.** The 1.5M → 165K reduction (89% discard rate) hides substantial
preprocessing engineering — deduplication, syntax checks, self-containedness, license filtering. The paper does not
quantify the human-time cost of building the filter pipeline. For Linus replicating the recipe on a different
low-resource domain, this is the actual unit of effort being amortized — analogous to QiMeng-GEMM's hint-repository cost
that the headline 240× development-cost claim glosses over.

**The synthesis toolchain itself is closed-source.** CodeV's downstream actor for production HDL is Synopsys Design
Compiler / Cadence Genus — proprietary tools with five-figure license costs and Linux-only support. For an Apple-Silicon
local Linus, the open-source path stops at Yosys + iverilog (synthesis to gate-level + simulation). This is fine for
prototyping and education but not for production ASIC tape-out. The "idea → reality" pipeline as currently realized has
a closed-source segment Linus cannot fully internalize.

**Training infrastructure is NVIDIA-only.** The training pipeline uses Magicoder (DeepSpeed + accelerate +
flash-attention 2 + bf16) on GPU clusters. None of this is directly viable on M1 Max for re-training; the realistic
Linus path is to use the **published HuggingFace checkpoints** as frozen Workers (compatible with Ollama / mlx-lm in
GGUF or MLX-converted form) and not attempt to reproduce training. The same caveat will apply to most QiMeng-family
artifacts.

**HDL data scarcity is real and bounds upside.** The paper's own framing — Verilog data is 40-80x scarcer than Python —
is the structural ceiling on how good HDL-generation LLMs can become without breakthrough synthetic-data techniques. The
QiMeng-CodeV-R1 follow-up [(2505.24183v5)](2505.24183v5.md) attacks exactly this ceiling with RLVR and an automated
testbench loop, but the underlying data-scarcity problem isn't solved by either paper — it is mitigated. For domains
Dan might extend the CodeV recipe to (KEGG SBML, WDL pipelines, _B. braunii_ assembly scripts), the data-scarcity
problem is even harder, and the recipe's effectiveness depends on Dan's own real-artifact corpus being large and clean
enough to bootstrap from.

## Open questions for Dan

1. **Recipe portability to Dan's domain corpora.** The CodeV recipe (collect real artifacts → multi-level summarize via
   teacher LLM → Chat-FIM-Tag SFT) is structurally domain-agnostic. Could it be applied to Dan's career corpus —
   _Botryococcus_ assembly scripts, LanzaTech metagenomics WDL pipelines, PacBio internal SOPs from the FAS / PO years —
   to bootstrap a "Dan-specialist" Worker that handles bioinformatics-pipeline authoring at a productivity level
   comparable to CodeV's Verilog gain? Or is the underlying data corpus too small / too proprietary / too unstructured
   to support a meaningful SFT? Worth a one-week scoping spike before any Phase 6 commitment.

2. **HDL skill viability for Linus.** Does Dan ever need Linus to write Verilog or Chisel? The realistic answer is
   probably no — Dan is a biochemist, not a chip designer, and the Phase 7 idea-to-reality skill class for him is
   bioprocess / chemistry / biotech instrumentation, not CPU microarchitecture. But CodeV's checkpoints are small enough
   (~4 GB int4) that hosting one as a "weekend curiosity" Worker is essentially free. Worth keeping the model on hand
   for occasional use; not worth investing in a CodeV-as-skill integration unless a specific Dan-task surfaces.

3. **Multi-level summarization as a Linus knowledge-base augmentation pattern.** The CodeV pipeline produces TWO
   summaries per code module — fine-grained implementation description + high-level functional spec. This is exactly the
   shape Dan's KnowledgeBase ingests for papers: title + abstract + section summaries + bullet TL;DR. Could Linus run a
   CodeV-style multi-level summarization over Dan's existing context corpus (papers, threads, notes) as a one-time
   augmentation pass, producing additional retrievable surface area for paper-qa (DEC-0044)? The cost is a single sweep
   with a strong teacher LLM (hosted Claude or DeepSeek-V3 via API); the benefit is a richer corpus for retrieval. Worth
   scoping as a Phase 2/3 KB augmentation experiment.

4. **HuggingFace checkpoint licensing.** CodeV is MIT-licensed, hosted at `yang-z/CodeV-*` on HuggingFace. The base
   models (DeepSeek-Coder, CodeLlama, CodeQwen, Qwen2.5-Coder) carry their own (mostly permissive) licenses. For Linus
   integrating any QiMeng-family checkpoint as a Worker, is there a Phase 7 license-audit checklist Dan wants to
   establish — analogous to the biosecurity-tier-control framework (DEC-0047) but for ML-supply-chain provenance? Not
   urgent; flag for the cluster synthesis or a future ADR.

5. **Where does CodeV anchor the g12 cluster synthesis.** Per
   [`qimeng-category-promotion.md`](../specs/qimeng-category-promotion.md) §"What remains to be done" Task 5.1, the new
   cluster synthesis (`g12-llm-hardware-design.md`) should treat CodeV + CodeV-R1 as the language-design thread, paired
   with QiMeng-GEMM / QiMeng-Kernel as the kernel-design thread and QiMeng-cpu-v1 / Push the Limits as the
   microarchitecture-design thread. Should the synthesis present these as three parallel substrates of the same
   idea-to-reality discipline, or as a maturity ladder (kernel < language < microarchitecture in increasing artifact
   complexity)? The framing affects how the synthesis presents the family's design space to readers.

6. **CodeV-R1 RLVR loop as a Linus self-improvement substrate.** The CodeV → CodeV-R1 progression is the cleanest
   instance in the corpus of "fine-tuned Worker improving via RL with verifiable rewards using its own SFT-data
   pipeline." For Linus's Phase 6+ self-improvement question (per CLAUDE.md North Star: "evolves toward hosting
   fine-tuned, Dan-specific models"), is the RLVR-with-automated-testbench pattern a candidate ADR seed
   (`_Seed: DEC-NNNN_`) for "Worker self-improvement via RLVR on verifiable Dan-tasks"? See CodeV-R1 paper-note Open
   question 6 for the parallel framing on the RLVR side.
