# Bitnet.cpp: Efficient Edge Inference for Ternary LLMs

## TL;DR

The runtime that makes the BitNet papers actually fast on real hardware. **bitnet.cpp** is a C++ inference library
specifically tuned for ternary-weight LLMs, with kernel innovations (TL series lookup tables and I2_S MAD kernels) that
achieve **up to 6.25× speedup over FP16** on x86 and **up to 4.91× on Apple M2 Ultra**, while preserving _lossless_
inference parity with the BitNet b1.58 training scheme. **Tested on Apple M2 Ultra** — the only paper in this set that
publishes Apple Silicon numbers directly relevant to Linus' hardware. Open source at `github.com/microsoft/BitNet`.

## The problem (in plain language)

Even if you have a beautifully trained 1.58-bit BitNet, you need to _actually run it_. Off-the-shelf inference engines
(llama.cpp, T-MAC) handle ternary weights via one of two strategies:

1. **MAD-based** — multiply-accumulate, treating ternary weights almost like regular ints. Simple but doesn't exploit
   the fact that weights only have 3 values.
2. **LUT-based** — pre-compute a lookup table of all possible products, then index into it. Faster in principle, but
   existing implementations are _bit-wise_ — they store ternary weights as 2-bit values for memory alignment, wasting
   ~21% of the compression and producing slow misaligned access patterns. They also use per-block quantization for
   activations, which doesn't match BitNet b1.58's per-tensor INT8 training scheme — so inference output diverges from
   training output.

The non-integer "1.58 bits per weight" doesn't fit cleanly into 2- or 4-bit memory layouts. You either pad (waste space)
or pack creatively (slow). And if you don't perfectly mirror the training quantization scheme, your "BitNet" inference
produces different tokens than your "BitNet" training did. That's not a deployable artifact — it's a separate model.

## What they propose

**bitnet.cpp**, with two new mpGEMM kernel families:

**TL (Ternary Lookup table)** — element-wise LUT-based mpGEMM. The key trick is **element-wise mirror consolidation**:
half the entries in any LUT are negations of the other half, so you can store a "sign weight" (1 bit) separately from a
"magnitude index" (4 bits) and reconstruct via XOR+ADD. This squeezes 3 weights into 5 bits (1.67 bpw) instead of 6 (2
bpw). Two variants:

- **TL1** (g=2 weights per index): K dimension only needs to be even; works on most LLM shapes.
- **TL2** (g=3 weights per index): tighter packing but K must be a multiple of 3 — handled via _block-fitting weight
  splitting_ (statically partition the matrix so most of K is computed via TL2, the remainder via TL1).

For the 8-bit overflow problem during table lookup accumulation, they introduce **pack-and-unpack**: store partial sums
as int16 (split across two table lookups), then merge — fully lossless, no extra quantization step.

**I2_S (Int2 with Scale)** — element-wise MAD-based kernel that strictly mirrors BitNet b1.58 training (per-tensor INT8
activation, per-tensor ternary weight quantization). Lossless by construction. Slightly larger storage (2 bpw) but more
general K compatibility (multiples of 128 vs llama.cpp TQ2_0's multiples of 256).

## Key results

**Apple M2 Ultra (the directly relevant numbers for Linus):**

- **I2_S vs FP16 (llama.cpp)**: 2.15× → 4.91× speedup as model size scales from 700M to 70B.
- **TL2_0 vs T-MAC (2-bit LUT)**: ~1.05–1.19× speedup at most sizes.
- **TL2_0 vs TQ1_0 (llama.cpp ternary, 1.69 bpw)**: 1.05–1.65× speedup.

**Intel i7-13700H (laptop x86):**

- **I2_S vs FP16**: up to **6.25× speedup** at 13B and 30B.
- Can run **100B BitNet** on a laptop CPU at meaningful speeds; FP16 can't even fit.

**Quality**: I2_S, TL1_1, and TL2_1 are _bit-exact_ with FP16 BitNet b1.58 on WikiText2, HellaSwag, WinoGrande. TL1_0
and TL2_0 lose <0.01 PPL.

Scale: production-relevant. They benchmark up to 100B ternary LLMs running locally on consumer hardware.

## What's reusable in Linus

**Direct (this is the deployment story):** This is _the_ paper for "how does Linus actually run a BitNet". On M2 Ultra
(close cousin of M1 Max — both Apple Silicon, both ARM), I2_S delivers 2–5× speedup over FP16 inference. That's a 7B
BitNet model running ~5× faster than its FP16 equivalent on Dan's hardware. The codebase at
[repos/BitNet/](../../repos/BitNet/) (Microsoft) bundles bitnet.cpp.

**Direct (Linus orchestration backend):** Bitnet.cpp exposes a llama.cpp-compatible CLI, which means Ollama-style
harnesses can swap a BitNet for a Llama with minimal work. This is the cleanest path to using BitNet b1.58 / Bonsai /
future ternary checkpoints as Workers in Phase 2/3 without writing inference code from scratch.

**Indirect (kernel design lessons):** The element-wise LUT + signed-unsigned splitting + pack-and-unpack pattern is a
non-trivial piece of low-level kernel work. If we ever need to write _Apple-Metal-native_ ternary kernels for the
GPU/ANE (e.g. for [pmetal/](../../repos/pmetal/) integration), this is the reference design.

**Indirect (M1 vs M2 Ultra):** M2 Ultra has more cores than M1 Max, but the per-core architecture is similar. Speedups
should transfer roughly; absolute throughput will be ~half. A simple benchmark run on M1 Max would give Linus its actual
numbers.

## What's NOT applicable

- **CPU-only**: bitnet.cpp targets CPU SIMD (AVX2 on x86, NEON on ARM). It does _not_ use Metal/MPS or the ANE. On Apple
  Silicon, this is leaving substantial silicon on the table — but it's also why the speedups are so striking even
  without GPU.
- **Per-tensor INT8 activation quantization** is hard-coded to BitNet b1.58's training scheme. BitNet v2 (4-bit
  activation, Hadamard) and BitNet a4.8 (sparsity) need different kernels, not yet integrated as of this paper.
- The 100B model demo is striking but uses synthetic ternary weights, not a trained 100B BitNet. There is no released
  100B BitNet checkpoint as of paper publication.
