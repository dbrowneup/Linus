# Towards Revealing the Mystery behind Chain of Thought: A Theoretical Perspective

## TL;DR
Feng, Zhang, Gu, Ye, He, and Wang (NeurIPS 2023) prove the formal expressivity gap that the [Garrison memory thesis](../../context/notes/garrison_memory_makes_computation_universal.md) leans on: a bounded-depth Transformer of polynomial size **cannot** solve basic arithmetic, linear-equation solving, or general dynamic-programming problems by direct prediction, but a **constant-size** autoregressive Transformer **can** solve all of them when allowed to emit chain-of-thought derivations. The paper is the constructive companion to Merrill & Sabharwal's TC0 impossibility line — same wall, opposite side: it shows what the recursive feedback of CoT actually buys you in expressive power.

## The problem (in plain language)
Empirically, prompting an LLM with "let's think step by step" turns a model that fails arithmetic into one that succeeds. That gap is enormous and has been treated as a psychological trick of prompting. The theoretical question is sharper: is the gap real in the sense of changing what the *architecture* can compute, or is it just a sampling/optimization artifact? If a single forward pass through a fixed-depth Transformer is computationally limited, then no amount of clever prompting of a single pass can rescue it; conversely, if recursion across generated tokens lifts the system into a strictly larger complexity class, then chain-of-thought is not a prompt trick but an architectural unlock that happens to be controlled by prompts.

The paper picks problems where the answer is unambiguous and the complexity well-understood — evaluating arithmetic expressions, solving systems of linear equations by Gaussian elimination, longest-increasing-subsequence (LIS), edit distance — and asks whether a Transformer of fixed depth and polynomial width can produce the answer directly, versus whether a far smaller Transformer can produce a step-by-step derivation that ends in the answer.

## What the paper proves
The technical core lives in the contrast between two regimes for the same Transformer architecture. In the **direct-answer** regime, the model must emit the final answer in a single forward pass; depth is bounded, so the entire computation lives inside a TC0-style bounded-parallel circuit. In the **CoT** regime, the model emits intermediate tokens that are then fed back as input to the next forward pass; this recursion lets the "effective depth" of the computation grow with the number of generated steps rather than being capped by the architectural depth L.

The key mechanism the authors isolate is that CoT raises *effective depth proportional to generation length*. Each emitted token gives the model another L layers of computation to apply to the now-extended input. A constant-depth, constant-width Transformer that runs for n CoT steps therefore commands O(n) effective depth — which is what lets it simulate inherently sequential algorithms like Gaussian elimination or DP recurrences that no fixed-depth circuit can compute.

## Key results
**Theorems 3.1 and 3.2 (impossibility for direct prediction).** No bounded-depth log-precision Transformer of polynomial size can directly output the value of an arithmetic expression or the solution of a linear-equation system, unless model size grows *super-polynomially* in input length. The lower bound is a circuit-complexity argument: these problems are not in TC0 (assuming standard complexity-theoretic separations), and bounded-depth Transformers with log-precision are.

**Theorems 3.3 and 3.4 (constructive CoT solutions).** A *constant-size* autoregressive Transformer suffices to solve both arithmetic and linear-equation tasks by emitting CoT derivations in a standard math format. The construction is explicit: the authors describe attention patterns and FFN weights that implement a step of the underlying sequential algorithm per generated token.

**Theorem 4.7 (CoT solves general DP).** For a broad class of dynamic-programming problems — including LIS and edit distance — a constant-size autoregressive Transformer with CoT can generate the full DP table and produce the correct answer.

**Theorem 4.8 (counterexample for direct DP).** Bounded-depth Transformers of polynomial size *cannot* directly solve Context-Free Grammar Membership Testing, a classic DP problem. So the impossibility is not specific to arithmetic — it is a structural statement about what fixed-depth attention can express.

**Empirical complement.** Across all four tasks, direct-prediction accuracy stays below ~60% even with abundant training data; the same architecture trained to emit step-by-step derivations learns the algorithm cleanly and *generalizes to longer inputs than it saw in training* — strong evidence the model has learned the recursive procedure rather than memorized input-output pairs.

## What's reusable for Linus
This paper is theory; what it gives Linus is a precise vocabulary for *why scratchpad space is computational substrate, not cosmetic output.* Read against the [Garrison memory thesis](../../context/notes/garrison_memory_makes_computation_universal.md), it supplies the missing formal half of the "single forward pass = TC0; CoT = UTM" claim — Merrill & Sabharwal hand you the upper-bound side, this paper hands you the constructive lower-bound side. Together they establish that **reasoning tokens are not a UX feature; they are the memory mechanism that lifts a Worker out of TC0**.

The design implications for Linus's short-term memory pillar are direct. First, the orchestration layer must treat the **scratchpad as a first-class durable object**, not as ephemeral output. If a Worker emits 200 tokens of derivation before its final answer, those 200 tokens *are* the computation — discarding them between turns (the o1 failure mode) deletes the very thing that made the answer reachable. The session store should retain reasoning tokens by default, and the Maestro/Worker protocol should specify that intermediate derivations are part of the artifact, not a side effect of generation.

Second, the result reframes what "small local model" means in Phase 2. A constant-depth Transformer with adequate CoT length is provably more expressive than a much larger one forced to answer in one pass. This means a 7B Worker (Qwen2.5-Coder, Mistral) running with generous output budgets and full reasoning-token retention may meaningfully outperform a 70B model run terse — and the cost-per-correct-answer comparison should be measured in CoT-token throughput, not parameter count. This connects directly to the verbosity-vs-throughput tension surfaced in [2502.16721v1](2502.16721v1.md): for tasks that are inherently sequential (math, code reasoning, DP-shaped problems), terseness is not just a style preference, it is an expressivity downgrade.

Third, the "effective depth grows with generation length" mechanism is the formal justification for an **explicit reasoning-token budget per task class** in the router. Algorithmic tasks (math evaluation, dependency resolution, multi-step tool sequences) require depth budget; lookup-style tasks (extract, classify) do not. The router can use this distinction to set generation caps and KV-cache retention policies that match the task's expressivity demand rather than applying a uniform output limit.

Fourth, the construction is *autoregressive* — the recursion comes from feeding generated tokens back as input. This favors any Linus inference path that preserves and reuses prior reasoning cheaply: KV-cache continuity across turns, sticky session state, append-only scratchpad logs. Architectures or harness behaviors that silently truncate reasoning prefixes are deleting expressive power.

## What's NOT applicable / hype filter
The expressivity results say a Transformer *can* implement these algorithms with the right weights; they do not guarantee that gradient descent will *find* those weights from typical training data. The empirical section shows learnability on tasks with explicit step-by-step training data — the in-the-wild claim that any pretrained LLM emits algorithmically correct CoT is much weaker, and the literature on unfaithful CoT is well documented. Linus should not assume reasoning tokens are *correct* just because they are *expressive*.

The complexity separations rely on standard but unproven assumptions (TC0 ⊊ P-style separations). Inside theoretical CS this is uncontroversial; outside it the impossibility theorems should be quoted as conditional, not absolute.

The "constant size suffices" result is asymptotic. The hidden constants in the explicit construction are not small, and the paper does not claim they correspond to deployable model sizes. The right takeaway is qualitative — *CoT changes the complexity class, not just the constants* — not "we can replace 7B with 100M parameters."

Finally, the paper studies textbook DP and arithmetic problems with clean inputs. Real Linus tasks (scientific Q&A, code generation, tool use) are messier, and the formal expressivity argument does not directly bound their reachable accuracy. The result motivates *retaining* reasoning tokens; it does not predict how much accuracy that retention will recover on Dan's actual benchmark suite.

## Connections
This is the constructive twin of the Merrill & Sabharwal TC0 impossibility line cited throughout the [Garrison memory note](../../context/notes/garrison_memory_makes_computation_universal.md), and one of the two formal anchors for Garrison's "single forward pass = TC0, chain-of-thought = UTM" framing. The companion paper at [2310.07923v5.pdf](../../context/papers/2310.07923v5.pdf) (forthcoming note) supplies the matching log-precision upper-bound machinery. Reading the three together gives Linus a complete formal picture: TC0 ceiling on a single pass, UTM-class capability with recursion, memory as the architectural primitive that implements the recursion.

The result also gives theoretical teeth to Kojima et al.'s [Zero-Shot Reasoners (arXiv:2205.11916)](https://arxiv.org/abs/2205.11916) — the "let's think step by step" paper. That paper showed empirically that prompting unlocks dramatic capability gains; Feng et al. show *why* such an unlock is even possible: it moves the system between complexity classes, not just between sampling regimes.

For Linus phasing, this paper most directly informs **Phase 2 (MVP)** memory architecture — specifically the boundary between session/scratchpad memory and episodic memory called out in the Garrison roll-up — and **Phase 3 (parallel agents)** dispatch policy, where the router must decide reasoning-token budgets per task class. It is also a justification, not just a preference, for inference paths that retain KV cache across turns.

## Open questions for Dan
1. **Scratchpad as first-class memory object.** Should the Phase 2 session store treat reasoning tokens as durable artifacts by default — versioned, addressable, replayable — or as ephemeral generation byproducts that the Worker can discard once a final answer is produced? The expressivity argument here says the former; the latter is the path of least implementation resistance and is what most chat harnesses do today.

2. **Reasoning-token budget per task class.** Does the orchestration router need a task-class-to-CoT-budget mapping (e.g., "DP-shaped tasks get up to 4096 reasoning tokens with full retention; lookup tasks get 256 with truncation") as a Phase 2 router primitive, or is uniform budgeting good enough until benchmarks force the issue?

3. **Worker-size vs CoT-length tradeoff.** Given finite 32 GB unified memory, is it worth empirically comparing a 7B Worker with generous CoT budget against a 14B Worker with terse output on Dan's task suite? The theory predicts the small-with-CoT configuration should win on inherently sequential tasks; this is a falsifiable Phase 1 benchmark.

4. **KV-cache continuity as an architectural constraint.** Should the Linus inference layer commit to preserving KV cache across Worker turns within a session as a hard requirement — given that recursion-via-feedback is what the expressivity result hinges on — or treat it as an optimization to add later? The answer affects which inference servers (Ollama, mlx-lm, future pmetal) are viable Worker backends.

5. **Faithfulness check for retained reasoning.** If reasoning tokens are stored as durable artifacts and surfaced to Dan, the system implicitly endorses them. Should there be a Phase 3 component that audits CoT for self-consistency (does the chain actually compute what it claims?), or is that out of scope until specific failure modes appear?
