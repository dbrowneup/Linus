# A multi-agent platform for assessment and improvement of bioinformatics software documentation

## TL;DR

BioGuider is a four-module multi-agent system (collection, assessment, report, optional correction) that treats
bioinformatics software documentation as a first-class testable object rather than ancillary text. It evaluates README,
installation, user-guide/API, and tutorial content against task-oriented criteria, benchmarks itself with controlled
error injection across general / biology-specific / configuration error categories, and shows that GPT-OSS edges out
GPT-4o, Claude Sonnet, and Qwen3 on correction quality and runtime. Applied to 47 widely-used bioinformatics tools,
BioGuider scores correlate positively with citations-per-year — making documentation a measurable driver of software
adoption.

## The problem (in plain language)

The paper opens with a diagnosis the bioinformatics community has lived with for two decades: tools proliferate faster
than they can be made usable. Mangul et al. estimated 36,702 omics tools published 2005–2017, with 28% no longer
accessible at the time of survey, 28% of a random subsample failing to install at all, and 57% failing to install when
users followed only the provided installation tutorial. The single largest contributor to this gap, the authors argue,
is documentation — the connective tissue that turns a published method into a tool another lab can actually run.

Bioinformatics documentation is unusual in that it must encode biological assumptions (which species, which protocol,
which preprocessing convention), evolving experimental conventions, and deeply parameter-rich pipelines whose correct
usage is rarely obvious from the code alone. Generic documentation generators built for industrial software engineering
— RepoAgent, Repograph, and similar — can describe code structure but cannot judge whether a tutorial's example data is
biologically meaningful, whether dependency pinning is realistic for a 2024-era omics stack, or whether installation
instructions actually work end-to-end. The community has best-practice guidelines from journals, but they are applied
unevenly and there is no standardized, agent-driven way to grade or repair existing documentation.

BioGuider is the authors' answer: a system that ingests a GitHub repo, finds the documentation, classifies it by type,
evaluates it against a hierarchical rubric, writes a structured QC report, and (optionally) edits the documentation
under tight constraints — without rewriting code blocks, expanding scope, or removing sections.

## What they propose

BioGuider is organized as four modules, each implemented by specialized agents with clearly scoped roles. The
**collection module** ingests a repository and runs three agents in sequence: a planning agent that analyzes the repo
structure to predict where documentation lives; an execution agent that walks directories and loads candidate files; and
a categorize agent that classifies each retrieved file into one of four standardized types (README, installation, user
guide / API reference, tutorial / vignette). The output is a curated documentation corpus that smooths over the layout
heterogeneity typical of academic software repos.

The **assessment module** has two agents. An optional testing agent actually executes installation and tutorial
workflows in a controlled environment, capturing error messages and system-level failures — this is the part that
distinguishes BioGuider from pure-text reviewers. An assessment agent then fuses execution outcomes with content-level
analysis to produce structured scores on completeness, clarity, correctness, and consistency with the underlying
codebase.

The **report module** runs a reporting agent that aggregates per-category evaluations into a standardized QC report
linking each identified issue to the specific file and section that produced it. The **correction module** is opt-in: a
correction agent translates feedback into targeted edits, but is constrained by explicit prompt-level rules — preserve
all code blocks, do not expand content, do not remove sections, only fix typographical / formatting /
biological-terminology / configuration errors. This conservatism is the design choice that makes the system usable on
real published tools without producing hallucinated rewrites.

The evaluation rubric (Fig. 1b, Supplementary Table S1) decomposes documentation quality by document type rather than
scoring a repo with a single number. README files are graded on availability, readability, project purpose, system
requirements, dependencies, licensing, and contributor info. Installation docs emphasize accessibility, OS
compatibility, hardware requirements, dependency complexity, and whether installation actually completes from the doc
alone. User guides / API docs check API availability, contextual explanation, argument clarity, error handling,
return-value interpretation, examples, and consistency with code. Tutorials get the strictest treatment — executability,
reproducibility, setup completeness, performance considerations, navigation, result verification, and coverage of core
functionality.

To benchmark themselves, the authors built a controlled error-injection framework: starting from clean baseline docs, a
constraint-based LLM prompt injects a minimum number of errors per category while preserving code-fence boundaries and
high token-level overlap with the original. Project-specific terminology is auto-extracted from source code so injected
errors look like the kinds of mistakes real bioinformatics docs actually have. Only the corrupted documents are passed
to BioGuider — the system never sees the baseline or the error manifest.

## Key results

GPT-OSS achieved the highest correction performance across most error categories among the four LLMs tested (GPT-OSS,
GPT-4o, Claude Sonnet, Qwen3). GPT-OSS and GPT-4o had comparable runtime, but GPT-OSS was more stable across
heterogeneous error profiles, so the authors selected it as the default backend. The paper does not publish absolute
fix-rate numbers in the main text — they live in Fig. 1c (a heatmap) and Fig. 1d (runtime bars).

A stress test progressively increased the number and density of injected errors per document; BioGuider maintained high
correction performance and F1 score even under heavy degradation (Fig. 1e), suggesting it is usable on legacy or
fragmented documentation, not just on already-clean repos.

Applied to 47 published bioinformatics tools, overall BioGuider scores correlate positively with citations-per-year
(Fig. 1f, with a 95% confidence band on a linear fit). The authors are explicit that this is correlation, not causation
— biological relevance, methodological novelty, and timing all matter — but argue documentation quality is a previously
under-quantified contributor to software impact.

The platform is deployed as a hosted web service at https://bmblx.bmi.osumc.edu/bioguider; users supply a GitHub URL and
get a downloadable QC report, with optional iterative correction in the same UI.

## What's reusable in Linus

**Direct, Phase 7 (Skills & Autonomy):** BioGuider is a clean reference design for what a Linus _skill_ looks like — a
bounded, multi-agent workflow with named roles (planner, executor, categorizer, tester, assessor, reporter, corrector),
tight constraints on what the corrector is allowed to do, and a structured output format. The four-module pipeline
(collect → assess → report → correct) is a template Dan can borrow when defining the first non-trivial Linus skill, e.g.
a "documentation review" skill that runs over the Linus repo itself or over Dan's own scientific software. The role
decomposition is also small enough (~6 agents) to fit comfortably in 32 GB with a single Worker model running the loop.

**Direct, Phase 2 (Linus MVP) and Phase 3 (Knowledge & Parallel Agents):** The agent-spawner design in the orchestration
layer needs concrete patterns for agent role separation. BioGuider's split between a _planning_ agent that decides where
to look, an _execution_ agent that does I/O, and a _categorize_ agent that labels what it found, is exactly the kind of
three-step pattern the spawner should support natively. It also illustrates why role separation is worth the
orchestration overhead: each agent has a tight prompt, fewer ways to fail, and produces structured output the next agent
can consume — easier to audit-log and easier to retry on a single failed step.

**Direct, Phase 1 (Recon & Baselines):** The error-injection benchmarking methodology is reusable for any Linus skill
that does correction or refactoring. Take a clean reference (a README, a code file, a configuration), inject controlled
errors with a constraint prompt, run the skill on the corrupted version, and measure fix rate and F1. This is a much
cheaper signal than waiting for real bug reports and gives Dan a way to compare a Qwen2.5-Coder-backed Linus skill
against a Claude-backed baseline on the same task. It maps cleanly onto `benchmarks/dan_tasks/` as a new task family.

**Indirect, semantic-knowledge memory layer:** The hierarchical evaluation rubric (Supplementary Table S1) is a
domain-specific knowledge artifact: a structured grading schema for bioinformatics documentation. If Linus eventually
serves Dan's scientific software work, this kind of rubric is precisely the content the semantic-knowledge layer should
ingest — not as raw PDF text, but as a structured object the orchestration layer can query when running a
documentation-related skill. KnowledgeBase ingestion of papers like this should preserve the rubric structure, not just
the prose.

**Indirect, Maestro/Worker discipline:** The choice of GPT-OSS over Claude Sonnet and GPT-4o is interesting for Worker
selection on Apple Silicon. GPT-OSS is open-weights and runnable locally, which means the entire BioGuider pipeline
could in principle run as a Linus skill without hitting any paid API — a strong proof point that complex multi-agent
workflows do not require a hosted Maestro for every step.

## What's NOT applicable

The system is hosted as a web service backed by GPT-OSS / GPT-4o / Claude Sonnet / Qwen3 calls; the paper says nothing
about local-only deployment, MLX, or Apple Silicon. Adapting BioGuider's design for Linus means borrowing the
architecture, not the deployment.

The domain is bioinformatics-specific: rubric items like "biological terminology errors" and "command-line /
configuration errors typical of omics tools" don't generalize to, say, a Rust crate review skill. The _shape_ of the
rubric (per-document-type, hierarchical, task-oriented) generalizes; the specific items don't.

The correction module's conservatism (no content expansion, no section removal) is a sensible default for
already-published tools where preserving authorial voice matters, but it would be the wrong default for a Linus skill
that drafts new documentation from scratch — the constraint set is task-specific, not universal.

The 47-tool / citation-rate correlation is a single observational study with substantial scatter and self-acknowledged
confounds (novelty, timing, biological relevance). It is suggestive, not a quantitative model Dan can plug into
anything. It is also worth noting the paper does not benchmark against simpler baselines (e.g. a single-prompt LLM doing
the whole job in one shot), so the marginal value of the multi-agent decomposition over a strong monolithic prompt is
not directly quantified.

The paper is short — a brief preprint with most quantitative detail in supplementary materials and figures rather than
tables in the main text. Some claims (fix-rate numbers, runtime values) are visible in the figures but not extractable
as exact numbers from text.

## Connections

BioGuider is structurally similar to what the [memory architecture spec](../specs/memory-architecture.md) and
[memory synthesis](../memory-synthesis.md) imply for the orchestration layer: a skill is a small graph of agents with
clear roles and constrained outputs, not a monolithic prompt. The four-module pipeline maps neatly onto the
orchestration-layer / agent-spawner / tool-registry split — collection is tool use, assessment is judgment, reporting is
structured output, correction is constrained edit.

The paper's evaluation methodology echoes the [task-completion-time argument in 2502.16721v1](2502.16721v1.md): both
papers argue that the right way to evaluate an LLM-based system is to measure what it actually does on a controlled task
suite, not to trust upstream metrics (tok/s in the 2502 paper, raw model benchmarks here). For Linus, both papers
reinforce the case for `benchmarks/dan_tasks/` containing real, end-to-end task suites with measurable success criteria
— not just micro-benchmarks.

The hosted-web-service deployment model contrasts directly with the local-first ethos in [VISION.md](../../VISION.md):
BioGuider chose hosted convenience and a closed-source default model, where Linus chooses local sovereignty. Both
choices are defensible for their respective audiences; the architectural lessons transfer regardless.

There is no obvious connection to the BitNet / quantization or hypercube-geometry threads. This is an applications and
orchestration paper, not an inference or representation paper.

## Open questions for Dan

1. **First-skill candidate:** Should the first non-trivial Linus skill (Phase 7 territory, but specifiable now) be a
   documentation-review skill modeled on BioGuider, run against the Linus repo itself as the smoke test? It would
   exercise the agent spawner, the tool registry (file I/O, classification, optional code execution for the testing
   agent), and the audit log all at once on a self-contained target.
2. **Error-injection benchmark for Linus:** Worth adopting BioGuider's error-injection methodology as a
   `benchmarks/dan_tasks/` family for any Linus skill that does correction or refactoring? It is concrete, cheap, and
   gives a comparable F1 number across Worker model choices — useful for the Phase 1 recon goal.
3. **Rubric as structured KB content:** When KnowledgeBase ingests papers like this, should it extract structured
   artifacts (like Supplementary Table S1's rubric) into a separate semantic layer queryable by skills, rather than only
   embedding the prose? If yes, what is the minimal extraction schema?
4. **Worker model for multi-agent skills:** GPT-OSS beat Claude Sonnet and GPT-4o on this task. Does Dan want a Phase 1
   spike that pulls GPT-OSS via Ollama and runs a small BioGuider-shaped pipeline locally, as a stress test for "complex
   multi-agent skill on M1 Max with no paid APIs" — and if so, what target task would best exercise it?
5. **Constraint-based prompting as a first-class pattern:** BioGuider's correction agent works because of strict
   prompt-level constraints (no code rewrites, no expansion, no removal). Should Linus's tool / skill registry have a
   formal slot for "behavioral constraints" attached to each agent role, separate from the task prompt — and is that
   worth specifying in the orchestration-layer ADRs now, before the spawner is built?
