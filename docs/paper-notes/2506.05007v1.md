# QiMeng: Fully Automated Hardware and Software Design for Processor Chip

## TL;DR

QiMeng is a three-layer LLM-driven system (bottom: LPCM domain model, middle: Hardware/Software Design Agents, top:
applications) that automates processor chip design end-to-end. Addresses the Church Problem by replacing manual
expertise with learned domain knowledge and multi-agent orchestration—relevant to Linus as a model for domain-specific
AI orchestration and knowledge encoding.

## The problem (in plain language)

Processor chip design is slow, expensive, and requires deep domain expertise. Current automated design methods handle
only isolated steps (logic synthesis, placement, routing); no system bridges free-form hardware descriptions, code
generation, and design verification in a unified workflow. LLMs offer promise but lack domain grounding and struggle
with correctness guarantees.

## What they propose

A three-layer architecture:

1. **Bottom: Large Processor Chip Model (LPCM)**: Domain-specialized LLM encoding processor chip knowledge via
   multimodal inputs (text specifications, circuit diagrams, chip layouts, code) and hierarchical prompting
2. **Middle: Hardware/Software Design Agents**: Dual-loop feedback system (Hardware Design Agent for physical layout,
   Software Design Agent for functional correctness) with verification and repair loops
3. **Top: Applications**: Domain-specialized agents for specific chip design tasks (front-end, HDL generation, OS
   configuration optimization)

Uses Joint Expression Modeling (Stage I) to capture CRUX (Core Refined Understanding eXpression) + Dual-Space
Optimization (Stage II) to refine code generation.

## Key results

- Successfully applied to automated front-end design, automated HDL generation, automated OS configuration
- Demonstrated significant advantages in reducing manual intervention and shortening design cycles
- Agents autonomously repair functional failures and ensure design correctness via feedback mechanisms
- Cross-stage collaboration (top-down + bottom-up) enables self-evolving framework

## What's reusable in Linus

**Phase 2–3 relevance (Orchestration layer, knowledge substrate, agent design)**:

- **Dual-agent pattern**: Hardware/Software Design Agents model maps to Linus Workers. Feedback-driven repair (outer
  loop: performance optimization, inner loop: functional correctness) is directly applicable to agent task execution.
- **LPCM as domain knowledge substrate**: The multimodal embedding approach (text + graphs + layouts) is a blueprint
  for KnowledgeBase v2. Linus could encode bioinformatics domain via analogous LPCM-style models trained on Dan's
  papers + experiment notes + protocol annotations.
- **Verification feedback**: QiMeng's automated verification → repair pattern could inform Linus phase 3 safety
  mechanisms and tool sandbox policy.

## What's NOT applicable / hype filter

- Processor chip design domain knowledge doesn't transfer; very specialized to hardware design
- Relies on high-quality labeled chip design datasets; Linus lacks equivalently curated datasets for biology/genomics
- QiMeng's CUDA-based training and scaling; Linus is Metal/ANE-only, 32 GB ceiling
- LLM sizes (GPT-4, Qwen) are much larger than the 7–14B models feasible on Apple Silicon LoRA
