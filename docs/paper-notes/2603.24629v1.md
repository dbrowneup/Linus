# Sketch2Simulation: Automating Flowsheet Generation via Multi-Agent Large Language Models

## TL;DR

The paper builds a multi-agent LLM system that converts process flow diagrams (images) directly into executable Aspen
HYSYS simulation flowsheets via the simulator's COM API. The architecture decomposes the task across three coordinated
layers — diagram parsing and interpretation, simulation model synthesis, and multi-level validation — each composed of
specialized agents (multimodal LLM for vision, code-LLMs for synthesis, rule-based normalizer, execution-fixer loop).
Implementation uses Gemini 3 Flash in the cloud for multimodal interpretation and locally-served Qwen2.5-Coder-7B /
Qwen3-Coder-30B (via Ollama, orchestrated through LangGraph) for code synthesis. On four case studies the system
achieves F1 = 1.00 on the two simpler cases and ≥0.93 connection / ≥0.96 stream consistency on the harder ones;
ablations show each layer contributes meaningfully. For Linus, the chemical-engineering domain is incidental; the
**interesting payload is the architectural pattern** — a layered multi-agent workflow whose effectiveness comes from
execution-grounding against an external tool with a non-trivial API.

## The problem (in plain language)

A process flow diagram is a chemical engineer's blueprint: boxes for reactors and columns, arrows for material streams.
To actually simulate the process, an engineer manually rebuilds the diagram inside Aspen HYSYS, a commercial simulator
that exposes its model through a COM API. Two parallel research lines have nibbled from opposite sides: computer-vision
people built diagram-to-graph extractors that stop at the topology graph; text-to-simulation people built LLM agents
that produce HYSYS code only when handed a clean structured spec. Nobody bridged the symbol-to-parameter gap end-to-end.

Why is bridging hard? Diagrams omit things humans infer (a pump has one inlet; a distillation column is really three
coupled objects in HYSYS; a "mix here" arrow needs an explicit Mixer node). Simulators reject things diagrams allow.
Component names need to map to exact entries in the HYSYS database. Each layer has its own characteristic failure mode —
vision agents hallucinate equipment, code agents misuse COM object hierarchies, execution failures need to be localized.
This is fundamentally a **tool-grounding problem dressed in chemistry clothing**: how do you bind probabilistic LLM
outputs to the strict, brittle interface of an industrial software API while keeping failures attributable to specific
stages?

## What they propose

A three-layer pipeline orchestrated as a LangGraph state machine where each agent is a state transformer
`s_{k+1} = A_k(s_k)`.

**Layer 1 — Diagram parsing and interpretation** turns an image into a structured Intermediate Representation (IR) — a
directed graph G = (V, E) serialized as JSON. The Descriptor Agent (A1, multimodal LLM) produces a description with a
self-evaluation prompt grounding each claim in visual evidence. A separate Description Validation Agent (A1.1) re-runs
the multimodal model as an independent evaluator. The Extractor Agent (A2) takes both image (ground truth) and
description (semantic prior) and runs two passes — first identifying all units, then extracting streams that can only
attach to units from the first pass. The two-pass discipline limits hallucination by constraining the second-pass action
space. A rule-based Normalization Agent (A3) inserts implicit Mixer/Splitter nodes where multi-stream convergence
violates simulator constraints, and consolidates multi-symbol units (column + condenser + reboiler) into single template
objects.

**Layer 2 — Simulation model synthesis** uses four code-oriented agents to progressively build a Python script targeting
the COM API. The Basis Agent (B1) sets up the case file, picks the property package, and uses **RAG over a curated
component database** to map free-form material names to exact HYSYS database entries. The Instantiation Agent (B2) walks
IR nodes and emits unit-creation calls. The Configuration Agent (B3) walks edges and emits stream-connection calls. The
Execution Agent (B4) is hybrid: a deterministic Python runner executes the script, and on failure a code LLM reads the
trace and patches before retrying. Each agent is constrained by per-unit-type instruction files encoding COM-API
conventions.

**Layer 3 — Multi-level validation** is woven through layers 1 and 2: description validation, JSON-schema enforcement on
every IR transition, the rule-based normalizer (itself validator-and-rewriter), and the execution-and-fix loop. Putting
validation at every interface is what makes failures localizable.

**Deployment is hybrid.** Multimodal interpretation runs on Gemini 3 Flash in the cloud (open-weight multimodals they
tested underperformed). Code synthesis runs locally on Qwen2.5-Coder-7B (B1, B2) and Qwen3-Coder-30B (B3, B4) via
Ollama, because automation scripts touch proprietary process IP. Cloud for hard perception, local for sensitive code.

## Key results

The system runs end-to-end on all four case studies and produces an executable HYSYS model in every case. F1 reported
across Unit, Stream, Connection, and Material Consistency.

- **CS1 (Desalting, low complexity):** F1 = 1.00 across all metrics, no fixing-loop intervention. Normalizer correctly
  inserted a Mixer where the diagram showed an implicit junction.
- **CS2 (Merox, moderate, one recycle):** F1 = 1.00. The system inferred a connection (caustic prewash → Merox reactor)
  that was _omitted_ from the source diagram — semantic completion.
- **CS3 (Atmospheric crude distillation):** UC = 1.00, SC = 1.00, CC = 0.93, MC = 1.00. The connection drop was traced
  to the COM API: HYSYS distillation columns are template objects whose side-draw streams aren't fully programmable
  through the Python COM interface, even though the IR represented them correctly.
- **CS4 (Aromatic production, industrial scale, multiple recycles):** UC = 1.00, SC = 0.96, CC = 0.98, MC = 1.00.
  Failures were one omitted recycle and one mis-routed quench stream in the densest part of the recycle network. Model
  still executed.

The **ablation study** is the most interesting part for an architectural reader. C1 removes the descriptor, C2 removes
the normalizer, C3 merges the four coding agents into one, C4 disables RAG. On Merox, C4 (no RAG) causes complete
execution failure — sequential model construction can't recover from unresolved component names. C3 (merged coder)
substantially reduces connection consistency, evidence that **modular task decomposition is doing real work, not
cosmetic**. On the harder Aromatic case, removing the descriptor (C1) and merging coders (C3) significantly degrade
connection accuracy; removing normalization (C2) introduces hallucinated structural elements. **Architectural
sensitivity grows with diagram complexity** — the simple case tolerates degradation, the complex one doesn't. Each layer
earns its keep, and the value of decomposition rises non-linearly with task complexity.

## What's reusable in Linus

**The three-layer pattern (parsing → synthesis → validation) generalizes well beyond chemistry.** Many Linus tasks have
the same shape: take an unstructured artifact (PDF, screenshot, idea), produce a structured IR, synthesize an executable
artifact, validate by running it. The Phase 3 agent-spawner should treat this triad as a default workflow template
rather than something each Worker re-invents. IR-as-JSON-graph between stages is the contract that lets you swap out the
parsing layer without retouching synthesis.

**Multi-level validation as a first-class orchestration primitive.** The paper's strongest architectural argument is
that validation at every interface makes the system debuggable. The Linus agent spawner should expose validation hooks
per stage (schema on IR transitions, sanity checks on tool inputs, runtime execution checks with localized fix-loop),
not just an end-of-pipeline pass/fail. The "execution validation with bounded LLM-fix retry" pattern (B4) is directly
portable.

**Hybrid cloud/local deployment with model-class routing.** Their split between cloud multimodal and local code is what
Linus's router will need: route by modality and sensitivity. Multimodal vision is currently weak in open-weight
Apple-Silicon-tractable models, so the router needs a clean "send to hosted Claude / Gemini" path for image-heavy tasks
while keeping code generation and sensitive context on-device.

**Tool binding via per-tool instruction files.** B3 uses unit-specific instruction files encoding COM API patterns for
each object class — essentially a structured prompt-template registry keyed by tool. Linus's tool registry can borrow
this directly: each registered tool ships with a per-tool instructions document the spawner injects into Worker context.
More flexible than function-calling JSON schemas because it can encode preferred patterns, gotchas, and worked examples
— closer to a tool-specific README than a signature.

**RAG-for-vocabulary-grounding.** B1 uses RAG just to map free-form material names to canonical database entries —
narrower and crisper than "answer questions about my corpus." Linus's KnowledgeBase will have many such cases (gene
names, paper IDs, chemical names), and the C4 ablation shows this can be load-bearing for the entire downstream
pipeline.

## What's NOT applicable

The chemical-engineering domain is not Dan's. Aspen HYSYS is a $20K+/seat Windows-only commercial product Linus will
never directly call. None of the case studies are in genomics or biochemistry. Don't take the F1 numbers as
transferable.

The hardware deployment is opposite to Linus's: NVIDIA RTX A4000 + Windows 11 + CUDA 12.8 + 256 GB RAM. Qwen models will
run on Apple Silicon under MLX/Ollama, but the paper offers no benchmarks Linus can use to predict throughput on M1 Max.

The cloud dependency on Gemini 3 Flash is the most uncomfortable architectural choice and Linus should not import it
wholesale. For a private local assistant, "the multimodal layer is too hard so we send images to a hosted frontier
model" is exactly the failure mode Phase 4 (Data Sovereignty) needs to design against. Eventually Linus's answer
involves either better local multimodals or a per-task cloud-tier budget the user explicitly opts into.

The deterministic-orchestrator-with-fixed-DAG pattern is appropriate for their task (pipeline shape is known) but not
for general agentic work. Linus's longer-term agent-spawner needs to support both deterministic DAGs and more open-ended
planner-driven graphs.

The four-case evaluation is small. Treat the F1 numbers as existence proofs, not benchmarks.

## Connections

The closest neighbors are the **multi-agent papers being noted in parallel** — Kosmos, Boiko/Gomes, BioGuider,
TradingAgents, the Fundamentals of LLM Agents survey. Sketch2Simulation sits on the **tool-grounded execution end** of
that cluster — its agents are not deliberating peers, they are pipeline stages bound to a specific external tool with
strict execution semantics. The contrast with TradingAgents (LLM personas debating) and Boiko/Gomes (planner driving an
open action space) is informative: when the output is an executable artifact in a constrained environment, the
architecture collapses toward a pipeline with validation gates rather than blossoming into a debate.

It connects to the **tool-registry and sandbox themes** in `ARCHITECTURE.md` and `SAFETY.md`: the per-unit-type
instruction files are the same idea as a per-tool prompt fragment in Linus's tool registry, and the bounded-fix-loop is
the kind of primitive the agent spawner needs once it's running Worker-generated scripts in a sandbox. It connects to
the [Speed and Conversational LLMs paper (2502.16721v1)](2502.16721v1.md) on routing: their hybrid cloud/local
deployment is exactly the kind of decision that needs task-completion-time profiles, not raw tok/s, to make sensibly. It
does not meaningfully connect to the BitNet/quantization, hypercube, or inference-streaming papers — this is an
architecture paper, not an efficiency one.

## Open questions for Dan

1. **Is multi-level validation a first-class agent-spawner primitive?** The ablations make a strong case (removing the
   normalizer or validation pass degrades the harder case noticeably). Should Phase 3 expose validation hooks per stage
   in the spawner API — `pre_dispatch`, `post_synthesis`, `post_execution` — rather than letting Workers reinvent it? My
   read: yes, and the spawner's contract should require at least an execution-validator, even if it's a no-op for tasks
   that don't produce executable artifacts.

2. **Should Linus formalize a "diagram → executable artifact" Worker archetype?** The pattern (unstructured input → IR →
   code → execute → fix-loop) covers plot-replication, schema-from-docs, screenshot-to-Selenium, paper-figure-to-numpy,
   and a lot of likely scientific-computing tasks. Worth defining as an archetype in the spawner with a default
   skeleton, or kept ad-hoc per-task?

3. **Per-tool instruction files vs. function-calling schemas in the tool registry.** Their per-unit instruction-file
   approach is more flexible than typed function signatures but harder to validate. Do we want both — a typed signature
   for input validation and a free-form prose document for the Worker's prompt context?

4. **Cloud vision as temporary crutch or permanent tier?** Are we comfortable with Phase 5+ Linus offering an explicit,
   opt-in "send to hosted multimodal" tier, or do we hold the line on local-only and live with weaker vision until
   open-weight multimodals catch up? Values question more than technical.

5. **How aggressive should the execution-fix loop be?** B4's "execute, capture trace, patch, retry" is powerful but
   unbounded versions are a footgun. Right policy default — N retries, escalation to Maestro after N, or per-tool
   tunable?

6. **Does the IR-as-JSON-graph contract belong as an orchestration-layer primitive?** The paper's evidence is that a
   strongly-typed IR between stages is what makes the pipeline debuggable. Worth a small ADR on whether Linus's standard
   inter-Worker message format should be a typed graph/JSON schema rather than free-form text.
