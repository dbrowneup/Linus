# The Expressive Power of Transformers with Chain of Thought

## TL;DR

"The Expressive Power of Transformers with Chain of Thought," by William Merrill and Ashish Sabharwal (ICLR 2024),
characterizes how the number of intermediate "scratchpad" decoding steps changes what a decoder-only transformer can
compute. A transformer with no intermediate steps is stuck inside the circuit class TC0 — too weak to simulate a finite
automaton, decide directed graph connectivity, or solve linear equalities. Add a logarithmic number of decoding steps
and the upper bound grows only slightly to L; add a linear number and the model can recognize all regular languages; add
polynomially many and, with strict causal attention plus projected pre-norm, the model recognizes exactly P. This paper
is the formal partner to Garrison's memory-makes-computation-universal thesis: it is the constructive
complexity-theoretic statement of why a single transformer pass is bounded and why recursion through generated tokens is
the mechanism that lifts it.

## Plain-language problem statement

TC0 is the class of problems solvable by a uniform family of constant-depth, polynomial-size Boolean circuits with
AND/OR/NOT and unbounded-fan-in threshold (majority) gates. Informally, it is "what fast, massively parallel pattern
matching can do in a fixed number of layers" — it can do arithmetic on integers of bounded length and a great deal of
associative pattern recognition, but it provably cannot count past a circuit-encoded threshold, cannot simulate an
arbitrary finite-state machine on long inputs, cannot decide whether two nodes in a long graph are connected, and cannot
recognize the language of balanced parentheses at unbounded depth. Prior work by Merrill, Sabharwal and others had
already shown that a standard log-precision decoder-only transformer producing its answer immediately after reading the
input lives inside TC0. That is the negative result: a single forward pass is, formally, a fixed-depth parallel pattern
matcher with no recurrence and therefore no general state.

This matters because the argument is architectural, not parametric. No amount of width, depth scaling within a constant,
training data, or RLHF moves a single-pass transformer outside TC0. The wall is in the model class. If a problem
requires honest sequential state — counting, simulating, recipe-following, multi-step reasoning over arbitrary input
length — then a one-shot answer from any standard transformer cannot reliably solve it at scale.

## What the paper proves

The formal device is to study a decoder-only transformer that, after consuming an input of length n, is allowed t(n)
intermediate decoding steps before committing to an answer. Each intermediate token is appended to the context and
conditioned on by the next step. This generalizes "chain of thought" and "scratchpad" prompting into a clean
computational resource — number of decoding steps as a function of input length — and asks how that resource trades
against expressive power.

The central inclusion (their Equation 1) sandwiches the language class CoT(t(n)) recognized by a transformer using t(n)
decoding steps:

TIME(t(n)) ⊆ CoT(t(n)) ⊆ SPACE(t(n) + log n) ⊆ TIME-tilde(t(n)² + n²)

The lower bound is constructive: the authors build a decoder-only transformer that simulates t steps of a multi-tape
Turing machine using t intermediate tokens, with no external memory, no nonstandard positional encoding, just strict
causal masking and projected pre-norm. The key technical device is a "layer-norm hash" — a compact module that lets the
transformer use its own previously generated tokens as an addressable, content-distinguishable, temporally ordered
store. The upper bound follows from simulating t(n) decoding steps in O(t(n) + log n) workspace.

## Key results

Three regimes carry the message. With a logarithmic number of intermediate steps, t(n) = Θ(log n), the upper bound rises
from TC0 only as far as L (deterministic logspace); transformers still cannot solve NL-complete problems like directed
graph connectivity or P-complete problems like linear equalities. With a linear number of steps, t(n) = Θ(n),
transformers with projected pre-norm can simulate arbitrary automata (NC1-complete), placing them above TC0 (assuming
the standard non-collapse) and within the context-sensitive languages of the Chomsky hierarchy. With a polynomial number
of steps, t(n) = O(n^c), transformers with strict causal masking and projected pre-norm recognize exactly P — the first
exact equivalence between a transformer family and a standard complexity class.

The corollary that matters for system design is that the length of the chain of thought is itself a complexity-theoretic
resource. Logarithmic scratchpad buys very little. Linear scratchpad is where qualitatively new abilities appear.
Polynomial scratchpad turns the transformer into a strong reasoner in principle, though the wall-clock cost of a
polynomial number of forward passes through a frontier model is the practical ceiling.

## What is reusable for Linus — the memory lens

Read against
[garrison_memory_makes_computation_universal.md](../../context/notes/garrison_memory_makes_computation_universal.md),
this paper is the formal half of Garrison's two-step. Garrison's thesis is that universality requires recursive state
maintenance and reliable history access; Merrill and Sabharwal's construction is what those two requirements look like
inside a transformer. The intermediate tokens are the maintained state. The strict causal attention plus the layer-norm
hash is the reliable history access. The reason a single-pass transformer is stuck in TC0 is precisely the reason
Garrison's framework predicts: no recurrence, no memory, no escape.

For Linus this is the negative result that makes the architectural pillar non-negotiable. A Worker that produces its
answer in one pass is, by theorem, a TC0 device — a pattern matcher, not a general computer. Any task that needs honest
sequential reasoning has to be carried by intermediate tokens that the model can later attend over. Three concrete
consequences for Linus:

First, scratchpad retention is not a UX nicety. The orchestration layer must treat the model's intermediate tokens as
load-bearing artifacts that survive across tool calls inside a session. Truncating them — the o1 failure mode Garrison
flags — collapses the model back toward TC0 on whatever residual remains. Phase 2 session state should preserve
scratchpad alongside final answers, not in addition to them.

Second, recurrence is the cheaper escape. The paper shows that linear chain-of-thought lifts transformers above TC0, but
the cost is t(n) extra forward passes through the model. For a local M1 Max budget that is expensive. Recurrent or
state-space architectures (Mamba, S4, the "were RNNs all we needed?" line) achieve the same recursion through hidden
state at lower per-step cost. The complexity argument here is independent of model family — what matters is that
_something_ carries state across steps. This is direct support for Phase 6's bias toward inference paths that hold
rolling state cheaply.

Third, episodic memory has the same theoretical justification as scratchpad, one level up. Within a session,
intermediate tokens lift the model from TC0. Across sessions, an episodic store of prior reasoning, decisions, and tool
outputs plays the same role for the assistant-as-a-whole. Without it, every session is an independent TC0 episode no
matter how clever the inside-session scratchpad design is. The four sub-requirements Garrison extracts (addressability,
disambiguation, temporal order, integrity) translate directly into design constraints for the episodic substrate (SQLite
plus content hashes, git as an append-only log, vector store with metadata, or some hybrid).

## What is NOT applicable / hype filter

The complexity-class results are asymptotic and assume idealized log-precision arithmetic, saturated attention, and
projected pre-norm. Real transformers run at bf16 or quantized precision with softmax attention; the constructions are
lower-bound existence proofs, not claims that any specific trained model actually realizes the upper limit of its class.
A linear chain of thought _can in principle_ recognize all regular languages; whether a given Qwen or Mistral checkpoint
actually does so on a specific task is empirical and unrelated to the proof.

The paper says nothing about _which_ chain-of-thought prompts elicit the latent expressive power, nor about training
procedures that make a model use its scratchpad efficiently. The upper bounds are worst-case over all transformers in
the class; specific models may waste their scratchpad capacity. Treat the framework as a ceiling argument and a
motivation, not as a recipe for prompt design.

The polynomial-step result is theoretically clean (CoT = P) but practically punishing — n^c forward passes through a
frontier-size model is intractable. The result motivates investment in cheap-per-step recursion, not in heroic test-time
compute. The o3-style brute force Garrison critiques is exactly the wasteful path this theorem makes possible but does
not endorse.

The constructions assume the model can read its own previously emitted tokens cleanly. Any deployment that hides,
compresses, or summarizes intermediate reasoning between steps (o1 style) leaks expressive power in ways the proofs do
not bound.

## Connections

The companion piece is
[garrison_memory_makes_computation_universal.md](../../context/notes/garrison_memory_makes_computation_universal.md) and
the proof paper it summarizes ([../../context/papers/2412.17794v1.pdf](../../context/papers/2412.17794v1.pdf)). Garrison
cites Merrill and Sabharwal for the TC0 limit; this paper supplies both that limit (via prior work it builds on) and the
constructive escape (chain of thought lifts decoder-only transformers to UTM-class behavior). Note that the strict TC0
result for single-pass transformers is from the earlier Merrill and Sabharwal "parallelism tradeoff" paper (TACL 2023);
the present paper restates it as Prior Work and characterizes what scratchpad steps add on top.

The "let's think step by step" empirical phenomenon ([2205.11916](https://arxiv.org/abs/2205.11916), Kojima et al.) is
what this paper formalizes: those prompts work because they create the very intermediate-token recursion the proofs
require. The minimal-RNN line ([2410.01201](https://arxiv.org/abs/2410.01201), Feng et al., "Were RNNs all we needed?")
matters because it shows that recurrent architectures with rolling hidden state can recover sequence-model performance
at lower per-step cost — a more efficient way to satisfy the same recursion requirement Merrill and Sabharwal isolate.
Together these three threads (Kojima, Merrill-Sabharwal, Feng) span empirical phenomenon, formal characterization, and
architectural alternative.

Phase mapping: Phase 2 (MVP) — scratchpad and session state design must respect the recursion-is-load-bearing argument
from this paper. Phase 3 (Knowledge & Parallel Agents) — episodic memory inherits the same theoretical motivation. Phase
6 (Fine-Tuning) — bias toward recurrent / state-space inference paths is supported by the cost analysis implicit in
linear vs. polynomial step counts.

   argument here says yes — discarding them is the o1 failure mode and collapses the Worker back toward TC0 on follow-up
   turns. What is the smallest viable schema (turn id, scratchpad blob, hash, timestamp) that satisfies the four
   sub-requirements from Garrison's framework?

   _Resolved (DEC-0030, see [answered-questions.md](../questions/answered-questions.md)): Scratchpad is a first-class

2. **Linear vs. polynomial scratchpad budget per Worker call.** The paper distinguishes log, linear, and polynomial step
   budgets as qualitatively different regimes. Should Linus's router enforce per-call scratchpad budgets, and if so, on
   what basis — task type, deadline, energy budget? A "this task gets up to n decoding steps of scratchpad" knob is
   cheap to implement and would let Linus exploit the linear-steps regime where it pays off without burning into
   polynomial test-time compute.

   _Resolved (DEC-0031, see [answered-questions.md](../questions/answered-questions.md)): Router cot\_budget primitive
   (logarithmic/linear/polynomial regimes) added as a Phase 2 first-class primitive._

3. **Recurrence as a memory-cost optimization.** Given that the same recursion can be carried by intermediate tokens
   (expensive, attention-quadratic) or by hidden state (cheap, constant per step), should Phase 6 actively prefer
   state-space or hybrid architectures for the default Linus Worker, on the grounds that they implement the
   Garrison/Merrill-Sabharwal recursion requirement at lower per-step cost on a 32 GB unified-memory budget? This
   connects to the Bonsai / pmetal / mlx-flash thread.

   _Partially resolved (DEC-0038): minGRU MLX port spike scheduled for Phase 1f; Phase 6 bias toward recurrent
   architectures pending spike results._

4. **Cross-session episodic memory as the "outer scratchpad."** If within-session scratchpad is what lifts a single
   Worker call out of TC0, is cross-session episodic memory the analogous lift for the Linus assistant as a whole? What

   hypothesized expressivity gap rather than just to convenience?
