# BitNet b1.58 2B4T Technical Report

## TL;DR

The first **open-weights, native 1.58-bit LLM at 2B-parameter scale** — released on Hugging Face along with bitnet.cpp
inference code for both CPU and GPU. Trained from scratch on 4 trillion tokens, instruction-tuned via SFT + DPO. **Beats
Qwen2.5-1.5B-INT4-GPTQ/AWQ** on MMLU/GSM8K/IFEval at ~half the memory (0.4 GB non-embedding vs 0.7 GB). Beats other
native 1-bit models (Bonsai 0.5B, OLMo-Bitnet 1B, Falcon3-1.58bit 7B, Llama3-8B-1.58) by wide margins. **0.4 GB memory,
29ms TPOT on CPU**. This is the model to actually try in Linus.

## The problem (in plain language)

For two years, the BitNet papers showed that 1-bit LLMs _could_ match FP16 performance — but the published checkpoints
were small, demo-scale, or proprietary. The open-source 1-bit landscape was: Bonsai (0.5B, weak), OLMo-Bitnet (1B,
weak), and various post-training-quantization conversions of Llama/Falcon (which the BitNet papers had specifically
argued would underperform).

To actually validate "1-bit LLMs are deployment-ready," Microsoft needed to release a serious, instruction-tuned, native
1.58-bit checkpoint at the size where general-purpose use becomes plausible (1–3B). 2B4T is that release: 2 billion
parameters trained on 4 trillion tokens, the same scale-and-data class as LLaMA-3.2-1B, Qwen2.5-1.5B, and SmolLM2-1.7B.

## What they propose

A **trained 2B BitNet b1.58 model** (not a new technique — the techniques are inherited from prior BitNet papers). The
relevant details:

**Architecture:**

- BitLinear layers (1.58-bit weights, 8-bit activations) per the original BitNet papers.
- **SubLN** normalization (matches BitDistill's stability fix).
- **Squared-ReLU GLU** in FFN (not SwiGLU) — drives sparsity per BitNet a4.8.
- **RoPE** positional embeddings.
- **No bias** in any linear or norm layer.
- **LLaMA-3 tokenizer** (BPE, 128,256 vocab) — direct ecosystem compatibility.

**Training (3 phases):**

1. **Pre-training, 4T tokens**: two-stage learning rate (high-LR initial + cooldown), two-stage weight decay (peak 0.1 →
   0). DCLM web crawls + FineWeb-EDU + synthetic math data.
2. **Supervised fine-tuning (SFT)**: WildChat, LMSYS-Chat-1M, WizardLM Evol-Instruct, SlimOrca + GLAN/MathScale
   synthetic. Loss summed (not averaged) per token. Larger LR than typical FP fine-tuning.
3. **Direct Preference Optimization (DPO)**: 2 epochs, LR 2e-7, β 0.1. UltraFeedback + MagPie. Optimized via Liger
   Kernel.

**Released artifacts** (Hugging Face):

- `bitnet-b1.58-2B-4T` — packed 1.58-bit weights for inference.
- `bitnet-b1.58-2B-4T-bf16` — bf16 master weights for further training/fine-tuning.
- `bitnet-b1.58-2B-4T-gguf` — GGUF format for bitnet.cpp.

## Key results

**Memory and compute** (vs full-precision peers):

- **0.4 GB non-embedding memory** (LLaMA-3.2-1B is 2 GB; Qwen2.5-1.5B is 2.6 GB; MiniCPM-2B is 4.8 GB).
- **29ms TPOT** (time per output token) on CPU (LLaMA-3.2-1B 48ms; Qwen2.5-1.5B 65ms; MiniCPM-2B 124ms).
- **0.028 J/token** estimated energy (LLaMA-3.2-1B 0.258 J; ~10× saving).

**Average over 17 benchmarks** (one shared evaluation pipeline):

- BitNet b1.58 2B4T: **54.19** average
- Qwen2.5 1.5B: 55.23 (best in class)
- SmolLM2 1.7B: 48.70
- LLaMA-3.2 1B: 44.90
- Gemma-3 1B: 43.74
- MiniCPM 2B: 42.05

**Highlights:** wins ARC-Challenge (49.91), BoolQ (80.18), WinoGrande (71.90), PIQA (77.09), MATH-500 (43.40), GSM8K
(58.38).

**vs Post-training-quantized Qwen2.5 1.5B** (INT4 GPTQ/AWQ, the natural deployment-comparable baseline):

- BitNet 2B4T: 0.4 GB, 55.01 avg on MMLU/GSM8K/IFEval
- Qwen2.5-1.5B-GPTQ-INT4: 0.7 GB, 52.15 avg
- Qwen2.5-1.5B-AWQ-INT4: 0.7 GB, 51.17 avg

**vs other 1-bit open models**: dominates Bonsai 0.5B (41.06 avg), OLMo-Bitnet 1B (32.22), Falcon3-1.58bit 7B (50.76),
Llama3-8B-1.58 (49.75).

## What's reusable in Linus

**Direct, immediate, the highest-priority paper here:** This is _the_ released model. Pull the GGUF from Hugging Face,
run it through [bitnet.cpp](2502.11880v1.md), benchmark on M1 Max. End of analysis — this is what a Linus Worker model
would be in Phase 2.

**Concrete next-step**: a 2B model at 0.4 GB memory leaves **31.6 GB free** out of 32 GB unified memory for context, KV
cache, parallel agents, KnowledgeBase indexes, and other Workers. This is the first checkpoint that makes "Linus runs N
parallel Workers in unified memory" stop being aspirational and start being immediate.

**Indirect (architecture choices to inherit):** SubLN + squared-ReLU GLU + LLaMA-3 tokenizer + no biases is a config
recipe. If we ever distill our own BitNet (per BitDistill), we'd want to start from this exact architecture.

**Indirect (instruction-tuning recipe):** SFT-then-DPO with summed (not averaged) loss is the documented recipe. The
fact that they used UltraFeedback + MagPie for DPO and got coherent improvements gives us a template for any future
Linus-specific alignment runs.

## What's NOT applicable / hype filter

- **Average score is just below Qwen2.5-1.5B (54.19 vs 55.23)**. Not winning on raw quality. The pitch is the dominant
  point on the _Pareto frontier_ (memory × quality), not best-in-class quality. If raw quality is the only thing that
  matters for a given Linus task, Qwen2.5 still wins.
- **HumanEval+ at 38.40** lags Qwen2.5 (50.60). Code-heavy tasks (generating Python for Dan's bioinformatics work) may
  want a code-specialized Worker rather than this general one.
- **No reinforcement-learning-with-policy-optimization stage** (PPO/GRPO). Reasoning ability is good for size but not at
  the level of late-2025 models that did RL at scale.
- **Training was on Microsoft infrastructure** — cannot reproduce. We're consumers of the released checkpoint, not
  trainers of a 2B BitNet.
