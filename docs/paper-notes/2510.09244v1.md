# Fundamentals of Building Autonomous LLM Agents

## TL;DR

A short survey that decomposes "an LLM agent" into four subsystems — **perception**, **reasoning/planning**, **memory**,
**execution** — and walks through the implementation patterns at each layer. Perception covers text-only, multimodal
(VLM/MM-LLM), structured (HTML/A11y trees), and tool-mediated inputs. Reasoning covers task decomposition (sequential,
interleaved, DPPM), multi-plan generation and selection (CoT-SC, ToT, GoT, LLM-MCTS, RAP), reflection (Reflexion,
Self-Refine, anticipatory "Devil's Advocate"), and multi-agent specialization. Memory is split into long-term (embodied
weights, RAG, SQL) and short-term (context window) with a typology of what to store (experiences, procedures, knowledge,
user info). Execution covers tool/API calls, GUI control, code generation, and physical actuation. **The value of this
paper for Linus is not its individual claims — most are familiar — but the taxonomy. It is a checklist against which to
audit Linus's existing architecture decisions and notice gaps.**

## The problem (in plain language)

The agentic-LLM literature is sprawling, much of it published as method papers ("here is our particular agent"), and
there is no canonical decomposition that lets a designer reason about an agent's parts independently. Someone building
an LLM agent today has to read a hundred papers and infer the shared structure. The authors set out to write that shared
structure down — explicitly distinguishing **agents** from **workflows** (workflow = LLM following a designer's pre-set
plan; agent = LLM that generates and adapts its own plan in response to environmental feedback) and then describing the
four subsystems every agent has, regardless of vendor or framework.

The motivation for Linus is exactly that audit problem: Linus's `ARCHITECTURE.md` already names an orchestration layer
with router, tool registry, agent spawner, sandbox, audit log, and a memory pillar with four layers. Does that
decomposition match what the survey literature converges on? If not, where does it differ deliberately, and where is it
missing something that the survey treats as standard?

## What they propose

The four-subsystem decomposition. Each subsystem gets a chapter.

**Perception** — the agent's sensor stack — comes in four flavours: pure text (no perception layer needed); multimodal
(VLM or MM-LLM with modality encoder → input projector → LLM backbone, optionally augmented by VCoder depth/segmentation
maps or Set-of-Mark image annotations to fix object-counting failures); structured-data (parsing HTML or accessibility
trees to extract semantic GUI elements); and tool-based (web search, specialized APIs, sensors, code execution). Real
agents combine several; the paper's running example fuses Set-of-Mark visual annotations with an A11y tree of the same
screen.

**Reasoning/planning** is the longest chapter, in three layers. _Task decomposition_: decomposition-first (HuggingGPT,
Plan-and-Solve), interleaved (CoT, ReAct), or DPPM (Decompose, Plan in Parallel, Merge — fans subtasks to independent
agents to avoid cascading errors). _Multi-plan generation and selection_: CoT-SC majority vote, Tree-of-Thought with
BFS/DFS over thought nodes, Graph-of-Thoughts allowing aggregation, LLM-MCTS and RAP using Monte Carlo tree search with
the LLM as a heuristic policy. _Reflection_: Reflexion's actor/evaluator/self-reflection triple, Self-Refine, and
"Devil's Advocate" anticipatory reflection (the agent argues against its own plan _before_ execution). The chapter
closes with multi-agent specialization — Planning, Reflection, Error Handling, Memory Management, Action, Coding,
Constraint, and Security experts coordinating through messages.

**Memory** is bipartite. Long-term covers embodied (weights modified by fine-tuning), RAG (vector retrieval over
external corpora), and SQL (text-to-SQL over structured stores). Short-term is the context window plus
chunking/summarization. Orthogonal content typology: experiences (instruction + observation-action trajectories,
including failures), procedures (Agent Workflow Memory induces reusable workflows from experiences), knowledge (external
facts), user information. Flagged limitations: context-window ceiling and "memory duplication" — consolidating
near-duplicate records, e.g., five successful action sequences for the same subgoal compressed into one canonical plan.

**Execution** is brief: structured-JSON tool/function calling, multimodal action spaces (GUI clicks via VLMs, code
generation as a universal action, robotic control), and the engineering challenges of latency, cross-modal error
propagation, and state synchronization.

## Key results

This is a survey, so there are no experimental results. What the survey _does_ establish, by aggregating roughly seventy
papers, is which design patterns have crystallized into accepted practice and which are still in flux. **Crystallized**:
function/tool calling with JSON, RAG for long-term knowledge, CoT for single-pass reasoning, ReAct for tool-augmented
reasoning, Reflexion for post-hoc self-correction, multimodal perception via VLM/MM-LLM with vision encoders. **Active
development**: multi-plan selection algorithms (CoT-SC vs ToT vs GoT vs MCTS — no clear winner; ToT/GoT are
computationally heavy, MCTS adds wall-clock cost from many LLM calls), anticipatory reflection (Devil's Advocate is one
paper, not yet a pattern), procedural memory (AWM is the only well-named example). **Recognized open problems**:
hallucination in perception, context-window limits eating long-trajectory memories, memory duplication consolidation,
the precision gap between language understanding and grounded GUI/physical action, latency in multimodal pipelines. The
**agent-vs-workflow distinction** is the survey's most repeated framing point: a system following a designer's fixed
plan is a workflow, not an agent, even if it calls tools and has memory. Adaptive re-planning is the dividing line.

## What's reusable in Linus

**Architecture cross-validation.** The four-subsystem taxonomy maps onto Linus's orchestration layer as follows. The
router occupies the reasoning-system entry point, though Linus has not yet committed to an explicit _task-decomposition_
role. The tool registry is execution-system tool/API integration. The agent spawner is multi-agent specialization. The
sandbox is a Linus concern the survey does not address (a deliberate Linus addition). The audit log loosely supplies the
trajectory data Reflexion-style self-reflection needs. The memory pillar's four layers map cleanly onto the survey's
bipartite split — intra-step latent and scratchpad as short-term; episodic and semantic-knowledge as long-term, with
episodic ↔ "experiences" and semantic-knowledge ↔ RAG + SQL + "knowledge"/"user info". The one mismatch is the survey's
"procedures" content type (AWM-style induced workflows), which sits between episodic (raw experience) and semantic
(generalized rule) and has no clean home in the current taxonomy.

**Default Worker reasoning pattern.** The survey lays out CoT, CoT-SC, ToT, GoT, ReAct, Reflexion, MCTS-RAP in roughly
that order of complexity and cost. For Linus Phase 2 the reasonable default is **ReAct + Reflexion** — ReAct because
Workers will be tool-augmented from day one and the interleaved think-act-observe pattern is the cheapest way to drive
that; Reflexion because the audit log already supplies the trajectory data a self-reflection step needs. ToT/GoT/MCTS
belong to Phase 3 (parallel agents), where the multi-plan selection cost is amortized across Worker fan-out.

**Anticipatory reflection inside the sandbox.** "Devil's Advocate" — make the Worker argue against its own plan _before_
execution — is cheap to add to a ReAct loop and a natural fit for any plan touching the autonomy-tier boundary. Worth
considering as a default for actions that cross the allowlist.

**Multi-agent specialization as a Phase 3 template.** The survey's expert taxonomy (Planning, Reflection, Error
Handling, Memory Management, Action, Coding, Constraint, Security) is a concrete enumeration of roles Linus could assign
to specialized Workers. The Security "expert" is a placeholder in the survey; Linus's sandbox already plays that role at
the orchestration layer rather than at the agent layer — a deliberate and probably better choice.

**Memory duplication as a real concern.** The survey flags duplication consolidation (collapse five similar trajectories
into one canonical workflow) as a problem every persistent-memory agent eventually hits. Linus's episodic-memory design
should have an answer before Phase 3 trajectory volume makes the problem unavoidable.

**Perception as an explicit subsystem — probably not yet.** Linus currently treats perception implicitly: tool calls are
how Workers see the world. The survey makes perception its own subsystem and distinguishes structured-data (HTML/A11y)
from tool-based perception. For Phase 2 the implicit collapse is fine; the question reopens when GUI or multimodal
Workers arrive (Phase 5+).

## What's NOT applicable

Multimodal perception (VLM/MM-LLM, VCoder, Set-of-Mark) is not Phase 2 work — no vision encoder, no modality projector,
no GUI screenshot processing for the MVP. Bookmarked for Phase 5 or later. Robotic and physical-system control is
irrelevant; Linus runs on a MacBook. The survey's multi-agent treatment is shallow — it enumerates expert roles without
addressing the coordination protocols, message formats, or failure modes that actually determine whether a multi-agent
system works; Linus will look to the application papers in this wave (BioGuider, TradingAgents, Boiko/Gomes) for
concrete coordination patterns. The Embodied Memory section conflates fine-tuning with memory in a way that is
operationally misleading at Linus's scale: fine-tuning is a Phase 6 deliverable, not a runtime memory mechanism. The
"Possible Extensions" section ("humans as assistants to LLM agents, 10x productivity") is hand-wave futurism. Skip.

## Connections

The reasoning chapter connects directly to the memory-pillar CoT papers. CoT-SC and ToT both rest on the mechanism
analysed in [Chain-of-Thought as Polynomial Recursion (2305.15408v5)](2305.15408v5.md) and
[Expressive Power of Transformers with Chain of Thought (2310.07923v5)](2310.07923v5.md): the reasoning trace turns an
inherently TC0 single-pass transformer into a UTM-class computation by externalizing intermediate state. The survey
treats CoT as a prompting trick; the [memory-synthesis](../syntheses/memory-synthesis.md) establishes it as a
complexity-class escape. [Zero-Shot Reasoners (2205.11916v4)](2205.11916v4.md) is the empirical anchor for everything in
the survey's reasoning chapter.

The Memory chapter's bipartite split (long-term / short-term) is shallower than Linus's four-layer pillar but
compatible: short-term ⊆ {intra-step latent, scratchpad}, long-term ⊆ {episodic, semantic-knowledge}. The survey's
content typology (experiences / procedures / knowledge / user info) exposes one gap: the Linus taxonomy has no explicit
home for _procedures_ (induced workflows like AWM).

The agent-vs-workflow distinction connects to the
[skills-and-practices synthesis](../syntheses/skills-and-practices-synthesis.md) and to the ADRs distinguishing
harnesses from orchestration. Anthropic's "Building Effective Agents" (ref [3]) is the canonical statement; the survey
adopts and amplifies it. The other agentic-systems papers in this wave — Kosmos, Boiko/Gomes, BioGuider, Sketch2Sim,
TradingAgents — are all _applications_ of this survey's taxonomy. Reading them through the four-subsystem lens should
make their architectural choices easier to compare.

## Open questions for Dan

1. **Perception layer commitment.** Does Linus need an explicit "perception" abstraction in the orchestration layer,
   separate from the tool registry, or is "perception = tool calls returning structured observations" the right framing
   for the foreseeable future? The survey treats them as distinct subsystems; Linus could deliberately collapse them and
   revisit if/when GUI-control Workers arrive.
2. **Default Worker reasoning pattern for Phase 2.** Should the MVP commit to ReAct + Reflexion as the default Worker
   loop, with ToT/GoT/MCTS as opt-in for tasks the router classifies as benefiting from multi-plan selection? Or should
   the default be plain CoT and reflection added only after measurement justifies the cost?
3. **Procedural memory layer.** The survey's "procedures" content type (Agent Workflow Memory) does not map cleanly onto
   the four-layer Linus memory pillar. Should Linus add a fifth layer for induced workflows, fold them into episodic
   with a "generalized" flag, or fold them into semantic-knowledge with a "procedural" type? The decision affects what
   Phase 3 fan-out is allowed to learn from past runs.
4. **Anticipatory reflection in the sandbox.** Should "Devil's Advocate" anticipatory reflection be a mandatory
   checkpoint for any Worker action that crosses the autonomy-tier boundary (e.g., touches the network, writes outside
   the allowlist)? It is cheap and aligns with SAFETY.md's tiered-graduation philosophy.
5. **Memory duplication policy.** What is Linus's commitment for handling duplicate or near-duplicate episodic records
   before Phase 3 trajectory volume makes ad-hoc handling untenable? Options the literature offers: FIFO eviction,
   LLM-driven consolidation into canonical workflows, count-aggregation. Worth picking before the problem appears, not
   after.
6. _Partially resolved (DEC-0050, DEC-0051, see [answered-questions.md](../questions/answered-questions.md)): Role as a
   first-class type (capability_set, memory_access_tier, critic_eligible); AgentReport as typed inter-agent message;
   generalist-vs-specialist Worker granularity still open._
