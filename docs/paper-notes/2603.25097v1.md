# ElephantBroker: A Knowledge-Grounded Cognitive Runtime for Trustworthy AI Agents

## TL;DR

Lupascu & Lupascu's 2026 paper (Elephant Broker, Bucharest, Romania; arXiv preprint, ~30+ pages with 2,200+
referenced tests) describes **ElephantBroker** — an open-source cognitive runtime that pairs a Neo4j knowledge graph
with a Qdrant vector store via the Cognee SDK to provide durable, verifiable agent memory. The system implements a
**complete cognitive loop** with eight named mechanisms: (1) a **hybrid five-source retrieval pipeline** (structural
graph queries, lexical matching, semantic vector search, ego-net graph expansion, artifact retrieval) with
configurable per-source weights and actor-level isolation; (2) an **eleven-dimension competitive scoring engine**
evaluating turn relevance, session-goal relevance, global-goal relevance, recency, use history, confidence with
verification multipliers, evidence strength, novelty, redundancy, contradiction, and token cost, then assembling a
working set via budget-constrained greedy selection with a (1 − 1/e) submodularity guarantee; (3) a **four-state
evidence verification model** (Unverified → Self-Supported → Tool-Supported → Supervisor-Verified, plus Rejected)
where verification status directly multiplies scoring priority (0.5 → 0.7 → 0.9 → 1.0); (4) a **five-stage context
lifecycle** (bootstrap → ingest → assemble → compact → afterTurn) managing profile resolution, four-block prompt
assembly across two surfaces, and six-signal successful-use detection; (5) a **six-layer cheap-first guard pipeline**
enforcing safety through autonomy-domain classification, static-rule matching, semantic similarity, structural
validation, forced constraint reinjection, and optional LLM escalation; (6) a **nine-stage consolidation engine**
running during idle periods to cluster near-duplicates, canonicalize, strengthen useful facts, decay unused ones,
prune ineffective auto-recall, promote recurring episodic-to-semantic memories, refine procedures, identify
verification gaps, and recompute salience priors; (7) an **AI firewall** transforming advisory guard verdicts into
physically enforceable tool-call interception across three defense layers (agent hook, proxy guardrails, SDK hooks)
with six-tier safety scanning (regex, fine-tuned classifiers, prompt injection, jailbreak, PII, toxicity); and (8) a
**numeric authority model** with a four-tier permission hierarchy (regular actor → team lead → organization admin →
system admin) plus eleven configurable authority rules with matching-exempt semantics analogous to Unix privilege
escalation. Architecturally validated with 2,200+ tests spanning unit/integration/end-to-end. Three deployment tiers
(`memory-only` / `context-only` / `full`), five profile presets (personal-assistant / research / coding / managerial /
worker), multi-gateway isolation, a web management dashboard for human oversight, and OpenTelemetry/Prometheus
observability across 100+ custom metrics. **For Linus, ElephantBroker is the closest published direct-competitor /
cousin to Linus's Phase 2+ Layer C episodic memory substrate** (DEC-0029) — overlapping enough that the design
decisions ElephantBroker locks in are directly load-bearing for Linus's own substrate choices, and *different
enough* that the design points where ElephantBroker diverges from Linus's commitments are themselves the most
informative places to compare.

## The problem (in plain language)

Three interlocking problems frame the paper. First, **contemporary agent memory systems treat all stored facts as
equally trustworthy**. The standard pattern — flat vector store with cosine-similarity retrieval — has no mechanism
to distinguish a verified tool output (e.g., a database query result with provenance) from an unverified user
assertion ("I think the meeting is Tuesday") from a superseded earlier fact ("the meeting was Tuesday, now moved to
Thursday"). When the agent's context window fills with retrieved memories, the unverified hearsay competes on equal
footing with verified facts. In high-stakes domains — medical advisory agents that may retrieve contradictory drug
interactions; legal research agents that may surface unverified precedent as if it were verified — this equality is
not a minor inefficiency; it is a correctness failure with potential for material harm. The authors are explicit
about the medical and legal framing in §1.1 Motivation, citing the regulatory environment (EU AI Act high-risk
systems requirements, §2.5) as the load-bearing argument for why durable provenance + verification state matter.

Second, **vector similarity alone cannot capture relational structure**. The canonical example from §1.1: "Alice
manages the Berlin office" plus "the Berlin office handles European compliance" should be traversable as a two-hop
implicit relationship between Alice and European compliance. A pure vector store cannot do this — the cosine
similarity between the query "who handles European compliance" and either source fact in isolation will not surface
the chain. GraphRAG-style approaches address this but typically **construct graphs offline from static documents
once**, not incrementally from live conversational turns. The paper's framing is that persistent agent memory
specifically demands **incremental graph construction**: each conversational turn potentially adds facts, updates
confidence scores, attaches evidence, or supersedes prior assertions.

Third, **retrieval is typically goal-blind**. The agent's current task — what session-level goal is active, what
persistent goal the user has expressed — does not shape retrieval. Goal-relevant memories compete on equal footing
with tangentially-related-but-higher-similarity entries. The authors frame this as a context-budget allocation
problem: the LLM context window is scarce real estate, and the retrieval-to-context pipeline must compete-allocate
that real estate against multiple competing dimensions (relevance to the current turn, relevance to session goals,
relevance to persistent goals, recency, evidence strength, novelty, redundancy, contradiction, token cost). The
"who should occupy context" question turns out to be a constrained-submodular-optimization problem, not a
similarity-ranking problem.

These three failures compound in **multi-agent settings** where users, sub-agents, and supervisors contribute
knowledge with differing authority levels. A research agent that synthesizes findings from three sub-agent
investigations cannot trust those findings equally if one sub-agent had broader source access than another. The
authority model that governs who can assert what — and who can verify whose assertions — must be part of the memory
substrate itself, not bolted on at the application layer.

## What they propose

The architecture (Figure 1 of the paper) is a **four-layer stack** with explicit separation of concerns. **Layer A —
Agent Integration** is thin TypeScript plugins that integrate with the OpenClaw agent platform (an open-source
TypeScript LLM-agent runtime) and expose twenty-four tools across six groups: five memory operations (search, get,
store, forget, update), five session-goal management tools, four procedure-lifecycle tracking tools, two artifact
handling tools, two guard inspection tools, and six administrative operations (organization, team, actor, member,
profile-override, actor-merge). These plugins are HTTP clients with W3C trace propagation; all intelligence lives in
Layer B. **Layer B — Cognitive Runtime** is a Python service backed by FastAPI with **seventeen modules** wired
together by a dependency-injection container: actor registry, goal manager, memory store facade, working set
manager, context assembler, compaction engine, procedure engine, evidence and verification engine, guard engine,
tool artifact store, retrieval orchestrator, rerank orchestrator, scoring engine, consolidation engine, profile
registry, statistics engine, trace ledger, and scoring tuner. The DI container instantiates only modules enabled by
the selected business tier — `memory-only` / `context-only` / `full`. **Layer C — Knowledge Plane** is the Cognee SDK
providing unified graph + vector storage and search through a `DataPoint` abstraction with automated indexing
pipelines (the Cognify pipeline performs entity extraction, relationship extraction, and triplet embedding).
**Layer D — Infrastructure** comprises Neo4j for graph storage, Qdrant for dense-embedding vector search, Redis for
caching and session goals, and OpenTelemetry + Prometheus for observability across 100+ custom metrics (store
operations, retrieval performance, scoring pipeline latency, embedding cache hit rates, guard outcomes,
consolidation progress, and goal/procedure lifecycle events).

The runtime implements a **complete cognitive loop** — store, retrieve, score, compose, protect, learn — in
§3:

**Storage (§3.7 Memory Management).** The memory facade provides the primary interface for storing facts. Before
persisting a new fact, the facade performs vector similarity dedup against the closest existing entry; if cosine
similarity exceeds a configurable threshold (per profile), the store is skipped and a deduplication trace event is
emitted. Otherwise: storage **atomically creates a graph node via merge-by-ID** and indexes the fact's text in a
Qdrant collection with dense embeddings (the Cognee SDK's dual-write pattern guarantees consistency). After storage,
the facade creates provenance edges (authorship, actor-about, goal-serving) on a best-effort basis; edge-creation
failures are logged but never block the store. The Cognee enrichment pipeline then builds rich graph indexes for
multi-hop retrieval.

**The data model (§3.2).** The knowledge base is a **property graph** G = (N, E, φ) where nodes denote facts,
actors, goals, procedures, claims, evidence, and artifacts; edges denote typed directed relationships (authorship,
supersession, contradiction, goal-serving, ownership, parent-child); and each node and edge carries a property map.
The core entity is a **fact assertion** — a unit of agent memory with fields for content text, a **twelve-category
taxonomy** (identity, preference, event, decision, system, relationship, trait, project, general, constraint,
procedure reference, verification) plus custom category strings, a scope, a memory class, a confidence score,
source/target actors, goal associations, and goal relevance tags. Pydantic v2 schema classes define the full data
model and serve as the **single source of truth across all layers**.

The **scope system** is one of the more distinctive details: **eight scope values** govern visibility,
consolidation behavior, and promotion trajectory. GLOBAL (visible to all gateways in org, decays monthly, promotes
to Terminal) → ORGANIZATION (matching org_id, weekly decay, promotes to GLOBAL) → TEAM (matching team_id, weekly,
promotes to ORGANIZATION) → ACTOR (owner only, daily, promotes to TEAM) → SESSION (current session only, hourly,
promotes to ACTOR) → TASK (current task, hourly, promotes to SESSION) → SUBAGENT (child session, hourly, promotes
to SESSION) → ARTIFACT (tool output ref, daily, promotes to ACTOR). The promotion trajectory is what makes the
hierarchy load-bearing: a fact starts narrow (TASK or SUBAGENT scope), and if it survives consolidation across N
sessions, it gets promoted to a broader scope where it becomes visible to more retrieval contexts. Combined with
the decay cadence (which removes facts that aren't accessed), this is a structural answer to "what becomes durable
knowledge vs. what stays transient context."

The **five memory classes** (§3.2.2) follow the cognitive-science taxonomy: EPISODIC (events, decisions, decays
over time, promotable to SEMANTIC after 3+ cross-session recurrence, summarizable on compaction); SEMANTIC (durable
knowledge, decays slowly, terminal scope, mergeable on compaction); PROCEDURAL (workflows, SOPs, no decay, terminal
scope, preserved on compaction); POLICY (constraints, rules, no decay, terminal, always-injected); WORKING_MEMORY
(transient context, session-only lifetime, dropped at session end). Classification uses a fast rule table at ingest
time with LLM fallback for ambiguous cases, and a second-pass reclassification during consolidation promotes
recurring EPISODIC facts to SEMANTIC based on cross-session recurrence (three or more sessions).

**Retrieval (§3.3 Hybrid Five-Source Retrieval).** The retrieval-to-context pipeline is formalized as the function
composition f(α) = σ(ρ(φ(α))), where α is the query string, φ is the multi-source search, ρ is the four-stage
reranker, and σ is the eleven-dimension scoring and budget selection function. The output f(α) is a
budget-constrained working set. The composition is the paper's central engineering claim: it decomposes recall (φ),
precision (ρ), and value optimization (σ) into independently tunable stages.

The **search function φ concurrently dispatches to five sources**: *structural* (graph pattern-matching queries
against Neo4j, filtering by session key, actor, memory class, scope; default weight w_struct = 0.4); *keyword*
(BM25-like lexical matching through Cognee's lexical chunk search; w_kw = 0.3); *semantic* (Cognee's semantic chunk
search or direct vector queries; w_sem = 0.5, the highest single weight); *graph-expansion* (ego-net traversal of
matching entities via Cognee's graph-completion mode; w_graph = 0.2); *artifact* (tool output summaries in dedicated
vector collections; w_art = 0.5). All five sources dispatch concurrently. The orchestrator multiplies each
candidate's raw score by its source weight and merges results by fact ID, retaining the maximum weighted score when
a candidate appears in multiple sources. The merged list is deduplicated, sorted descending, and capped at a
configurable maximum. The pipeline enforces a **five-level graceful degradation hierarchy**: source-skip (coarsest
— STRICT isolation disables keyword search since it leaks across scope boundaries), method-level fallback (Cognee
search falls back to direct vector queries), pipeline-level degradation (reranker failure returns input list
unchanged), operation-level degradation (cache failure proceeds without caching), edge-level degradation (graph
edge creation failure logs a warning without blocking).

**Reranking (§3.4 Four-Stage Reranking).** Before scoring, retrieved candidates pass through a four-stage refinement
that progressively spends more compute. Stage 1 is a cheap blended score (50% token overlap + 50% original retrieval
score), discarding clearly irrelevant matches without embedding computation. Stage 2 performs semantic reranking via
cosine similarity between query and candidate embeddings, blended with retrieval score at a configurable α. Stage 3
invokes a configurable external cross-encoder via HTTP, with normalized sigmoid output and graceful timeout fallback
that returns the semantic-rerank order plus a degraded-operation trace event. Stage 4 merges near-duplicate
candidates via union-find at cosine similarity > 0.95, retaining the highest-scored cluster representative and
union-ing relations onto it.

**Scoring (§3.5 Eleven-Dimension Competitive Scoring).** Pass 1 computes nine dimensions independently: *turn
relevance* (cosine to current turn embedding); *session goal relevance* (direct match = 1.0, indirect = 0.7,
fallback to maximum cosine over session goal embeddings, with parent-chain walk-up awarding 0.7×child_relevance for
hierarchical goal structures); *global goal relevance* (max cosine to persistent goals); *recency* via exponential
decay r(t) = exp(−ln 2 / t_{1/2} × Δt) where the half-life is per-profile (12 hours for fast-paced workers up to
720 hours for personal assistants); *successful use prior* (ratio of successful uses to total uses); *confidence*
(raw confidence × verification multiplier m_v from the four-state evidence model — Unverified m_v=0.5, Self-Supported
0.7, Tool-Supported 0.9, Supervisor-Verified 1.0); *evidence strength* (fraction of evidence references relative to
per-profile max, ranging 2 for coding to 5 for research); *novelty* (binary: 0 if fact ID in compaction state, 1
otherwise); *cost penalty* (candidate's token count divided by remaining budget, growing as budget fills). Pass 2
computes two interaction-dependent dimensions during the greedy selection loop: *redundancy penalty* (max cosine
similarity to already-selected items; penalty −0.7 if exceeds per-profile threshold) and *contradiction penalty* (a
two-layer detection — first checks explicit graph edges, distinguishing supersession 1.0 from contradiction 0.9
edges with bidirectional check; second checks semantic contradiction via high-similarity-plus-large-confidence-gap,
penalty 0.7; final negative weight bound −1.0). Budget-constrained selection then sorts remaining candidates by
Pass 1 weighted sum descending, processes each greedily, checks budget fit, recomputes Pass 2 penalties against the
remaining budget, and includes the candidate if final score exceeds zero. Mandatory items (constraints, blocked
goals, evidence-required procedures) are pre-allocated unconditionally — safety requirements take precedence over
budget even when it causes overallocation. The greedy submodular approach yields a **(1 − 1/e) ≈ 0.63 approximation
guarantee** for the eleven-dimension objective.

**Context Assembly (§3.8–3.14, the five-stage lifecycle).** A *bootstrap* stage resolves profile and session
identity, *ingest* runs the turn-extraction pipeline (which produces typed structured output — see "What's reusable
in Linus" — including supersession indices, contradiction indices, and goal relevance arrays with strength
annotations), *assemble* renders the working set into a four-block prompt across two surfaces (system surface
carries policy and high-confidence facts; user surface carries scoped working memory and recent turns), *compact*
fires near the budget threshold, *afterTurn* hooks delegate workflows and detect six successful-use signals (LLM
inspection on assistant message, repeated reference to a fact, fact appears in tool call, fact appears in user
follow-up, fact survives N turns without contradiction, fact promoted to longer-lived scope).

**Safety (§3.15+, the six-layer cheap-first guard pipeline and AI firewall).** Guards enforce safety through six
ordered layers: (1) **autonomy domain classification** — every tool call is classified into a domain (filesystem,
network, finance, personal-data, code-execution, etc.); per-profile autonomy rules either permit, require
confirmation, or deny by domain; (2) **static rule matching** — exact-match patterns flag forbidden actions (e.g.,
`rm -rf /` or `DROP TABLE`); (3) **semantic similarity** — embedded similarity to known-malicious patterns from
the static-rule corpus; (4) **structural validation** — schema checks on tool arguments and outputs; (5) **forced
constraint reinjection** — for tool calls in sensitive domains, the runtime injects all active POLICY-class facts
(constraints, red-lines) into the prompt and re-prompts the LLM to verify the call complies; this addresses the
"agent has forgotten its constraints by turn 30" problem; (6) **optional LLM escalation** — for novel or ambiguous
cases, escalate to a larger LLM for adversarial inspection. The pipeline is *cheap-first*: layers 1–5 are local and
sub-millisecond; layer 6 is escalated only when cheaper layers cannot decide. **Procedure-bound red-line
constraints** dynamically compose the active rule set based on which workflows are running and *near-miss tracking*
auto-tightens strictness under boundary-probing behavior.

The **AI Firewall (§3.16)** transforms advisory guard verdicts into **physically enforceable tool-call
interception** across three defense layers — agent hook (preflight in Layer A's TypeScript plugins), proxy
guardrails (middleware between Layer B and external APIs), SDK hooks (instrumentation of the Cognee SDK and tool
implementations themselves). The firewall also performs **six-tier safety scanning** on inbound content from tool
outputs and user messages: (a) regex patterns for low-cost detection of obvious threats; (b) fine-tuned
classification models for prompt injection detection; (c) jailbreak attempt detection; (d) PII leakage detection;
(e) toxicity classification; (f) escalation tier for novel threats. Critically, **the firewall blocks adversarial
content from being stored as trusted facts at the ingest boundary** — addressing the indirect prompt-injection
threat where adversarial instructions embedded in retrieved context could otherwise persist into agent memory and
influence future turns.

**Consolidation (§3.15, the nine-stage engine).** Runs during idle periods. Stage 1: cluster near-duplicates by
cosine similarity exceeding per-class threshold. Stage 2: canonicalize cluster members onto a single representative.
Stage 3: strengthen useful facts (boost confidence based on successful-use signals). Stage 4: decay unused (apply
class-specific decay function based on time-since-last-recall). Stage 5: prune ineffective auto-recall (remove
facts that consistently fail to make it into the working set). Stage 6: promote recurring EPISODIC to SEMANTIC
based on cross-session recurrence. Stage 7: refine procedures from cross-session patterns. Stage 8: identify
verification gaps (high-confidence facts with no evidence references). Stage 9: recompute salience priors. The
consolidation engine writes its state to Redis so the novelty dimension in scoring can check whether a candidate's
fact ID is in the compaction state set.

**Authority (§3.2.3, the authority model).** Twelve actor types span human roles (coordinator, operator, external),
agent roles (manager, worker, reviewer, supervisor, peer, service, external), and organizational units (teams,
organizations). Actors form authority chains through twelve relationship types (`delegates_to`, `supervises`,
`reports_to`, `collaborates_with`, `trusts`, `blocks`, `owns_goal`, `owns_artifact`, `requested_by`, `approved_by`,
`verified_by`, `prohibited_by`). Trust levels (0–1) on actor references affect evidence weighting: a
supervisor-verified claim from a high-trust actor produces a stronger confidence signal than the same claim from a
low-trust source. Four authority tiers (regular actor → team lead → organization admin → system admin) plus eleven
configurable authority rules with matching-exempt semantics produce a Unix-privilege-escalation-analogous hierarchy.
Organization and team entities are first-class graph nodes; platform-qualified actor handles produce stable identity
across sessions; scope-aware goal visibility enforces who can see what. A web management dashboard provides human
oversight per EU AI Act Article 14.

**Deployment.** The runtime supports three tiers: *memory-only* (actors, goals, memory, retrieval, evidence,
procedures — six modules); *context-only* (actors, goals, working set, assembly, compaction, guards — six modules);
*full* (all seventeen modules). The DI container instantiates only tier-enabled modules, with granular feature
flags (subagent context packets, consolidation, guard preflight checks) beyond module-level enablement. Five
profile presets (`personal-assistant` uses STRICT/SESSION_KEY isolation for privacy; `research` uses NONE/GLOBAL
isolation for maximum recall; `coding`, `managerial`, `worker` use LOOSE/SESSION_KEY) configure per-domain
defaults. The three deployment tiers, profile inheritance chain (base → preset → organization override → personal
override), and dependency-injection container together let a lightweight memory-only deployment skip the cost of
context assembly and guard evaluation, while a full deployment gets the complete cognitive loop.

**Validation.** Architectural validation through a comprehensive **test suite of 2,200+ tests** spanning unit,
integration, and end-to-end levels confirms subsystem correctness. The paper does not report performance benchmarks
(latency / throughput / accuracy on standard benchmarks) — this is an **architecture and engineering paper**, not
an empirical comparison paper. The comparison table (Table 1) sets ElephantBroker's capability surface against
Zep, Mem0, MemGPT, Lossless-Claw, QMD, and Supermemory, with ElephantBroker uniquely checking 18 of 18 capabilities
(knowledge graph, vector store, evidence tracking, goal-aware scoring, working set budget, multi-agent actors,
cross-session memory, temporal reasoning, session goals, procedure tracking, guard pipeline, consolidation,
authority model, enforceable guards, AI safety scanning, management dashboard, etc.). The implicit comparison
position: Zep optimizes for retrieval accuracy and latency (state-of-the-art on Deep Memory Retrieval and
LongMemEval benchmarks per §2.4); ElephantBroker extends retrieval with evidence verification, multi-dimensional
scoring with budget competition, goal-aware context assembly, a guard pipeline, and consolidation-based learning.

## Key results

The paper does not report empirical results in the standard ML sense — there are no benchmark numbers comparing
ElephantBroker to baselines on a public dataset. The validation is **architectural**: 2,200+ tests across the
seventeen modules, the four-layer stack, and the integration boundaries with Cognee, Neo4j, and Qdrant. Three
classes of "results" are reported:

**Architectural validation via comprehensive test coverage.** The 2,200+ test suite spans unit tests (per-module
behavior), integration tests (cross-module interactions, e.g., scoring engine consuming retrieval orchestrator
output), and end-to-end tests (full cognitive loop from store → retrieve → score → assemble → afterTurn). The paper
treats this as evidence of subsystem correctness — every named mechanism in §3 has corresponding tests, and the
test pass rate is the proxy for "the architecture is internally consistent and works as specified." This is a
weaker claim than empirical benchmarks against baselines but a stronger claim than a typical architecture paper
that ships code without test commitments. For Linus, the relevance is that **a memory substrate with 17 modules
and a 4-layer stack is a substantial test surface**, and the per-module testability discipline ElephantBroker
demonstrates is a useful reference for what Phase 2 Linus's memory implementation should commit to.

**Capability matrix comparison (Table 1).** ElephantBroker is compared against six contemporary agent-memory
systems (Zep, Mem0, MemGPT, Lossless-Claw, QMD, Supermemory) across 18 capability axes. ElephantBroker checks all
18; the next best (Zep) checks 5; Mem0 checks 4 (with three partial); MemGPT checks 2 (working set budget,
cross-session). The argument is **not** "we have the best of everything" but rather "we are explicitly the
super-set system": every prior agent-memory approach optimizes a subset, and ElephantBroker integrates all of
them into a coherent runtime. This is a defensible framing for the paper's contribution — it is an
*integration architecture* paper, not a *novel mechanism* paper.

**OpenTelemetry + Prometheus observability over 100+ metrics.** The runtime ships with full observability:
per-store operation metrics, retrieval performance (per-source latency, per-source recall, fusion overhead),
scoring pipeline latency (Pass 1, Pass 2, greedy loop), embedding cache hit rates, guard outcomes (per-layer pass /
block / escalate rates), consolidation progress (per-stage runtime, candidate counts), goal lifecycle events
(creation, status changes, completion), procedure lifecycle events. This is a **deployability** result: the system
is designed to run in production with measurable behavior, not as a research artifact. The 100+ metric surface is
the most thorough observability layer in any of the cloned-corpus agent-memory systems.

The paper explicitly closes (§6 Conclusion, paraphrased from the abstract): "Architectural validation through a
comprehensive test suite of over 2,200 tests spanning unit, integration, and end-to-end levels confirms subsystem
correctness. The modular design supports three deployment tiers, five profile presets with inheritance,
multi-gateway isolation, and a management dashboard for human oversight, enabling configurations from lightweight
memory-only agents to full cognitive runtimes with enterprise-grade safety and auditability." The proof-of-concept
posture is honest — the paper is a *design and engineering* artifact, not a *measurement* artifact, and the next
natural step for ElephantBroker would be empirical benchmarks against Zep / Mem0 / MemGPT on Deep Memory Retrieval,
LongMemEval, and any forthcoming agent-memory benchmark suite. The implementation is open-source (referenced in the
abstract and §3 Conclusion as a public release); Linus's evaluation can in principle bench it directly once the
substrate decisions are finalized.

## What's reusable in Linus

ElephantBroker is the most direct and informative cousin to Linus's Phase 2+ Layer C episodic memory substrate
(DEC-0029) in the cloned-corpus collection. The architecture overlaps substantially with Linus's commitments —
the major divergence is the substrate choice (Linus = SQLite + content hashes + git per DEC-0029; ElephantBroker =
Neo4j + Qdrant + Cognee SDK). That divergence is itself informative: the design points where ElephantBroker locks
in different choices are precisely where Linus should compare alternatives.

**Phase 2 — four-state evidence verification model as the canonical confidence-with-provenance pattern (DEC-0029
extension candidate).** ElephantBroker's verification state machine (Unverified m_v=0.5 → Self-Supported 0.7 →
Tool-Supported 0.9 → Supervisor-Verified 1.0, plus Rejected as terminal) directly maps onto Linus's planned
provenance discipline for Layer C episodic memory. The Linus equivalent: a fact stored in `~/.linus/episodic.db`
gets a verification state that multiplies its retrieval-time confidence. Unverified facts are user assertions
without supporting evidence; Self-Supported facts have at least one cross-reference to another fact; Tool-Supported
facts have at least one trace event linking them to a tool output that produced them; Supervisor-Verified facts
have been explicitly sign-off by Maestro (Dan, in the single-user case). This is a directly portable pattern that
sharpens Linus's Layer C beyond the current "content hash + git history" provenance into a **graded trust signal**
that scoring can use. Worth a Phase 2 ADR; the multipliers can be lifted directly (0.5 / 0.7 / 0.9 / 1.0) as a
starting point, calibrated against Linus's task suite as data accumulates. The MemGPT paper-note's "function-call
memory API" reusability (per [Letta-2310.08560](Letta-2310.08560.md) §"What's reusable in Linus") sets the *verb*
surface for memory operations; ElephantBroker's verification-state model sets the **quality dimension** on the
stored objects themselves. Both belong in the Phase 2 spec.

**Phase 2 — eleven-dimension scoring as the canonical context-window-budget allocation pattern (DEC-0032 extension).**
Linus's DEC-0032 in-context-window-cap policy commits to a 16K-token default cap and audit-logged bypass. The
mechanism that decides *which 16K* is currently implicit. ElephantBroker's eleven-dimension competitive scoring
plus budget-constrained greedy selection with (1 − 1/e) submodularity guarantee is the most thorough published
reference for what that decision should look like. The dimensions translate to Linus's setting directly:
*turn relevance* (cosine to current dispatch query); *session goal relevance* (per the dispatch struct's
`memory_mode` field per DEC-0031, the session-scoped task); *global goal relevance* (the long-running project
Linus is operating on); *recency* (with per-profile half-life — for Linus, a worker-profile half-life of 12-24
hours, a research-profile half-life of 168 hours seems calibrated); *successful use prior* (Linus's audit log
already records per-fact uses; computing the ratio is straightforward); *confidence-with-verification-multiplier*
(adopt the four-state model above); *evidence strength* (fraction of references to per-profile max); *novelty*
(binary against compaction state); *cost penalty* (token-count proportional). Pass 2 *redundancy penalty* (cosine
to already-selected) and *contradiction penalty* (graph-edge-first then semantic-fallback). The Phase 2+ Linus
dispatch layer should adopt this scoring framework as the canonical context-selection mechanism, with Linus-specific
profile presets (Worker / Maestro / personal-assistant / research) tuning the recency half-life, evidence-strength
max, redundancy threshold, contradiction sensitivity. The (1 − 1/e) submodularity guarantee is a theoretical
backstop worth preserving — it provides a worst-case bound on how far greedy selection can deviate from optimal,
which is a useful property when Worker dispatch behavior is auditable.

**Phase 3 — eight-value scope system as a memory-promotion trajectory pattern (DEC-0029 extension).** ElephantBroker's
scope hierarchy (GLOBAL → ORGANIZATION → TEAM → ACTOR → SESSION → TASK / SUBAGENT / ARTIFACT) with promotion
trajectory and per-scope decay cadence is a structural answer to the question Linus's Phase 3 multi-agent spawner
will need to answer: when a Worker (or sub-Worker) writes a fact during a task, what scope does it get, and under
what conditions does it survive into a more durable scope? The ElephantBroker model says: TASK starts narrow (hourly
decay, promotion to SESSION on success), SESSION promotes to ACTOR after one session-survival, ACTOR promotes to
TEAM after multi-actor visibility, TEAM to ORGANIZATION to GLOBAL via increasingly slow promotion criteria.
For Linus's setting (single-user, multi-Worker, single-organization-of-one), the simplified hierarchy is plausibly
**TASK → SUBAGENT → SESSION → ACTOR (=Dan) → SEMANTIC (durable knowledge)**, with the EPISODIC-to-SEMANTIC
promotion criterion (3+ cross-session recurrence) as a load-bearing rule. The pattern is directly applicable to
Layer C in Linus's memory model: a fact starts as scratchpad-scoped during a Worker turn, gets promoted to session
on Worker completion, gets promoted to actor on session close, and gets promoted to semantic on 3+ recurrence.
The five-memory-class taxonomy (EPISODIC / SEMANTIC / PROCEDURAL / POLICY / WORKING_MEMORY) plus its per-class
decay-and-compaction policy is the right *cognitive-science-grounded* substrate for the Phase 2+ memory
implementation. Worth a Phase 2 ADR that names the Linus-side simplified hierarchy and the per-class policy.

**Phase 3 — six-layer cheap-first guard pipeline as the canonical safety-enforcement pattern (SAFETY.md extension).**
SAFETY.md commits to autonomy-tier graduation and a sandbox policy; the *mechanism* that enforces these decisions
is currently per-tool-call permission gating. ElephantBroker's six-layer guard pipeline (autonomy-domain
classification → static-rule matching → semantic similarity → structural validation → forced constraint reinjection
→ optional LLM escalation) is the most thorough published reference for what cheap-first defense in depth looks
like for an LLM-tool-call surface. Each layer is sub-millisecond and locally evaluable except layer 6 (LLM
escalation). The **forced constraint reinjection** layer is the most non-obvious and is the answer to a real
problem: by turn 30 of a long conversation, the agent has forgotten the system-prompt-level constraints; reinjecting
them right before a sensitive tool call (filesystem write, network request, code execution) is the only practical
mechanism for re-anchoring the agent's behavior. Linus's Phase 3+ guard implementation should adopt the six-layer
pipeline as the spec, with Layer 6 LLM escalation routing to a Linus-side larger Worker (Qwen3-32B or hosted
Claude when the threat genuinely demands it). The procedure-bound red-line constraints and near-miss tracking
(auto-tightening on boundary-probing behavior) are additional refinements worth incorporating; the latter is
particularly relevant for the autonomy-tier graduation discipline since boundary-probing is exactly the signal
SAFETY.md is trying to detect.

**Phase 3 — AI firewall as the canonical tool-call interception pattern (SAFETY.md extension).** ElephantBroker's
firewall pattern — advisory guard verdicts become physically enforceable tool-call interception across three defense
layers (agent hook, proxy guardrails, SDK hooks) — generalizes to Linus's setting as a multi-point enforcement
discipline: the harness-side hook (analogous to pi's `beforeToolCall` per [pi.md](pi.md) §3), the orchestration-side
guardrail (Linus's dispatch layer evaluates each Worker tool call against the active rule set), and the SDK-side
instrumentation (Linus's MCP servers per DEC-0045 enforce their own argument validation). The three-layer pattern
is **defense in depth**: a compromised harness layer doesn't bypass the orchestration guard; a compromised
orchestration guard doesn't bypass SDK-level instrumentation. For Linus's Phase 3+ skill surface (and Phase 7+
external skill manifests), this is the right enforcement architecture. The six-tier safety scanning (regex →
classifiers → prompt injection → jailbreak → PII → toxicity) is independently usable — particularly the **prompt
injection scanning at the ingest boundary** (blocking adversarial content from being stored as trusted facts in
Layer C). This addresses the indirect prompt-injection threat that any KnowledgeBase-style RAG system exposes Linus
to, where adversarial content embedded in a retrieved document could otherwise persist into agent memory and
influence future turns. Worth a Phase 7+ ADR; the threat model is concrete and the mitigation pattern is portable.

**Phase 2+ — four-layer architecture stack as a reference partitioning for Linus's memory substrate.** ElephantBroker's
A/B/C/D layering (TypeScript plugins / Python runtime / Cognee SDK / Neo4j+Qdrant+Redis infrastructure) is a clean
example of "separate the policy from the mechanism from the storage." For Linus, the analogous layering would be:
Layer A (harness-side memory tool calls — `linus.memory.*` MCP tools, the verb surface); Layer B (Linus
orchestration runtime — the Python policy implementing scoring, retrieval orchestration, guard pipeline,
consolidation); Layer C (Linus knowledge plane — the implementation of SQLite + content hashes + git per DEC-0029,
possibly augmented with a graph-shaped sidecar for relational queries); Layer D (Linus infrastructure — the
SQLite files themselves, the git history, the embedding index, audit log). The cleanly-layered partitioning is
what makes ElephantBroker's seventeen-module decomposition tractable; for Linus, the equivalent partitioning would
limit the Phase 2+ memory implementation to a smaller set of modules (estimate 6–10 for Linus's reduced scope and
single-user setting) but the **principle of layered separation** is directly portable. The dependency-injection
container plus three deployment tiers (memory-only / context-only / full) is the substrate that lets Linus's
Phase 2 ship a `memory-only` tier first (Layer C SQLite substrate only, no scoring, no guard pipeline) and
iteratively add the context-assembly and guard layers in Phase 3+. Worth committing to the tiered-deployment
discipline as a Phase 2 architectural convention.

**Phase 2+ — Cognee SDK (Layer C) as a candidate evaluation, with DEC-0029 substrate caveat.** Cognee is the SDK
that provides unified graph + vector storage with the `DataPoint` abstraction and automated Cognify enrichment
pipeline (entity extraction → relationship extraction → triplet embedding). It is a Python library, MIT-licensed,
designed to abstract over graph + vector backends. Linus's DEC-0029 commits to SQLite + content hashes + git as
the Layer C substrate, which is **deliberately not** graph-oriented. Two stances are possible: (a) Linus's substrate
is intentionally simpler than Cognee — accept the trade-off that multi-hop graph traversal is harder, in exchange
for the auditability and durability that git + content-hashes provide; (b) Linus's substrate evolves toward a
graph-augmented form in Phase 3+ if the scoring pipeline empirically demonstrates that graph-traversal queries
materially improve Worker performance. The ElephantBroker paper is the strongest argument so far for stance (b) —
the multi-hop traversal use case ("Alice manages Berlin; Berlin handles European compliance → Alice handles
European compliance") is concrete and the Cognee SDK is the most polished published reference for the abstraction.
For Phase 2, stance (a) is the safer commitment per DEC-0029; for Phase 3+, the evaluation of "does Linus's Layer
C need graph-traversal augmentation?" should explicitly reference ElephantBroker and Cognee as the candidate
architecture. Worth a Phase 3+ open question for the memory-architecture spec.

**Phase 2+ — 100+ OpenTelemetry observability metrics as a Linus audit-log schema reference.** Linus's audit log is
committed as append-only JSONL at `~/.linus/audit.jsonl` per ARCHITECTURE.md. ElephantBroker's 100+ Prometheus
metrics surface (per-store operations, retrieval performance by source, scoring pipeline latency, embedding cache
hit rates, guard outcomes per layer, consolidation per-stage runtime, goal/procedure lifecycle events) is the most
thorough published reference for what an agent-memory observability layer should cover. For Linus's audit log, the
equivalent fields are: per-call dispatch metadata (`memory_mode`, `cot_budget` per DEC-0031), per-call retrieval
latency, scoring pipeline runtime, per-Worker tool-call counts, guard outcomes (which layer blocked / escalated /
passed), context-cap-bypass events (per DEC-0032), consolidation events when they fire. The PostHog/OTel
distinction in the [goose repo-note](goose.md) §4 plus the ElephantBroker 100+-metric surface together set the
upper bound on what audit-log granularity could look like; Linus's v0 should pick a substantial subset (estimate
20-40 distinct event types) and grow as needed. Worth a Phase 2 audit-log-schema spec.

## What's NOT applicable / hype filter

**The Neo4j+Qdrant+Cognee substrate is heavier than Linus needs (DEC-0029 commits otherwise).** ElephantBroker's
Layer D infrastructure assumes Neo4j (running as a separate service, typically Docker-containerized — though the
paper notes Cognee can use alternate backends), Qdrant (separate service for vector search), Redis (caching),
plus the Cognee SDK as the unifying abstraction. This is **substantial infrastructure** for a single-user system:
Neo4j alone is 800MB+ of JVM and graph storage with its own configuration burden; Qdrant adds another service to
maintain; Redis is lightweight but is another moving part. Linus's DEC-0029 substrate (SQLite + content hashes +
git) is **deliberately lighter** and the difference is load-bearing: Linus is single-user, runs locally on a
MacBook Pro M1 Max, must operate offline, and the file-based git substrate gives auditability and reproducibility
that a Neo4j+Qdrant+Redis stack does not (the latter requires explicit backup discipline; git is the backup). The
ElephantBroker substrate is the right choice for a multi-tenant SaaS deployment with EU AI Act compliance
requirements; it is the wrong choice for a single-user local-first system. Linus's substrate choice is correct;
ElephantBroker is the design reference for *what to build on top of* the substrate, not a model for the substrate
itself.

**The multi-tenant authority model is over-scoped for Linus's single-user setting.** ElephantBroker's twelve actor
types, twelve relationship types, four authority tiers, and eleven configurable authority rules are the right
shape for a multi-tenant enterprise deployment with multiple organizations, teams, and users. Linus is single-user
(Dan); the equivalent authority surface is a small subset: Dan as actor, Workers as delegated sub-actors,
Maestro as supervisor. The trust-level mechanism (0–1 on actor references) is plausibly worth keeping for the
sub-Worker case (high-trust Workers like Qwen3-coder vs. lower-trust general Workers), but the rest of the
authority surface is over-scoped. The pattern is inspiration only; the implementation is too heavy for the use
case.

**The 17-module FastAPI service architecture is heavier than Linus's Python orchestration needs.** ElephantBroker's
Layer B runtime is 17 modules wired by a DI container, exposing 24 tools via 6 administrative groups. For a
single-user Linus system, the equivalent module count is plausibly 6-10 (memory facade, scoring engine, guard
pipeline, retrieval orchestrator, consolidation engine, context assembler, plus 1-3 infrastructure modules). The
DI container with three deployment tiers is a sensible engineering discipline at any scale; the 17-module
decomposition is over-engineered for Linus's scope. Lift the partitioning principle, not the module count.

**The web-management dashboard is over-scoped for single-user operation.** ElephantBroker's management dashboard
provides a memory browser, guard-event review, cross-session approval queue, and 100+ Prometheus metrics
visualization. This is the right surface for an enterprise deployment with human supervisors reviewing agent
behavior; for Linus, the equivalent is **Maestro himself reviews the audit log**, with no dedicated dashboard.
The pattern of "human-in-the-loop oversight via a dashboard" is correct in principle; the implementation is
infrastructure Linus does not need.

**The EU AI Act Article 14 compliance framing is not Linus's threat model.** ElephantBroker's evidence verification,
trace ledger, GDPR-compliant forget operation, and "effective oversight by natural persons" dashboard are explicitly
positioned as EU AI Act high-risk-system compliance mechanisms (§2.5 Trustworthy AI and Verification). Linus is
single-user, private, and not deployed against EU regulatory scrutiny. The mechanisms (evidence tracking, audit
trail, deletion semantics, oversight) are independently useful for Linus's own quality discipline, but the
compliance framing is not. Don't import the regulatory language into Linus's ADRs.

**The TypeScript-side OpenClaw plugin layer assumes a TypeScript harness.** ElephantBroker's Layer A is twenty-four
TypeScript tools that integrate with the OpenClaw agent platform (a TypeScript LLM-agent runtime). Linus's harness
ecosystem (openclaw, claw-code-local, future Rust harnesses, future Python tooling) is multi-language per
DEC-0027. The TypeScript-specific plugin design is inspiration only — the equivalent in Linus is the MCP server
contract (DEC-0045), which is language-agnostic. The pattern of "thin harness-side plugins that proxy to the
orchestration runtime via HTTP" is correct; the TypeScript specifics are not.

**Cognee SDK as a black-box dependency.** The paper presents Cognee as a battle-tested abstraction (per §2.1
Knowledge Graphs for Language Model Grounding, Luo et al. cited at [22]), but the SDK is a third-party dependency
maintained outside the ElephantBroker codebase. For Linus's substrate decisions (DEC-0029), adopting Cognee would
mean adopting both the SDK and the upstream maintenance burden. The unified `DataPoint` abstraction is elegant but
the cost — a non-trivial dependency on a Python SDK whose roadmap Linus does not control — is at odds with the
"public APIs only, audited supply chain" discipline (DEC-0024, CLAUDE.md §"Public APIs only"). Worth considering
Cognee as a reference architecture rather than as an adoption target.

**The 2,200+ test count is not a quality signal in itself.** A test count is a coarse proxy for surface-area
coverage; it does not measure coverage *per module* or *per code path*. ElephantBroker's claim "2,200+ tests
confirm subsystem correctness" is plausible but Linus should be skeptical of test-count claims in general — the
right metric is "every named mechanism in §3 has at least one test in the suite, and every branch in the scoring
pipeline + guard pipeline is exercised." Test count is downstream of test quality. Lift the discipline (tests per
mechanism), not the count.
