# Linus — Roadmap

This roadmap is written in phases with target durations, not fixed deadlines. Blitzscaling
mindset: ship rough, learn, iterate. The Algorithm check applies at every phase boundary —
question whether the next phase's scope can be deleted or simplified.

## Guiding targets

- **Time targets are aggressive-but-plausible.** Slip without guilt; meeting them means
  the Algorithm is working.
- **Phase boundaries are porous.** Parallel work across phases is expected when it
  doesn't introduce dependency inversions.
- **Every phase produces a demo-able outcome.** If a phase doesn't end in something Dan
  can point at and say "this works," the phase is too big.

## Phase 1 — Recon and Baselines

**Target duration:** 2 weeks (in progress; ~halfway through as of 2026-05-01)

**Status snapshot (2026-05-01):**

- **1a — Repo synthesis notes:** largely complete. Twelve original repo notes written.
  Likely scope expansion to add a small set of new repos surfaced in the LLM wiki and
  skills syntheses (omega-memory, keppi, agentmemory, openaugi, fastmcp, Task Master AI,
  claude-squad, etc.) — to be folded in either as 1a addenda or as a new 1f bucket
  (decided in this planning refresh).
- **1b — pmetal deep evaluation:** in progress. Built from source successfully; smoke
  tests pass; initial impressions strongly positive. LoRA trial, serve trial, and
  comparative benchmark vs. Ollama still to run before the ADR is written.
- **1c, 1d, 1e:** not yet started.

**Goal:** know what each repo is, know how our worker models perform on real tasks, and
have the first Maestro/Worker loop on record.

**Deliverables, in dependency order:**

**1a — Repo synthesis notes** in `docs/repo-notes/`. One page per cloned repo:

- Purpose and scope
- Architecture summary (if code) or content overview (if not)
- What's reusable in Linus
- What's inspiration only
- What's incompatible or out of scope
- Decision: integrate / study / ignore

Fan out across multiple sessions; this is embarrassingly parallel. Each note is ~500
words, generated by a Worker against a spec written by the Maestro. Expected repos:
BitNet, Bonsai-demo, ANE, flash-moe, mlx-flash, pmetal, claw-code, claw-code-local,
autoresearch, openclaw, cline, project-nomad. Twelve notes total. The claw-code and
claw-code-local notes should explicitly compare the two — upstream is inspiration only
(Anthropic-API-bound), the fork adds an Ollama backend and is the more practical lead
for a future local CLI harness.

**1b — pmetal deep evaluation.** This is the gate decision for Phase 2's serving layer.

- Build pmetal from source (`cargo build --release` in `repos/pmetal/`)
- Smoke test: `pmetal info`, `pmetal tui`
- LoRA trial: train a tiny LoRA on a toy dataset (e.g., 100 ShareGPT pairs) against
  Qwen3-0.6B. Verify adapter saves and loads.
- Serve trial: `pmetal serve --model Qwen/Qwen3-0.6B --port 8080`. Hit the endpoint with
  curl; verify OpenAI compatibility.
- Comparative benchmark vs. `ollama serve` at matched model/quantization:
  - Tokens/second sustained
  - Time to first token
  - Memory footprint (RSS)
  - Throughput under small concurrent request load (5 parallel)
- Verdict one-pager as a new ADR in `docs/adr/` (id assigned at authoring time —
  the next free `DEC-NNNN`, slug `inference-backend`): adopt pmetal, adopt ollama,
  or use both (e.g., pmetal for serving, ollama as fallback).

**1c — lm-evaluation-harness standup.**

- Install `lm-eval` in the `linus` conda env
- Configure Ollama backend
- Run MBPP, HumanEval, MMLU (5-shot), GPQA (main) against `qwen2.5-coder:7b` and
  `mistral:7b-instruct`
- Results to `benchmarks/results/public_baseline_YYYY-MM-DD.json`

**1d — Private "Dan task" suite** at `benchmarks/dan_tasks/`.

- 5–10 tasks pulled from real work. Examples:
  - "Summarize this paper and extract 3 key findings" (KnowledgeBase paper input)
  - "Write a Python script that reads a FASTA file and computes GC content"
  - "Given this matplotlib figure code, add log-scale axes and a heartbeat indicator"
  - "Given these 50 paper titles, cluster them into 5 topics and name each"
  - "Given this bioinformatics pipeline error traceback, diagnose and propose a fix"
- Each task: input, expected output schema, rubric (what counts as success)
- Run against hosted Claude (reference), Ollama+Qwen2.5-Coder, Ollama+Mistral, and
  pmetal+something if adopted in 1b
- Results to `benchmarks/results/dan_tasks_baseline_YYYY-MM-DD.json`
- This suite becomes the primary evaluation measure for every future phase

**1e — First Maestro/Worker loop recorded.**

- Claude (Maestro, via Claude Code) writes a concrete spec in `experiments/first-loop.md`
- Worker (Qwen2.5-Coder via Ollama, driven by Cline or a minimal orchestration script)
  executes the spec
- Smoke test gate before any full run
- Atomic commit with clean message
- Full protocol documented in `docs/maestro-worker-protocol.md`

**Gate to Phase 2:** we know (a) which inference engine to use for the serving layer,
(b) our baseline numbers on public and private benchmarks, (c) the Maestro/Worker
pattern works end-to-end on a real task.

## Phase 2 — Linus MVP

**Target duration:** 3 weeks

**Goal:** a minimal but real Linus orchestration backend + a chat UI, grounded in
KnowledgeBase, demo-able to a peer in under 5 minutes.

**2a — Orchestration backend (first implementation) at `src/linus/`:**

- FastAPI app exposing `/v1/chat/completions` with OpenAI ChatCompletions schema
- Router: single engine (Ollama or pmetal per Phase 1b verdict), no dispatch intelligence
- Tool registry with initial set:
  - `linus.knowledge.search_papers(query, limit)` — SPECTER2 search via KnowledgeBase
  - `linus.knowledge.get_paper(sha256)` — fetch paper metadata + abstract
  - `linus.fs.read(path)` — sandboxed read
  - `linus.fs.write(path, content)` — sandboxed write
  - `linus.shell.run_sandboxed(command)` — allowlisted command execution
- Sandbox policy enforced per SAFETY.md
- Session store: SQLite at `~/.linus/sessions.db`
- Audit log: JSONL at `~/.linus/audit.jsonl`
- Settings loaded from `~/.linus/config.toml`

**2b — Chat UI at `src/linus/app/main.py`:**

- Streamlit, inheriting patterns from KnowledgeBase's app
- Sidebar: session selector, model selector
- Main area: chat with streaming
- One panel: retrieved context from RAG, with clickable paper references

**2c — KnowledgeBase integration (v1):**

- `src/linus/knowledge/` adapter package
- Loads metadata.db read-only
- Lazy-loads SPECTER2 embeddings
- Exposes `search_papers`, `get_paper` tools
- Does NOT mutate KnowledgeBase state

**2d — First real use, smoke-tested:**

- Ask Linus "what papers have I been reading about 1-bit LLMs?"
- Linus routes to local model, model calls `linus.knowledge.search_papers`, returns
  grounded answer with citations
- Compare to hosted Claude on the same query (Dan task suite)
- Record the delta

**Explicit non-goals for Phase 2:**

- Multi-agent spawning (Phase 3)
- openclaw integration (Phase 5)
- Voice, Canvas, native app (later phases)
- Fine-tuning (Phase 6)
- Skills (Phase 7)
- Memory beyond session history

**Gate to Phase 3:** MVP demo-able end-to-end. Paper-grounded answers measurably better
than the same local model without Linus's RAG on the Dan task suite.

## Phase 3 — Knowledge Integration and Parallel Agents

**Target duration:** 3 weeks (can run partly in parallel with Phase 4)

**Goal:** Linus becomes meaningfully smarter on Dan's work, and the Maestro/Worker
pattern extends to parallel multi-agent fan-out.

**Branching model graduation:** Phase 3 is the gate for adopting full Driessen gitflow
(see BRANCHING.md). At the start of Phase 3:
- Create `develop` branch from `main` as the integration point for features
- Feature branches branch from `develop` instead of `main`
- Introduce `release/*` and `hotfix/*` branches for release management
- Tag releases on `main` with semantic version numbers
- This graduation will be documented as a new ADR in `docs/adr/` (and indexed in
  DECISIONS.md) with rationale and migration steps.

**3a — KnowledgeBase deepening:**

- All current KnowledgeBase tools exposed: `linus.knowledge.neighbors_of`,
  `linus.knowledge.query_kg`, `linus.knowledge.cluster_context`,
  `linus.knowledge.specter2_nearest`
- Agents 3–5 from KnowledgeBase's plan (per-paper summarizer, cluster synthesizer,
  writer/editor) built under Linus's orchestration, reusing the Maestro/Worker pattern
- Context-packing strategy: multi-paper queries return summaries + key snippets +
  citation graph, not raw text dumps

**3b — Parallel agent spawning:**

- `linus.agent.spawn(spec, subtasks)` tool
- Each spawned agent gets: scoped prompt, scoped tool allowlist, shared results store
- First test use case: audit `src/linus/` for style/bug issues — one agent per file
- Second test use case: literature-review synthesis across a cluster of KB papers —
  one agent per paper, then a merger agent
- LangGraph evaluated here as the state machine; adopt only if measurably better than
  rolling our own

**3c — Dan task suite re-run:** Linus-with-RAG vs. Linus-without-RAG vs. hosted Claude.
Goal: measurable closing of the gap on domain-specific tasks.

**3d — Memory (first pass, only if needed):**

- Only build if a concrete use case demands it. The Algorithm check: delete before adding.
- Candidate implementation: embedding-based retrieval over past session turns, indexed by
  topic

**Gate to Phase 4:** Linus answers domain questions measurably better than raw local
model, and can execute multi-subtask jobs in parallel with observable speedup.

## Phase 4 — Data Sovereignty Layer

**Target duration:** 2 weeks (can run partly in parallel with Phase 3)

**Goal:** Linus's knowledge scope extends beyond the personal library to curated external
corpora, all stored locally and accessible completely offline. Phase 4 makes explicit what
has always been implicit: Linus must operate without network access, with every data source
under Dan's physical control.

**Core:**

- Kiwix running natively on macOS
- ZIMs installed: English Wikipedia (full or curated selection), Simple English Wikipedia;
  topical ZIMs (medical, genomics/biochem if available) evaluated at install time
- Kolibri running natively on macOS for structured Khan Academy courseware; Kolibri serves
  curriculum-depth content (worked explanations, exercises, learning trees) where Kiwix
  serves reference lookups — both belong, as distinct tools
- ProtoMaps PMTiles with OSM data for offline geographic queries
- Dataset version tracking: `~/.linus/data_packages.json` registry
- Update check tool: `linus.data.check_updates` polls upstream, reports available
  updates, user approves downloads
- `linus.kiwix.search(zim, query)` tool wrapping Kiwix's HTTP API
- `linus.kolibri.search(topic)` tool wrapping Kolibri's local API
- `linus.maps.query(bbox)` tool wrapping PMTiles
- `linus.data.list` tool for listing installed packages

**Optional Docker:**

- Qdrant in Docker (if KnowledgeBase's native search hits scalability limits)
- Any other stateless services nomad-inspired: CyberChef, ArchiveBox, etc.
- Rule from earlier discussion: Docker for stateless services only; inference stays native

**Obsidian vault integration (stretch):**

- `linus.obsidian.search(query)` over Dan's local Obsidian vault, treated as a personal
  note corpus alongside the paper KnowledgeBase; Obsidian replaces FlatNotes from the
  NOMAD stack as the notes layer

**Explicit non-goal:** cloning project-nomad's full scope. Borrow what serves Linus;
delete the rest.

**Gate to Phase 5:** Dan can ask Linus "what's in this region of Oregon," "summarize the
Wikipedia article on electronegativity," or "explain Needleman-Wunsch step by step" and
get an offline-served answer from the appropriate local source — with no network required.

## Phase 6 — Fine-Tuning

**Target duration:** 4 weeks

**Goal:** produce a Linus-flavored model that measurably outperforms the stock base on
Dan's work.

**6a — Continued pretraining LoRA (1 week):**

- LoRA adapter trained on ~1M tokens of genomics/biochem papers from KnowledgeBase
- Base: Qwen2.5-7B or 14B (pick based on eval)
- Pipeline via pmetal or mlx-lm-ft (per Phase 1b verdict)
- Validation: held-out perplexity on KB papers + Dan task suite
- First "Linus model" adapter, saved to `experiments/adapters/linus-pretrain-v1/`

**6b — Instruction-tuning LoRA (1 week):**

- Q&A pairs synthesized from KB abstracts and summaries using Maestro/Worker pattern:
  - Maestro (Claude) writes diverse question templates
  - Worker (Qwen) generates answers against paper content
  - Dan spot-checks a random 5% sample for quality
- LoRA trained on top of 6a's adapter
- Dan task suite evaluation

**6c — Preference alignment (1 week, optional):**

- ORPO or DPO on preference pairs where Dan has rated outputs (could be generated via
  pairwise comparison tool in the chat UI)
- Aligns output style/length/formality to what Dan actually wants

**6d — Flash-streaming evaluation (1 week):**

- Use mlx-flash to run a larger fine-tuned model that exceeds RAM (e.g., LoRA'd
  32B Qwen variant)
- Inference-only path (the "b" option from planning discussion)
- Benchmark on Dan task suite: does larger model + LoRA + streaming outperform smaller
  model + LoRA at native speed?
- Apply autoresearch methodology: set a target tok/s, iterate until hit

**Explicit open-research non-goal for Phase 6:** streaming-based training (gradients
and optimizer states on SSD). Deferred — this is unsolved research territory.

**Later-phase model candidates** for fusion/flash-inference experiments but NOT for
fine-tuning on 32GB: Kimi K2 (~1T / 32B active), DeepSeek V3 (~671B / 37B active),
Llama 4 Scout (~109B). Re-evaluate as inference targets once flash-streaming is proven.

**Gate to Phase 7:** a fine-tuned Linus-branded adapter that beats the base model on Dan
task suite by a measurable margin.

## Cross-phase themes

### Benchmarks are continuous

The Dan task suite is re-run at every phase boundary. Regression is not acceptable
without a documented reason. Public benchmarks (MBPP, HumanEval, MMLU, GPQA) re-run on
any model change.

### ADRs are append-only

Every significant decision gets its own ADR file at `docs/adr/NNNN-<slug>.md` and is
indexed in DECISIONS.md. Each ADR has a `DEC-NNNN` id, a date, a status, a context
paragraph, a decision paragraph, and a consequence/rationale paragraph. See
[`docs/adr/README.md`](docs/adr/README.md) for the format and authoring procedure.

### Known Library Quirks grows

Every resolved gotcha goes into CLAUDE.md's "Known Library Quirks" section the same
session. No quirks lost to memory.

### The Algorithm is applied at every phase boundary

Before starting phase N+1, walk through its scope with the Algorithm:
- Question every requirement: who needs this? why?
- Delete every possible step: what can be cut?
- Simplify: after deletion, can the remaining be simpler?
- Accelerate cycle time: are feedback loops short?
- Automate: last, not first.

### Parallel by default

Work that fans out (repo notes, paper summaries, benchmark sweeps, multi-file audits)
fans out to parallel agents or Workers. Sequential is the exception.

### Maestro budget discipline

Hosted Claude is expensive attention, not expensive tokens. Arrive with context
gathered, questions sharp, task well-specified. Offload implementation to Workers.
